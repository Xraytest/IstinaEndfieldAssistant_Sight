# 自动采集准确性分析报告

> **生成时间**：2026-07-23
> **任务**：分析 AutoCollect 执行结果判断精准度，识别"未正确处理但判断完成"的风险点

## 1. 根因分析

### 1.1 AutoCollect 执行架构

AutoCollect 采用两层架构：
- **MaaFW Pipeline 层**（`assets/pipelines/auto_collect.json`）：负责传送（SceneEnterWorld*）和背包整理（StashBackpackMain）
- **Python 编排器层**（`runtime.py::_auto_collect_run`）：VLM 导航 + 采集验证 + 多轮重试

### 1.2 采集验证机制

当前验证逻辑（`_verify_collect_success`）：
1. 等待 1.5s 让飘字/弹窗动画完成
2. 调用 `scene.elements` OCR 全屏识别
3. 检查 OCR 文本是否包含以下关键词之一：`获得`、`采集成功`、`收取成功`、`已采集`

### 1.3 潜在误判场景

| 场景 | 现象 | 当前行为 | 风险 |
|------|------|----------|------|
| 飘字未出现/未捕获 | 采集成功但 OCR 未捕捉到飘字 | 判定为失败，触发重试 | 低（重试机制补偿） |
| 飘字被遮挡 |  UI 弹窗遮挡飘字 | 判定为失败 | 低 |
| 关键词不匹配 | 游戏更新飘字文本 | 判定为失败 | 低（重试） |
| 采集点无资源 | 资源已被其他玩家采集 | 判定为失败（正确） | 无 |
| VLM 导航到达但未交互 | F 键交互失败 | 判定为失败（正确） | 无 |

### 1.4 结论

当前 AutoCollect 的验证机制**不存在"未正确处理但判断完成"的系统性风险**：
- 验证失败 → 重试（最多 5 轮）→ 仍失败才标记路线失败
- 每条路线独立判断，不会因为一条路线失败而掩盖其他路线的成功
- 汇总逻辑：`overall_ok = all(routes.success)`，任何失败都返回 `error`

唯一改进空间：关键词列表可扩展（如添加 `获得物品`、`采集完毕` 等变体），但当前 4 个关键词已覆盖实测飘字文本。

## 2. 修改方案

### 2.1 无需修改（当前机制已足够）

AutoCollect 的采集验证 + 多轮重试 + 逐路线独立判断机制，已在架构层面避免了"假成功"风险。

### 2.2 可选增强（如后续需要）

1. 增加 OCR ROI 聚焦：对采集飘字常见区域（屏幕右侧）做 ROI OCR 而非全屏，提高捕获率
2. 增加延迟重试：采集后等待 3s 再做 OCR（当前仅 1.5s），给飘字更多出现时间
3. 关键词扩展：添加 `获得物品`、`采集完毕`、`资源获取` 等变体

## 3. 影响面

当前分析仅涉及 `runtime.py::_verify_collect_success` 和 `runtime.py::_auto_collect_run`，不涉及 MaaFW pipeline 修改。

## 4. 非期待变化

无代码修改，无副作用。
