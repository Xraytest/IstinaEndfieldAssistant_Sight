### NEW-RECOGNIZER-1（Medium）— 角色元素中心 y 坐标被宽度归一化（复制粘贴错误）

**位置**：`src/core/capability/element_recognition/recognizer.py:150-152`

**根因分析**
`recognize()` 的 Phase 2.5 把 3D 游戏场景检测到的「角色」追加为 `character` 元素：

```python
147:            deduped.append(ElementInfo(
148:                element_type="character",
149:                label="character",
150:                bbox=(char["cx"] / w_img, char["cy"] / h_img,
151:                      (char["cx"] + char["w"]) / w_img, (char["cy"] + char["h"]) / h_img),
152:                center=(char["cx"] / w_img, char["cy"] / w_img),   # ← 第二个分量误用 w_img
```

`bbox` 的 y 分量正确使用了 `h_img`，但同块 `center` 的 y 分量却除以了 **`w_img`**（宽度）。下行 `region`（物体）块（:164-165）`center=(obj["cx"] / w_img, obj["cy"] / h_img)` 是正确的，二者对照即可确认 :152 为复制粘贴错误。

直接原因：归一化公式把纵向坐标按图像宽度缩放。
根本原因：`character` 与 `region` 两段相似代码分别编写，未统一抽出归一化辅助函数，导致其中一个被错写为 `w_img`。
影响面：所有 `char["cy"]/w_img` 计算出的 y 中心会被错误缩放约 `w_img/h_img` 倍（典型 16:9 下约 1.78×）。该 `center` 被 `_deduplicate()`（`:385 _is_nearby` 用 `center` 做空间邻近判定）及任何下游使用 `center[1]` 的点击/导航/推理消费方读取，会导致角色元素去重错位与潜在的误点击。

**修改方案**
`recognizer.py:152` 改为：
```python
                center=(char["cx"] / w_img, char["cy"] / h_img),
```
（与 :151 `bbox` 的 y 口径及 :165 `region` 的 `center` 一致。）

**影响面**
- 函数：`EndfieldElementRecognizer.recognize`（仅 `character` 分支）。
- 下游：去重 `_is_nearby`、场景理解/导航点击定位、CLI `scene.identify/elements` 返回的 `center`。
- 调用点：`recognize_text`/`recognize` 全链路；`_scene_identify`、GUI 预览取元素中心。

**非期待变化**
- 纯数学修正，仅改变 `character` 元素的 y 中心数值，无副作用。
- 回退策略：如误改可还原为 `char["cy"] / w_img`（原错误值），但不建议保留。

### NEW-RECOGNIZER-2（Medium）— 必需元素 OCR 关键词匹配用集合精确成员而非子串（与同函数 Tier-3 自相矛盾）

**位置**：`src/core/capability/element_recognition/recognizer.py:292-296`

**根因分析**
`_score_page()` 的 Tier-1「必需元素」评分中，OCR 关键词分支：

```python
276:        element_labels = {e.label.lower() for e in elements}   # 标签集合
...
292:            ocr_kws = catalog_entry.get("ocr_keywords", [])
293:            if any(kw.lower() in element_labels for kw in ocr_kws):
294:                score += 1.5
295:                required_matched += 1
296:                continue
```

`element_labels` 是一个 **`set`**，`kw.lower() in element_labels` 对集合做的是**精确相等**成员判断，而非子串匹配。因此当目录中 `ocr_keywords` 是部分词（如 `"确认"`）而 OCR 实际识别出的标签为 `"确认战斗"` 时，`"确认" in {"确认战斗"}` 为 `False`，该必需元素分支**永不命中**，导致页面分类评分偏低、可能误判页面。

矛盾证据：同函数 Tier-3（:312-318）的 OCR 关键词匹配写法是 `any(kw.lower() in elem.label.lower() for kw in ocr_keywords)`（**子串匹配**，正确）。两处语义不一致，说明 :293 是错误写法（作者本意是子串，Tier-3 才是正确实现）。

直接原因：对 set 使用了 `in` 期望子串，实际是精确成员。
根本原因：页面签名关键词常以「部分词」声明，但匹配逻辑未统一为子串。
影响面：依赖 `page_signatures` 中 `required_elements` + `ocr_keywords` 做页面分类的所有场景；与 `recognizer NEW-2`（YOLO 分支死代码，见审计 A2）叠加后，必需元素评分更脆弱，可能把 gameplay/战斗/对话等关键页面错分。

**修改方案**
将 :293 改为与 Tier-3 一致的子串判定：
```python
            ocr_kws = catalog_entry.get("ocr_keywords", [])
            if any(kw.lower() in lbl for kw in ocr_kws for lbl in element_labels):
                score += 1.5
                required_matched += 1
                continue
```
（或等价地 `any(any(kw.lower() in lbl for lbl in element_labels) for kw in ocr_kws)`。）

**影响面**
- 函数：`EndfieldElementRecognizer._score_page`（Tier-1 OCR 关键词分支）。
- 下游：`_classify_page` 页面打分 → `recognize()` 返回的 `page_info.page_type`/`confidence`；进而影响导航/任务进入判定。
- 调用点：GUI 场景识别、CLI `scene.identify`、自动导航的前置页面判断。

**非期待变化**
- 改为子串后，部分签名因「关键词被更长标签包含」而更频繁命中，可能使某些页面的 `confidence` 略升；属预期修正方向。
- 回退策略：保留原 `kw.lower() in element_labels` 即回到精确匹配（不推荐）。
