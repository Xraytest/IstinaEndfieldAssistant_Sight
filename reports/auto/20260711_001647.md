## 1. 代码错误（功能失效类）

### [D1] `_force_stop` 将 `"am force-stop"` 作为单个参数传递，导致强制停止从未生效

**优先级：High**

**文件**：`src/core/capability/device/recovery.py:72`

**根因**：
```python
def _force_stop(self, serial: Optional[str]) -> None:
    try:
        self._run(["shell", "am force-stop", self._package], serial)
```

`_run` 方法将 `args` 与 `_resolve_adb(serial)` 拼接后传给 `subprocess.check_output`。由于 `"am force-stop"` 是一个字符串，最终 ADB 命令为：
```
[adb_path, "-s", serial, "shell", "am force-stop", "com.hypergryph.endfield"]
```
Android 的 mksh 会将 `"am force-stop"` 解释为单个命令名，而非 `am` 加子命令 `force-stop`。系统找不到名为 `am force-stop` 的可执行文件，该命令始终失败。

**影响面**：`restart()` 方法中的 `_force_stop` 步骤从未正确执行。旧应用进程未被终止，直接启动新进程，可能导致两个实例同时运行或应用状态异常。同时，`_force_stop` 的 `except` 块仅记录 warning 后吞掉异常继续执行，调用方无法感知强制停止失败。

**修改建议**：将 `"am force-stop"` 拆分为 `"am"` 和 `"force-stop"` 两个独立参数：
```python
self._run(["shell", "am", "force-stop", self._package], serial)
```

**非期待变化**：修复后 `_force_stop` 将真正终止应用进程。需验证游戏在强制停止后能否正常启动。

## 2. 漏洞 / 安全风险

### [D2] `shell()` 方法接受用户直接输入但未做任何过滤，存在命令注入

**优先级：High**

**文件**：`src/core/capability/device/adb_manager.py:71-82（shell）`、`104-110（_shell_via_subprocess）`
**调用链**：`src/cli/handlers.py:467-473`（`_handle_shell` 处理 `istina shell <cmd>` 命令）

**根因**：
`ADBDeviceManager.shell(cmd)` 接受任意字符串 `cmd`，直接传递给 `adb shell`。虽然 `subprocess.check_output` 使用列表形式避免主机侧 shell 注入，但 `adb shell` 会将整个字符串转发到设备端的 mksh，由设备 shell 解释。若 `cmd` 包含 shell 元字符（`;`、`|`、`&`、`$(...)` 等），设备端会执行任意命令。

**当前暴露路径**：
`handlers.py:467-473` 的 `_handle_shell` 处理 CLI 命令 `istina shell <cmd>`：
```python
def _handle_shell(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    try:
        output = android.shell(args.cmd)
        return {"status": "success", "output": output}
```
`args.cmd` 完全来自用户输入，未经过任何过滤或白名单检查。用户可执行 `istina shell "dumpsys meminfo; rm -rf /sdcard/*"` 在设备上执行任意命令。

**影响面**：任何有权限执行 CLI 命令的用户可在已连接的 Android 设备上执行任意 shell 命令，包括删除文件、修改系统设置、安装恶意应用等。当前白名单机制（`android_runtime.py` 的 `_is_allowed_shell_cmd`）仅约束守护进程 RPC 路径，`adb_manager.shell()` 的 subprocess 路径完全绕过了该检查。

**修改建议**：
1. 在 `_handle_shell` 层面增加命令白名单或参数校验，仅允许安全的诊断命令
2. 或在 `shell()` 方法内增加命令过滤，与 `android_runtime.py` 的 `_is_allowed_shell_cmd` 保持一致
3. 至少将 `;`、`|`、`&`、`$`、```、`\`、`$(` 等 shell 元字符加入黑名单

**非期待变化**：过滤后可能影响合法诊断命令（如 `dumpsys meminfo` 包含无危害的文本）。应使用精确拒绝而非泛化黑名单。

## 3. 逻辑 / 健壮性问题

### [D3] `get_devices()` 裸 `except Exception: pass` 吞掉 ImportError

**优先级：Medium**

**文件**：`src/core/capability/device/adb_manager.py:47-55`

**根因**：
```python
try:
    import adbutils
    adb = adbutils.AdbClient(host="127.0.0.1", port=5037)
    for device in adb.devices():
        devices.append(ADBDeviceInfo(serial=device.serial, state=device.state))
    return devices
except Exception:
    pass
```

`except Exception` 捕获所有异常类型，包括 `ImportError`（adbutils 未安装）、`ConnectionError`（ADB 服务器未运行）、`AdbError`（ADB 协议错误）等。所有异常均被静默吞掉，直接回退到 subprocess 路径。如果 adbutils 已安装但因配置错误（如端口冲突）导致连接失败，该失败被隐藏，调用方无法区分"adbutils 不可用"和"ADB 服务器未运行"。

**影响面**：设备扫描失败时诊断困难。如果 adbutils 和 subprocess 路径都失败（如 adb.exe 不存在），最终返回空列表，调用方看到"无设备"但实际是 ADB 基础设施故障。

**修改建议**：至少记录异常信息：
```python
except Exception as e:
    self._logger.debug(LogCategory.ADB, "adbutils 路径失败，回退到 subprocess", error=str(e))
    pass
```

### [D4] `screencap()` 不验证返回数据有效性

**优先级：Low**

**文件**：`src/core/capability/device/adb_manager.py:84-95（screencap）`、`120-130（_screencap_via_subprocess）`

**根因**：`screencap()` 返回 `bytes`，但两路实现（adbutils 和 subprocess）均未验证返回数据是否为有效 PNG。如果 screencap 失败（如设备离线、权限不足），adb 可能返回错误文本（如 `error: device offline`）而非 PNG 数据。调用方收到非 PNG 数据后继续处理，可能导致后续图像分析崩溃或产生错误结果。

**影响面**：低——screencap 失败通常是设备断连等严重问题，但返回无效数据而不报错会掩盖真实故障。

**修改建议**：在 `_screencap_via_subprocess` 中验证 PNG 签名：
```python
data = subprocess.check_output(args, timeout=self._timeout)
if data[:4] != b"\x89PNG":
    raise RuntimeError(f"screencap 返回无效数据（前4字节：{data[:4]!r}）")
if data[:4] == b"\x89PNG" and b"\x0d\x0a" in data:
    data = data.replace(b"\x0d\x0a", b"\x0a")
return data
```

### [D5] `_first_device_serial()` 非确定性多设备选择

**优先级：Low**

**文件**：`src/core/capability/device/adb_manager.py:97-102`

**根因**：当多个设备处于 `device` 状态时，`_first_device_serial()` 返回 adb 列表中的第一个设备。adb 的设备列表顺序取决于设备连接时间和 ADB 服务器内部状态，不具备确定性。在多设备环境下，操作可能落到非预期设备上。

**影响面**：低——当前项目通常只连接单个设备，但多设备场景下行为不可预测。

**修改建议**：返回列表让调用方选择，或支持配置 `preferred_serial`。

## 4. 资源泄漏 / 性能

### [D6] `_screencap_via_subprocess` 的 CRLF 修复可能过度替换，损坏合法 PNG 数据

**优先级：Low**

**文件**：`src/core/capability/device/adb_manager.py:127-129`

**根因**：
```python
if data[:4] == b"\x89PNG" and b"\x0d\x0a" in data:
    data = data.replace(b"\x0d\x0a", b"\x0a")
```

该修复针对"ADB on Windows 可能将 LF 转换为 CRLF"的假设。但 `data.replace(b"\x0d\x0a", b"\x0a")` 会替换 PNG 数据中**所有** CRLF 序列，包括 IDAT 图像数据块和 tEXt 文本块中合法存在的 `\x0d\x0a`。替换后 PNG 结构被破坏，解码失败。

**影响面**：低——仅在 screencap 数据恰好包含 CRLF 时触发，但一旦触发会导致截图完全不可用。

**修改建议**：仅在数据末尾的附加内容（如 adb 错误消息）中替换 CRLF，或改为逐行处理：
```python
if data[:4] == b"\x89PNG":
    # 仅在 PNG 末尾查找并清理 CRLF（adb 可能附加的错误消息）
    png_end = data.find(b"IEND")
    if png_end != -1:
        iend_crc = data[png_end:png_end + 12]
        if len(iend_crc) == 12:
            after = data[png_end + 12:]
            after = after.replace(b"\x0d\x0a", b"\x0a")
            data = data[:png_end + 12] + after
```

## 5. 触控管理器新增发现

### [D7] `back()` 无错误处理与日志记录

**优先级：Low**

**文件**：`src/core/capability/device/touch_manager.py:53-56`

**根因**：
```python
def back(self, serial: Optional[str] = None) -> None:
    import subprocess
    args = self._adb_base_args(serial) + ["shell", "input", "keyevent", "KEYCODE_BACK"]
    subprocess.check_output(args, timeout=10)
```

与 `tap()`、`swipe()`、`long_press()` 不同，`back()` 未将 subprocess 调用包裹在 try/except 中，也未记录任何日志。如果命令失败（如设备断连、adb 未响应），`CalledProcessError` 或 `TimeoutExpired` 直接传播到调用方，没有任何诊断上下文（哪个设备、什么错误）。

**修改建议**：保持与其他触控方法一致的错误处理模式：
```python
def back(self, serial: Optional[str] = None) -> None:
    try:
        import subprocess
        args = self._adb_base_args(serial) + ["shell", "input", "keyevent", "KEYCODE_BACK"]
        subprocess.check_output(args, timeout=10)
    except Exception as e:
        self._logger.error(LogCategory.EXECUTION, "返回键失败", serial=serial, error=str(e))
        raise
```

### [D8] 所有触控操作硬编码 10 秒超时，无重试机制

**优先级：Low**

**文件**：`src/core/capability/device/touch_manager.py:24-56`

**根因**：`tap()`、`swipe()`、`long_press()`、`back()` 均使用 `timeout=10`。对于应在毫秒级完成的触控输入事件，10 秒超时过于宽松。更重要的是，没有任何重试机制——瞬时的设备忙碌或 ADB 抖动会导致操作失败并直接抛出异常。

**影响面**：低——触控操作通常很快，但偶发的 ADB 延迟会导致自动化任务中断。

**修改建议**：为简单操作使用更短超时（如 3 秒），并在 `TouchManager` 层添加可配置的重试次数。

### [D9] `device_address` 参数已存储但从未使用

**优先级：Low**

**文件**：`src/core/capability/device/touch_manager.py:19-21（init）`、`40-44（_adb_base_args）`

**根因**：`TouchManager.__init__` 接受 `device_address` 参数并存储到 `self._device_address`，但 `_adb_base_args` 只使用 `serial` 参数，从未回退到 `device_address`。`get_instance` 也接收 `device_address` 但同样不生效。

**影响面**：低——当前所有调用方都直接传 `serial`，`device_address` 形同虚设。若调用方期望 `device_address` 作为默认值，该期望被违背。

**修改建议**：在 `_adb_base_args` 中回退到 `self._device_address`，或移除该参数以消除歧义。

### [D10] 单例 `_instance` 永不重置，配置变更后返回过期实例

**优先级：Low**

**文件**：`src/core/capability/device/touch_manager.py:58-64`

**根因**：`TouchManager._instance` 在首次 `get_instance()` 调用时创建，此后永不重置。若后续调用传入不同的 `adb_path` 或 `device_address`，方法仍返回首次创建的实例，配置变更被忽略。

**影响面**：低——当前代码中 `get_instance` 的调用参数一致，但这是一个隐患。

**修改建议**：添加 `@classmethod reset_instance(cls)` 方法，在配置变更时重置。

## 6. 恢复策略新增发现

### [D11] `_clear_canvas` 完全吞掉异常，关键清理步骤失败不可见

**优先级：Low**

**文件**：`src/core/capability/device/recovery.py:81-94`

**根因**：
```python
def _clear_canvas(self, serial: Optional[str]) -> None:
    try:
        self._run(["shell", "wm", "dismiss-keyguard"], serial)
    except Exception:
        pass
    try:
        self._run(["shell", "input", "keyevent", "KEYCODE_WAKEUP"], serial)
    except Exception:
        pass
    try:
        self._run(["shell", "input", "keyevent", "KEYCODE_HOME"], serial)
    except Exception:
        pass
```

三个子步骤（解除锁屏、唤醒、回主页）的异常均被 `except Exception: pass` 静默吞掉。如果设备处于锁屏状态且 `dismiss-keyguard` 失败（如设置了密码锁），或屏幕处于关闭状态且 `KEYCODE_WAKEUP` 失败，调用方和日志均无任何提示。后续 `_launch` 可能在锁屏或黑屏状态下执行，导致启动失败或行为异常。

**修改建议**：至少记录 warning 日志：
```python
except Exception as exc:
    self._logger.warning(LogCategory.MAIN, "清理画布步骤失败", step="dismiss-keyguard", error=str(exc))
```

### [D12] `_launch()` activity 失败时 monkey 回退的异常掩盖原始错误

**优先级：Low**

**文件**：`src/core/capability/device/recovery.py:96-111`

**根因**：
```python
def _launch(self, serial: Optional[str]) -> None:
    if self._activity:
        try:
            self._run(["shell", "am", "start", "-n", f"{self._package}/{self._activity}"], serial)
            return
        except Exception as exc:
            self._logger.warning(...)
    self._run(["shell", "monkey", "-p", self._package, "-c", "android.intent.category.LAUNCHER", "1"], serial)
```

如果 activity 启动失败且 monkey 回退也失败，最终抛出的异常只包含 monkey 的错误信息，activity 启动失败的根本原因被掩盖。

**修改建议**：chain 异常或聚合两阶段错误信息：
```python
except Exception as monkey_exc:
    raise RuntimeError(f"activity 启动失败({exc}) 且 monkey 回退失败({monkey_exc})") from monkey_exc
```
