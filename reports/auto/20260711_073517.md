## 2. 新发现漏洞与错误

### 2.1 🟢 中危问题

#### E01 - `handlers.py:_handle_device_status` 未验证 `default_client` 为 None

**文件**: `src/cli/handlers.py:387-399`
**风险等级**: 🟢 中危
**影响**: 未连接设备时调用 `device status` 命令崩溃，CLI 返回非预期异常

**问题代码**:
```python
def _handle_device_status(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    try:
        client = android.default_client      # ❌ 可能为None
        server_version = client.version()    # ❌ AttributeError
        devices = android.get_devices()
        ...
    except Exception as exc:
        return {"status": "error", "message": str(exc)}
```

**问题分析**:
- `android.default_client` 在无设备连接时返回 `None`
- 直接调用 `client.version()` 触发 `AttributeError: 'NoneType' object has no attribute 'version'`
- 异常被 try/except 捕获，返回 `{"status": "error", "message": "'NoneType' object has no attribute 'version'"}`
- 错误信息不具可读性，用户无法理解是"未连接设备"导致的
- 与 `_handle_device_screenshot`、`_handle_device_tap` 等未做 `default_client` 校验的 handler 同病

**修复建议**:
```python
def _handle_device_status(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    try:
        client = android.default_client
        if client is None:
            return {"status": "error", "message": "no device connected"}
        server_version = client.version()
        devices = android.get_devices()
        return {
            "status": "success",
            "adb_server": {"version": server_version},
            "devices": [{"serial": d.serial, "state": d.state} for d in devices],
        }
    except Exception as exc:
        return {"status": "error", "message": str(exc)}
```

### 2.2 🟢 低危问题

#### E02 - `handlers.py:_handle_device_screenshot` `_logger` 未定义

**文件**: `src/cli/handlers.py:291-296`
**风险等级**: 🟢 低危
**影响**: screenshot handler 中 `_logger` 未定义，调用时触发 `NameError`

**问题代码**:
```python
def _handle_screenshot(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    _logger = get_logger(__name__)   # ❌ 缺失！未定义 _logger
    _logger.info("CLI handler: 开始 screenshot", out=...)   # ❌ NameError
    data = runtime.execute("screenshot", {})
    _logger.info("CLI handler: screenshot 完成", ...)       # ❌ NameError
    return _write_or_base64(data, ...)
```

**问题分析**:
- 第292行和第295行使用了 `_logger` 变量
- 但函数内从未调用 `_logger = get_logger(__name__)`
- 每次执行 `istina screenshot` 命令时，都会触发 `NameError: name '_logger' is not defined`
- 这导致 screenshot 功能完全不可用
- 同文件的其他 handler 函数（如 `_handle_task_run`）也存在同样缺失

**修复建议**:
```python
def _handle_screenshot(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    _logger = get_logger(__name__)   # ✅ 添加
    _logger.info("CLI handler: 开始 screenshot", out=getattr(args, "out", None))
    data = runtime.execute("screenshot", {})
    _logger.info("CLI handler: screenshot 完成", data_type=type(data).__name__, ...)
    return _write_or_base64(data, getattr(args, "out", None))
```

#### E03 - `device_settings_page.py:_read_config` 损坏配置无备份

**文件**: `src/gui/pyqt6/pages/device_settings_page.py:301-307`
**风险等级**: 🟢 低危
**影响**: 设备配置损坏时，设备历史、自动重连设置等丢失

**问题代码**:
```python
def _read_config(self) -> Dict[str, Any]:
    if not self._config_path.exists():
        return {}
    try:
        return json.loads(self._config_path.read_text(encoding="utf-8"))
    except json.JSONDecodeError:
        return {}  # ❌ 静默返回空dict，损坏文件未被备份
```

**问题分析**:
- 与批次28 D05（`settings_page.py`）完全相同的问题
- 设备配置包含：`last_connected`、`serial`、`history`、`auto_reconnect`、`adb_restart_on_timeout`
- 损坏时用户丢失设备连接历史和重连设置
- `QueueState` 类已有 `_backup_corrupt_state` 方法实现备份逻辑，但两个 settings 页面均未复用

**修复建议**:
```python
def _read_config(self) -> Dict[str, Any]:
    if not self._config_path.exists():
        return {}
    try:
        return json.loads(self._config_path.read_text(encoding="utf-8"))
    except json.JSONDecodeError:
        self._backup_corrupt_config()  # ✅ 备份损坏文件
        return {}

def _backup_corrupt_config(self) -> None:
    """备份损坏的配置文件。"""
    try:
        backup = self._config_path.with_suffix(".json.bak")
        counter = 0
        while backup.exists():
            backup = self._config_path.with_suffix(f".json.bak.{counter}")
            counter += 1
        import os
        os.replace(self._config_path, backup)
    except Exception:
        pass
```

#### E04 - `prts_full_intelligence_page.py:_stop_llm` 异步停止后立即查询状态

**文件**: `src/gui/pyqt6/pages/prts_full_intelligence_page.py:183-188`
**风险等级**: 🟢 低危
**影响**: 停止 LLM 后 UI 可能短暂显示错误状态

**问题代码**:
```python
def _stop_llm(self) -> None:
    self._append_chat("系统", locale.tr("llm_stopping", "Stopping LLM..."))
    self._bridge.execute("llm stop", {})  # ❌ 异步，不等待结果
    if self._startup_timer is not None:
        self._startup_timer.stop()
    self._update_llm_status()             # ❌ 立即查询状态，可能仍为ready
```

**问题分析**:
- `self._bridge.execute("llm stop", {})` 是异步的，不阻塞等待结果
- `_update_llm_status()` 立即发送 `llm status` 查询
- 如果 `llama-server` 进程尚未完全停止，`health_check` 仍返回 True
- UI 显示 "Ready" 而非 "Not Ready"，与用户刚点击"停止"的预期矛盾
- 用户可能误以为停止操作未生效

**修复建议**:
```python
def _stop_llm(self) -> None:
    self._append_chat("系统", locale.tr("llm_stopping", "Stopping LLM..."))
    self._status_label.setText(locale.tr("llm_stopping", "Stopping..."))
    self._status_label.setStyleSheet(RED_STYLE)
    self._bridge.execute("llm stop", {})
    # 延迟查询状态，给 llama-server 进程时间完全停止
    if self._startup_timer is not None:
        self._startup_timer.stop()
    QTimer.singleShot(1500, self._update_llm_status)  # ✅ 延迟1.5秒
```

#### E05 - `prts_full_intelligence_page.py:_on_command_finished` 未校验命令来源

**文件**: `src/gui/pyqt6/pages/prts_full_intelligence_page.py:209-224`
**风险等级**: 🟢 低危
**影响**: 其他页面或并发实例的 llm 命令结果可能错误更新 PRTS 页面状态

**问题代码**:
```python
def _on_command_finished(self, command: str, result: dict) -> None:
    cmd_parts = command.split()
    if len(cmd_parts) >= 2 and cmd_parts[0] == "llm" and cmd_parts[1] == "status":
        ready = bool(result.get("ready"))
        self._finalize_startup_status(ready)  # ❌ 接受任何来源的 llm status 结果
```

**问题分析**:
- `commandFinished` 信号是全局的，所有页面的 CLI 命令结果都通过此信号分发
- 如果 SettingsPage 或 MaaEndControlPage 也发送了 `llm status` 命令
- 其结果会触发 PRTS 页面的 `_finalize_startup_status`
- 可能导致 PRTS 页面状态被其他页面的 LLM 查询结果干扰
- 尤其在多页面快速切换时更易触发

**修复建议**:
```python
def _on_command_finished(self, command: str, result: dict) -> None:
    # 仅处理本页面发起的命令（通过请求ID或时间戳匹配）
    cmd_parts = command.split()
    if len(cmd_parts) >= 2 and cmd_parts[0] == "llm" and cmd_parts[1] == "status":
        # 验证命令是否由本页面发起（通过检查当前worker或启动状态）
        if self._startup_timer is None or not self._startup_timer.isActive():
            return  # 忽略非本页面发起的查询
        ready = bool(result.get("ready"))
        self._finalize_startup_status(ready)
```
