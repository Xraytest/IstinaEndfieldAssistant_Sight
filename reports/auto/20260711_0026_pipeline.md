## 1. 代码错误（功能失效类）

### [PN-1] `PipelineNode.from_dict` 对 `action` 字符串分支冗余且存在逻辑死区
**优先级：Medium**
**位置**：`pipeline_node.py:67-72`

```python
if isinstance(action_raw, str):
    if action_raw in ("StopTask", "DoNothing"):
        action = action_raw
    else:
        action = action_raw   # ← 与 if 分支结果完全相同
    if action_raw == "Custom":   # ← 此时 action_raw 不是 "StopTask"/"DoNothing"
        ...
```

**根因**：`else` 分支对非 `StopTask`/`DoNothing` 的字符串执行 `action = action_raw`，但 `if` 分支也执行 `action = action_raw`。两个分支产生相同结果，`else` 是死代码。更重要的是，当 `action_raw == "Custom"` 时，上面的 if/else 走的是 `else` 分支，`action` 被赋值为 `"Custom"`，但随后 `if action_raw == "Custom"` 才提取自定义参数——该逻辑正确但可读性差。真正的问题是：当 `action_raw` 不是字符串而是 dict 时，`action = action_raw.get("type", "DoNothing")`，若 dict 缺少 `type` 键则默认 `DoNothing`，用户可能误认为自定义动作已配置但实际上回退到无操作。

**影响面**：JSON 配置中 action 使用 dict 格式但忘记写 `"type"` 键时，静默回退到 `DoNothing`，自动化完全跳过动作。

**修改建议**：
```python
if isinstance(action_raw, str):
    action = action_raw
    if action_raw == "Custom":
        custom_action = data.get("custom_action")
        custom_action_param = data.get("custom_action_param")
elif isinstance(action_raw, dict):
    action = action_raw.get("type")   # 不设默认，None 在下层处理
    if action == "Custom":
        ...
```

### [PN-2] `from_dict` 对 `recognition` 字段解析存在三段式冗余与隐含错误
**优先级：Low**
**位置**：`pipeline_node.py:59-62`

```python
raw = data.get("recognition", "DirectHit")
recognition = raw if isinstance(raw, RecognitionType) else RecognitionType(raw) if isinstance(raw, str) and raw in RecognitionType._value2member_map_ else RecognitionType.DirectHit
if recognition == RecognitionType.DirectHit and not raw:
    recognition = RecognitionType.DirectHit
```

**根因**：
1. 三元表达式嵌套可读性差。
2. `data.get("recognition", "DirectHit")` 的默认值 `"DirectHit"` 会在 `raw = ""` 时回退到默认字符串，但如果用户显式写 `"recognition": ""`，`raw = ""`，第一段 `isinstance(raw, str) and raw in RecognitionType._value2member_map_` 为 `False`（空串不在枚举值映射中），fallback 到 `RecognitionType.DirectHit`。随后 `not raw` 为 `True`，`recognition` 再次被设为 `RecognitionType.DirectHit`——冗余。
3. 如果 `raw` 是非枚举字符串（如 `"FooBar"`），第一段 `RecognitionType(raw)` 会抛出 `ValueError`，因为 `RecognitionType.__new__` 不接受非枚举值。代码假设 `raw in RecognitionType._value2member_map_` 会过滤，但嵌套三元表达式中 `RecognitionType(raw)` 在 `isinstance(raw, str) and raw in ...` 为 `False` 时不会执行，因此安全——只是可读性差。

**影响面**：低。当前逻辑功能正确但难维护。

**修改建议**：
```python
raw = data.get("recognition", "DirectHit")
if isinstance(raw, RecognitionType):
    recognition = raw
elif isinstance(raw, str) and raw in RecognitionType._value2member_map_:
    recognition = RecognitionType(raw)
else:
    recognition = RecognitionType.DirectHit
```

## 2. 漏洞 / 安全风险

### [PL-1] `_load_file` 使用 `open(path, "r")` 无编码保护，中文 JSON 在 Windows 上崩溃
**优先级：Medium**
**位置**：`pipeline_loader.py:72`

```python
with open(path, "r", encoding="utf-8") as f:
    data = json.load(f)
```

**根因**：虽然指定了 `encoding="utf-8"`，但若 pipeline JSON 文件实际编码为 UTF-8 with BOM（Windows 记事本默认），`json.load` 会因 BOM 字符（`\ufeff`）在第一个键前而抛出 `JSONDecodeError`。`_load_file` 的 `except Exception` 捕获后仅记录 debug 日志并返回，整批节点静默丢失。

**影响面**：用户在 Windows 上用记事本编辑 pipeline JSON 后，文件带 BOM，重新加载时所有节点丢失，pipeline 图变为空。

**修改建议**：
```python
import codecs
with codecs.open(path, "r", encoding="utf-8-sig") as f:
    data = json.load(f)
```

### [PL-2] `load_all` 的 `glob("*.json")` 不递归，子目录中的 pipeline JSON 被静默忽略
**优先级：Medium**
**位置**：`pipeline_loader.py:45`

```python
for fpath in sorted(self._pipelines_root.glob("*.json")):
```

**根因**：`glob("*.json")` 只匹配 `_pipelines_root` 直接子文件，不递归匹配子目录中的 JSON。但 `load_module` 方法明确检查了子目录结构（`_pipelines_root / module_name / f"{module_name}.json"`）。这意味着：
- `load_all()` 只加载根目录的 JSON
- `load_module("X")` 可能从 `assets/pipelines/X/X.json` 加载
- 如果用户的 pipeline 文件放置在子目录中但文件名与目录名不同，`load_all` 不会加载它，而 `load_module` 也不会匹配到（因为按 `{module_name}/{module_name}.json` 查找）

**影响面**：pipeline 配置不完整，节点缺失导致 `get_node_or_entry` 回退到默认入口，执行错误的 pipeline。

**修改建议**：使用 `rglob("*.json")` 或显式递归子目录。

## 3. 线程安全

### [PN-3] `PipelineGraph` 的 `merge` 方法非线程安全，并发合并损坏节点字典
**优先级：Medium**
**位置**：`pipeline_node.py:135-137`

```python
def merge(self, other: PipelineGraph) -> None:
    self.nodes.update(other.nodes)
    self.entry_points.extend(other.entry_points)
```

**根因**：`dict.update()` 和 `list.extend()` 在 CPython 的 GIL 下是原子的单条字节码操作（对于小字典/列表）。但如果 `other.nodes` 在迭代期间被其他线程修改（如另一个线程正在向 `other` 添加节点），`dict.update()` 可能抛出 `RuntimeError: dictionary changed size during iteration`。此外，`entry_points` 的 `extend` 与 `nodes` 的 `update` 之间非原子——另一个线程可能在两个操作之间看到半合并状态。

**影响面**：多线程场景下（如并行加载多个 pipeline 模块后合并），极端情况下崩溃或状态不一致。

**修改建议**：
```python
def merge(self, other: PipelineGraph) -> None:
    with threading.Lock():  # 或上层调用者持有锁
        self.nodes.update(other.nodes)
        self.entry_points.extend(other.entry_points)
```

### [PL-3] `_loaded_modules` 集合在多线程 `load_module` 下无锁保护
**优先级：Low**
**位置**：`pipeline_loader.py:37` + `21`

```python
self._loaded_modules.add(module_name)
```

**根因**：`set.add()` 在 CPython 下是原子的（单条字节码），但 `_loaded_modules` 在 `load_module` 中被读写（检查是否已加载未直接做，但 `add` 发生在方法尾部），如果两个线程同时加载不同模块，`add` 操作本身安全。但如果未来加入"if module in loaded: skip"的检查-执行逻辑，就会形成 TOCTOU 竞态。当前代码不检查是否已加载，每次调用都执行加载并 `add`，重复添加无害（set 语义）。

**影响面**：当前无实际 bug，但模式脆弱。如果有代码路径做 `if module_name not in self._loaded_modules` 判断，会引入竞态。

**修改建议**：添加注释说明当前无竞态，或提前使用锁保护。

## 4. 资源泄漏

### [PL-4] `_load_file` 的 JSON 解析异常被 `except Exception` 吞掉后无计数反馈
**优先级：Low**
**位置**：`pipeline_loader.py:71-90`

```python
try:
    with open(path, "r", encoding="utf-8") as f:
        data = json.load(f)
except Exception as e:
    logger.error(f"Failed to load pipeline file {path}: {e}")
    return
```

**根源**：`except Exception` 捕获所有异常（`JSONDecodeError`、`PermissionError`、`MemoryError` 等），记录 error 日志后直接 `return`。这是合理的单文件容错，但有一个问题：`_load_file` 被 `load_module` 调用时，如果文件加载失败，`loaded = True` 不会被设置，`_loaded_modules.add(module_name)` 仍然执行——模块被标记为已加载但图为空。

**影响面**：调用方看到 `_loaded_modules` 包含该模块名，但 `graph.nodes` 为空。如果 `load_module` 被重复调用，每次都重新尝试加载（因为 `_loaded_modules` 不阻止加载），重复读取失败文件浪费 I/O。

**修改建议**：将 `_loaded_modules.add` 移至 `if loaded:` 块内：

```python
if loaded:
    self._loaded_modules.add(module_name)
```

### [PN-4] `PipelineNode` 的 `metadata` 字段存储原始 dict 副本，占用双倍内存
**优先级：Info**
**位置**：`pipeline_node.py:108`

```python
metadata=data,
```

**根因**：`from_dict` 将整个输入 `data` dict 存入 `metadata` 字段，而 `PipelineNode` 本身已将 `data` 中的字段提取为独立属性。结果是每个节点持有两份相同的数据：一份在各字段中，一份在 `metadata` 中。对于大型 pipeline（数百个节点），内存开销翻倍。

**影响面**：内存效率低，但非 bug。

**修改建议**：`metadata` 应存储子集（如非标准字段），或删除该字段。

## 5. 设计问题

### [PN-5] `to_dict` 返回 `self.metadata` 引用而非深拷贝，外部修改污染节点内部状态
**优先级：Medium**
**位置**：`pipeline_node.py:111-112`

```python
def to_dict(self) -> Dict[str, Any]:
    return self.metadata
```

**根因**：返回内部 `metadata` 字典的直接引用。调用方修改返回的 dict（如 `node.to_dict()["threshold"] = 0.9`）会直接修改节点的内部状态，绕过 `PipelineNode` 的字段约束。同时 `to_dict` 的名字暗示"序列化"，但实际返回的是不可靠的引用。

**影响面**：调用方意外修改节点状态，导致后续 pipeline 执行使用被污染的值。

**修改建议**：返回深拷贝或仅包含标准字段的新 dict：
```python
def to_dict(self) -> Dict[str, Any]:
    return {
        "name": self.name,
        "recognition": self.recognition.value if isinstance(self.recognition, RecognitionType) else self.recognition,
        ...
    }
```

### [PN-6] `get_node_or_entry` 的 fallback 仅在子节点缺失时触发，不在链路断裂时触发
**优先级：Info**
**位置**：`pipeline_node.py:126-130`

```python
def get_node_or_entry(self, name: str) -> Optional[PipelineNode]:
    node = self.nodes.get(name)
    if node is None and self.entry_points:
        node = self.nodes.get(self.entry_points[0])
    return node
```

**根因**：该方法在 `pipeline_runner.py:52` 的 `run()` 方法中用于解析入口节点。如果入口节点名称在 `graph.nodes` 中不存在，回退到 `entry_points[0]`。但如果 `entry_points` 为空或 `entry_points[0]` 也不在 `nodes` 中，返回 `None`。`run()` 正确检查了 `current is None` 并返回错误。问题是 `get_node_or_entry` 的 fallback 语义过于宽松——它应该只在 `name` 是入口的别名时才回退，而不是在任何节点缺失时都回退。

**影响面**：低。可能导致错误的节点被作为入口执行。

### [PL-6] `extract_module_nodes` 使用字符串前缀匹配，可能误匹配模块名子串
**优先级：Info**
**位置**：`pipeline_loader.py:65`

```python
if name.startswith(module_prefix):
```

**根因**：`startswith` 匹配前缀。如果模块前缀为 `"Main"`，会匹配 `"MainChapter"` 和 `"Main"` 自身。如果模块命名约定中有层级（如 `"CombatMain"` 和 `"Main"`），前缀 `"Main"` 会错误包含 `"CombatMain"` 的节点。

**影响面**：低。取决于实际模块命名。
