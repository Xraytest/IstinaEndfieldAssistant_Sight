## §1 审计纠错（对批次 56 · 1730 报告）

### [A1 — 推翻] 批次 56 `SRV-01`(High) + `AUDIT-2` 复活的 D1 为假阳性，force-stop 实际生效

**批次 56 论断**：`recovery.py:72` 的 `["shell","am force-stop",pkg]` 把 `"am force-stop"` 作为"单个参数"传给 adb shell，Android shell 把它当作"不含空格的命令名"而非 `am` 的子命令 `force-stop`，故"force-stop 永远不会生效"，并据此在 `AUDIT-2` 称"批次 7 D1 仍未修复"。

**代码复核（决定性证据）** —— `recovery.py:66-79`：

```python
def _run(self, args: list[str], serial: Optional[str]) -> None:
    cmd = self._resolve_adb(serial) + args          # 拼接为 argv 列表
    subprocess.check_output(cmd, text=True, timeout=30)

def _force_stop(self, serial: Optional[str]) -> None:
    self._run(["shell", "am force-stop", self._package], serial)
```

`subprocess.check_output(list)` 把列表元素当作 **argv 令牌（token）**。adb 进程收到的 argv 为：
`[adb, (可含 -s serial), "shell", "am force-stop", pkg]`

**关键点（批次 56 混淆了两层语义）**：
- **本地层**：`"am force-stop"` 只是 Python 列表里的一个 argv 元素，adb 客户端**不会**把它原样当成"一个命令名"传给设备。
- **设备层**：`adb shell` 的实现会把 `"shell"` 之后的**所有 argv 令牌用单个空格重新拼接**成一个命令字符串发给设备。因此设备实际执行的是：
  `am force-stop com.hypergryph.endfield`
  这正是 `am` 工具 `force-stop` 子命令的**规范语法**，force-stop **实际生效**。

批次 56 把"本地 argv 里是一个元素"错误地等同于"设备端看到的是不能拆分的命令名"，从而得出错误结论。

**佐证 1 — 安全白名单自证规范语法**：`shell_security.py:13-23` 的 `ALLOWED_SHELL_PREFIXES` 明确包含 `"am force-stop "`（带尾随空格的前缀）：

```python
ALLOWED_SHELL_PREFIXES: tuple[str, ...] = (
    ...
    "am start ",
    "am force-stop ",      # ← 规范命令前缀
    "wm ",
    ...
)
```

该安全模块把 `am force-stop <pkg>` 视为**合法允许前缀**，反证批次 56 "该命令永远无效"的论断在事实与项目自身契约上均不成立。

**佐证 2 — 早期审计结论一致**：本记忆文件记录批次 0659·A1、1017、1337·A1、1442 **均独立得出 D1/C01 为假阳性**（`adb shell` 对 post-shell 参数按空格拼接，`am force-stop <pkg>` 正确生效，且 `self._package` 为硬编码常量无注入面）。批次 56 在未重新核对 `_run` 实现的情况下**复古了已被撤销的 D1**，构成历史报告的自我矛盾。

**结论**：`SRV-01`(High) 与 `AUDIT-2`("D01 仍未修复")均为**错误论断**，应从活跃风险台账移除。`SRV-01` 建议的"修复"（`["shell","am","force-stop",pkg]`）与现状在行为上**完全等价**（adb 同样会以空格拼接），进一步说明 D1 从来不是真实缺陷。

### [A2 — 降级] 批次 56 `SRV-02`(Medium) 关于 `_start_agent` "逻辑倒置/靠巧合工作" 的过评

**批次 56 论断**：`maa_end/runtime.py:386-400` 的轮询循环"逻辑倒置""靠巧合工作"。

**代码复核**（`maa_end/runtime.py:387-400`）：

```python
ready = False
for _ in range(10):
    if process.poll() is not None:   # 进程已立即退出 → fail-fast
        break
    time.sleep(0.5)
    if process.poll() is None:      # 0.5s 后仍在运行 → 视为就绪
        ready = True
        break
if not ready:
    ... error ...; return
```

逐路径推演：
- **正常启动**（进程存活）：第 1 次 `poll()` 为 `None` → 跳过首个 if → `sleep(0.5)` → 仍 `None` → `ready=True` → `break` → 成功。✔
- **立即崩溃**（启动即退出）：第 1 次 `poll()` 非 `None` → `break`（保持 `ready=False`）→ 报错返回。这是**正确的 fail-fast** 语义。✔
- **睡眠期间崩溃**（首查存活、sleep 后死亡）：`if poll() is None` 为 `False`，不 break，循环进入第 2 次 → 首个 `if poll() is not None: break` 命中 → `ready=False` → 报错返回。✔

**结论**：逻辑**正确且语义清晰**（"立即退出=失败；存活 0.5s=就绪"），并非"倒置"。所谓"靠巧合工作"不成立。仅存在**风格层面的可改进点**（可用 `for…else` 表达 "10 次均未就绪" 的兜底）。原 Medium 评级**虚高**，建议降为 **Info/Style**，不作为代码错误计入。

## §2 新发现（确属历史未覆盖）

### [NEW-LOW] `llm/client.py:74` — `health_check` 在 `base_url` 不含 `/v1` 时生成错误端点

```python
def health_check(self) -> bool:
    url = f"{self._base_url.split('/v1', 1)[0]}/health"
    ...
```

`str.split('/v1', 1)[0]` 在 `/v1` **存在**时正确切出协议+主机端口（`http://127.0.0.1:9998/health`）；但当 `base_url` **不含** `/v1`（如误配置或被手工构造为 `http://host:port`）时，`split` 不命中、返回**整个原串**，`url` 退化为 `http://host:port/v1/health`——出现**重复 `/v1`**，命中错误端点，健康检查恒失败或请求 404。

**触发条件**：仅当 `_base_url` 缺 `/v1` 时（正常由 `LlamaServerRuntime.base_url` 拼接，默认含 `/v1`，常规路径不触发）。属 **Low / 健壮性边界**。

**建议**：改为显式解析并去除尾随 `/v1` 后再拼 `/health`，或统一以 `urljoin` 处理：
```python
base = self._base_url
if base.endswith("/v1"):
    base = base[:-3]
base = base.rstrip("/")
url = f"{base}/health"
```

> 说明：同一文件 `LlmClient._post`（:98）将 `LogCategory.MAIN` 作为首个 `%s` 值内联进消息串、丢失结构化分类元数据的问题，已被批次 56 `LLM05` 以 Low 计入，本批不重复。
