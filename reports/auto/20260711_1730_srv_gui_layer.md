### [SRV-01 High] `recovery.py:72` — shell 子命令格式错误导致 force-stop 永远不生效

```python
# recovery.py:70-72
def _force_stop(self, serial: Optional[str]) -> None:
    try:
        self._run(["shell", "am force-stop", self._package], serial)
```

**问题**：`_run` 将列表拼接为 adb 命令行参数：`["adb", "-s", serial, "shell", "am force-stop", pkg]`。`subprocess` 将 `"am force-stop"` 作为**单个**参数传递给 adb shell。Android shell 将 `"am force-stop"` 解释为单个命令名（不含空格的命令），而非 `am` 工具的子命令 `force-stop`。结果：force-stop **永远不会生效**，旧进程持续残留。

同理，`recovery.py:111` 的 `monkey` 启动命令也存在参数拼接问题：
```python
self._run(["shell", "monkey", "-p", self._package, "-c", "android.intent.category.LAUNCHER", "1"], serial)
```
此处 `-c` 和 `android.intent.category.LAUNCHER` 是分开的 list 元素，adb 会正确传递，但 `monkey` 命令本身已被废弃（Android 10+），存在兼容性风险。

**建议**：修复 force-stop 命令格式：
```python
self._run(["shell", "am", "force-stop", self._package], serial)
```

### [SRV-02 Medium] `maa_end/runtime.py:386-400` — `_start_agent` 就绪判断逻辑倒置

```python
# maa_end/runtime.py:386-400
ready = False
for _ in range(10):
    if process.poll() is not None:
        # 进程已退出
        break          # ← 直接 break，不设置 ready
    time.sleep(0.5)
    if process.poll() is None:
        ready = True
        break
if not ready:
    self.logger.error(...)
    self._agent_client = None
    self._agent_process = None
    return
```

**问题**：循环逻辑倒置。第一次迭代时，`process.poll()` 刚创建通常为 None（进程还在运行），所以第一个 `if process.poll() is not None` 不成立，进入 `time.sleep(0.5)`。sleep 后 `process.poll()` 仍为 None 时 `ready=True`。但如果进程在 sleep 期间退出，`process.poll()` 变为非 None，`ready` 保持 False。但**初始 break 逻辑错误**：第一次 `poll()` 为 None 时不 break，sleep 后若 poll 仍为 None 则 ready=True。实际上如果 go-service 在 sleep 后立即退出（如 DLL 加载失败），循环通过 `break`（第一个 if）退出，但此时 `ready` 仍为 False。

然而更严重的问题是：第一次 `if process.poll() is not None` 在 sleep 前不 break（因为刚创建的进程 poll 通常为 None），所以正常情况下 ready 会被设为 True。但逻辑上是依赖 sleep 后的 poll 状态，且首次迭代的 break 永远不会触发。整体逻辑**靠巧合工作**，而非清晰的意图表达。

**建议**：重写为清晰的逻辑：
```python
for _ in range(10):
    if process.poll() is not None:
        self.logger.error("go-service exited immediately")
        self._agent_client = None
        self._agent_process = None
        return
    time.sleep(0.5)
    if process.poll() is None:
        break  # 进程仍在运行，视为就绪
else:
    self.logger.error("go-service did not become ready in time")
    self._agent_client = None
    self._agent_process = None
    return
```

### [SRV-03 Medium] `runtime.py:740` — 日志记录格式字符串错误

```python
# runtime.py:740
self._logger.warning("VLM keyevent '%s' failed: %s", key, exc)
```

**问题**：`key` 是字符串（如 `"KEYCODE_W"`），`exc` 是 Exception 对象。logger 使用 `%s` 格式化，但 `exc` 作为单独参数传递。实际上 Python logging 会将 `(key, exc)` 作为 `%s` 格式化元组，正确输出 `"VLM keyevent 'KEYCODE_W' failed: [Errno 111] Connection refused"`。**此条格式字符串实际正确，标记为误报候选**——见 AUDIT-1。

然而，`exc` 作为 Exception 对象传递，`%s` 会调用 `str(exc)`，这与 `%r` 调用 `repr(exc)` 不同。对于大多数异常两者等价，但对于自定义异常类可能丢失类型信息。

### [SRV-05 Medium] `runtime.py:745` — `_decode_image` 无错误处理


### [SRV-07 Low] `llm/runtime.py:85` — `get_instance` 无条件覆盖已有实例配置

```python
# llm/runtime.py:78-86
@classmethod
def get_instance(cls, config: Dict[str, Any]) -> LlamaServerRuntime:
    port = int(config.get("port", 9998))
    with cls._lock:
        if port not in cls._instances:
            cls._instances[port] = cls(config)
        instance = cls._instances[port]
```

**问题**：`get_instance` 在单例已存在时无条件覆盖 `instance._config`。如果两个不同的调用方（如 GUI 和 CLI）以不同 config 调用 `get_instance`，后调用的 config 会覆盖先调用的。这意味着：
- 如果 GUI 以 `model_path=A` 启动实例，然后 CLI 以 `model_path=B` 调用，实例的 config 变为 B
- 但 GUI 的实例已引用旧 config，可能导致 model_path 不一致

**建议**：首次创建后不应覆盖 config，或至少记录 config 变更警告。

### [SRV-09 Info] `maa_end/runtime.py:48-56` — 猴子补丁 `AgentClient.__del__` 风险

```python
# maa_end/runtime.py:48-56
if AgentClient is not None:
    _original_agent_client_del = AgentClient.__del__

    def _safe_agent_client_del(self):
        try:
            _original_agent_client_del(self)
        except Exception:
            pass

    AgentClient.__del__ = _safe_agent_client_del
```

**问题**：在类定义后猴子补丁 `__del__` 存在以下风险：
1. **对象复活**：如果原始 `__del__` 异常导致对象复活（self 被引用），补丁版本吞掉异常后可能产生不可预期的对象状态
2. **线程安全**：`__del__` 可能在任意线程的 GC 中被调用，原始实现可能有线程局部状态
3. **不可逆**：一旦补丁，全局生效，影响所有 AgentClient 实例

虽然意图是防止 MaaFramework 的 `__del__` 在解释器关闭时抛出异常导致进程崩溃，但更好的方案是在 `MaaEndRuntime._cleanup_partial` 中显式调用 `agent_client.disconnect()` 而非依赖 `__del__`。

**建议**：移除 `__del__` 补丁，确保所有资源通过显式 `disconnect()` / `_cleanup_partial()` 释放。

### [AUDIT-1] 批次 56 SRV-03 误报候选 — `runtime.py:740` 格式字符串实际正确

分析 `runtime.py:740`：
```python
self._logger.warning("VLM keyevent '%s' failed: %s", key, exc)
```

Python logging 模块的 `logger.warning(msg, *args)` 使用 `msg % args` 格式化。`key` 是 `str`，`exc` 是 `Exception`。`%s` 对两者都调用 `str()`，输出格式正确：`"VLM keyevent 'KEYCODE_W' failed: [Errno 111] Connection refused"`。**格式字符串无错误**。

但发现该行位于 `except Exception as exc` 块中，`exc` 作为 Exception 对象传递。虽然 `%s` 等价于 `str(exc)`，但如果调用方需要 traceback 信息，应使用 `self._logger.exception(...)` 或 `self._logger.warning(..., exc_info=True)`。

**结论**：SRV-03 降级为 Info 级——格式字符串本身无误，但缺少 `exc_info` 导致调试信息不足。

### [AUDIT-3] 批次 45 N03 仍存活 — `runtime.py:86-88` `__getattr__` 单例

批次 45 N03 报告的 `AndroidRuntimeProxy.__getattr__` 单例问题**仍未修复**，现位于 `runtime.py:86-88`：

```python
def __getattr__(self, name: str) -> Any:
    client = self._client_for(None)
    return getattr(client, name)
```

此 `__getattr__` 将所有属性访问委托给内部 client，如果 client 尚未初始化（如 `_client_for` 抛出异常），`__getattr__` 会递归调用自身导致 `RecursionError`。此外，`__getattr__` 仅在**正常属性查找失败**时触发，如果 `AndroidRuntimeProxy` 有同名的类属性，会优先返回类属性而非 client 的同名属性。

**建议**：在 `__getattr__` 开头添加递归保护，并考虑使用显式代理方法代替魔法方法。
