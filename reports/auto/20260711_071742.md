### 2.1 🟢 中危问题

#### D01 - `_resolve_input_tokens` 使用 `json.loads(json.dumps(...))` 做深拷贝丢失非JSON类型数据

**文件**: `src/core/service/maa_end/runtime.py:512`
**风险等级**: 🟢 中危
**影响**: pipeline_override payload 中的非JSON可序列化值（tuple、set、自定义对象、bytes等）在深拷贝后静默丢失

**问题代码**:
```python
def _resolve_input_tokens(self, payload: Dict[str, Any], value: Any) -> Dict[str, Any]:
    if not isinstance(value, dict):
        return payload
    resolved = json.loads(json.dumps(payload))  # ❌ 非JSON类型静默丢失
    for key, val in resolved.items():
        if isinstance(val, str):
            resolved[key] = self._replace_tokens(val, value)
        elif isinstance(val, dict):
            resolved[key] = self._resolve_input_tokens(val, value)
    return resolved
```

**问题分析**:
- `json.loads(json.dumps(payload))` 作为深拷贝手段只能处理JSON可序列化类型
- 如果 `payload` 中的值包含 `tuple`（坐标对）、`set`（枚举值）、`bytes`（二进制数据）或自定义对象
- 这些值在序列化/反序列化过程中会变成 `list`（tuple）或丢失（set、bytes、自定义对象）
- 调用方期望保留原始类型，但实际收到的是被转换或丢失的数据
- 这可能导致 pipeline 执行时参数类型不匹配

**修复建议**:
```python
import copy

def _resolve_input_tokens(self, payload: Dict[str, Any], value: Any) -> Dict[str, Any]:
    if not isinstance(value, dict):
        return payload
    resolved = copy.deepcopy(payload)  # ✅ 使用 copy.deepcopy 保留所有类型
    for key, val in resolved.items():
        if isinstance(val, str):
            resolved[key] = self._replace_tokens(val, value)
        elif isinstance(val, dict):
            resolved[key] = self._resolve_input_tokens(val, value)
    return resolved
```

#### D02 - `VlmWalkNavigator._is_stuck` 使用绝对阈值与目标距离尺度不一致

**文件**: `src/core/service/navigation/vlm_walk_navigator.py:303-311`
**风险等级**: 🟢 中危
**影响**: VLM导航可能在缓慢接近目标时误判为"卡住"，触发不必要的 navmesh 回退

**问题代码**:
```python
def _is_stuck(self, cx: float, cy: float) -> bool:
    self._last_positions.append((cx, cy))
    if len(self._last_positions) < self._config.stuck_threshold:
        return False
    xs = [p[0] for p in self._last_positions]
    ys = [p[1] for p in self._last_positions]
    spread = (max(xs) - min(xs)) + (max(ys) - min(ys))
    return spread < 2.0  # ❌ 硬编码绝对阈值，与 target_radius 尺度无关
```

**问题分析**:
- `spread < 2.0` 是硬编码的绝对阈值
- `target_radius` 是 12.0（单位：地图坐标单位）
- 如果角色以每步0.5单位的速度稳定接近目标，4步后 spread=2.0，恰好触发卡住判定
- 但角色实际上正在接近目标（距离仍可能>12.0），不应判定为卡住
- 不同的地图坐标尺度（大世界 vs 室内）对"缓慢移动"的定义不同
- 绝对阈值无法适应不同场景

**修复建议**:
```python
def _is_stuck(self, cx: float, cy: float) -> bool:
    self._last_positions.append((cx, cy))
    if len(self._last_positions) < self._config.stuck_threshold:
        return False
    xs = [p[0] for p in self._last_positions]
    ys = [p[1] for p in self._last_positions]
    spread = (max(xs) - min(xs)) + (max(ys) - min(ys))
    # 使用相对阈值：与当前距离目标的距离成比例
    return spread < 2.0 and self._last_distance is not None and self._last_distance > self._config.target_radius * 2
```

### 2.2 🟢 低危问题

#### D03 - `execute()` 静默丢弃多余点号分段且无日志

**文件**: `src/core/service/runtime.py:275-281`
**风险等级**: 🟢 低危
**影响**: 超过2个点号的命令被静默标记为"unknown"，调试困难

**问题代码**:
```python
def execute(self, command: str, params: Optional[Dict[str, Any]] = None) -> Any:
    params = params or {}
    parts = command.split(".")
    if len(parts) == 2:
        domain, action = parts
    elif len(parts) == 1:
        domain, action = parts[0], ""
    else:
        domain, action = "unknown", command  # ❌ 静默丢弃，无日志
```

**问题分析**:
- 超过2个点号的命令（如 `"nav3.walk.target"`）被静默处理
- `domain="unknown"` 导致后续路由失败，返回 `None`
- 调用方收到 `None` 后可能因缺少 `get("status")` 检查而崩溃
- 无任何日志记录该错误路由
- 在调试多级子命令时无法追踪问题

**修复建议**:
```python
else:
    self._logger.warning(LogCategory.MAIN, "命令点号分段过多", command=command, parts_count=len(parts))
    domain, action = "unknown", command
```

#### D04 - `_try_recover` 不验证 `_reconnect()` 结果即返回 True

**文件**: `src/core/service/maa_end/runtime.py:740-743`
**风险等级**: 🟢 低危
**影响**: 应用重启成功但设备未连接，导致后续任务在断开状态下执行

**问题代码**:
```python
def _try_recover(self, task_name: str) -> bool:
    ...
    try:
        restart_policy = AndroidAppRestartPolicy(...)
        ok = restart_policy.restart(serial=self._device_address)
        if ok:
            self._reconnect()  # ❌ 不检查返回值
            return True       # ❌ 假定连接成功
    except Exception as exc:
        ...
    return False
```

**问题分析**:
- `restart_policy.restart()` 只负责重启应用，不负责验证设备连接
- `self._reconnect()` 可能因设备未就绪（重启需要时间、ADB延迟等）而失败
- 但方法仍返回 `True`，导致调用方认为设备已恢复
- 后续任务在 `_connected=False` 状态下尝试执行，所有截图和操作都会失败

**修复建议**:
```python
if ok:
    self._reconnect()
    if self._connected:  # ✅ 验证连接状态
        return True
    self.logger.warning(LogCategory.MAIN, "重启后连接未恢复", task=task_name)
```

#### D05 - `SettingsPage._read_config` 静默丢弃损坏配置无备份

**文件**: `src/gui/pyqt6/pages/settings_page.py:207-214`
**风险等级**: 🟢 低危
**影响**: 配置文件损坏时，用户所有设置丢失且无法恢复

**问题代码**:
```python
def _read_config(self) -> Dict[str, Any]:
    if not self._config_path.exists():
        return {}
    try:
        return json.loads(self._config_path.read_text(encoding="utf-8"))
    except json.JSONDecodeError as exc:
        QMessageBox.warning(...)
        return {}  # ❌ 返回空dict，原始损坏文件仍覆盖在原位置
```

**问题分析**:
- 配置文件损坏时（如磁盘故障、写入中断、手动编辑错误），返回空 `{}`
- 用户所有LLM参数、预览设置等丢失
- 损坏文件未被备份，无法恢复原始数据
- `QueueState` 类已有 `_backup_corrupt_state` 方法实现备份逻辑，但 `SettingsPage` 未复用

**修复建议**:
```python
def _read_config(self) -> Dict[str, Any]:
    if not self._config_path.exists():
        return {}
    try:
        return json.loads(self._config_path.read_text(encoding="utf-8"))
    except json.JSONDecodeError as exc:
        self._backup_corrupt_config()
        QMessageBox.warning(...)
        return {}

def _backup_corrupt_config(self) -> None:
    """备份损坏的配置文件，避免每次启动都失败。"""
    try:
        backup = self._config_path.with_suffix(".json.bak")
        counter = 0
        while backup.exists():
            backup = self._config_path.with_suffix(f".json.bak.{counter}")
            counter += 1
        import os
        os.replace(self._config_path, backup)
    except Exception:
        pass
```
