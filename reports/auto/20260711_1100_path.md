### [MAA01 Medium] `main.py:8-11,21` 路径设置的冗余与脆弱的鸡-蛋问题

**位置**：`src/gui/pyqt6/main.py:8-11`（手动 sys.path）和 `:21`（ensure_src_path）

```python
# line 8-11: 手动计算路径，用于早期导入
_SRC_DIR = Path(__file__).resolve().parent.parent.parent
_src_str = str(_SRC_DIR)
if _src_str not in sys.path:
    sys.path.insert(0, _src_str)

# line 13-19: 导入 core.* 和 gui.* 模块（依赖上述路径）
from core.foundation.paths import ensure_src_path
from gui.pyqt6.i18n import get_locale_manager
...

# line 21: 再次调用 ensure_src_path（与 line 8-11 功能重复）
ensure_src_path(__file__)
```

**根因分析**：这是一个经典的"鸡-蛋"问题——需要先设置 sys.path 才能导入 `core.foundation.paths`，但 `ensure_src_path()` 本身就在 `core.foundation.paths` 中。当前解决方案是双重路径设置：line 8-11 手动计算，line 21 调用函数验证。

**影响面**：
- **直接**：main.py 的两段路径计算必须保持同步。如果 `main.py` 移动（如从 `src/gui/pyqt6/` 移到 `src/gui/`），line 8 的 3 级 parent 链和 line 21 的 `ensure_src_path(__file__)` 会自动适应（因为 ensure_src_path 从传入路径计算），但 line 8 的手动计算不会更新。
- **间接**：任何修改 main.py 位置的开发者必须同时更新两处路径计算。

**修改方案**：
```python
# 方案1：将 ensure_src_path 移到独立模块（如 paths.py 本身），
#        使其不依赖 core.* 导入
# 方案2：在 main.py 顶部使用硬编码的 3 级 parent 链，
#        移除 line 21 的冗余 ensure_src_path() 调用
# 方案3：创建一个 bootstrap 模块（如 src/_bootstrap.py）
#        只做 sys.path 设置，不依赖任何 core.* 导入
```

**位置**：
- `src/gui/pyqt6/pages/maaend_control_page.py:127`
  ```python

  ```

  ```python
  base = Path(__file__).resolve().parent.parent.parent.parent.parent / "cache"
  ```

**根因分析**：`maaend_control_page.py` 位于 `src/gui/pyqt6/pages/`，向上 5 级才能到达项目根。这是项目中**最深的 parent 链**（其他文件最多 4 级）。5 级链比 4 级链多一层脆弱性——任何目录结构调整（如在 `pages/` 下新增子目录）都会破坏此路径。

**对比**：
| 文件 | parent 链深度 | 使用 get_project_root()? |
|------|-------------|----------------------|
| paths.py:22 | 4 级 | ✅ 是（定义者） |
| logger.py:118 | 4 级 | ❌ 否 |
| qt_log_filter.py:59 | 4 级 | ❌ 否 |
| queue_state.py:29 | 4 级 | ❌ 否 |
| scripting/recorder.py:56 | 4 级 | ❌ 否 |
| scripting/scripting_page.py:39 | 4 级 | ❌ 否 |
| **maaend_control_page.py:127** | **5 级** | ❌ 否 |
| **maaend_control_page.py:1165** | **4 级（fallback）** | ❌ 否 |
| **maaend_control_page.py:1183** | **5 级** | ❌ 否 |

**影响面**：`_OPTION_LOCALE_PATH` 用于加载任务选项的中文翻译。如果路径断裂，选项标签会回退到英文原文（`_resolve_label` 中的 `return raw` fallback）。`_metadata_cache_path` 用于缓存元数据，路径断裂会导致缓存无法读写。

**修改方案**：

```python
from core.foundation.paths import get_project_root

_OPTION_LOCALE_PATH = get_project_root() / "3rd-part" / "maaend" / "locales" / "interface" / "zh_cn.json"
# ...
base = get_project_root() / "cache"
```

### [MAA04 Low] `paths.py:61` `ensure_src_path()` 内重复计算项目根

**位置**：`src/core/foundation/paths.py:58-61`

```python
def ensure_src_path(path: Optional[str] = None) -> None:
    if path is not None:
        source = Path(path).resolve()
        project_root = source.parent.parent.parent.parent  # ← 重复计算
    else:
        project_root = get_project_root()
```

**根因分析**：当传入 path 参数时，`ensure_src_path()` 独立计算 project_root（4 级 parent 链），而非调用 `get_project_root()`。这意味着如果项目目录结构调整，`get_project_root()` 和 line 61 的两处计算必须同步更新。

**影响面**：低——`get_project_root()` 和 line 61 的计算逻辑相同（都是 4 级 parent），但维护时容易遗漏一处。

**修改方案**：
```python
def ensure_src_path(path: Optional[str] = None) -> None:
    if path is not None:
        project_root = get_project_root()  # 复用已有函数
    else:
        project_root = get_project_root()
    src_dir = project_root / "src"
    ...
```

### [MAA05 Low] `maaend_control_page.py:1165` fallback 路径使用 4 级 parent 链

**位置**：`src/gui/pyqt6/pages/maaend_control_page.py:1161-1165`

```python
try:
    from core.foundation.paths import get_project_root
    base = Path(get_project_root()) / "config"
except Exception:
    base = Path(__file__).resolve().parent.parent.parent.parent / "config"  # ← fallback
```

**根因分析**：`_state_path` 方法在 `get_project_root()` 导入失败时，fallback 到 4 级 parent 链。虽然 fallback 逻辑合理（处理极端导入失败情况），但 fallback 路径本身也应使用 `get_project_root()` 的等效计算，而非硬编码 parent 链。当前 `except Exception` 过于宽泛——它会吞掉 `ImportError`（合理）但也吞掉 `ModuleNotFoundError`、`SyntaxError` 等。

**影响面**：低——fallback 仅在极端导入失败时触发。但 `except Exception` 过于宽泛，可能掩盖非路径相关的导入错误。

**修改方案**：
```python
except ImportError:
    base = Path(__file__).resolve().parent.parent.parent.parent / "config"
```

### [MAA07 Info] 全局 197 处 bare `except Exception:` 模式

**统计**：`src/` 目录下共 197 处 `except Exception:`（无 `as e`，无处理逻辑）和 86 处 `except Exception as e:`（仅记录日志）。

**根因分析**：这是项目中最普遍的结构性反模式。虽然批次 37-45 已逐批报告了大量静默吞错的具体实例（N01, S02, S05, D03, MCP06 等），但全局统计显示此模式的普遍性远超已报告数量。

**典型模式分类**：
| 模式 | 数量 | 示例 |
|------|------|------|
| `except Exception: pass` | ~30 | template_backend.py:248, yolo_backend.py 等 |
| `except Exception: return []` | ~25 | 多个后端 recognize 方法 |
| `except Exception: return None` | ~20 | 多个加载/解析方法 |
| `except Exception as e: logger.debug` | ~86 |  widespread |
| `except Exception: return {}` | ~10 | 特征提取、配置加载 |
| 其他 | ~26 | 各种 fallback 模式 |

**影响面**：系统性风险。静默吞错使调试极度困难——生产环境中识别器/导航器/设备连接的异常被完全隐藏，用户只看到"功能失效"但无日志线索。
