### [SCR01 Medium] `recognizer.py:70` 直接修改另一类的内部状态

**位置**：`src/core/capability/element_recognition/recognizer.py:70`

```python
if self._catalog:
    self._template_backend._catalog = self._catalog
```

**根因分析**：`EndfieldElementRecognizer.__init__` 在初始化后，直接赋值修改 `TemplateBackend` 的私有属性 `_catalog`。这是一种脆弱的紧耦合——`TemplateBackend` 的实现细节（`_catalog` 是否存在、类型是什么、何时加载）被 `recognizer` 侵入式修改。如果未来 `TemplateBackend` 将 `_catalog` 改为 property、改为延迟加载、或改为只读属性，此行代码会静默失败（属性不存在时抛 AttributeError，property setter 不存在时抛 AttributeError）。

**影响面**：`TemplateBackend` 的所有调用方——包括 `recognize()`、`recognize_templates()`、`get_available_templates()`——都依赖 `_catalog` 的正确性。如果此赋值失败，模板匹配会使用空目录，导致所有模板匹配返回空结果。

**修改方案**：
```python
# 方案1：通过公开 API 传递 catalog
self._template_backend.set_catalog(self._catalog)

# 方案2：在 TemplateBackend.__init__ 中接受 catalog 参数
# （已在 batch 44 报告中建议过 TemplateBackend 改进方向）
```

### [SCR02 Medium] `matcher.py:27, 102` `cv2.cvtColor` 对 BGRA 输入抛异常

**位置**：`src/core/capability/element_recognition/backends/matcher.py:27`（`match` 方法）和 `:102`（`match_all_instances` 方法）

```python
screen_gray = cv2.cvtColor(screen, cv2.COLOR_BGR2GRAY) if len(screen.shape) == 3 else screen
```

**根因分析**：条件 `len(screen.shape) == 3` 只检查维度数。对于 BGRA 截图（4 通道，shape=(H, W, 4)），维度数为 3，条件为 True，进入 `cvtColor` 分支。但 `COLOR_BGR2GRAY` 不接受 4 通道输入，会抛 `cv2.error: Assertion failed (CN > 0 && CN <= 4)`。

scrcpy 截图在某些配置下可能返回 BGRA（4 通道），而 ADB screencap 通常返回 BGR（3 通道）。此代码假设所有 3 维输入都是 BGR，忽略了 BGRA 的可能性。

**影响面**：当 scrcpy 返回 BGRA 截图时，`match` 和 `match_all_instances` 都会崩溃。由于此方法被 `TemplateBackend._match_single` 调用（batch 44 N01 已报告该处静默吞错），实际影响是被 `except Exception: pass` 掩盖——模板匹配静默返回 None。但吞错掩盖了真正的病根（BGRA 通道不匹配），使调试困难。

**修改方案**：
```python
channels = screen.shape[2] if len(screen.shape) == 3 else 1
if channels == 4:
    screen_gray = cv2.cvtColor(screen, cv2.COLOR_BGRA2GRAY)
elif channels == 3:
    screen_gray = cv2.cvtColor(screen, cv2.COLOR_BGR2GRAY)
else:
    screen_gray = screen
```

### [SCR03 Medium] `minimap_locator.py:167-179` `break` 仅退出内层循环

**位置**：`src/core/service/navigation/minimap_locator.py:167-179`

```python
if "Tier" in tile_class:
    parts = tile_class.split("Tier")
    if len(parts) == 2:
        tier_str = parts[1]
        grid_data = self._data.load_grid_tiers()
        for gk, gc in grid_data.items():
            for tier_id in gc.items.values():
                if tier_id.endswith(f"tier_{tier_str}"):
                    grid_cell = gk
                    cx, cy = gc.center
                    break          # ← 仅退出内层 for tier_id 循环
            if grid_cell:
                break           # ← 退出外层 for gk, gc 循环
```

**根因分析**：双层嵌套循环中，内层 `break`（line 178）仅退出 `for tier_id in gc.items.values()`，不退出外层 `for gk, gc in grid_data.items()`。外层通过 `if grid_cell: break`（line 179）补偿。逻辑上正确，但代码可读性差——读者可能误以为 line 178 的 break 同时退出两层循环。更严重的是：如果 `grid_data` 中有多个匹配项（同一 tier 在不同 grid_cell 中），代码会取第一个匹配，而非最接近的匹配。

**影响面**：当同一 tier 在多个 grid cell 中出现时，可能取到错误的 grid_cell，导致导航定位偏移。在大型地图中，这种偏移可能使角色走到错误的位置。

**修改方案**：
```python
for gk, gc in grid_data.items():
    for tier_id in gc.items.values():
        if tier_id.endswith(f"tier_{tier_str}"):
            grid_cell = gk
            cx, cy = gc.center
            break
    if grid_cell:
        break
# 添加注释明确 break 的作用域
```

### [SCR04 Low] `element_info.py:64-73` `__post_init__` 静默篡改数据

**位置**：`src/core/capability/element_recognition/element_info.py:64-73`

```python
def __post_init__(self) -> None:
    if self.element_type not in ELEMENT_TYPES:
        self.element_type = "unknown"       # 静默篡改
    if self.action not in ("tap", "swipe", "none", "unknown"):
        self.action = "unknown"             # 静默篡改
    self.confidence = max(0.0, min(1.0, float(self.confidence)))
    ...
```

**根因分析**：`ElementInfo` 的 `__post_init__` 在构造后静默修改 `element_type` 和 `action`。如果调用方传入 `element_type="foobar"`，不会收到任何警告——值被静默改为 `"unknown"`。这使得上游代码无法区分"原始数据就是 unknown"和"原始数据无效被修正为 unknown"，掩盖了数据质量问题。

**影响面**：所有创建 `ElementInfo` 的代码路径（5 个后端 + 识别器 + 场景服务）。如果某个后端返回了无效的 element_type（如拼写错误），静默修正使其在日志中看起来正常，但实际丢失了原始类型信息。

**修改方案**：
```python
def __post_init__(self) -> None:
    if self.element_type not in ELEMENT_TYPES:
        logger.warning("Invalid element_type '%s', falling back to 'unknown'", self.element_type)
        self.element_type = "unknown"
    if self.action not in ("tap", "swipe", "none", "unknown"):
        logger.warning("Invalid action '%s', falling back to 'unknown'", self.action)
        self.action = "unknown"
```

### [SCR05 Low] `scene_service.py:56-58` 异常处理丢失诊断上下文

**位置**：`src/core/service/scene_service.py:56-58`

```python
try:
    page = self.recognizer.recognize(screen)
except Exception as exc:
    logger.warning("场景识别异常: %s", exc)
    return PageInfo(page_type="unknown", confidence=0.0)
```

**根因分析**：`identify` 方法捕获所有异常后返回一个通用的 "unknown" 结果。虽然异常被记录为 warning，但返回的 `PageInfo` 丢失了所有诊断信息——调用方无法区分"识别器正常返回 unknown"和"识别器崩溃了"。对于自动化系统，这种区分很重要：前者可能触发重试，后者可能需要重启识别器。

**影响面**：所有调用 `identify` 的自动化任务。当识别器因内存不足、OpenCV 错误等崩溃时，调用方收到一个看似正常的 "unknown" 结果，可能继续执行后续步骤而不是报告故障。

**修改方案**：
```python
except Exception as exc:
    logger.warning("场景识别异常: %s", exc)
    return PageInfo(
        page_type="error",
        confidence=0.0,
        metadata={"error": str(exc), "error_type": type(exc).__name__}
    )
```

### [SCR06 Low] `player.py:62, 84` 信号真值检查恒为 True

**位置**：`src/gui/pyqt6/scripting/player.py:62` 和 `:84`

```python
if self.playback_started:    # line 62
    self.playback_started.emit()

if self.playback_stopped:    # line 84
    self.playback_stopped.emit()
```

**根因分析**：`playback_started` 和 `playback_stopped` 是 `pyqtSignal` 对象，不是 Python 的可调用函数。`pyqtSignal` 对象在布尔上下文中永远为 True（它们不是 None，也不是空列表）。因此这两个条件判断恒为 True，与是否存在槽连接无关。

**影响面**：运行时无实际影响（信号恒为 True，emit 正常执行），但此代码模式是技术债——如果未来将信号改为可空类型（如 `Optional[pyqtSignal]`），此检查会静默失效。

**修改方案**：
```python
# 移除恒为 True 的条件判断
self.playback_started.emit()
```
