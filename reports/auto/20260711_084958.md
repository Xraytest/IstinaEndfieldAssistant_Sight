### 2.1 🔴 致命问题

#### M01 - `_vlm_keyevent` 发送的字母键被 `_is_valid_keyevent` 静默拒绝，VLM 步行导航完全失效

**文件**: `src/core/service/runtime.py:712-730`（调用方）、`src/core/capability/device/android_runtime.py:75-84`（拦截方）
**风险等级**: 🔴 致命
**影响**: VLM 步行导航（nav3 walk）的所有按键操作全部失败

**完整调用链**:

1. `VlmWalkNavigator._execute_action`（`vlm_walk_navigator.py:264-282`）将 VLM 返回的 action 映射为字母键：
```python
def _execute_action(self, action):
    act = action["action"]
    if act == "forward":
        self._input_fn("w", duration)      # 字母 "w"
    elif act == "backward":
        self._input_fn("s", duration)      # 字母 "s"
    elif act == "left":
        self._input_fn("a", duration)      # 字母 "a"
    elif act == "right":
        self._input_fn("d", duration)      # 字母 "d"
    elif act == "turn_left":
        self._input_fn("q", None)          # 字母 "q"
    elif act == "turn_right":
        self._input_fn("e", None)          # 字母 "e"
    elif act == "interact":
        self._input_fn("f", None)          # 字母 "f"
```

2. `_input_fn` 在 `IstinaRuntime._nav3_walk`（`runtime.py:689-700`）中绑定为 `_vlm_keyevent`：
```python
def _nav3_walk(self, params):
    nav = self.navigator()
    return nav.to_coords_vlm(
        ...,
        keyevent_fn=self._vlm_keyevent,  # 绑定
    )
```

3. `_vlm_keyevent`（`runtime.py:712-730`）将字母键原样传递给 `android.keyevent()`：
```python
def _vlm_keyevent(self, key: str, duration):
    android = self.android()
    if duration is not None and duration > 0.3:
        for _ in range(repeats):
            android.keyevent(key)  # 传入 "w"
    else:
        android.keyevent(key)     # 传入 "q"/"e"/"f"
```

4. `_is_valid_keyevent`（`android_runtime.py:75-84`）只接受纯数字或 17 个 KEYCODE_ 常量名：
```python
_KNOWN_KEYEVENT_NAMES = frozenset({
    "KEYCODE_HOME", "KEYCODE_BACK", "KEYCODE_APP_SWITCH",
    ...
    "KEYCODE_DPAD_CENTER",
})

def _is_valid_keyevent(key):
    s = str(key).strip()
    if s.isdigit():
        return True
    return s in _KNOWN_KEYEVENT_NAMES  # "w" 不在其中！
```

**分析**:
- VLM 返回的 `"w"/"a"/"s"/"d"/"q"/"e"/"f"` 是 PC 游戏风格的按键命名（WASD 移动 + QE 视角 + F 交互）
- `_is_valid_keyevent` 基于 Android KEYCODE 规范设计，只接受 `KEYCODE_*` 常量名或纯数字
- 字母键被 `_is_valid_keyevent` 拒绝后，`_Daemon._dispatch` 返回错误响应，但 `_vlm_keyevent` 的 try/except 只记录了 warning，不抛出异常
- **结果**: 导航步骤执行时按键操作静默失败，VLM 看到的前进/转向操作未生效，角色不移动
- 与批次23 W1 的区别：W1 报告了 whitelist 本身的缺陷，M01 发现了**集成层的致命影响**——VLM 导航完全不可用
- 此 bug 是导航失效的主要原因，与批次34 J01（伪造坐标）叠加形成双重失效：角色既不移动（M01），位置坐标也可能是假的（J01）

**修复建议**（两选一）:

方案A：在 `_vlm_keyevent` 中做按键映射
```python
_KEY_MAP = {
    "w": "KEYCODE_DPAD_UP",
    "s": "KEYCODE_DPAD_DOWN",
    "a": "KEYCODE_DPAD_LEFT",
    "d": "KEYCODE_DPAD_RIGHT",
    "q": "KEYCODE_DPAD_LEFT",  # 或自定义
    "e": "KEYCODE_DPAD_RIGHT",
    "f": "KEYCODE_ENTER",
}

def _vlm_keyevent(self, key, duration=None):
    mapped = _KEY_MAP.get(key, key)
    ...
```

方案B：在 `_is_valid_keyevent` 中添加 WASD 映射
```python
def _is_valid_keyevent(key):
    s = str(key).strip()
    if not s:
        return False
    if s.isdigit():
        return True
    _WASD_MAP = {
        "w": "KEYCODE_DPAD_UP",
        "a": "KEYCODE_DPAD_LEFT",
        "s": "KEYCODE_DPAD_DOWN",
        "d": "KEYCODE_DPAD_RIGHT",
        "q": "KEYCODE_DPAD_LEFT",
        "e": "KEYCODE_DPAD_RIGHT",
        "f": "KEYCODE_ENTER",
    }
    mapped = _WASD_MAP.get(s)
    if mapped:
        key = mapped  # 需修改调用链传递方式
    return s in _KNOWN_KEYEVENT_NAMES
```

推荐方案A，更符合单一职责原则。

### 2.2 🟡 中危问题

#### M02 - `execute()` 对未知命令返回 None，破坏数据契约

**文件**: `src/core/service/runtime.py:348-349`
**风险等级**: 🟡 中危
**影响**: 调用方期望返回 dict 时获得 None，导致 AttributeError

**问题代码**:
```python
# runtime.py:348-349
self._logger.warning(LogCategory.MAIN, "未知命令", command=command)
return None  # ❌ 调用方期望 dict
```

**分析**:
- `execute()` 的返回值契约：有 `@property` 的 `config` 返回 dict，所有子方法返回 dict 或 bool
- 但未知命令分支返回 `None`——与 `{"status": "error", "message": ...}` 约定不一致
- GUI 的 `_sync_execute` 和 CLI 的 `execute_command` 通常调用 `result.get("status", "error")`
- `None.get(...)` 抛出 `AttributeError: 'NoneType' object has no attribute 'get'`
- 这会导致前端显示异常，而非优雅的错误消息
- 虽然已记录 warning，但返回结构应始终一致

**修复建议**:
```python
self._logger.warning(LogCategory.MAIN, "未知命令", command=command)
return {"status": "error", "message": f"unknown command: {command}", "command": command}
```

#### M03 - `reload_config` 直接写入 MaaEndRuntime 的"私有"属性

**文件**: `src/core/service/runtime.py:473`
**风险等级**: 🟡 中危
**影响**: MaaEndRuntime 内部属性改名时，配置重载静默失败

**问题代码**:
```python
# runtime.py:468-474
def reload_config(self) -> None:
    self._config = self._load_config()
    device_cfg = self._config.get("device", {}) or {}
    adb_restart_on_timeout = bool(device_cfg.get("adb_restart_on_timeout", True))
    for runtime in self._maaend_clients.values():
        runtime._adb_restart_on_timeout = adb_restart_on_timeout  # ❌ 访问私有属性
```

**分析**:
- `_adb_restart_on_timeout` 在 `MaaEndRuntime` 中以 `_` 前缀声明

- 如果 MaaEndRuntime 将来将属性改为 `adb_restart_on_timeout` 或 property，`reload_config` 静默失败
- `MaaEndRuntime` 应提供 `set_adb_restart_on_timeout()` 公共方法
- 当前代码依赖实现细节，维护成本高

#### M04 - `_ensure_game_in_world` 中 `shell("pidof")` 调用无超时控制

**文件**: `src/core/service/runtime.py:514-521`
**风险等级**: 🟡 中危
**影响**: 设备断连或 shell 服务异常时，调用可能长时间阻塞

**问题代码**:
```python
# runtime.py:514-521
def _ensure_game_in_world(self, runtime, serial, client_version):
    try:
        android = self.android(serial)
        pid = android.shell("pidof com.hypergryph.endfield").strip()  # ❌ 无超时
        if pid:
            self._logger.info(...)
        else:
            self._logger.info(...)
    except Exception as exc:
        self._logger.warning(...)
```

**分析**:
- `android.shell()` 的底层是 ADB shell 命令或 JSON-RPC 调用
- 如果设备已断开连接但 proxy 未清除，shell 调用可能无限等待
- 虽有外层 try/except 捕获异常，但无超时机制意味着可能阻塞数分钟
- `_wait_for_in_world` 有超时（120秒），但 shell 调用在它之前，没有独立的超时
- 如果 `pidof` 命令因 ADB 问题而挂起，后续 `AndroidOpenGame` 任务永远不会启动

**修复建议**:
```python
import concurrent.futures
...
try:
    with concurrent.futures.ThreadPoolExecutor(max_workers=1) as executor:
        future = executor.submit(android.shell, "pidof com.hypergryph.endfield")
        pid = future.result(timeout=5).strip()  # 5秒超时
except concurrent.futures.TimeoutError:
    self._logger.warning("pidof 超时，跳过进程检查")
    pid = ""
```

#### M05 - `_wait_for_in_world` 无连续失败计数，循环可能跑满超时

**文件**: `src/core/service/runtime.py:531-543`
**风险等级**: 🟢 低危（从上次的中危降级，考虑实际场景影响）
**影响**: EnterGame pipeline 不存在时无意义地跑满 120 秒

**问题代码**:
```python
# runtime.py:531-543
def _wait_for_in_world(self, runtime, timeout=120, interval=2):
    deadline = time.time() + timeout
    while time.time() < deadline:
        try:
            if runtime.run_pipeline("EnterGame", {}):
                return True
        except Exception as exc:
            self._logger.warning(...)
        time.sleep(interval)
    return False
```

**分析**:
- 如果 `EnterGame` pipeline 始终返回 False，循环以 `interval=2` 秒间隔跑满 `(120/2)=60` 次
- 每次 `run_pipeline` 调用有 overhead（序列化、传输、MAA 内部执行）
- 如果 pipeline 不存在，每次调用都会返回 False 并消耗时间
- 应该设置连续失败阈值（如连续 5 次失败后提前退出）

**修复建议**:
```python
def _wait_for_in_world(self, runtime, timeout=120, interval=2):
    deadline = time.time() + timeout
    consecutive_failures = 0
    while time.time() < deadline:
        try:
            if runtime.run_pipeline("EnterGame", {}):
                return True
            consecutive_failures = 0
        except Exception as exc:
            self._logger.warning(...)
            consecutive_failures += 1
        time.sleep(interval)
        consecutive_failures += 1
        if consecutive_failures >= 15:  # 30秒连续失败后退出
            break
    return False
```

#### M06 - `_llm_client_instance` property 在 LLM runtime 启动失败时静默设置 None

**文件**: `src/core/service/runtime.py:129-132`
**风险等级**: 🟢 低危
**影响**: LLM 运行时不可用时，后续调用抛出 AttributeError

**问题代码**:
```python
# runtime.py:129-132
@property
def _llm_client_instance(self) -> Any:
    if self._llm_client is None:
        self._llm_client = _get_llm_client(self._llm_runtime_instance)
    return self._llm_client
```

**分析**:
- `_llm_runtime_instance` property 内部调用 `_get_llama_runtime` → `LlamaServerRuntime.get_instance()`
- 如果 llama-server 未安装或端口被占用，`_get_llama_runtime` 可能返回异常实例或抛出异常
- `_get_llm_client` 调用 `LlmClient(base_url=...)`，如果 runtime 未启动，`base_url` 可能是无效值
- 初始化失败时 `_llm_client` 保持 None，后续 `_llm_client_instance.chat()` 调用 `None.chat()` 抛出 AttributeError
- 虽然 `_llm_run` 有 try/except，但 `_llm_status` 等其他调用方没有保护

**修复建议**:
```python
@property
def _llm_client_instance(self) -> Any:
    if self._llm_client is None:
        try:
            self._llm_client = _get_llm_client(self._llm_runtime_instance)
        except Exception as exc:
            self._logger.error("LLM client 初始化失败: %s", exc)
            raise RuntimeError(f"LLM client init failed: {exc}") from exc
    return self._llm_client
```

### 2.3 🟢 低危问题

#### M07 - `AndroidRuntimeProxy.__getattr__` 创建客户端后无缓存清理

**文件**: `src/core/service/runtime.py:86-100`
**风险等级**: 🟢 低危
**影响**: 断连后代理仍持有无效客户端引用

**问题代码**:
```python
class AndroidRuntimeProxy:
    def __getattr__(self, name: str) -> Any:
        client = self._client_for(None)
        return getattr(client, name)  # ❌ 客户端断连后仍返回同一实例

    def _client_for(self, serial):
        resolved = serial or self._device_address or "default"
        client = self._clients.get(resolved)
        if client is None:
            client = _get_android_runtime()(serial=resolved, adb_path=self._adb_path)
            self._clients[resolved] = client
        return client
```

**分析**:
- `_clients` dict 缓存客户端实例，断连后实例的 `connected=False`
- 但 `_client_for` 只检查 `client is None`，不检查客户端是否有效
- 断开设备后，代理持有的客户端引用指向已断连的 AndroidRuntime
- 调用 `android.keyevent()` 时，底层已断连的操作会静默失败或抛异常
- 应添加客户端有效性检查或清理机制

**修复建议**:
```python
def _client_for(self, serial):
    resolved = serial or self._device_address or "default"
    client = self._clients.get(resolved)
    if client is None or not getattr(client, '_connected', False):
        if client is not None:
            try:
                client.disconnect()
            except Exception:
                pass
        client = _get_android_runtime()(serial=resolved, adb_path=self._adb_path)
        self._clients[resolved] = client
    return client
```

#### M12 - `_connect_with_timeout` 超时后 Daemon 线程持续运行

**文件**: `src/core/service/maa_end/runtime.py:267-284`
**风险等级**: 信息
**影响**: 超时后资源未释放，后台线程继续占用 ADB 连接

**问题代码**:
```python
def _connect_with_timeout(self, timeout):
    result = {"success": False}
    def target():
        try:
            result["success"] = self._connect_once()
        except Exception as exc:
            result["success"] = False

    thread = threading.Thread(target=target, daemon=True)
    thread.start()
    thread.join(timeout=timeout)

    if thread.is_alive():
        self.logger.error("ADB 连接超时")
        self._cleanup_partial()
        return False  # ❌ daemon 线程仍在运行
```

**分析**:
- `_connect_once` 内部创建了 `AdbController`、`Tasker`、Agent 进程等资源
- 超时后 daemon 线程继续运行，`_connect_once` 可能继续创建资源
- `_cleanup_partial()` 只清理 `_controller/_tasker/_resource/_agent_process/_agent_client`
- 但 daemon 线程可能正在执行中间步骤（如等待 `post_connection().wait()`），清理后继续操作已被释放的资源
- daemon 线程在程序退出时自动终止，但在长期运行的服务中可能持续占用

**修复建议**: 使用 Event 或 threading.Timer 实现真正的超时中断，而非简单的 `join(timeout)`。

## 6. 结论

本次审计发现10个新问题，其中1个致命、3个中危、4个低危、2个信息。历史报告零修正。

**关键发现**:
1. **M01（致命）**: `_vlm_keyevent` 发送的字母键（w/a/s/d/q/e/f）被 `_is_valid_keyevent` 静默拒绝——VLM 步行导航完全失效。这是批次23 W1 的致命升级：W1 指出 whitelist 本身限制，M01 发现此限制导致导航功能完全不可用。
2. **M02（中危）**: `execute()` 对未知命令返回 None，破坏数据契约。
3. **M03（中危）**: `reload_config` 直接写入私有属性，违反封装。
4. **M04（中危）**: shell 调用无超时控制。
5. **M07-M10**: 低危问题涵盖代理客户端管理、legacy 路径残留、重试机制缺失、重复实例创建。
6. **M06/M11/M12**: 信息级问题涵盖 LLM 初始化异常处理、连接状态管理、线程超时。

**建议修复顺序**:
1. **M01**: 添加 WASD→KEYCODE 映射（致命，导航功能基础）
2. **M02**: 修复 execute() 返回值（中危，数据契约）
3. **M03**: 添加公共 setter（中危，代码规范）
4. **M04**: 添加 shell 超时（中危，稳定性）
5. **M07**: 添加客户端有效性检查
6. **M08-M10**: 代码质量改进
7. **M06/M11/M12**: 代码质量改进

**关联分析**: M01 与批次34 J01、批次36 L01 形成"三重失效链"：VLM 导航无法发送有效按键（M01）→ 角色不移动 → 定位依赖小地图（L01 硬编码 bbox 可能裁剪失败）→ 即使定位成功，坐标可能是伪造的（J01）。修复 M01 是恢复导航功能的首要步骤。
