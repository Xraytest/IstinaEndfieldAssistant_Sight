## 4. 任务/预设名不一致

### CFG-07（High）任务文件内部名与文件名不一致，导致按名加载失败
**文件**：
- `assets/tasks/CreditShopping.json` → 内部任务名为 `CreditShoppingN2`
- `assets/tasks/Weapon.json` → 内部任务名为 `WeaponUpgrade`

**根因**：
- `MaaEndRuntime.load_tasks()`（`src/core/service/maa_end/runtime.py:133`）通过扫描所有 JSON 文件并按内部 `name` 字段索引，因此 `CreditShoppingN2` 和 `WeaponUpgrade` 能进入 `_tasks` 字典。
- 但 `TaskLoader.load_task(task_name)`（`src/core/capability/element_recognition/tasks/task_loader.py:25`）按 `assets/tasks/{task_name}.json` 做文件名查找。调用 `load_task("CreditShoppingN2")` 时，会在 `assets/tasks/` 下寻找 `CreditShoppingN2.json`，该文件不存在，返回 `None`。

**触发路径**：
1. 预设 `DailyFull` / `QuickDaily` 中引用了 `CreditShoppingN2`
2. 若任何代码路径通过 `TaskLoader.load_task("CreditShoppingN2")` 加载该任务，会返回 `None`
3. 调用方可能因此认为任务不存在，而实际上任务定义在 `CreditShopping.json` 内

**影响**：
- 当前 `MaaEndRuntime` 的 `run_preset` → `run_queue` → `run_task` 使用 `load_tasks()` 扫描索引，因此能执行；
- 但 `TaskLoader`（识别/任务元数据模块）无法按名加载这两个任务，导致 GUI 的任务详情面板、选项编辑器等功能对这两个任务失效。

**修复建议**：
1. 将 `CreditShopping.json` 内部名改为 `CreditShopping`，或重命名文件为 `CreditShoppingN2.json`；
2. 将 `Weapon.json` 内部名改为 `Weapon`，或重命名文件为 `WeaponUpgrade.json`；
3. 建立文件名与内部 `name` 一致的约束，并在 CI 中校验。

## 5. 配置加载安全

### CFG-09（Medium）`IstinaRuntime` 接受任意 config_path，存在路径遍历
**文件**：`src/core/service/runtime.py:109-110, 447-466`

**根因**：
```python
def __init__(self, config_path: Optional[str] = None):
    self._config_path = Path(config_path).resolve() if config_path else None
```
`Path.resolve()` 会规范化 `..` 和符号链接，但不会限制结果必须位于项目 `config/` 目录内。CLI 入口（`src/cli/istina.py:42-46`）暴露了 `--config` 参数，直接将用户输入传入。

**触发路径**：
```
istina --config ../../../../etc/passwd.json daily run
```
`_resolve_config_path()` 会返回解析后的 `/etc/passwd.json`，`_load_config()` 尝试以 JSON 格式读取它，`save_config()` 甚至可以向任意路径写入。

**影响**：
- 信息泄露：可读取项目外部的敏感 JSON 文件内容（虽然会因 JSON 解析失败而返回 `{}`，但错误日志可能泄露路径）
- 任意文件写入：`save_config()` 在 `_config_path` 父目录不存在时会 `mkdir(parents=True, exist_ok=True)`，然后覆写目标文件

**修复建议**：
1. 在 `_resolve_config_path()` 中校验：
   ```python
   def _resolve_config_path(self) -> Path:
       if self._config_path is not None:
           resolved = self._config_path
           try:
               resolved.relative_to(get_project_root() / "config")
           except ValueError:
               raise ValueError(f"Config path must be under project config/: {self._config_path}")
           return resolved
       return get_project_root() / "config" / "client_config.json"
   ```
2. 或者移除 `--config` 参数，仅允许内建路径。

### CFG-10（Medium）`_resolve_model_path` 允许绝对路径逃逸项目目录
**文件**：`src/core/capability/llm/runtime.py:240-247`

**根因**：
```python
def _resolve_model_path(self, model_path: str) -> str:
    p = Path(model_path)
    if p.exists():
        return str(p)
    candidate = get_project_root() / model_path
    if candidate.exists():
        return str(candidate)
    return model_path
```
当 `model_path` 为绝对路径（如 `C:\malicious\model.gguf` 或 `/models/test.gguf`）且文件存在时，直接返回该路径。`client_config.json` 中的 `model_path: "/models/test.gguf"` 在 Windows 上解析为 `\models\test.gguf`（当前盘符根目录）。

**影响**：
- LLM 可以加载项目目录之外的模型文件；
- 攻击者若能控制 `client_config.json`，可指定任意路径的模型，可能用于加载恶意模型或消耗系统资源。

**修复建议**：
```python
def _resolve_model_path(self, model_path: str) -> str:
    p = Path(model_path)
    if p.is_absolute():
        # 限制绝对路径必须在已知安全目录内
        allowed = [Path("/models"), get_project_root() / "models"]
        if not any(str(p).startswith(str(a.resolve())) for a in allowed):
            raise ValueError(f"Model path outside allowed directories: {model_path}")
        return str(p)
    candidate = get_project_root() / model_path
    if candidate.exists():
        return str(candidate)
    return model_path
```

## 6. 配置值安全与健壮性

### CFG-11（Medium）`client_config.json` 含不安全默认值与类型错误风险
**文件**：`config/client_config.json`

**发现的问题值**：

| 键 | 当前值 | 问题 |
|---|---|---|
| `llm.model_path` | `"/models/test.gguf"` | Windows 上为非法绝对路径；文件不存在导致 LLM 启动必失败 |
| `llm.mmproj_path` | `"models/LLM/mmproj-F16.gguf"` | 相对路径，依赖 cwd；可能不存在 |
| `llm.n_gpu_layers` | `999` | 极端值，超出常见 GPU 显存容量，易导致 OOM |
| `llm.enabled` | `true` | 配合上述无效路径，启动时会产生误导性失败 |
| `device.serial` | `"192.168.1.12:16512"` | 硬编码默认设备 IP，内网环境变化时自动连接会失败 |
| `device.history` | `["192.168.1.12:16512", "localhost:8888", "localhost:9999"]` | 历史记录包含 localhost 映射，泄露测试环境痕迹 |

**根因**：
- `LlamaServerRuntime._build_args()` 对 `context_size`、`threads`、`temperature`、`n_gpu_layers` 直接调用 `int()` / `float()` 无范围校验。基线报告 `20260710_2350_llm.md` 已指出 `TypeError` 风险（JSON `null` → `None` → `int(None)` 崩溃），本报告补充具体危险值。

**影响**：
- 首次运行 LLM 功能时，因 `model_path` 无效而启动失败，用户需手动排查；
- `n_gpu_layers=999` 在显存不足时导致进程崩溃；
- `auto_connect_last=true` + 硬编码 serial 导致每次启动自动尝试连接已下线设备。

**修复建议**：
1. `model_path` 改为空字符串或相对于 `models/` 的相对路径；
2. `n_gpu_layers` 改为 `-1`（自动）或合理上限如 `99`；
3. 增加启动前配置校验：
   ```python
   def _validate_llm_config(cfg):
       ngl = int(cfg.get("n_gpu_layers", -1))
       if ngl > 99:
           raise ValueError(f"n_gpu_layers {ngl} exceeds safe limit 99")
       # ... 其他校验
   ```
4. `device.serial` 默认改为空字符串，避免硬编码 IP。

### CFG-12（Medium）配置加载无 schema 验证，malformed JSON 静默回退默认值
**文件**：
- `src/core/service/runtime.py:447-455`
- `src/core/capability/llm/runtime.py`（无校验）
- `src/core/capability/element_recognition/tasks/task_loader.py:33-76`

**根因**：
```python
def _load_config(self) -> Dict[str, Any]:
    try:
        with open(path, "r", encoding="utf-8") as f:
            return json.load(f)
    except Exception as e:
        self._logger.warning(...)
    return {}
```
所有配置加载点均使用 `except Exception` 吞掉所有错误（JSON 语法错误、权限错误、磁盘错误），静默返回空字典。调用方获得 `{}` 后使用各种 `.get(key, default)`，导致配置变更被无声忽略。

**影响**：
- 用户手动编辑 `client_config.json` 时写错一个逗号，所有配置（包括设备 serial、LLM 参数）全部回退到代码默认值，且无明确告警；
- 自动化部署中若配置文件损坏，系统继续运行但行为不符合预期。

**修复建议**：
1. 区分错误类型：`json.JSONDecodeError` 提示配置语法错误；`PermissionError` 提示权限；其他异常向上抛；
2. 引入轻量 schema 校验（如 `jsonschema` 或手写验证函数），在加载后校验必填键和类型；
3. 至少记录原始异常堆栈，而非仅 `str(e)`。
