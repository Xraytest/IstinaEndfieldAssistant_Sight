## 新增发现（2 项）

### LLM-CUDA-FLAG — `_cuda_failed` 标记在 CPU 回退成功后仍为 True，后续启动跳过 GPU 尝试

**等级**: 代码质量 / 中
**位置**: `src/core/capability/llm/runtime.py:140-142`

**问题代码**:

```python
# llm/runtime.py:139-143
ok = self._try_start(exe, model_path, llm_cfg)
if not ok and not self._cuda_failed:
    self._cuda_failed = True          # ← 标记设为 True，但后续 CPU 回退成功后不重置
    return self._try_start(exe, model_path, llm_cfg, force_cpu=True)
return ok
```

**根因分析**:

`LlamaServerRuntime.start()` 在首次 GPU 启动失败后将 `_cuda_failed` 设为 True，然后尝试 CPU 回退。如果 CPU 回退也失败（`ok=False`），`_cuda_failed` 仍保持 True。后续任何 `start()` 调用都会：

1. 先尝试 GPU（line 139: `self._try_start(exe, model_path, llm_cfg)`）
2. GPU 失败后，因 `_cuda_failed=True`，跳过 CPU 回退
3. 直接 `return ok`（False）

这意味着：

- 用户修复 CUDA 环境（驱动更新、重装等）后，`start()` 仍不会尝试 GPU
- 唯一的恢复路径是调用 `reset_cuda_state()`（line 72-75），但该方法在代码库中**无任何调用点**
- 用户必须重启整个 CLI 进程才能重置标记

**与同类代码的对比**:

| 启动方法 | CUDA 失败处理 |
|---------|-------------|
| `LlamaServerRuntime.start()` (line 140-142) | 标记为 True，CPU 回退成功后不重置 |
| `reset_cuda_state()` (line 72-75) | 手动重置标记，但无人调用 |
| CLI `llm start` handler | 仅调用 `warmup_llm()`，不感知 CUDA 状态 |

**调用链**:
```
用户点击"启动 LLM"
  → CLI handler: warmup_llm()
    → LlamaServerRuntime.start()
      → _try_start(GPU) 失败
      → _cuda_failed = True
      → _try_start(CPU) 失败
      → return False
  用户修复 CUDA 后再次点击"启动 LLM"
  → LlamaServerRuntime.start()
    → _try_start(GPU) 失败
    → _cuda_failed=True → 跳过 CPU 回退
    → return False
  ← 用户困惑：明明已修好 CUDA，为何仍无法启动
```

**影响面**:
- **中**：CUDA 环境可变（驱动更新、系统重启），标记永驻导致 GPU 路径被永久跳过。CPU 回退仅有一次机会，失败后 LLM 功能完全不可用。

**修复建议**:

```python
# llm/runtime.py:139-143
ok = self._try_start(exe, model_path, llm_cfg)
if not ok and not self._cuda_failed:
    self._cuda_failed = True
    cpu_ok = self._try_start(exe, model_path, llm_cfg, force_cpu=True)
    if cpu_ok:
        self._cuda_failed = False  # CPU 成功则重置标记，允许后续 GPU 重试
    return cpu_ok
return ok
```

或更简洁的方案：始终尝试 CPU 回退，不依赖 `_cuda_failed` 标记：

```python
ok = self._try_start(exe, model_path, llm_cfg)
if ok:
    self._cuda_failed = False
    return True
if not self._cuda_failed:
    self._cuda_failed = True
    cpu_ok = self._try_start(exe, model_path, llm_cfg, force_cpu=True)
    if cpu_ok:
        self._cuda_failed = False
    return cpu_ok
return False
```

### PRTS-CHAT-ERROR — PRTS 页面仅连接 `commandFinished`，无 `commandError` 处理器，聊天错误时 UI 冻结

**等级**: 代码质量 / 中
**位置**: `src/gui/pyqt6/pages/prts_full_intelligence_page.py:165,230-244`

**问题代码**:

```python
# prts_full_intelligence_page.py:165
self._bridge.commandFinished.connect(self._on_command_finished)
# ← 仅连接 commandFinished，commandError 无处理器

# prts_full_intelligence_page.py:230-244
self._worker = LlmChatWorker(self._bridge, text, image_b64, self)
self._worker.start()
# ...
self._send_btn.setEnabled(False)   # 发送时禁用按钮
self._prompt_input.setEnabled(False)
```

**根因分析**:

`LlmChatWorker` 在后台线程执行 `self._bridge.execute("llm chat", params)`。发送时禁用按钮和输入框（line 239-240）。只有在 `commandFinished` 信号触发时，`_on_command_finished`（line 209-224）才会重新启用按钮：

```python
elif len(cmd_parts) >= 2 and cmd_parts[0] == "llm" and cmd_parts[1] == "chat":
    ...
    self._send_btn.setEnabled(True)   # 仅在此处启用
    self._prompt_input.setEnabled(True)
    self._worker = None
```

`CLIBridge` 在以下场景会发出 `commandError` 而非 `commandFinished`：

| 场景 | 信号 |
|------|------|
| CLI 子进程崩溃 5 次 | `commandError`（line 224-228） |
| 进程启动超时 | `commandError`（line 141） |
| 业务错误（非零退出码） | **两者都发**（line 246-253） |
| 正常完成 | `commandFinished` |

但 PRTS 页面**未连接 `commandError`**。所以：

1. 如果聊天命令在子进程崩溃场景（`_restart_pending` 为 True）下失败，`commandFinished` 不会发出，按钮永远禁用
2. 如果进程启动超时，`commandFinished` 不会发出，按钮永远禁用
3. 业务错误场景（非零退出码）同时发出两个信号，按钮会重新启用——但 `_on_command_finished` 只处理 `result.get("status") == "success"` 和 `else`（错误消息），不检查 `_worker` 是否为旧引用

**与同类代码的对比**:

| 页面 | 连接的信号 | 错误处理 |
|------|----------|---------|
| `PrtsFullIntelligencePage` | 仅 `commandFinished` | 缺失 |
| `MaaEndControlPage` | `commandFinished` + `commandError` | 完整 |
| `DeviceSettingsPage` | `commandFinished` + `commandError` | 完整 |

**调用链**:
```
用户输入聊天消息
  → _send_chat() 禁用按钮
  → LlmChatWorker.run() 执行 bridge.execute("llm chat", ...)
  → CLI 子进程崩溃（连续 5 次）
  → _on_finished: commandError 发出，_show_crash_dialog()
  → commandFinished 不发出
  → _send_btn.setEnabled(True) 永远不执行
  → UI 冻结，用户无法继续聊天
```

**影响面**:
- **中**：PRTS 页面是实验性功能（施工中），但聊天交互是其核心功能。UI 冻结后用户必须重启应用才能继续。

**修复建议**:

```python
# prts_full_intelligence_page.py:165 后添加
self._bridge.commandError.connect(self._on_chat_error)

# 新增处理器

def _on_chat_error(self, command: str, message: str) -> None:
    cmd_parts = command.split()
    if len(cmd_parts) >= 2 and cmd_parts[0] == "llm" and cmd_parts[1] == "chat":
        self._send_btn.setEnabled(True)
        self._prompt_input.setEnabled(True)
        self._worker = None
        self._append_chat("系统", locale.tr("llm_error", "Error: {msg}").format(msg=message))
```

或在 `_send_chat` 中使用 `QTimer.singleShot` 作为兜底：

```python
# _send_chat 末尾添加兜底定时器，超时后强制启用按钮
QTimer.singleShot(30000, lambda: self._ensure_chat_ready())
```
