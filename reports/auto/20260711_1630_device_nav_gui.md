### [GUI54-01 High] `qt_log_filter.py:64-66` 安装失败时设置 `_INSTALLED = True`，永久阻塞重试

**位置**：`src/gui/pyqt6/qt_log_filter.py:61-66`

```python
def install_qt_message_filter() -> None:
    global _INSTALLED
    if _INSTALLED:
        return

    project_root = Path(__file__).resolve().parent.parent.parent.parent
    log_dir = project_root / "logs"
    try:
        log_dir.mkdir(parents=True, exist_ok=True)
    except OSError:
        # 无法建目录时不阻塞启动，Qt 日志退回默认 stderr 行为
        _INSTALLED = True      # ← 问题：设置 True 后永久无法重试
        return
```

**根因分析**：
`install_qt_message_filter()` 函数入口处检查 `if _INSTALLED: return` 实现幂等性。但当 `mkdir` 失败时（如磁盘满、权限不足），代码在 `except OSError` 块中设置 `_INSTALLED = True` 后返回，**未实际安装消息处理器**。此后任何重试调用都会因 `_INSTALLED == True` 立即返回，Qt 日志过滤永久失效。

同样的模式在 `except OSError`（line 82-85）也存在——`FileHandler` 创建失败时也设置 `_INSTALLED = True`。

**直接原因**：异常处理中过早设置 `_INSTALLED` 标记，将"安装失败"与"已安装"混淆。

**根因**：`_INSTALLED` 变量是二值布尔标记，无法区分"已安装"、"未安装"、"安装失败"三种状态。

**影响面**：
- 高——若应用启动时 `logs/` 目录创建失败（如首次运行在只读目录），Qt 字体警告将永久污染 stderr 和 main.log，且无法通过任何方式恢复。Qt 字体探测噪声（如 `qt.qpa.fonts` 警告）会在每次启动时混入控制台。

**修改方案**：
```python
# 方案1：失败时不设置 _INSTALLED，允许重试
def install_qt_message_filter() -> None:
    global _INSTALLED
    if _INSTALLED:
        return

    project_root = Path(__file__).resolve().parent.parent.parent.parent
    log_dir = project_root / "logs"
    try:
        log_dir.mkdir(parents=True, exist_ok=True)
    except OSError as exc:
        # 无法建目录，不设置 _INSTALLED，允许后续重试
        logger.warning("qt log filter: cannot create log dir: %s", exc)
        return

    qt_logger = logging.getLogger("qt")
    if not qt_logger.handlers:
        ...
    try:
        handler = logging.FileHandler(log_dir / "qt.log", encoding="utf-8")
        ...
    except OSError as exc:
        logger.warning("qt log filter: cannot create file handler: %s", exc)
        return  # 同样不设置 _INSTALLED

    qInstallMessageHandler(_qt_message_handler)
    _INSTALLED = True
```

### [DEVICE-01 Medium] `touch_manager.py:58-64` 线程不安全单例，并发构造可能创建多个实例

**位置**：`src/core/capability/device/touch_manager.py:58-64`

```python
_instance = None

@classmethod
def get_instance(cls, adb_path="3rd-part/adb/adb.exe", device_address=None):
    if cls._instance is None:           # ← 无锁检查
        cls._instance = cls(adb_path, device_address)
    return cls._instance
```

**根因分析**：
`if cls._instance is None` 检查与赋值之间无原子性保证。Python 的 GIL 在 CPython 中使简单赋值操作具有原子性，但 `cls(adb_path, device_address)` 的 `__init__` 调用 `get_logger()` 可能触发 I/O（日志文件创建），GIL 在线程切换时释放，两个线程可同时进入 `if` 分支，创建两个实例。

**直接原因**：未使用 `threading.Lock` 保护单例构造。

**根因**：项目中使用多个单例模式（`TouchManager`、`LocaleManager`、`AndroidRuntimeProxy` 等），均缺少线程安全保护。CLI 模式下单线程运行无问题，但 GUI 模式下多个页面可能并发访问。

**影响面**：
- 中——CLI 模式下单线程运行，无实际问题；GUI 模式下若未来在后台线程中调用 `TouchManager.get_instance()`，可能导致两个实例并行发送 ADB 指令，产生竞态输入。

**修改方案**：
```python
_instance = None
_instance_lock = threading.Lock()

@classmethod
def get_instance(cls, adb_path="3rd-part/adb/adb.exe", device_address=None):
    if cls._instance is None:
        with cls._instance_lock:
            if cls._instance is None:
                cls._instance = cls(adb_path, device_address)
    return cls._instance
```

- 中——多候选场景下用户体验差——全部失败后仅收到泛泛的 "failed to navigate"，无法判断是哪个实体路径不通。调试困难。

**修改方案**：
```python
for m in matches[:limit]:
    ok = self._nav_to_entity(m)
    if ok.get("status") == "success":
        return ok
    self._logger.info("nav to '%s' (%s/%s) failed: %s",
                      entity_name, m.key_name, m.map_level_id, ok.get("message"))
return {
    "status": "error",
    "message": f"failed to navigate to '{entity_name}' "
               f"({len(matches)} candidates tried)",
}
```

### [AUDIT-2] 批次 53 CLI02 修正："死代码"描述不准确，修复方案存在错误

**位置**：`reports/auto/20260711_1600_cli_handlers.md:102-119`

**原描述**：批次 53 称 `handlers.py:733` 的 `elif mem >= 2 * GB` 分支为"死代码"。

**当前代码核验**（`handlers.py:733-738`）：
```python
if mem >= 4 * GB:
    recommendation = "GPU (>=4GB)"
elif mem >= 2 * GB:
    recommendation = "GPU (>=2GB)"
else:
    recommendation = "CPU"
```

**审计结论**：`elif mem >= 2 * GB` 分支**不是死代码**——它处理 2GB ≤ mem < 4GB 的区间，是可达分支。批次 53 的分析混淆了"覆盖"与"互斥分支"概念——`mem >= 4 * GB` 确实覆盖了 `mem >= 2 * GB` 的上界，但 `elif` 结构确保了 2GB-4GB 区间由第二分支处理。

**实际存在的问题**：条件阈值存在**重叠**——4GB 以上的 GPU 被标记为 "GPU (>=4GB)"，2GB-4GB 的 GPU 被标记为 "GPU (>=2GB)"。但推荐逻辑中两个分支的**推荐行为完全相同**（均推荐 GPU），标签区分对用户无实质意义。

**批次 53 修复方案错误**：
报告中"修改方案"（第 104-109 行）和"简化为两分支"（第 112-118 行）提供的代码**仍保留 `elif mem >= 2 * GB` 分支**，与原始代码逻辑相同，未消除重叠。

**正确修复方案**：
```python
# 方案1：调整阈值，使三个分支互不重叠且行为不同
if mem >= 6 * GB:
    recommendation = "GPU (>=6GB)"
elif mem >= 3 * GB:
    recommendation = "GPU (>=3GB)"
else:
    recommendation = "CPU"

# 方案2：简化为两分支（推荐行为相同，仅区分是否推荐 GPU）
if mem >= 4 * GB:
    recommendation = "GPU"
else:
    recommendation = "CPU"
```

## 5. 关键发现总结

1. **[GUI54-01 High]**：`qt_log_filter.py` 在 `mkdir` 或 `FileHandler` 创建失败时设置 `_INSTALLED = True`，导致后续无法重试安装。Qt 日志过滤永久失效——这是 P1 级问题，一旦触发无法自动恢复。

2. **[GUI54-02/03 High]**：`main_window.py` 中 hero 标题硬编码英文、导航默认页硬编码中文，均绕过 i18n 系统。切换语言后 UI 语言不一致且默认页失效。

3. **[COLOR-01 Medium]**：颜色匹配的 RecognitionEngine 路由使用裸 `except: pass`，Engine 异常时完全静默降级到 OpenCV，对生产和调试均造成盲区。是最需要修复的 backend 问题。

4. **[SCENE-02 Medium]**：`SceneUnderstandingService.analyze_elements` 硬编码 `yolo=False`，使得 `__init__(enable_yolo=True)` 的初始化意图在 `analyze_elements` 路径下被完全无视，属于参数传递逻辑 bug。

5. **[DEVICE-01 Medium]**：`TouchManager` 为线程不安全单例，与 `LocaleManager`（i18n/__init__.py:93-98）同模式——项目中有多个单例缺少线程安全保护。

6. **[NAV-03 Low]**：`to_coords_vlm` 对 `_teleport_to` 的返回值丢弃，是 `to_coords` 和 `to_coords_vlm` 之间的行为不一致。传送失败时 VLM 路径继续在错误地图上导航。

7. **[批次 53 审计修正]**：
   - CLI02 的"死代码"描述不准确——`elif mem >= 2 * GB` 是可达分支（处理 2GB-4GB 区间），实际问题是条件阈值存在重叠。
   - CLI02 修复方案存在错误——提供的代码仍保留相同分支结构，未实际消除重叠。
   - CLI01/CLI03/CLI04/CLI05 发现成立。
