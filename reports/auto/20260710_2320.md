## 0. 边界与需求简述

IEA 是 Python 层「上层智能协调器」：GUI/CLI → `IstinaRuntime`（门面）→ `MaaEndRuntime`（MaaFW 桥）/ `AndroidRuntime`（scrcpy + ADB 守护）/ `LlamaServerRuntime` / `Navigator`。其中 `nav3`（VLM 行走）是一条独立可达链路：CLI `nav3 walk/to_entity` → `runtime.execute("nav3.*")` → `Navigator.to_coords_vlm/to_entity_vlm` → `VlmWalkNavigator.walk_to` 循环（截图 → VLM 决策 → 键盘动作 → 再截图）。设备键位由 `IstinaRuntime._vlm_keyevent` 经 `AndroidRuntime.keyevent()` 下发。

第三方 `MaaEnd/`、`3rd-part/` 不被本仓库追踪，审查仅限本仓库 Python 源码。

## 1. 代码错误（优先）

### [W1] VLM 行走导航的所有移动指令完全失效（整条 nav3 walk 特性无效）
**优先级：High**
**链路位置**：
- `src/core/service/navigation/vlm_walk_navigator.py:264-282`（`_execute_action`）
- `src/core/service/runtime.py:712-730`（`_vlm_keyevent`）
- `src/core/capability/device/android_runtime.py:75-84 / 606-612`（`_is_valid_keyevent` + `keyevent` dispatch）

**根因**：`_execute_action` 将动作映射为**键盘字母**下发：
```python
if act == "forward":   self._input_fn("w", duration)
elif act == "backward":self._input_fn("s", duration)
elif act == "left":    self._input_fn("a", duration)
elif act == "right":   self._input_fn("d", duration)
elif act == "turn_left":  self._input_fn("q", None)
elif act == "turn_right": self._input_fn("e", None)
elif act == "interact":   self._input_fn("f", None)
```
`_input_fn` 实际是 `IstinaRuntime._vlm_keyevent`，其内部 `android.keyevent("w")`。而守护进程 `_dispatch` 的 keyevent 分支强制校验：
```python
def _is_valid_keyevent(key):
    s = str(key).strip()
    if s.isdigit(): return True
    return s in _KNOWN_KEYEVENT_NAMES   # 仅 17 个 KEYCODE_* 常量
```
`"w"/"a"/"s"/"d"/"q"/"e"/"f"` 既非纯数字，也不在白名单内 → 一律返回 `{"error": "invalid keyevent: 'w'"}`。即使**忽略**白名单，`input keyevent w` 也不是合法 ADB 语法（Android 需要数字 keycode 或 `KEYCODE_W` 形式）。因此每一步移动都被丢弃。

**为何完全静默**：`AndroidRuntimeProxy.keyevent`（`android_runtime.py:784-786`）在收到 error 时**不抛异常**，仅 `return response.get("result", "")`（返回空串）；`_vlm_keyevent` 只有 `try/except`、不检查返回值，故 except 永不触发、无任何告警。结果：`walk_to` 循环空转到 `stuck`/`max_steps`，最终回退 navmesh —— 用户以为 VLM 在走路，实际角色从未移动。

**影响面**：`nav3 walk`、`nav3 to_entity` 两条 CLI 命令、`Navigator.to_coords_vlm/to_entity_vlm` 全部失效；VLM 决策、LLM 调用、截图开销全部白费；仅靠 fallback navmesh 兜底（若 fallback 也失败则彻底无效）。

**修改建议**：在 navigator 层建立「语义动作 → Android keycode」映射，而非透传字母：
```python
_KEY_TO_KEYCODE = {"w":"51","s":"47","a":"29","d":"32","q":"45","e":"33","f":"34"}
# 或使用常量名 KEYCODE_W(51)/KEYCODE_S(47)/KEYCODE_A(29)/KEYCODE_D(32)...
self._input_fn(_KEY_TO_KEYCODE[act], duration)
```
并同步在 `_KNOWN_KEYEVENT_NAMES` 中放行对应常量（若用常量名形式）。**注意**：仅按前次报告 N4 建议「放行所有 KEYCODE_*」并不能修复本问题，因为病根在 navigator 发的是字母而非 keycode（详见 §4 审计）。

**非期待变化**：修复后按键将真实下发到设备；需评估游戏是否响应 ADB 单次 keyevent（移动类操作通常需持续按压/摇杆，单击 keyevent 可能仍不足以驱动角色，属更深层设计问题，本报告仅修复「指令被丢弃」这一确定错误）。

### [W2] `_vlm_keyevent` 丢弃 keyevent 的错误返回，失败不可见
**优先级：Medium**
**位置**：`src/core/service/runtime.py:712-730`
**根因**：`android.keyevent()` 对非法/被拒键位**返回错误 dict/空串而非抛异常**（见 W1）。`_vlm_keyevent` 仅用 `try/except` 包裹、忽略返回值，因此任何下发失败都不会记录日志。这是 W1「完全静默」的直接放大器，也会掩盖真实设备离线、ADB 断连等问题。
**修改建议**：检查返回值并在异常时告警：
```python
res = android.keyevent(key)
if isinstance(res, dict) and res.get("error"):
    self._logger.warning("VLM keyevent '%s' rejected: %s", key, res["error"])
```
**影响面**：仅诊断可见性提升，无行为副作用。

### [W3] `walk_to` 在 `max_steps=0` 时 `step_idx` 未定义 → NameError
**优先级：Low**
**位置**：`src/core/service/navigation/vlm_walk_navigator.py:154 / 253`
**根因**：`max_steps` 由 CLI `nav3 walk --max-steps 0` 直接透传（`handlers.py:832` → `runtime.py:698` → `walk_to(max_steps=0)`）。`for step_idx in range(0)` 从不进入循环，随后返回值构造：
```python
"total_decisions": step_idx + 1,   # step_idx 从未绑定 → NameError
```
命令以「未捕获异常 → 通用错误」告终，信息误导。
**修改建议**：循环前 `step_idx = -1`，或在入口 `steps = max(1, max_steps or self._config.max_steps)`。
**影响面**：仅边界输入路径；修复无副作用。

## 2. 漏洞 / 健壮性问题

### [W4] VLM 动作 `duration` 未做范围与类型钳制，单步可阻塞数十秒
**优先级：Medium**
**位置**：`vlm_walk_navigator.py:264-282`（透传 `action.get("duration")`）+ `runtime.py:720-726`（`repeats = max(1, int(duration/0.15))`）
**根因**：系统提示要求 `duration` 为 1-5s，但代码对 VLM 返回值**不校验、不钳制**。若模型幻觉返回 `duration: 100`，则 `_vlm_keyevent` 执行 `int(100/0.15)=666` 次 tap，每次 `sleep(0.12)` ≈ 80s 阻塞单步；返回非数字（如字符串）则 `int()` 直接抛异常。此外 `VlmWalkConfig.step_timeout_s=30` 字段**声明后从未被使用**（全仓无引用），所谓「每步超时」保护实际不存在。
**修改建议**：解析动作时钳制 `duration = min(5.0, max(1.0, float(duration or 1.5)))` 并 try/except 兜底；或真正实现 `step_timeout_s` 看门狗。
**影响面**：`_execute_action`、`_vlm_keyevent`；修复提升鲁棒性，无功能回退。

### [W5] `minimap_locator._parse_tile` 的 level_id 提取被错误嵌套在 `Tier` 分支内
**优先级：Low（需结合模型类名确认）**
**位置**：`src/core/service/navigation/minimap_locator.py:167-181`
**根因**：`level_id` 抽取语句
```python
if tile_class.startswith(("Map01Lv", "Map02Lv")):
    level_id = tile_class[5:11].lower()
```
位于 `if "Tier" in tile_class:`（167 行）**内部**。对于不含 `Tier` 的地图瓦片类名（如纯 `Map01Lv001...`），`level_id` 恒为空串。而 `Navigator.where_am_i`（`navigator.py:128`）与附近实体查询均要求 `pos.level_id` 非空，空 level 会导致 `nearby_entities` 恒为空、VLM 上下文缺 level。
**修改建议**：将 level 抽取移到与 `Tier` 判断同级（先解析 map_id/level_id，再按需解析 Tier/grid）。
**影响面**：`classify` → `locate` → `where_am_i` / VLM 上下文。因无法离线验证 `cls.json` 实际类名，标注为 Low，修复前建议先核对模型类命名。

### [W6] `to_coords_vlm` 回退 navmesh 时使用循环前的陈旧 level
**优先级：Low**
**位置**：`src/core/service/navigation/navigator.py:242 / 275`
**根因**：`current_level` 在 VLM 行走**开始前**解析一次；VLM 走完（可能已跨层/传送）后若回退 navmesh，仍用旧 `current_level` 拼 `target_level`，与真实当前层可能不符。
**修改建议**：回退前重新 `self._locator.locate(self._get_frame())` 刷新 level。
**影响面**：仅 VLM 失败 → navmesh 兜底路径；低频。
