## 0. 边界与需求简述

LLM 集成层由两个核心类组成：
- `LlamaServerRuntime`（`runtime.py`）：管理本地 `llama-server` 子进程的生命周期（启动/停止/健康检查/优雅关闭）。
- `LlmClient`（`client.py`）：OpenAI 兼容的 HTTP 客户端，封装 `/v1/chat/completions` 调用与健康检查。

消费方通过 `IstinaRuntime` 门面按需初始化。`LlamaServerRuntime` 采用基于端口的单例注册表（同一端口仅一个实例）。`LlmClient` 由消费方实例化并传入 `base_url`。

## 1. 连接管理

### [LLM-01] `_try_start` 丢弃 stderr/stdout，启动失败完全不可诊断

**文件**：`src/core/capability/llm/runtime.py`
**行号**：320
**严重性**：Medium
**类别**：连接管理 / 错误处理

**根因**：
```python
self._process = subprocess.Popen(
    args,
    cwd=str(get_project_root()),
    stdout=subprocess.DEVNULL,
    stderr=subprocess.DEVNULL,
)
```

`llama-server.exe` 的 stdout 和 stderr 均被重定向到 `DEVNULL`。如果启动失败（例如：DLL 缺失、模型文件损坏、CUDA 驱动不兼容、端口被占用但 health endpoint 未就绪等），进程可能在启动后 0.1 秒崩溃并输出错误到 stderr，但这些信息全部丢失。

**影响面**：
- 调用方只能通过 60 秒超时得知失败，无法区分“模型缺失”与“显存不足”或“DLL 加载失败”。
- 用户需要手动用命令行复现启动命令并捕获 stderr 才能诊断，严重拖慢排障。

**修复建议**：
1. 将 stderr 捕获到 `subprocess.PIPE`，启动失败时将 `proc.stderr.read()` 记录到日志：
   ```python
   self._process = subprocess.Popen(
       args,
       cwd=str(get_project_root()),
       stdout=subprocess.DEVNULL,
       stderr=subprocess.PIPE,
   )
   ```
2. 或在 `_try_start` 的 except / 超时路径中，将启动参数记录到 debug 日志，方便用户复现：
   ```python
   self._logger.debug("llama-server start args: %s", " ".join(args))
   ```

### [LLM-02] `_cuda_failed` 跨调用持久化，首次 CPU fallback 后永久禁止 GPU

**文件**：`src/core/capability/llm/runtime.py`
**行号**：130-133
**严重性**：Medium
**类别**：连接管理 / 状态管理

**根因**：
```python
ok = self._try_start(exe, model_path, llm_cfg)
if not ok and not self._cuda_failed:
    self._cuda_failed = True
    return self._try_start(exe, model_path, llm_cfg, force_cpu=True)
return ok
```

`_cuda_failed` 是实例级状态，标记后永不重置。场景：
1. 用户首次启动时 GPU 驱动异常，`_cuda_failed` 被设为 `True`，回退 CPU 启动成功。
2. 用户修复 GPU 驱动或更换模型后再次调用 `start()`。
3. `start()` 仍走 CPU 路径，即使 GPU 现已可用。

**影响面**：
- GPU 模型在首次 CPU fallback 后无法再尝试 GPU 加速。
- 用户必须重启整个 IEA 进程（创建新的 `LlamaServerRuntime` 实例）才能恢复 GPU 推理。

**修复建议**：
- 将 `_cuda_failed` 改为每次 `start()` 的局部变量，而非实例状态。每次启动独立决策是否 fallback：
  ```python
  def start(self) -> bool:
      ...
      cuda_failed = False
      for attempt in range(2):
          ok = self._try_start(exe, model_path, llm_cfg, force_cpu=cuda_failed)
          if ok:
              return True
          if not cuda_failed:
              cuda_failed = True
              continue
          return False
  ```

### [LLM-03] `start()` 在 60 秒等待期间无任何进度日志

**文件**：`src/core/capability/llm/runtime.py`
**行号**：327-334
**严重性**：Low
**类别**：连接管理 / UX

**根因**：
```python
for _ in range(60):
    if self._process.poll() is not None:
        return False
    if self.health_check():
        self._ready = True
        return True
    time.sleep(1)
return False
```

循环共 60 次、每次 1 秒，但全程无日志。对于大型模型（首次加载需 30-60 秒），用户和开发者完全无法区分“正在加载”与“卡死”。

**修复建议**：
- 在循环内每 N 秒记录一次进度：
  ```python
  for i in range(60):
      if self._process.poll() is not None:
          return False
      if self.health_check():
          self._ready = True
          return True
      if i % 10 == 0:
          self._logger.debug("waiting for llama-server health (%ds elapsed)", i)
      time.sleep(1)
  ```

## 2. 错误处理

### [LLM-04] `_try_start` 裸 except 吞掉所有异常且不记录日志

**文件**：`src/core/capability/llm/runtime.py`
**行号**：335-336
**严重性**：Medium
**类别**：错误处理

**根因**：
```python
except Exception:
    return False
```

`_try_start` 最后的 except 块捕获所有异常（包括 `MemoryError`、`OSError`、`ValueError` 等）并静默返回 `False`。调用方 `start()` 无法区分“CUDA fallback 后仍失败”与“参数构造阶段就崩溃”。

**影响面**：
- 配置错误（如 `context_size` 为负数导致 `int()` 失败）被吞掉，用户只看到“启动失败”而不知道原因。
- 与 [LLM-01] 叠加：进程崩溃信息 + Python 异常信息全部丢失。

**修复建议**：
```python
except Exception as exc:
    self._logger.error("llama-server startup error: %s", exc, exc_info=True)
    return False
```

### [LLM-05] `chat()` 对 `temperature` / `max_tokens` 无边界校验

**文件**：`src/core/capability/llm/client.py`
**行号**：53-56
**严重性**：Low
**类别**：输入验证 / 配置处理

**根因**：
```python
if temperature is not None:
    payload["temperature"] = float(temperature)
if max_tokens is not None:
    payload["max_tokens"] = int(max_tokens)
```

`float(temperature)` 和 `int(max_tokens)` 不校验边界：
- 负温度（如 `-1.0`）被原样转发，llama-server 可能拒绝或行为异常。
- `max_tokens=0` 或 `max_tokens=10**9` 可能导致空响应或内存耗尽。

**影响面**：
- 调用方传参错误时，错误表现为服务端 HTTP 400 或超时，Python 侧无早期拦截。
- `LlmClientError` 会抛出，但错误消息是服务端返回的原始文本，不够友好。

**修复建议**：
```python
if temperature is not None:
    t = float(temperature)
    if not 0.0 <= t <= 2.0:
        raise LlmClientError(f"temperature must be in [0.0, 2.0], got {t}")
    payload["temperature"] = t
if max_tokens is not None:
    m = int(max_tokens)
    if m <= 0 or m > 32768:
        raise LlmClientError(f"max_tokens must be in (0, 32768], got {m}")
    payload["max_tokens"] = m
```

## 3. 线程安全

### [LLM-06] `_owned_pids` 集合在并发 start/stop 下无锁保护

**文件**：`src/core/capability/llm/runtime.py`
**行号**：42, 325, 186
**严重性**：Medium
**类别**：线程安全 / 资源泄漏

**根因**：
```python
self._owned_pids: Set[int] = set()  # 行 42
...
self._owned_pids.add(self._process.pid)  # 行 325（_try_start）
...
self._owned_pids.discard(pid)  # 行 186（_kill_tracked_process）
```

`set.add()` 和 `set.discard()` 在 CPython 的 GIL 下看似安全，但：
1. `_owned_pids` 的读写分散在 `start`（通过 `_try_start`）和 `stop`（通过 `_shutdown_owned` → `_kill_tracked_process`）中。
2. 如果 GUI 主线程触发 `stop()` 而工作线程正在 `start()` 的 `_try_start` 中执行 `self._owned_pids.add(...)`，两者并发操作同一个 `set` 对象。
3. GIL 保证单个字节码指令的原子性，但 `set.add()` 涉及哈希计算和内部表扩容，可能被拆分为多个字节码。

**影响面**：
- 极端情况下可能触发 `RuntimeError: Set changed size during iteration`（如果 atexit 清理时正在 add/discard）。
- 更可能的结果是 `_owned_pids` 状态不一致：PID 未被正确清理，atexit 阶段重复 kill 已死亡的进程。

**修复建议**：
- 在 `_owned_pids` 上加模块级锁，或改用 `threading.Lock` 保护所有修改：
  ```python
  self._pids_lock = threading.Lock()
  ...
  with self._pids_lock:
      self._owned_pids.add(self._process.pid)
  ...
  with self._pids_lock:
      self._owned_pids.discard(pid)
  ```

### [LLM-07] `get_instance` 静默覆盖已有实例的 `_config`

**文件**：`src/core/capability/llm/runtime.py`
**行号**：76
**严重性**：Low
**类别**：线程安全 / 配置处理

**根因**：
```python
if port not in cls._instances:
    cls._instances[port] = cls(config)
instance = cls._instances[port]
instance._config = config  # 静默覆盖
return instance
```

`get_instance` 在获取已有实例后无条件覆盖其 `_config`。如果两个调用方以不同配置请求同一端口，后到达的配置会覆盖先到达的配置。当前使用场景（单端口、单配置）下不触发，但 API 设计脆弱。

**影响面**：
- 未来多模型或多 profile 场景下，配置切换会静默生效，可能导致正在运行的进程使用过期配置。

**修复建议**：
- 仅在配置实际变更时更新，并记录日志：
  ```python
  import copy
  if copy.deepcopy(instance._config) != copy.deepcopy(config):
      self._logger.warning("LlamaServerRuntime config changed for port %s", port)
      instance._config = config
  ```

### [LLM-08] `_atexit_cleanup` 遍历 `_instances` 时无锁保护

**文件**：`src/core/capability/llm/runtime.py`
**行号**：52-58
**严重性**：Low
**类别**：线程安全 / 资源泄漏

**根因**：
```python
def _atexit_cleanup(self) -> None:
    for instance in list(LlamaServerRuntime._instances.values()):
        try:
            instance._shutdown_owned()
        except Exception:
            pass
```

`atexit` 在进程退出时执行，此时其他线程可能仍在运行（特别是 PyQt 的事件循环线程可能在关闭阶段仍在调用 `get_instance`）。`list(cls._instances.values())` 创建快照，但 `_shutdown_owned()` 可能触发 `_kill_tracked_process()` → `subprocess` 调用，在极端情况下与另一线程的 `get_instance` 内部写操作冲突。

**影响面**：
- 理论上的竞态；实际触发概率低（atexit 时其他线程通常已终止）。
- 若触发，可能导致 `RuntimeError: dictionary changed size during iteration`（因为 `_instances` 可能被 `get_instance` 修改）。

**修复建议**：
- 在遍历前持有 `_lock`：
  ```python
  def _atexit_cleanup(self) -> None:
      with LlamaServerRuntime._lock:
          instances = list(LlamaServerRuntime._instances.values())
      for instance in instances:
          ...
  ```

## 4. 资源泄漏

### [LLM-09] `_kill_tracked_process` 的 `proc.wait(timeout=5)` 不处理 `ValueError`

**文件**：`src/core/capability/llm/runtime.py`
**行号**：179, 182
**严重性**：Low
**类别**：资源泄漏

**根因**：
```python
try:
    proc.wait(timeout=5)
except subprocess.TimeoutExpired:
    proc.kill()
    proc.wait(timeout=5)
```

如果 `proc` 在两次 `wait()` 之间被外部终止（例如，用户手动 taskkill），第二次 `proc.wait()` 可能抛出 `ValueError`（"process is already stopped" 或 "process is not a child"）。该异常未被捕获，会传播到 `_kill_tracked_process` 的调用者。

**影响面**：
- `stop()` 或 `_shutdown_owned()` 可能因此崩溃，导致 `_owned_pids` 未被清理。
- 但 `finally` 块中的 `self._owned_pids.discard(pid)` 仍会执行，所以 PID 集合本身不会泄漏。

**修复建议**：
- 将第二次 `wait()` 也包裹在 try/except 中：
  ```python
  try:
      proc.wait(timeout=5)
  except (subprocess.TimeoutExpired, ValueError):
      pass
  ```

## 5. 输入验证

### [LLM-10] `chat()` 未验证 `image` 参数的真实格式

**文件**：`src/core/capability/llm/client.py`
**行号**：36
**严重性**：Low
**类别**：输入验证

**根因**：
```python
image_url = image if image.startswith("data:") else f"data:{image_mime_type};base64,{image}"
```

如果 `image` 既不是 `data:` URI，也不是有效的 base64 字符串（例如包含换行、非 base64 字符、或长度极短），客户端会构造一个格式错误的 data URI 并发送给 llama-server。服务端通常返回 400，但客户端侧无提前校验。

**影响面**：
- 图像识别请求因格式错误被服务端拒绝，`LlmClientError` 抛出的错误消息可能不明确。
- 在自动任务链路中，图像识别失败可能导致导航或场景理解中断。

**修复建议**：
- 在构造 payload 前验证 base64 格式：
  ```python
  import base64
  if not image.startswith("data:"):
      try:
          base64.b64decode(image, validate=True)
      except Exception:
          raise LlmClientError("image is not valid base64")
  ```

### [LLM-11] `_build_args` 不验证数值型配置参数的合法性

**文件**：`src/core/capability/llm/runtime.py`
**行号**：249-311
**严重性**：Low
**类别**：配置处理

**根因**：
```python
args.extend([
    "--port", str(self._default_port),
    "-c", str(int(llm_cfg.get("context_size", 2048))),
```

## 9. 修复优先级建议

**第一波（建议立即处理）**：
- **LLM-01**：启动诊断透明化。用户遇到 llama-server 启动失败时，当前完全无法自助排障。
- **LLM-02**：GPU 状态持久化 bug。一旦触发，用户只能重启进程才能恢复 GPU 加速，且无法感知。
