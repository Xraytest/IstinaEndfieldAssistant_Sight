### [N02 Low] `pipeline_node.py:60` 使用内部枚举属性 `_value2member_map_`

**位置**：`src/core/capability/element_recognition/pipeline/pipeline_node.py:60`

```python
recognition = raw if isinstance(raw, RecognitionType) else RecognitionType(raw) if isinstance(raw, str) and raw in RecognitionType._value2member_map_ else RecognitionType.DirectHit
```

**根因分析**：`_value2member_map_` 是 Python `Enum` 的内部实现细节（以 `_` 前缀标识），非公开 API。虽然当前 Python 3.12 中稳定可用，但：
1. 未来 Python 版本可能重命名或移除该属性
2. 类型检查器（mypy/pyright）会将其标记为私有成员访问
3. 应使用 `RecognitionType.__members__` 或 `RecognitionType._value2member_map_` 的公开替代

**修改方案**：
```python
# 使用 __members__ 或 try/except
if isinstance(raw, str):
    try:
        recognition = RecognitionType(raw)
    except ValueError:
        recognition = RecognitionType.DirectHit
elif isinstance(raw, RecognitionType):
    recognition = raw
else:
    recognition = RecognitionType.DirectHit
```

**影响面**：仅影响 `PipelineNode.from_dict()` 的枚举转换逻辑，不影响运行时行为。

### [N04 Medium] `yolo_backend.py:102-112` `_is_available` 首次加载失败后永不重试

**位置**：`src/core/capability/element_recognition/backends/yolo_backend.py:102-112`

```python
def _is_available(self) -> bool:
    if self._model is not None:
        return True
    try:
        from ultralytics import YOLO
        self._model = YOLO(self._model_path)
```

**根因分析**：

- 用户看到的体验：YOLO 识别从未出现，且无任何提示（仅 debug 日志）
- 与批次 23:50 REC-3（YOLO 惰性加载无锁，并发时重复加载模型）互补——REC-3 关注并发安全，N04 关注失败后永久失效

**修改方案**：
```python
def _is_available(self) -> bool:
    if self._model is not None:
        return True
    # 每次调用都尝试加载，避免首次失败后永久不可用
    try:
        from ultralytics import YOLO
        self._model = YOLO(self._model_path)
        logger.info(f"YOLO model loaded: {self._model_path}")
        return True
    except Exception as e:
        logger.debug(f"YOLO not available: {e}")
        return False
```

**影响面**：`YOLOBackend.recognize()` → `EndfieldElementRecognizer.recognize()` → 所有调用 YOLO 识别的路径。

### [N06 Medium] `logger.py:37-66` `ProjectLogger._format` 破坏 `extra` 机制

**位置**：`src/core/foundation/logger.py:37-66`

```python
def _format(self, args, kwargs):
    logging_kwargs = {}
    for key in ("exc_info", "stack_info", "stacklevel", "extra"):
        if key in kwargs:
            logging_kwargs[key] = kwargs.pop(key)  # ← extra 被弹出
    ...
    return msg, fmt_args, logging_kwargs
```

**根因分析**：
- `logging.Logger` 的 `extra` 参数是标准机制——允许调用方附加自定义字段到 LogRecord
- `ProjectLogger._format` 将 `extra` 从 `kwargs` 中弹出并放入 `logging_kwargs`，然后返回
- 调用方（line 70-91）将 `logging_kwargs` 传递给底层 `self._logger.debug(msg, *fmt_args, **logging_kwargs)`
- 表面上看这似乎正确——`extra` 最终被传递给了底层 logger
- **但**：`_format` 在弹出 `extra` 后，line 62-64 还将剩余 `kwargs` 拼接为消息后缀：
  ```python
  if kwargs:
      suffix = " ".join(f"{k}={v}" for k, v in kwargs.items())
      msg = f"{msg} {suffix}" if msg else suffix
  ```
  这意味着 `extra` 字典的字段（如 `extra={"task_id": 42}`）会先被弹出，**不会**出现在消息后缀中。但 `extra` 本身仍会被传递——所以功能上 `extra` 仍然有效。

  **真正的 bug**：当调用方使用 `extra` 传递自定义字段时，这些字段被传递给了底层 logger。但 `ProjectLogger.__getattr__`（line 93-94）将未识别的属性委托给底层 logger。如果调用方通过 `ProjectLogger` 实例调用 `logger.info("msg", extra={"key": "val"})`，`extra` 被正确处理。但如果调用方通过 `ProjectLogger` 实例调用 `logger.info("msg", key="val")`（非标准用法），`key=val` 会被拼接到消息字符串中，而非作为 `extra` 传递。

  更严重的问题是：`_format` 的 `kwargs.pop(key)` 会修改调用方的 kwargs 字典（因为 dict 是引用传递）。如果调用方复用了同一个 kwargs 字典（如通过 `**kwargs` 展开），弹出操作会导致后续使用中缺失 `extra` 字段。

- 与 K11（LocaleManager 单例无锁）、G01（ThemeManager 双路径单例无锁）同属系统性并发设计缺陷

**影响面**：所有通过 `ProjectLogger` 记录日志并使用了 `extra` 参数的路径。当前代码中 `extra` 使用较少，但一旦扩展（如日志过滤、审计追踪），此 bug 会导致自定义字段丢失或 kwargs 污染。

**修改方案**：
```python
def _format(self, args, kwargs):
    logging_kwargs = {}
    for key in ("exc_info", "stack_info", "stacklevel", "extra"):
        if key in kwargs:
            logging_kwargs[key] = kwargs[key]  # 使用索引而非 pop，避免修改调用方字典
    ...
```
