## 1. 代码错误（功能失效类）

### [D1] `_force_stop` 将 `"am force-stop"` 作为单个参数传递，导致强制停止从未生效
**优先级：High**
**位置**：`src/core/capability/device/recovery.py:72`

```python
self._run(["shell", "am force-stop", self._package], serial)
```

`_run` 方法将参数拼接后传给 `subprocess.check_output`。由于 `"am force-stop"` 是一个字符串，Android mksh 将其解释为单个命令名，而非 `am` + 子命令 `force-stop`。系统找不到名为 `am force-stop` 的可执行文件，该命令始终失败。`_force_stop` 的 `except` 块仅记录 warning 后吞掉异常继续执行，调用方无法感知。

**影响面**：`restart()` 中的强制停止步骤从未正确执行，旧应用进程未被终止，可能两个实例同时运行。

**修改建议**：将 `"am force-stop"` 拆分为 `"am"` 和 `"force-stop"` 两个独立参数：
```python
self._run(["shell", "am", "force-stop", self._package], serial)
```

### [CFG-07] 任务文件内部名与文件名不一致，导致按名加载失败
**优先级：High**
**位置**：
- `assets/tasks/CreditShopping.json` → 内部任务名为 `CreditShoppingN2`
- `assets/tasks/Weapon.json` → 内部任务名为 `WeaponUpgrade`

**根因**：
- `MaaEndRuntime.load_tasks()` 通过扫描所有 JSON 文件并按内部 `name` 字段索引，因此 `CreditShoppingN2` 和 `WeaponUpgrade` 能进入 `_tasks` 字典。
- 但 `TaskLoader.load_task(task_name)` 按 `assets/tasks/{task_name}.json` 做文件名查找。调用 `load_task("CreditShoppingN2")` 时，会在 `assets/tasks/` 下寻找 `CreditShoppingN2.json`，该文件不存在，返回 `None`。

**触发路径**：预设 `DailyFull` / `QuickDaily` 中引用了 `CreditShoppingN2`。若任何代码路径通过 `TaskLoader.load_task("CreditShoppingN2")` 加载该任务，会返回 `None`，导致 GUI 的任务详情面板、选项编辑器等功能对这两个任务失效。

**修改建议**：
1. 将 `CreditShopping.json` 内部名改为 `CreditShopping`，或重命名文件为 `CreditShoppingN2.json`
2. 将 `Weapon.json` 内部名改为 `Weapon`，或重命名文件为 `WeaponUpgrade.json`
3. 建立文件名与内部 `name` 一致的约束，并在 CI 中校验

## 2. 漏洞 / 安全风险

### [D2] `shell()` 方法接受用户直接输入但未做任何过滤，存在命令注入
**优先级：High**
**位置**：`src/core/capability/device/adb_manager.py:71-82` + `src/cli/handlers.py:467-473`

**根因**：`ADBDeviceManager.shell(cmd)` 接受任意字符串 `cmd`，直接传递给 `adb shell`。虽然 `subprocess.check_output` 使用列表形式避免主机侧 shell 注入，但 `adb shell` 会将整个字符串转发到设备端的 mksh，由设备 shell 解释。`handlers.py:467-473` 的 `_handle_shell` 处理 CLI 命令 `istina shell <cmd>`，`args.cmd` 完全来自用户输入，未经过任何过滤或白名单检查。`android_runtime.py` 的 `_is_allowed_shell_cmd` 白名单仅约束守护进程 RPC 路径，`adb_manager.shell()` 的 subprocess 路径完全绕过了该检查。

**影响面**：任何有权限执行 CLI 命令的用户可在已连接的 Android 设备上执行任意 shell 命令，包括删除文件、修改系统设置、安装恶意应用等。

**修改建议**：
1. 在 `_handle_shell` 层面增加命令白名单或参数校验，仅允许安全的诊断命令
2. 或在 `shell()` 方法内增加命令过滤，与 `android_runtime.py` 的 `_is_allowed_shell_cmd` 保持一致
3. 至少将 `;`、`|`、`&`、`$`、`` ` ``、`\`、`$(` 等 shell 元字符加入黑名单

> **审计附注**：与 2345.md C1（shell 黑名单缺失反斜杠）互补——C1 关注守护进程 RPC 路径的注入，D2 关注 subprocess 路径的完全绕过。两者合起来覆盖了所有 shell 调用路径。

## 3. 漏洞 / 健壮性问题

### [LLM-02] `_cuda_failed` 跨调用持久化，首次 CPU fallback 后永久禁用 GPU
**优先级：Medium**
**位置**：`src/core/capability/llm/runtime.py:130-133`

`_cuda_failed` 是实例级状态，标记后永不重置。用户修复 GPU 后再次调用 `start()`，仍走 CPU 路径，必须重启进程才能恢复 GPU。

**修改建议**：将 `_cuda_failed` 改为每次 `start()` 的局部变量。

---

### [LLM-01] `_try_start` 丢弃 stderr/stdout，启动失败完全不可诊断
**优先级：Medium**
**位置**：`src/core/capability/llm/runtime.py:320`

`llama-server.exe` 的 stdout 和 stderr 均被重定向到 `DEVNULL`。启动失败时（DLL 缺失、模型损坏、CUDA 不兼容等），进程可能在启动后 0.1 秒崩溃并输出错误到 stderr，但这些信息全部丢失。调用方只能通过 60 秒超时得知失败，无法区分根本原因。

**修改建议**：将 stderr 捕获到 `subprocess.PIPE`，启动失败时将 `proc.stderr.read()` 记录到日志。

---

### [LLM-04] `_try_start` 裸 except 吞掉所有异常且不记录日志
