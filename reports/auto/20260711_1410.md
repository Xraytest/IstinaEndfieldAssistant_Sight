## 1. Python 启动钩子

### [SITE-01] `sitecustomize.py` 在每次 Python 解释器启动时修改全局环境变量，影响所有进程

**优先级：Medium**
**位置**：`sitecustomize.py:1-16`

```python
_tmp_root = Path(__file__).resolve().parent / ".tmp" / "python-temp"
_tmp_root.mkdir(parents=True, exist_ok=True)
os.environ["TMPDIR"] = str(_tmp_root)
os.environ["TEMP"] = str(_tmp_root)
os.environ["TMP"] = str(_tmp_root)

_maafw_dir = Path(__file__).resolve().parent / "3rd-part" / "maaend" / "agent" / "maafw"
if _maafw_dir.is_dir():
    os.environ["MAAFW_BINARY_PATH"] = str(_maafw_dir.resolve())
```

**根因**：
`sitecustomize.py` 是 Python 标准库在解释器启动时自动导入的模块（若存在于 `sys.path` 中）。本项目将其放在项目根目录，当 `src/` 被加入 `sys.path` 后，**任何**使用该项目 Python 环境的解释器（包括 `python -m pytest`、`python -c "print(1)"`、其他工具的脚本）都会自动执行此文件。

**具体问题**：

1. **全局临时目录劫持**：`TMPDIR`/`TEMP`/`TMP` 是操作系统级环境变量，被 `tempfile` 模块、`subprocess`、`open()` 的临时文件创建等多处使用。将所有 Python 进程的临时目录重定向到 `.tmp/python-temp` 意味着：
   - 其他项目或系统工具创建的临时文件也写入该项目目录
   - 若 `.tmp` 被清理，其他进程的未关闭临时文件丢失
   - 多项目共享同一 bundled Python 时，临时文件互相干扰

2. **MAAFW_BINARY_PATH 全局污染**：`MAAFW_BINARY_PATH` 是 MaaFW 库用于定位原生 DLL 的环境变量。将其设为项目特定路径后，同一 Python 环境中运行的其他 MaaFW 应用（如独立的 MaaEnd 实例）会加载本项目版本的原生库，可能导致版本不匹配。

3. **.tmp 目录创建副作用**：`mkdir(parents=True, exist_ok=True)` 在项目根目录创建 `.tmp/python-temp`，即使调用方只是 `python -c "import os"` 也会触发。

**影响面**：
- 开发环境中，`pytest`、`mypy`、`ruff` 等工具使用同一 Python 解释器时，临时文件行为改变。
- 若开发者同时使用多个项目，临时文件写入错误项目目录。

**修改建议**：
1. 将 `sitecustomize.py` 改为仅在项目入口点（`istina.py`、`main.py`）中显式调用，而非依赖自动导入。
2. 或改为 `usercustomize.py`（仅影响交互式 shell，不影响工具）。
3. 若确需全局设置，将环境变量修改移至 `init_logger()` 或 `IstinaRuntime.__init__()` 中，在应用启动时而非解释器启动时执行。

> **注意**：当前 `sitecustomize.py` 需位于 `sys.path` 中才能被自动导入。`conftest.py` 和 `test_istina_runtime.py` 均将 `src/` 加入 `sys.path`，因此 pytest 执行时也会导入此文件。但 `src/core/foundation/paths.py` 的 `ensure_src_path()` 仅在项目代码中调用，不影响系统 Python。

_maafw_dir = Path(__file__).resolve().parent / "3rd-part" / "maaend" / "agent" / "maafw"

## 3. 启动脚本

### [BAT-01] `start_gui.bat` 未验证子进程退出码为负数情况

**优先级：Low**
**位置**：`start_gui.bat:26-30`

```batch
if not "%EXITCODE%"=="0" (
    echo.
    echo [ERROR] Exited with code: %EXITCODE%
    pause
)
```

Python 进程被信号终止时（如 `Ctrl+C`、`SIGKILL`），`ERRORLEVEL` 为负数（Windows 上 `ERRORLEVEL` 范围通常为 0-255，但某些信号传递方式可能产生负值）。`if not "%EXITCODE%"=="0"` 对负数仍能匹配（字符串比较），因此不会遗漏错误。但 `exit /b %EXITCODE%` 对负数的处理取决于 cmd.exe 版本——某些版本会将负数截断为 256+n，导致父进程看到的退出码与 Python 实际返回值不同。

**影响面**：低。仅影响调试场景。

**修改建议**：对负数做截断处理：

```batch
if %EXITCODE% lss 0 (
    set /a EXITCODE=256+%EXITCODE%
)
```

## 5. 嵌套项目目录

### [MAAEND-01] `MaaEnd/` 目录包含完整的嵌套项目，与主项目形成双仓库结构

**优先级：Low**
**位置**：`MaaEnd/` 目录

`MaaEnd/` 包含完整的项目结构：
- `AGENTS.md`（AI Agent 编码指南）
- `.github/`（CI workflows、issue templates）
- `.gitignore`、`.gitmodules`、`.editorconfig`
- `agent/`（`cpp-algo/`、`go-service/`）
- `assets/`、`config/`、`docs/`、`tests/`、`tools/`

而主项目 `.gitignore` 第 3 行将 `MaaEnd/` 排除在版本控制之外。但 `3rd-part/maaend/` 也已排除（第 2 行）。这意味着项目依赖的两个 MaaEnd 目录均不在版本控制中。

**根因分析**：
`MaaEnd/` 可能是 MaaEnd 项目的独立克隆（用于开发 MaaFW agent 的 Go 服务或 C++ 算法模块），而 `3rd-part/maaend/` 是运行时依赖的预编译二进制版本。两者功能不同但目录名相似，容易混淆。

**影响面**：
- 新开发者看到两个 `MaaEnd` 目录，不清楚哪个是运行时依赖、哪个是开发仓库。
- `MaaEnd/` 的 `.git/` 可能存在于嵌套目录中，导致 `git status` 显示大量 untracked 文件。
- 若 `MaaEnd/` 的 `agent/` 子目录包含 Python 文件（当前为空），可能与主项目的 `3rd-part/maaend/` 产生 import 冲突。

**修改建议**：
1. 在 `README.md` 或 `CLAUDE.md` 中明确说明两个 MaaEnd 目录的用途和关系。
2. 若 `MaaEnd/` 为独立开发仓库，考虑使用 git submodule 而非裸目录复制。
3. 若不再需要，将 `MaaEnd/` 移至项目外部或删除。
