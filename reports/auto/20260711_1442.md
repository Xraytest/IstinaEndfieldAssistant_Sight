## §1 新发现（优先级：错误 > 漏洞 > 优化）

### NEW-PT-1（Medium，安全漏洞）`llm/runtime.py` `_resolve_model_path` 相对路径遍历仍未闭环

`1337` 批次 NEW-2（CFG-10）建议"对 candidate/under_models 统一做 `root not in .parents` 越界校验"。工作树已落地的修复**只覆盖了绝对路径分支**，相对路径分支仍逃逸项目根：

```python
# llm/runtime.py (当前工作树)
root = get_project_root().resolve()
p = Path(model_path)
if p.is_absolute():
    resolved = p.resolve()
    if root not in resolved.parents and resolved != root:
        self._logger.warning("model_path 绝对路径越界，已忽略: %s", model_path)
        p = Path(p.name)                       # 绝对分支已收敛
# 相对路径：项目内优先，其次 models/ 子目录
in_root = (root / str(p)).resolve()            # ← 若 p="../../../x.gguf"，此处解析到项目根外
if in_root.exists():
    return str(in_root)                        # 项目根外的文件被返回 → 路径遍历逃逸
```

- **触发**：`model_path` 取配置项 `llm.model` 或 CLI `--model`，传入相对 `..` 路径（如 `../../secret.gguf`）。`p.is_absolute()` 为 `False` 跳过校验，`(root / "../../secret.gguf").resolve()` 落到项目根外；若该文件存在即被返回并交给 `llama-server` 加载。
- **影响**：本地自动化工具（用户自跑）场景下面，恶意/错误配置可读取项目根外任意 `.gguf`（或任意名）文件；与 `1337` 原评级（Medium）一致。
- **修复**：在算出 `in_root`/`under_models` 后补越界判断，与绝对分支口径统一：

```python
in_root = (root / str(p)).resolve()
if root in in_root.parents or in_root == root:
    if in_root.exists():
        return str(in_root)
under_models = (root / "models" / str(p)).resolve()
if root in under_models.parents or under_models == root:
    if under_models.exists():
        return str(under_models)
return str(p)   # 越界或未找到，原样返回由上层兜底（不返回根外路径）
```

> 注：本项是对 `1337` NEW-2 **修复不完整**的确认（详见 §4-A2），但因其是"已报告缺陷未被真正闭环"的残留漏洞，单列于此并记入风险台账，避免被误判为"已修复"。

### NEW-PT-2（Low，安全/语义）绝对越界回退可能误服务项目内同名文件

`_resolve_model_path` 绝对路径越界时退化 `p = Path(p.name)`（仅保留文件名），随后 `(root / name)` 若存在即返回。例如越界输入 `/etc/passwd` → 回退为 `passwd` → 若 `<root>/passwd` 恰好存在，将**静默加载根内无关文件**而非报错。建议越界时直接 `return model_path`（保持原样、由调用方 `Popen` 失败暴露），避免"偷换文件"。
