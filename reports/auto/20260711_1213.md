## 1. Utility 脚本问题

### [SCR-02] `verify_llm.py` MODEL_PATH 硬编码 Windows 反斜杠路径

**优先级：Low**
**位置**：`scripts/verify_llm.py:19`

```python
MODEL_PATH = r"models\LLM\Qwen3.5-4B-UD-Q4_K_XL.gguf"
```

`r"models\..."` 在 Windows 上可工作（反斜杠被 Python 接受为路径分隔符），但在 WSL/Linux/macOS 上运行时，`\L` 被视为字面字符而非路径分隔符，导致文件找不到。虽为 Windows-only 项目，但脚本在 CI 或 WSL 开发环境中无法复用。

**修改建议**：使用 `Path(__file__).resolve().parent.parent / "models" / "LLM" / "Qwen3.5-4B-UD-Q4_K_XL.gguf"`。

### [SCR-03] `verify_llm_simple.py` 端口范围与 `LlamaServerRuntime` 默认端口 9998 重叠

**优先级：Low**
**位置**：`scripts/verify_llm_simple.py:122-123`

```python
port = 9998 + idx  # idx 从 1 开始，第一个候选用 9999
```

脚本 5 个候选使用端口 9999~10003。但 `LlamaServerRuntime` 默认端口为 9998，且 `IstinaRuntime` 在 GUI 启动时可能已在该端口上运行 llama-server。若应用实例与脚本同时运行，第一个候选（9999）可能与相邻实例冲突，后续候选也可能被防火墙/路由器规则影响。

`check_llm_cuda.py` 使用端口 10270，`verify_llm.py` 使用端口 10260~10265，两者之间无冲突但各自与 9998 系列不重叠。

**修改建议**：
1. 将脚本端口统一偏移至 10xxx 系列（如 10270+），避免与应用默认端口冲突。
2. 或在脚本启动前检测端口占用并自动跳过。

### [SCR-04] `check_llm_cuda.py` n_gpu_layers=999 重复传递两次

**优先级：Low**
**位置**：`scripts/check_llm_cuda.py:34-37`

```python
"-ngl",
"999",
"--n-gpu-layers",
"999",
```

`-ngl` 和 `--n-gpu-layers` 是 llama-server 的同名参数（短格式和长格式），重复传递相同值无害但冗余。此为 CFG-11（`n_gpu_layers=999` 不安全默认值）的下游症状——utility 脚本直接复用了该配置值，即使显存不足也会尝试 offload 999 层。

**修改建议**：使用合理的显存自适应值（如先查询 `nvidia-smi` 获取可用显存，再计算最大可 offload 层数），或至少改为 `-1`（自动）。

## 2. 测试文件问题

### [TST-03] `test_main_window.py` 使用 `importlib` 隔离加载模块，但 QProcess mock 为死代码

**优先级：Low**
**位置**：`tests/test_main_window.py:12-22, 26-48`

```python
def _load_main_window_module():
    spec = importlib.util.spec_from_file_location("main_window_under_test", module_path)
    module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(module)
    return module
```

三个测试函数（`test_main_window_can_be_instantiated`、`test_main_window_bridge_returns_cli_bridge`、`test_main_window_uses_left_navigation`）均通过 `importlib` 从文件路径重新加载 `main_window.py`，并在 `with patch("gui.pyqt6.cli_bridge.QProcess")` 上下文中执行。

**根因分析**：
`main_window.py:28` 在模块顶层执行 `from gui.pyqt6.cli_bridge import CLIBridge`。当 `_load_main_window_module()` 在 patch 上下文中执行 `spec.loader.exec_module(module)` 时，该 import 语句解析到被 patch 的 `gui.pyqt6.cli_bridge` 模块——此时 `QProcess` 已被替换为 `MagicMock` 类。`CLIBridge.__init__` 内的 `QProcess()` 调用创建 `MagicMock` 实例。

但 mock 为**死代码**的原因：
1. `MagicMock()` 默认返回 `MagicMock` 实例，任何方法调用（`waitForStarted`、`state`、`start` 等）均返回可预测的 mock 对象，不会失败。
2. 三个测试的断言仅验证 `window is not None`、`bridge is not None`、导航文本匹配——**均不验证 QProcess 的初始状态是否正确**（如 `_process.state()` 是否返回 `NotRunning`）。
3. 若将 `main_window.py` 中 `CLIBridge.__init__` 的 `self._process.state()` 改为 `self._process.foo()`，三个测试仍会通过，因为 `MagicMock.foo()` 返回另一个 MagicMock。

**影响面**：测试不验证 `MainWindow` 对 QProcess 初始化的实际依赖。若 CLIBridge 的构造函数逻辑变更（如增加对 `QProcess.ProcessError` 的初始化检查），这些测试不会 catching regression。

**修改建议**：
1. 若需要隔离加载，改用 `unittest.mock.patch.dict(sys.modules, ...)` 或更精确的 spy 模式验证 QProcess 的实际调用参数。
2. 或放弃 importlib 隔离，使用与 `test_gui_maaend_control.py` 一致的 `mock_bridge` fixture 模式（通过 `monkeypatch.setattr` 替换 `_sync_execute` 等关键方法）。

### [TST-04] `test_template_pipeline.py` 调用 `TemplateRegistry.clear()` 跨测试文件状态泄漏

**优先级：Low**
**位置**：`tests/test_template_pipeline.py:21,32,41,50,67,81,168,198`

```python
registry = TemplateRegistry()
registry.clear()  # 清除单例全局状态
```

`TemplateRegistry` 是单例（`__new__` 返回同一实例）。8 处 `registry.clear()` 清除该单例的 `_templates`、`_template_paths`、`_loaded_modules`。

**根因**：
pytest 按 `testpaths = ["tests"]` 收集所有测试文件并在同一进程中执行。`test_template_pipeline.py` 中每个测试函数先 `clear()` 再 `load_module()`，若 pytest 执行顺序为：
1. `test_template_pipeline.py` → `clear()` → `load_module("Common")` → 填充单例
2. `test_gui_maaend_control.py`（或其他引用 TemplateRegistry 的测试）→ 读取已填充的单例 → 正常

但若顺序为：
1. `test_gui_maaend_control.py` → `load_module("Common")` → 填充单例
2. `test_template_pipeline.py` → `clear()` → 单例被清空
3. `test_gui_maaend_control.py` 的后续测试 → 单例为空 → 匹配失败

当前 pytest 默认按文件名字典序收集（`test_cli_bridge.py` → `test_error_paths.py` → `test_istina_cli_commands.py` → `test_istina_runtime.py` → `test_llm_mmproj.py` → `test_llm_runtime_image.py` → `test_main_window.py` → `test_queue_state.py` → `test_template_pipeline.py`）。`test_template_pipeline.py` 靠后执行，可能在其之后无引用 TemplateRegistry 的其他测试，因此**当前实际风险低**。但模式脆弱——任何新增测试文件在 `test_template_pipeline.py` 之后引用 TemplateRegistry 即触发泄漏。

**修改建议**：
1. 使用 `@pytest.fixture(autouse=True)` 在每个测试函数后恢复单例状态，或在 `setUp`/`tearDown` 中保存/恢复。
2. 或避免 `clear()`，改用子集操作（如仅加载/卸载单个模块）。

### [TST-05] `test_template_pipeline.py` 顶层 import cv2/numpy，依赖缺失时 pytest collection 阶段崩溃

**优先级：Low**
**位置**：`tests/test_template_pipeline.py:4-5`

```python
import cv2
import numpy as np
```

`cv2`（OpenCV）和 `numpy` 在顶层 import，不在任何 `try/except` 或 fixture 内。若开发环境缺少任一依赖（如新开发机 `pip install -r requirements.txt` 失败），pytest 在**收集阶段**即抛出 `ModuleNotFoundError`，**整个测试套件无法执行**——包括与 cv2/numpy 完全无关的测试（如 `test_cli_bridge.py`、`test_queue_state.py`）。

**影响面**：
- 新开发环境搭建时，cv2/numpy 安装失败会导致"所有测试都失败"的假象。
- CI 中若 cv2 包版本变更导致 import 失败，整个 CI pipeline 中断。

**修改建议**：
1. 将 `import cv2` 和 `import numpy as np` 移至各个测试函数内部（延迟 import），仅在实际需要的测试中加载。
2. 或在文件顶层包裹 `try/except ImportError`，在缺失时跳过所有 cv2/numpy 相关测试：

```python
try:
    import cv2
    import numpy as np
    _CV2_AVAILABLE = True
except ImportError:
    _CV2_AVAILABLE = False
```
