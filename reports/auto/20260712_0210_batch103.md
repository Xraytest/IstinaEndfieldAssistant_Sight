## 新增发现（1 项）

### LLM-CUDA-FLAG（修复不完整）— `reset_cuda_state()` 死代码，`_cuda_failed` 永久为 True

**等级**: 代码错误 / 中
**位置**: `src/core/capability/llm/runtime.py:72-75, 139-142`

**问题代码**:

```python
# llm/runtime.py:72-75
def reset_cuda_state(self) -> None:
    """LLM-02: 重置 CUDA 失败标记，允许环境变化后重新尝试 GPU 启动。"""
    self._cuda_failed = False
    self._logger.info("llama-server CUDA 失败标记已重置 (port=%s)", self._default_port)

# llm/runtime.py:139-142
ok = self._try_start(exe, model_path, llm_cfg)
if not ok and not self._cuda_failed:
    self._cuda_failed = True
    return self._try_start(exe, model_path, llm_cfg, force_cpu=True)
return ok
```

**根因分析**:

批次 96 已报告 LLM-CUDA-FLAG（GPU 启动失败后 `_cuda_failed` 标记为 True，后续启动跳过 CPU 回退）。随后代码添加了 `reset_cuda_state()` 方法作为修复，但该方法在整个 `src/` 范围内**零调用**：

1. `grep reset_cuda_state` → 仅定义处（line 72），无调用点
2. 用户无法通过任何 GUI 操作、CLI 命令或代码路径重置此标记
3. 一旦首次 GPU 启动失败，`_cuda_failed = True` 永久有效
4. 后续所有 `start()` 调用：GPU 尝试失败 → `_cuda_failed` 为 True → **不执行 CPU 回退** → 直接返回 False

**与批次 96 原报告的差异**:

- 批次 96 LLM-CUDA-FLAG：首次发现 `_cuda_failed` 标记在 CPU 回退成功后仍为 True
- 本报告：修复尝试（添加 `reset_cuda_state()`）已存在但为死代码，修复不完整

**调用链**:

```
场景 A：用户修复 CUDA 环境后再次启动
  → LlamaServerRuntime.start()
  → _try_start(GPU) → 成功（CUDA 已修复）
  → _ready = True → 返回 True ✓

场景 B：用户未修复 CUDA，首次启动 GPU 失败，CPU fallback 也失败
  → _cuda_failed = True
  → 用户修复 CUDA 环境后再次点击"启动 LLM"
  → LlamaServerRuntime.start()
  → _try_start(GPU) → 失败
  → _cuda_failed 为 True → 跳过 CPU fallback
  → 返回 False ✗（应 fallback CPU）
```

**影响面**:
- **中**：GPU 不可用时，首次启动后 CPU 回退永久失效。用户必须重启整个应用程序才能重新尝试 CPU 模式。如果用户先遇到 GPU 失败（即使 GPU 后来修复），应用也无法自动恢复。

**修复建议**:

```python
# 方案 1（最小修改）：在 stop() 中重置标记
def stop(self) -> None:
    self._ready = False
    self._cuda_failed = False  # 添加：停止后允许下次启动重新评估 GPU
    self._shutdown_owned()

# 方案 2：在 start() 入口处重置（更激进，允许运行时环境变化后重试）
def start(self) -> bool:
    self._cuda_failed = False  # 每次启动重新评估
    ...
```

方案 1 更安全：只在显式停止后重置，避免运行中环境变化导致意外行为。方案 2 更灵活：每次启动都重新评估 GPU 可用性。

**推荐方案 1**（最小可行修改）。
