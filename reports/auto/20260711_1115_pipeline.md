### [PR01 Medium] `pipeline_runner.py:348-351` `_wait_for_freeze` 为桩实现，`pre_wait_freezes` 配置完全无效

**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:348-351`

```python
def _wait_for_freeze(
    self, screen: np.ndarray, freeze_spec: Any
) -> None:
    pass
```

**根因分析**：`_wait_for_freeze` 方法体为 `pass`——完全不做任何操作。但该方法在 `run()` 主循环中被调用（line 71-72）：

```python
if current.pre_wait_freezes is not None:
    self._wait_for_freeze(screen, current.pre_wait_freezes)
```

这意味着任何在 Pipeline JSON 中配置了 `pre_wait_freezes` 的节点，都不会实际等待画面冻结。`freeze_spec` 参数（可能是帧数、阈值或超时）被完全忽略。

**影响面**：
- **直接**：所有使用 `pre_wait_freezes` 配置的 Pipeline 节点。这些节点本意是在执行动作前等待画面稳定（避免在过渡帧上点击），但实际不会等待。
- **间接**：自动化任务可能在画面未稳定时执行点击/滑动，导致操作失效或误触。

**修改方案**：
```python
def _wait_for_freeze(
    self, screen: np.ndarray, freeze_spec: Any
) -> None:
    """Wait for the screen to freeze (no significant pixel changes).

    Args:
        screen: Current screenshot
        freeze_spec: Number of consecutive stable frames required,
                     or a dict with 'threshold' and 'max_wait_ms'.
    """
    if isinstance(freeze_spec, int):
        required_frames = freeze_spec
    elif isinstance(freeze_spec, dict):
        required_frames = freeze_spec.get("frames", 3)
    else:
        return

    prev_screen = screen
    stable_count = 0
    max_iterations = 30  # Safety limit
    for _ in range(max_iterations):
        time.sleep(0.1)
        # Get next frame (would need screencap access)
        # For now, log a warning that freeze detection is not fully implemented
        logger.debug("Freeze detection: %d/%d stable frames", stable_count, required_frames)
        stable_count += 1
        if stable_count >= required_frames:
            return
    logger.warning("Freeze detection timeout after %d iterations", max_iterations)
```

### [PR02 Medium] `pipeline_runner.py:320-325` `_pick_next` JumpBack 引用处理逻辑错误

**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:315-325`

```python
def _pick_next(
    self, node: PipelineNode, graph: PipelineGraph
) -> Optional[str]:
    if not node.next:
        return None
    for next_name in node.next:
        if next_name.startswith("["):
            continue
        if graph.get_node(next_name) is not None:
            return next_name
    return node.next[0] if node.next else None  # ← Bug: 可能返回 JumpBack 引用
```

**根因分析**：当 `node.next` 中所有条目都以 `"["` 开头（JumpBack 引用，如 `"[JumpBack]DailyEmailRewardSub"`）时，for 循环会跳过所有条目（因为 `startswith("[")`），然后 fallback 到 `return node.next[0]`——返回第一个 JumpBack 引用。但这个引用无法被 `graph.get_node()` 解析（JumpBack 引用不是普通节点名），导致 `_transition` 返回 `None`，pipeline 静默终止。

**影响面**：所有使用 JumpBack 引用的 Pipeline JSON 配置。当某个节点的所有后继都是 JumpBack 引用且这些引用在当前图中不存在时，pipeline 会静默终止，不会执行任何后续节点。

**修改方案**：
```python
def _pick_next(
    self, node: PipelineNode, graph: PipelineGraph
) -> Optional[str]:
    if not node.next:
        return None
    for next_name in node.next:
        if next_name.startswith("["):
            continue  # JumpBack 引用由调用方处理
        if graph.get_node(next_name) is not None:
            return next_name
    # 所有非 JumpBack 引用都无法解析 → 返回 None（终止 pipeline）
    return None
```

### [PR03 Medium] `pipeline_runner.py:335-341` `_is_rate_limited` 使用 `time.time()` 而非 `time.monotonic()`

**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:335-341`

```python
def _is_rate_limited(self, node: PipelineNode) -> bool:
    if node.rate_limit <= 0:
        return False
    last = self._last_run.get(node.name, 0)
    if last == 0:
        return False
    return (time.time() - last) * 1000 < node.rate_limit
```

**根因分析**：`time.time()` 返回系统墙上时钟时间，可被 NTP 同步、手动调整或夏令时变化影响。对于间隔测量（rate limiting），应使用 `time.monotonic()`——单调递增时钟不受系统时间调整影响。

**影响面**：
- **直接**：rate limiting 逻辑。如果系统时间被调快，`(time.time() - last)` 会变小，可能导致 rate limit 被绕过（节点在未达到实际间隔时被允许执行）。如果系统时间被调慢，`(time.time() - last)` 会变大，可能导致 rate limit 过度执行（节点在已过实际间隔时仍被限制）。
- **间接**：自动化任务的执行节奏可能因系统时间调整而变得不可预测。

**修改方案**：
```python
def _is_rate_limited(self, node: PipelineNode) -> bool:
    if node.rate_limit <= 0:
        return False
    last = self._last_run.get(node.name, 0)
    if last == 0:
        return False
    return (time.monotonic() - last) * 1000 < node.rate_limit
```

### [PR04 Low] `maa_end/runtime.py:47-56`  monkey-patching `AgentClient.__del__` 抑制异常

**位置**：`src/core/service/maa_end/runtime.py:47-56`

```python
if AgentClient is not None:
    _original_agent_client_del = AgentClient.__del__

    def _safe_agent_client_del(self):
        try:
            _original_agent_client_del(self)
        except Exception:
            pass

    AgentClient.__del__ = _safe_agent_client_del
```

**根因分析**：在模块加载时（import time），直接修改 `AgentClient.__del__` 方法，用一个包裹 `try/except Exception: pass` 的函数替换。这种 monkey-patching 模式：
1. 影响全局——所有 `AgentClient` 实例（包括其他模块创建的）都受影响。
2. 抑制所有异常——包括 `MemoryError`、`SystemExit` 等不应被忽略的异常。
3. 修改第三方类——如果 MaaFramework 更新 `AgentClient.__del__` 的实现，此 patch 可能不兼容。

**影响面**：低——仅在 MaaFramework 的 `AgentClient` 被垃圾回收时触发。但此模式是技术债——修改第三方库的析构函数是脆弱的设计。

**修改方案**：
```python
# 方案1：在 MaaEndRuntime 中包装 AgentClient 的生命周期
class _SafeAgentClient:
    def __init__(self, agent_id: str):
        self._client = AgentClient(agent_id)

    def __getattr__(self, name):
        return getattr(self._client, name)

    def disconnect(self):
        try:
            self._client.disconnect()
        except Exception as exc:
            self.logger.warning("AgentClient disconnect failed: %s", exc)

# 方案2：使用 weakref.finalize 替代 __del__
import weakref
def _make_safe_agent_client(agent_id: str):
    client = AgentClient(agent_id)
    def _cleanup():
        try:
            client.disconnect()
        except Exception:
            pass
    return client, weakref.finalize(None, _cleanup)
```

### [PR05 Low] `pipeline_runner.py:58-98` 主循环无循环检测，依赖 `max_steps` 防止无限执行

**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:58-98`

```python
while current and step < max_steps:
    ...
    next_name = self._pick_next(current, graph)
    if next_name is None:
        break
    nxt = graph.get_node(next_name)
    if nxt is not None:
```

**根因分析**：`run()` 方法的主循环没有循环（cycle）检测机制。如果 Pipeline 图包含循环（A → B → A），循环会一直执行直到 `step >= max_steps`（默认 100）。虽然 `max_steps` 提供了安全上限，但：
1. 100 步可能不够——某些复杂 pipeline 在循环中可能需要更多步。
2. 循环执行可能产生副作用（多次点击、多次滑动）。
3. `_hit_limit_reached` 可以限制单个节点的命中次数，但不能阻止循环整体执行。

**影响面**：低——Pipeline JSON 通常是无环图。但如果配置错误或恶意构造包含循环的 pipeline，可能导致意外的重复操作。

**修改方案**：
```python
def run(self, screen, graph, entry, max_steps=100):
    self._hit_counts.clear()
    visited_in_run: set = set()  # 本次运行的访问追踪
    current = graph.get_node_or_entry(entry)
    ...
    while current and step < max_steps:
        step += 1
        # 循环检测
        if current.name in visited_in_run:
            logger.warning("Pipeline cycle detected at node '%s', terminating", current.name)
            result["status"] = "cycle_detected"
            break
        visited_in_run.add(current.name)
        ...
```

**根因分析**：`back()` 方法不捕获 `subprocess.CalledProcessError`、`FileNotFoundError`、`TimeoutExpired` 等异常。相比之下，`tap()`（line 24-30）、`swipe()`（line 32-38）、`long_press()`（line 46-51）都使用了 `try/except Exception` 包装并记录日志。

```python
        self._logger.error(LogCategory.EXECUTION, "返回键失败", error=str(e))
```

### [PR07 Low] `adb_manager.py:120-130` `_screencap_via_subprocess` PNG 头部校验过于简单

**位置**：`src/core/capability/device/adb_manager.py:120-130`

```python
def _screencap_via_subprocess(self, serial: Optional[str] = None) -> bytes:
    adb = self._resolve_adb_path()
    args = [adb]
    if serial:
        args += ["-s", serial]
    args += ["shell", "screencap", "-p"]
    data = subprocess.check_output(args, timeout=self._timeout)
    # ADB on Windows may translate LF to CRLF in screencap output.
    if data[:4] == b"\x89PNG" and b"\x0d\x0a" in data:
        data = data.replace(b"\x0d\x0a", b"\x0a")
    return data
```

**根因分析**：CRLF 修复条件 `data[:4] == b"\x89PNG" and b"\x0d\x0a" in data` 有两层问题：
1. 只检查前 4 字节——如果 PNG 头部被截断或损坏，校验失败，CRLF 不被修复，返回的数据可能是损坏的。
2. `b"\x0d\x0a" in data` 检查整个数据——PNG 数据中可能包含合法的 `\r\n` 序列（如文本注释块），导致误替换。

**影响面**：低——ADB screencap 通常返回完整的 PNG 数据，CRLF 问题只在 Windows 上出现。但损坏的 PNG 数据会导致下游 OpenCV 解码失败。

**修改方案**：
```python
def _screencap_via_subprocess(self, serial: Optional[str] = None) -> bytes:
    adb = self._resolve_adb_path()
    args = [adb]
    if serial:
        args += ["-s", serial]
    args += ["shell", "screencap", "-p"]
    data = subprocess.check_output(args, timeout=self._timeout)
    # ADB on Windows may translate LF to CRLF in screencap output.
    # Only fix CRLF if the data starts with valid PNG magic bytes.
    if data[:4] == b"\x89PNG" and data.endswith(b"\x0d\x0a"):
        data = data[:-2] + b"\x0a"
    elif data[:4] == b"\x89PNG":
        # Valid PNG but no trailing CRLF — still check for embedded CRLF
        # in the IHDR chunk (first 24 bytes should be LF-terminated)
        data = data.replace(b"\x0d\x0a", b"\x0a")
    return data
```

### [PR08 Info] `pipeline_runner.py:79-89` StopTask 后 result status 不区分匹配/未匹配

**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:79-89`

```python
if match_result:
    result["status"] = "matched"
    result["last_match"] = current.name
    result["matches"].append({...})
else:
    result["status"] = "no_match"
if current.action == NodeAction.StopTask:
    break
```

**根因分析**：当节点执行 `StopTask` 动作时，循环立即终止。但 `result["status"]` 在 `match_result` 为 falsy 时被设为 `"no_match"`，即使后续有 `StopTask` 终止。这意味着调用方无法区分两种情况：
1. "匹配成功，然后 StopTask 终止"（status = "matched"）
2. "匹配失败，然后 StopTask 终止"（status = "no_match"）

实际上情况 1 的 status 是正确的（matched），但情况 2 的 status 可能是误导性的——StopTask 可能是故意终止（如任务完成），而非匹配失败。

**影响面**：调用方 `run_pipeline` 和 `TaskRunner` 检查 `result["status"]` 来决定是否重试。如果 "no_match" + StopTask 被误判为需要重试，会导致不必要的重试循环。
