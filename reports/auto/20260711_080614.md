### 2.1 🟡 中危问题

#### H01 - `runtime.py` `_nav3_walk`/`_nav3_to_entity` 传递未初始化LLM客户端

**文件**: `src/core/service/runtime.py:697`、`src/core/service/runtime.py:706`
**风险等级**: 🟡 中危
**影响**: VLM导航功能（nav3.walk/nav3.to_entity）完全无法使用LLM，因为传递的是未初始化的引用

**问题代码**:
```python
def _nav3_walk(self, params: Dict[str, Any]) -> Dict[str, Any]:
    nav = self.navigator()
    return nav.to_coords_vlm(
        ...
        llm_client=self._llm,          # ❌ 传递未初始化属性
        keyevent_fn=self._vlm_keyevent,
    )

def _nav3_to_entity(self, params: Dict[str, Any]) -> Dict[str, Any]:
    nav = self.navigator()
    return nav.to_entity_vlm(
        ...
        llm_client=self._llm,          # ❌ 同上
        keyevent_fn=self._vlm_keyevent,
    )
```

**问题分析**:
- `IstinaRuntime.__init__` 初始化 `self._llm = None`（line 118）
- 只有通过 `_llm_client_instance` property 才延迟初始化（line 129-132）
- `self._llm` 直接访问的是原始属性，始终为 None
- `VlmWalkNavigator.__init__` 接收 `llm_client` 并赋值给 `self._llm`（vlm_walk_navigator.py:110）
- 当 VLM walk 执行到 `self._llm.chat()`（line 209）时，`self._llm` 为 None，调用 None.chat() 引发 AttributeError
- 与批次7/23的 W1 问题关联：即使 LLM 客户端正确初始化，vlm_walk_navigator 发送的字母键也会被 `_is_valid_keyevent` 拒绝
- **批次29 TASK_LOG 已记录 "N01扩展: _vlm_keyevent 接收字母键但 _is_valid_keyevent 拒绝"**，但根本原因未修复——VLM walk 功能从 LLM 客户端就无法工作

**修复建议**:
```python
def _nav3_walk(self, params: Dict[str, Any]) -> Dict[str, Any]:
    nav = self.navigator()
    return nav.to_coords_vlm(
        ...
        llm_client=self._llm_client_instance,  # ✅ 使用 property 获取初始化实例
        keyevent_fn=self._vlm_keyevent,
    )

def _nav3_to_entity(self, params: Dict[str, Any]) -> Dict[str, Any]:
    nav = self.navigator()
    return nav.to_entity_vlm(
        ...
        llm_client=self._llm_client_instance,  # ✅
        keyevent_fn=self._vlm_keyevent,
    )
```

#### H02 - `android_runtime.py:_is_valid_keyevent` 白名单拒绝字母键，VLM导航完全失效

**文件**: `src/core/capability/device/android_runtime.py:64-84`
**风险等级**: 🟡 中危
**影响**: VLM导航的"前进/后退/左右平移"功能完全无法执行

**问题代码**:
```python
_KNOWN_KEYEVENT_NAMES = frozenset({
    "KEYCODE_HOME", "KEYCODE_BACK", "KEYCODE_APP_SWITCH",
    "KEYCODE_DPAD_UP", "KEYCODE_DPAD_DOWN",
    "KEYCODE_DPAD_LEFT", "KEYCODE_DPAD_RIGHT",
    "KEYCODE_DPAD_CENTER",
})

def _is_valid_keyevent(key: Any) -> bool:
    if key is None:
        return False
    s = str(key).strip()
    if not s:
        return False
    if s.isdigit():
        return True
    return s in _KNOWN_KEYEVENT_NAMES
    # ❌ "w", "a", "s", "d", "q", "e", "f" 全部被拒绝
```

**问题分析**:
- `vlm_walk_navigator.py:_execute_action` 发送字母键（line 268-280）：`"w"`, `"a"`, `"s"`, `"d"`, `"q"`, `"e"`, `"f"`
- `_is_valid_keyevent` 仅接受数字或 KEYCODE_* 常量
- 字母键被拒绝后返回 `{"error": "invalid keyevent: 'w'"}`（android_runtime.py:610）
- **批次7 W1、批次23 W1、批次29 N01扩展** 均记录此问题，至今未修复
- VLM walk 功能（nav3）依赖这些按键控制角色移动，白名单使其完全不可用
- 安全角度：白名单设计正确防止注入，但列表不完整导致功能断裂

**修复建议**:
```python
_KNOWN_KEYEVENT_NAMES = frozenset({
    # 系统键
    "KEYCODE_HOME", "KEYCODE_BACK", "KEYCODE_APP_SWITCH",
    "KEYCODE_DPAD_UP", "KEYCODE_DPAD_DOWN",
    "KEYCODE_DPAD_LEFT", "KEYCODE_DPAD_RIGHT",
    "KEYCODE_DPAD_CENTER",
    # 游戏控制字母键（VLM导航必需）
    "w", "a", "s", "d", "q", "e", "f",
})
```

### 2.2 🟢 低危问题

#### H03 - `runtime.py:AndroidRuntimeProxy.__getattr__` 未初始化时可能返回 None 的方法

**文件**: `src/core/service/runtime.py:86-88`
**风险等级**: 🟢 低危
**影响**: 代理对象的任何方法调用在 `_client_for` 返回 None 时崩溃

**问题代码**:
```python
def __getattr__(self, name: str) -> Any:
    client = self._client_for(None)
    return getattr(client, name)  # ❌ 若 client 为 None，getattr(None, name) 引发 AttributeError
```

**问题分析**:
- `AndroidRuntimeProxy` 的 `__init__` 不创建任何 AndroidRuntime 实例
- `_client_for` 仅在 `_clients` 缓存未命中时才创建新实例
- 正常使用场景中 `IstinaRuntime.android()` 持有锁并管理生命周期，`client` 不会为 None
- 但 `AndroidRuntimeProxy` 也可独立使用（无 IstinaRuntime 包裹），此时 `_clients` 为空
- `_client_for(None)` 会创建新实例（adb_path/device_address 从 proxy 属性传递），理论上不会为 None
- 但如果创建过程中 `_get_android_runtime()` 导入失败（ImportError），`_client_for` 内的异常不会被捕获
- 更实际的场景：如果 `__getattr__` 被调用访问 `__class__`、`__dict__` 等 Python 内部属性（这些不在 `__init__` 中定义），会触发无限递归

**修复建议**:
```python
def __getattr__(self, name: str) -> Any:
    if name.startswith("_"):
        raise AttributeError(f"'{type(self).__name__}' has no attribute '{name}'")
    client = self._client_for(None)
    if client is None:
        raise AttributeError(f"'{type(self).__name__}' has no active client for method '{name}'")
    return getattr(client, name)
```

#### H04 - `runtime.py:_client_for` 创建实例时使用错误 device_address

**文件**: `src/core/service/runtime.py:94-100`
**风险等级**: 🟢 低危
**影响**: 代理可能创建与预期 device_address 不同的 AndroidRuntime 实例

**问题代码**:
```python
def _client_for(self, serial: Optional[str]) -> Any:
    resolved = serial or self._device_address or "default"  # ❌ 传入 serial 参数时忽略 self._device_address
    client = self._clients.get(resolved)
    if client is None:
        client = _get_android_runtime()(serial=resolved, adb_path=self._adb_path)
        # ↑ 将 resolved 作为 serial 传入 AndroidRuntime
        # resolved 可能是传入的 serial，而非 proxy 初始化的 device_address
        self._clients[resolved] = client
    return client
```

**问题分析**:
- `__getattr__` 调用 `self._client_for(None)`，使用 `self._device_address`
- 但如果外部代码直接调用 `_client_for("some-other-serial")`，会创建不同 serial 的实例
- `_clients` 字典以 serial 为键，意味着一个 proxy 可以管理多个不同 serial 的 AndroidRuntime
- 这与 proxy "单设备地址"的设计意图矛盾——`__init__` 只接受一个 `device_address`
- 实际调用中 `_client_for` 仅被 `__getattr__` 和 `default_client` property 使用，均传入 None
- 潜在风险：未来代码扩展直接调用 `_client_for(specific_serial)` 时行为不一致

**修复建议**:
```python
def _client_for(self, serial: Optional[str]) -> Any:
    # 始终使用 proxy 初始化的 device_address，忽略传入的 serial
    resolved = self._device_address or "default"
    client = self._clients.get(resolved)
    if client is None:
        client = _get_android_runtime()(serial=resolved, adb_path=self._adb_path)
        self._clients[resolved] = client
    return client
```

#### H05 - `theme_manager.py` ThemeManager 单例非线程安全（第四次出现）

**文件**: `src/gui/pyqt6/theme/theme_manager.py`
**风险等级**: 🟢 低危
**影响**: 多线程首次调用 `get_theme()` 时可能创建多个实例

**问题分析**:
- ThemeManager 使用相同的 `if cls._instance is None` 无锁模式（与批次29 SECPEN N-7/N-8、批次30 F03 完全相同）
- 当前为第四次报告同类问题：ThemeManager → TemplateRegistry → TouchManager → ThemeManager（确认）
- `gui/pyqt6/tray_icon.py:28` 在 `__init__` 中调用 `get_theme()`，若与 GUI 主线程同时首次调用可能竞态
- 全局 `COLORS` dict（theme_manager.py 模块级）被 `set_current_theme` 修改，race condition 下可能被部分更新

**修复建议**（与历史报告一致，提取线程安全基类）:
```python
# 提取到 core/foundation/singleton.py
import threading

class ThreadSafeSingleton:
    _instance = None
    _lock = threading.Lock()
    def __new__(cls, *args, **kwargs):
        if cls._instance is None:
            with cls._lock:
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
        return cls._instance

# ThemeManager 继承
class ThemeManager(ThreadSafeSingleton):
    ...
```

#### H07 - `tray_icon.py:show_message` 托盘不可见时静默失败

**文件**: `src/gui/pyqt6/tray_icon.py:80-83`
**风险等级**: 🟢 低危
**影响**: 用户收不到托盘通知（错误状态、任务完成等）

**问题代码**:
```python
def show_message(self, title: str, message: str) -> None:
    if self._tray is None:
        return
    self._tray.showMessage(title, message, QSystemTrayIcon.MessageIcon.Information, 3000)
    # ❌ 未检查 tray 是否实际可见（isVisible()）
```

**问题分析**:
- `_setup_tray` 中调用 `self._tray.show()` 后托盘应可见
- 但在某些桌面环境（如 GNOME、KDE 某些配置），系统托盘本身不可用或被禁用
- `QSystemTrayIcon.showMessage()` 在托盘不可见时静默失败，无任何信号或日志
- 调用方无法区分"通知已发送"和"通知丢失"
- 影响：任务完成通知、错误告警等关键用户反馈丢失

**修复建议**:
```python
def show_message(self, title: str, message: str) -> None:
    if self._tray is None or not self._tray.isVisible():
        self._logger.warning("托盘不可见，无法显示通知: %s - %s", title, message)
        return
    self._tray.showMessage(title, message, QSystemTrayIcon.MessageIcon.Information, 3000)
```

### 2.3 信息级问题

#### H08 - `runtime.py:_load_config` 无备份机制（D05 模式再次出现）

**文件**: `src/core/service/runtime.py:447-455`
**风险等级**: 信息
**影响**: 配置加载失败时无备份可恢复

**问题代码**:
```python
def _load_config(self) -> Dict[str, Any]:
    path = self._resolve_config_path()
    if path.exists():
        try:
            with open(path, "r", encoding="utf-8") as f:
                return json.load(f)
        except Exception as e:
            self._logger.warning(LogCategory.MAIN, "加载配置失败，使用默认值", error=str(e))
    return {}
    # ❌ 无备份文件回退
```

**问题分析**:
- 批次28 D05、批次30 F07、批次31 G05 均记录了同类"无备份"问题
- 此为本模式第四次出现（不同文件但相同缺陷模式）
- `_load_config` 使用 bare except 捕获 JSON 解析错误、权限错误等
- 失败后返回空 dict，意味着所有设备配置、LLM 配置丢失
- 调用方（如 `maaend()` 初始化）使用空配置可能导致异常默认值（如默认 device_address = "localhost:16512"）

**修复建议**:
```python
def _load_config(self) -> Dict[str, Any]:
    path = self._resolve_config_path()
    backup_path = path.with_suffix(".json.bak")
    for candidate in [path, backup_path]:
        if candidate.exists():
            try:
                with open(candidate, "r", encoding="utf-8") as f:
                    return json.load(f)
            except Exception as e:
                self._logger.warning(LogCategory.MAIN, "加载配置失败",
                                     path=str(candidate), error=str(e))
    return {}
```
