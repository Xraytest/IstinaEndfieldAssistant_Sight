### RECORDER-STALE-TIMER（防抖定时器替换不停止旧实例 + 死代码）— `recorder.py`

**等级**: 代码错误 / 中
**位置**: `src/gui/pyqt6/scripting/recorder.py:176-182, 64, 181`

**问题代码**:

```python
# recorder.py:176-182 — 防抖定时器创建
if obj_name not in self._text_timers:
    timer = QTimer(self)
    timer.setSingleShot(True)
    timer.timeout.connect(lambda: self._emit_text_action(obj_name, widget_type, action_type, text))
    self._text_timers[obj_name] = timer
self._pending_text[obj_name] = text
self._text_timers[obj_name].start(800)
```

**根因分析**:

1. **定时器替换不停止旧实例**：当用户快速连续输入时（如 200ms 间隔），每次 `_record_text_change` 被调用都会用新 timer 覆盖 `self._text_timers[obj_name]`，但**不停止旧 timer**。旧 timer 进入孤儿状态，仍会在其设定的 800ms 后触发。Qt 的 GC 不会主动停止已孤立的 QTimer。

2. **lambda 捕获 stale 值**：旧 timer 的 `timeout` 信号连接的 lambda 捕获了创建时的 `text` 变量值。当旧 timer 触发时，`_emit_text_action` 使用这个过时的文本值。

3. **死代码 `_pending_text`**：第 64 行 `self._pending_text: dict[str, str] = {}` 和第 181 行 `self._pending_text[obj_name] = text` 持续写入最新文本，但 `_emit_text_action` (line 184) **从不读取** `_pending_text`——它只使用 lambda 捕获的值。`_pending_text` 是设计意图与实现不一致的残留。

**调用链（错误场景）**:

```
用户快速输入 "hello world"（每 200ms 一个字符）
  t=0ms:  输入 "h" → timer_A 创建, 捕获 text="h", 启动 800ms
  t=200ms: 输入 "he" → timer_B 创建, 捕获 text="he", timer_A 未停止！
  t=400ms: 输入 "hel" → timer_C 创建, 捕获 text="hel", timer_B 未停止！
  t=600ms: 输入 "hell" → timer_D 创建, 捕获 text="hell", timer_C 未停止！
  t=800ms: timer_A 触发 → _emit_text_action(text="h") → 记录 "h"
  t=1000ms: timer_D 触发 → _emit_text_action(text="hell") → 记录 "hell"
  t=1200ms: timer_C 触发 → _emit_text_action(text="hel") → 但 "hel" ≠ "hell"，去重不生效 → 记录 "hel"
  ...
```

**去重为何失效**：`_emit_text_action` (line 189-191) 检查最后一条记录是否与当前 action 相同。但如果中间有其他 action 插入（如 click），去重链断裂。

**影响面**:
- **中**：快速输入时录制脚本包含错误的文本值（中间态而非最终态）。用户回放脚本时会点击/输入错误的文本。
- 概率：仅快速连续输入（<800ms 间隔）时触发，但该场景在脚本录制中很常见（快速填写表单）。

**修复建议**:

```python
# 方案 1（最小修改）：替换 timer 前停止旧实例
if obj_name in self._text_timers:
    old_timer = self._text_timers[obj_name]
    old_timer.stop()
timer = QTimer(self)
timer.setSingleShot(True)
# 方案 1a：lambda 仍捕获创建时的 text（ acceptable 因为旧 timer 已停止）
timer.timeout.connect(lambda: self._emit_text_action(obj_name, widget_type, action_type, text))
self._text_timers[obj_name] = timer
# 方案 1b（更精确）：lambda 读取 _pending_text 最新值
timer.timeout.connect(lambda obj_name=obj_name: self._emit_text_action_from_pending(obj_name))
self._text_timers[obj_name] = timer
```

**推荐方案 1a**（最小修改，停止旧 timer 即可消除孤儿触发）。

### LLM-HTTP-LEAK（HTTP 响应未关闭）— `llm/runtime.py`

**等级**: 代码质量 / 低
**位置**: `src/core/capability/llm/runtime.py:179`

**问题代码**:
```python
def _http_shutdown(self) -> None:
    try:
        import urllib.request
        req = urllib.request.Request(...)
        urllib.request.urlopen(req, timeout=2)  # ← HTTPResponse 未关闭
    except Exception:
        pass
```

**根因分析**:

`urllib.request.urlopen()` 返回 `http.client.HTTPResponse` 对象，其底层 socket 连接未显式关闭。虽然在 CPython 中 GC 最终会调用 `HTTPResponse.close()`，但：

1. 文件描述符泄漏延迟——在 GC 运行前，socket 保持 OPEN 状态
2. 在高频调用场景下（如多次 `_http_shutdown` 尝试），可能耗尽本地端口

`_http_shutdown` 是 `LlamaServerRuntime.stop()` 的最后一步，仅在停止 LLM 时调用一次，因此**实际影响极低**。但作为代码规范，应使用 `with` 语句。

**修复建议**:
```python
def _http_shutdown(self) -> None:
    try:
        import urllib.request
        req = urllib.request.Request(...)
        with urllib.request.urlopen(req, timeout=2) as resp:
            resp.read()  # 确保响应体消费完毕
    except Exception:
        pass
```

### LLM-SUBPROCESS-PIPE-LEAK（子进程管道未读取）— `llm/runtime.py`

**等级**: 代码质量 / 低
**位置**: `src/core/capability/llm/runtime.py:211, 238`

**问题代码**:
```python
# llm/runtime.py:211 — _kill_processes_on_port
p = subprocess.Popen(["taskkill", "/F", "/PID", str(pid)], stdout=subprocess.PIPE, stderr=subprocess.PIPE)
p.wait(timeout=5)

# llm/runtime.py:238 — _kill_processes_on_port (类似模式)
p = subprocess.Popen(["taskkill", "/F", "/PID", str(pid)], stdout=subprocess.PIPE, stderr=subprocess.PIPE)
p.wait(timeout=5)
```

**根因分析**:

使用 `subprocess.PIPE` 捕获子进程 stdout/stderr，但 `p.wait()` 之前从未读取管道内容。如果子进程输出超过 OS pipe buffer（Windows 约 4KB），子进程会阻塞在 `write()` 上，导致 `p.wait()` 永远等待——**死锁**。

当前场景是 `taskkill /F /PID <pid>`，该命令通常只输出 1-2 行，远低于 pipe buffer 上限。实际死锁概率极低。但该模式违反了 `subprocess` 最佳实践。

对比：`_try_start` 中启动 llama-server 后正确使用 `self._process.communicate()` 读取管道。

**修复建议**:

```python
# 方案 1（最小修改）：不使用 PIPE，让输出直接继承父进程
p = subprocess.run(["taskkill", "/F", "/PID", str(pid)], timeout=5)

# 方案 2（保守修改）：读取管道
p = subprocess.Popen(["taskkill", "/F", "/PID", str(pid)], stdout=subprocess.PIPE, stderr=subprocess.PIPE)
p.wait(timeout=5)
p.communicate()  # 读取残余输出
```

**推荐方案 1**（`subprocess.run` 更简洁，无需手动管理管道）。
