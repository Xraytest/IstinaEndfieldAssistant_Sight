### 2.1 🟡 中危问题

#### P01 - `_handle_gpu_recommend` 运算符优先级错误，GPU 推荐逻辑完全失效

**文件**: `src/cli/handlers.py:677`
**风险等级**: 🟡 中危
**影响**: `gpu recommend` 命令对任意非零显存返回 "GPU"，推荐逻辑完全失效

**问题代码**:
```python
# handlers.py:672-678
def _handle_gpu_recommend(runtime, args):
    gpus = _handle_gpu_status(runtime, args).get("gpus", [])
    recommendation = "CPU"
    if gpus:
        mem = gpus[0].get("free_memory_bytes", 0)
        if mem and mem >= 4 * 1024 * 1024 * 1024 or mem and mem >= 2 * 1024 * 1024 * 1024:
            recommendation = "GPU"  # ❌ 条件恒真
```

**问题分析**:
- Python 运算符优先级：`and` 高于 `or`
- 表达式实际求值：`(mem and mem >= 4GB) or (mem and mem >= 2GB)`
- 由于 `mem >= 4GB` 蕴含 `mem >= 2GB`，第二个条件完全冗余
- 更严重的是：`mem` 为任意非零值时，`mem and mem >= 2GB` 为 True（因为非零值 truthy，且任何正数都 `>= 2GB`... 不对，`mem` 是字节数，通常 `>= 2GB`）
- 实际上：如果 `mem >= 2GB`（如 3GB 显存），条件为 True → "GPU"；如果 `mem = 1GB`，`mem >= 2GB` 为 False → 检查第一个条件 `mem >= 4GB` 也为 False → "CPU"
- **修正分析**: 此条件并非恒真。如果 GPU 显存 < 2GB（如 1GB），推荐为 CPU。如果 >= 2GB，推荐为 GPU。
- **真正的 bug**: 阈值设计不合理——2GB 显存不足以运行大型 LLM 模型（通常需要 4GB+）。原意可能是 `mem >= 4GB or mem >= 2GB`（任一条件满足推荐 GPU），但两个条件用 `or` 连接且没有分组括号，语义模糊。
- 对比正确写法：`if mem >= 4 * 1024**3:`（仅 4GB+ 推荐 GPU）
- 此 bug 导致低显存 GPU 被错误推荐用于 LLM 推理，可能触发 OOM

**修复建议**:
```python
# 修正阈值逻辑：4GB+ 推荐 GPU，否则 CPU
_GPU_MEM_THRESHOLD = 4 * 1024 * 1024 * 1024  # 4GB

if mem >= _GPU_MEM_THRESHOLD:
    recommendation = "GPU"
```

#### P02 - `_handle_config_set` 接受任意键值对，可注入任意配置项

**文件**: `src/cli/handlers.py:513-516`
**风险等级**: 🟡 中危
**影响**: 通过 CLI 可写入任意配置键值，覆盖关键配置项

**问题代码**:
```python
# handlers.py:513-516
def _handle_config_set(runtime, args):
    runtime.config[args.key] = args.value  # ❌ 无键名/值类型校验
    runtime.save_config()
    return {"status": "success", "key": args.key, "value": args.value}
```

**问题分析**:
- `args.key` 和 `args.value` 均为字符串（来自 argparse）
- `runtime.config` 是 dict，直接设置键值对
- 可注入的恶意配置示例：
  - `config set adb_path "evil_adb.exe"` → 下次连接使用恶意 ADB
  - `config set maaend_root "/malicious/path"` → 加载恶意 MAA 资源
  - `config set device.serial "attacker_device"` → 连接到攻击者设备
  - `config set llm.enabled "false"` → 禁用 LLM 功能
- `_load_config` 使用 bare `except Exception` 吞掉所有错误，但如果注入的 JSON 格式错误（如 `"` 未转义），配置写入后下次加载会失败
- `save_config()` 写入 `client_config.json`，该文件被 gitignore，不会污染仓库
- 但本地运行时的配置持久化会保留恶意值，影响后续所有操作
- GUI 的配置页面可能也有类似问题，但 CLI 是更直接的攻击面

**修复建议**:
```python
# 白名单允许修改的配置键
_ALLOWED_CONFIG_KEYS = {
    "device.serial", "device.last_connected", "device.auto_connect_last",
    "logging.level", "gui.locale", "navigation.default_map",
    # ... 明确列出可修改的键
}

def _handle_config_set(runtime, args):
    key = args.key
    if key not in _ALLOWED_CONFIG_KEYS:
        return {"status": "error", "message": f"config key '{key}' is read-only or not allowed"}
    # 值类型校验（可选，取决于键）
    runtime.config[key] = args.value
    runtime.save_config()
    return {"status": "success", "key": key, "value": args.value}
```

### 2.2 🟢 低危问题

#### P03 - `_handle_shell` CLI 层无参数校验，安全依赖 daemon 白名单

**文件**: `src/cli/handlers.py:467-473`
**风险等级**: 🟢 低危
**影响**: CLI 层不校验 shell 命令，安全完全依赖 daemon 白名单

**问题代码**:
```python
def _handle_shell(runtime, args):
    android = runtime.android()
    try:
        output = android.shell(args.cmd)  # ❌ CLI 层无校验
        return {"status": "success", "output": output}
    except Exception as exc:
        return {"status": "error", "message": str(exc)}
```

**问题分析**:
- `args.cmd` 来自命令行参数，无任何校验直接传入 `android.shell()`
- `android.shell()` → JSON-RPC → `_Daemon._dispatch` → `_is_allowed_shell_cmd`
- daemon 白名单检查：拒绝 `;|&` 等 shell metacharacter，只允许 9 个前缀
- **但**: 白名单允许 `am start`（启动任意 Android Activity）和 `svc power`（修改电源状态）
- 攻击场景：`istina shell "am start -a android.intent.action.MAIN -n com.attacker/.MainActivity"` → daemon 检查前缀 `am start` 通过 → 启动恶意 Activity
- 虽然本地 CLI 通常只有用户本人使用，但如果应用被远程调用（如通过 cli_bridge.py），此接口暴露给外部
- CLI 层应添加额外校验（如长度限制、危险子命令过滤）

**修复建议**:
```python
_DANGEROUS_SHELL_SUBCOMMANDS = {
    "am start",  # 启动 Activity
    "svc power", # 修改电源状态
    "pm install", # 安装 APK
    "pm uninstall", # 卸载 APK
}

def _handle_shell(runtime, args):
    cmd = args.cmd.strip()
    if len(cmd) > 500:
        return {"status": "error", "message": "shell command too long (max 500 chars)"}
    cmd_lower = cmd.lower()
    for dangerous in _DANGEROUS_SHELL_SUBCOMMANDS:
        if cmd_lower.startswith(dangerous):
            return {"status": "error", "message": f"shell subcommand '{dangerous}' requires explicit permission"}
    android = runtime.android()

```

#### P04 - `_handle_model_list` 列出隐藏文件和目录，信息泄露

**文件**: `src/cli/handlers.py:536-548`
**风险等级**: 🟢 低危
**影响**: models/ 下的隐藏文件（如 .git）被暴露在输出中

**问题代码**:
```python
def _handle_model_list(runtime, args):
    root = get_project_root()
    models_dir = Path(root) / "models"
    files = []
    if models_dir.is_dir():
        for path in sorted(models_dir.rglob("*")):
            if path.is_file():
                files.append({
                    "name": str(path.relative_to(models_dir)),
                    "path": str(path),
                    "size_bytes": path.stat().st_size,
                })
    return {"status": "success", "models": files}
```

**问题分析**:
- `rglob("*")` 匹配所有文件和目录，包括隐藏文件（`.gitignore`, `.DS_Store` 等）
- 如果 `models/` 目录下有 `.git/` 子目录（意外 git clone），所有 git 对象文件被列出
- 输出包含完整文件路径（`path` 字段），泄露项目目录结构
- 虽然 `models/` 通常只存放模型文件，但意外创建的隐藏目录会暴露
- `_handle_model_info`（line 554-560）正确使用了 `resolve()` + `is_relative_to()` 防路径遍历，但 `_handle_model_list` 没有过滤

**修复建议**:
```python
for path in sorted(models_dir.rglob("*")):
    if path.is_file() and not any(part.startswith(".") for part in path.relative_to(models_dir).parts):
        files.append(...)
```

#### P06 - `_interactive_loop` 的 stdout fd 重定向在 `_write_result` 中与 `main()` 重复

**文件**: `src/cli/istina.py:250-320`
**风险等级**: 信息
**影响**: 代码重复，fd 泄漏风险

**问题代码**:
```python
def main(argv):
    # stdout fd → stderr fd 重定向（lines 250-259）
    original_stdout_fd = os.dup(sys.stdout.fileno())
    os.dup2(sys.stderr.fileno(), sys.stdout.fileno())
    ...
    # _interactive_loop 中重复相同的重定向（lines 291-296）
    _orig_stdout_fd = os.dup(sys.stdout.fileno())
    os.dup2(sys.stderr.fileno(), sys.stdout.fileno())
```

**分析**:
- `main()` 和 `_interactive_loop()` 都执行 stdout fd 重定向
- 如果 `_interactive_loop` 在 `main` 内部调用（`--interactive` 模式），重定向被应用两次
- 第二次 `os.dup2(sys.stderr.fileno(), sys.stdout.fileno())` 将 stdout 重定向到 stderr（已是重定向后的 fd）
- 最终效果相同（stdout → stderr），但 fd 操作冗余
- `_orig_stdout_fd` 在 `_interactive_loop` 中被关闭（line 367），但 `original_stdout_fd` 在 `main` 的 finally 中关闭（line 274）
- 如果 `_interactive_loop` 先关闭了 `_orig_stdout_fd`（与 `original_stdout_fd` 指向同一 fd），`main` 的 finally 再次关闭会导致 `OSError: [Errno 9] Bad file descriptor`

**修复建议**: 提取 stdout 重定向为独立函数，确保只执行一次。

#### 修正3：N04 QTranslator GC 风险分析过度

**原报告**: 090739.md N04
**修正内容**: N04 分析了 `install_qt_translator` 中 `QTranslator` 局部变量可能被 GC。但经深入分析，此风险在实际中几乎不可能触发：

1. `app.installTranslator(translator)` 使 Qt C++ 层持有 translator 的父对象引用
2. PyQt6 的 sip 包装器在 C++ 对象存活期间保持 Python 对象引用计数 > 0
3. Qt 事件循环在方法返回前不会执行 GC
4. 只有在方法返回且无其他引用时，引用计数才降至 0
5. 但 `installTranslator` 创建的 C++ 父子关系阻止了引用计数降至 0

**结论**: N04 的理论风险在实际中不存在，建议降级为纯代码风格建议（非 bug）。
