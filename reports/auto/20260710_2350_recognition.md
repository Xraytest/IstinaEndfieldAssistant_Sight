### [REC-1] OCR Route 1 结果绕过 Route 2/3 回退，低置信度结果导致完全无 OCR 输出

**文件**：`src/core/capability/element_recognition/backends/ocr_backend.py:76-98`  
**严重程度**：Medium

**根因**：Route 1（maafw 内置 OCR）在 `results` 列表非空时阻止 Route 2（MaaEndOCR）和 Route 3（OCRManager）执行：

```python
# Route 1
if self._maa_tasker is not None and MAAFW_OCR_AVAILABLE:
    try:
        results.extend(self._run_maafw_ocr(screen, roi, expected))
    except Exception as e:
        logger.debug(f"maafw OCR failed: {e}")

# Route 2
if not results and self._maaend_ocr is not None:    # ← results 非空则跳过
    results.extend(self._run_maaend_ocr(screen, roi, expected))
```

但 Route 1 的结果在行 90 被置信度过滤：
```python
results = [r for r in results if r.confidence >= self._confidence_threshold]
```

如果 Route 1 返回的结果置信度均低于阈值（如模糊文字、小字号），过滤后 `results` 变为空列表，Route 2/3 已在前一步被跳过。调用方收到空列表，无法区分"真的没有文字"和"Route 1 结果质量差"。

**影响面**：OCR 识别召回率下降——MaaEndOCR（EasyOCR 后端）或 OCRManager 可能识别出 Route 1 遗漏的文字，但永远不会被调用。

**修改建议**：将置信度过滤移至各 Route 内部，或在过滤后检查结果是否为空并触发回退：

```python
# Route 1
raw_results = []
if self._maa_tasker is not None and MAAFW_OCR_AVAILABLE:
    try:
        raw_results.extend(self._run_maafw_ocr(screen, roi, expected))
    except Exception as e:
        logger.debug(f"maafw OCR failed: {e}")

results = [r for r in raw_results if r.confidence >= self._confidence_threshold]

# Route 2/3: 即使 Route 1 有原始结果，过滤后为空也应回退
if not results and self._maaend_ocr is not None:
    results.extend(self._run_maaend_ocr(screen, roi, expected))
if not results and self._ocr_manager is not None:
    results.extend(self._run_ocr_manager(screen, roi, expected))
```

### [REC-2] `_adjust_tap_center` 亮度阈值硬编码，暗屏/亮屏下中心点校准失效

**文件**：`src/core/capability/element_recognition/backends/template_backend.py:304`  
**严重程度**：Medium

**根因**：

```python
_, bright = cv2.threshold(roi, 120, 255, cv2.THRESH_BINARY)
```

固定阈值 120 未根据屏幕亮度自适应。在暗场景（如夜晚关卡、loading 暗画面）中，大部分像素低于 120，二值化后前景几乎为空，`findContours` 找不到轮廓，回退到 `(tpl_x + tpl_w // 2, tpl_y + tpl_h // 2 + tpl_h // 4)`——下偏 1/4 模板高度，可能点中非交互区域。在亮场景（如白昼 UI）中，阈值 120 过低，大量背景区域被纳入前景，找到的轮廓可能是背景而非实际按钮。

**影响面**：模板匹配的中心点校准在极端亮度场景下偏移，导致点击位置偏离目标按钮。

**修改建议**：使用自适应阈值（Otsu 或基于 ROI 均值）：

```python
mean_val = float(np.mean(roi))
# 自适应阈值：均值 + 偏移，确保在暗/亮场景下都能检测到前景
adaptive_thresh = max(80, min(180, int(mean_val * 0.6)))
_, bright = cv2.threshold(roi, adaptive_thresh, 255, cv2.THRESH_BINARY)
```

### [REC-3] YOLO 惰性加载非线程安全，并发识别可创建重复模型实例

**文件**：`src/core/capability/element_recognition/backends/yolo_backend.py:102-112`  
**严重程度**：Medium

**根因**：

```python
def _is_available(self) -> bool:
    if self._model is not None:
        return True
    try:
        from ultralytics import YOLO
        self._model = YOLO(self._model_path)   # ← 无锁写入
        logger.info(f"YOLO model loaded: {self._model_path}")
        return True
    except Exception as e:
        logger.debug(f"YOLO not available: {e}")
        return False
```

`recognize()` 调用 `_is_available()` 后再使用 `self._model`。如果两个线程同时首次调用 `recognize()`：
1. 线程 A 检查 `self._model is None` → True
2. 线程 B 检查 `self._model is None` → True
3. 线程 A 执行 `self._model = YOLO(...)`（耗时 1-3 秒）
4. 线程 B 执行 `self._model = YOLO(...)`（覆盖线程 A 的引用）
5. 线程 A 持有旧引用，线程 B 持有新引用——两个模型实例共存于内存

更糟的是，如果加载过程中一个线程读取 `self._model`（已赋值但未完成初始化），另一个线程开始推理，可能触发段错误。

**影响面**：并发识别场景下（GUI 并行预览 + 自动化任务），YOLO 模型被重复加载，内存占用翻倍，极端情况下推理崩溃。

**修改建议**：添加模块级锁或使用双检锁：

```python
import threading
_load_lock = threading.Lock()

def _is_available(self) -> bool:
    if self._model is not None:
        return True
    with self._load_lock:
        if self._model is not None:          # 二次检查
            return True
        try:
            from ultralytics import YOLO
            self._model = YOLO(self._model_path)
            logger.info(f"YOLO model loaded: {self._model_path}")
            return True
        except Exception as e:
            logger.debug(f"YOLO not available: {e}")
            return False
```

### [REC-4] 游戏场景检测在每次识别调用中无条件执行，UI 页面下浪费计算

**文件**：`src/core/capability/element_recognition/recognizer.py:143` + `src/core/capability/element_recognition/backends/color_backend.py:97-192`  
**严重程度**：Low

**根因**：`EndfieldElementRecognizer.recognize()` 在 Phase 2.5（行 143）无条件调用 `recognize_gameplay_scene()`：

```python
# Phase 2.5: 3D 游戏场景检测
gameplay_info = self._color_backend.recognize_gameplay_scene(screen)
```

`recognize_gameplay_scene` 执行以下操作：
1. BGR → HSV 转换（行 120）
2. BGR → Gray 转换（行 121）
3. 蓝色像素遮罩 + 比率计算（行 125-127）
4. 肤色检测 + 轮廓查找（行 135-147）
5. 非蓝色物体检测 + 轮廓查找（行 150-163）
6. 亮度阈值 + 左右面板轮廓查找（行 166-187）

共 2 次颜色空间转换、4 次 `findContours` 调用、2 次 `threshold`。对于 1920x1080 截图，单次调用约 30-50ms。如果当前页面是纯 UI（如设置页、任务列表），蓝色比率检查（行 129）快速返回 `is_gameplay=False`，但前面的颜色转换和轮廓检测已经全部执行。

**影响面**：每次识别调用浪费 20-40ms 在无效的游戏场景检测上。高频识别（如导航循环 2Hz）累积浪费显著。

**修改建议**：将蓝色比率检查提前为独立的快速 pre-check，仅在比率超过阈值时才执行完整的轮廓分析：

```python
def recognize_gameplay_scene(self, screen: np.ndarray, min_blue_ratio: float = 0.6) -> Dict[str, Any]:
    result = {"is_gameplay": False, "blue_ratio": 0.0, ...}
    
    # 快速 pre-check：仅计算蓝色比率
    hsv = cv2.cvtColor(screen, cv2.COLOR_BGR2HSV)
    blue_mask = cv2.inRange(hsv, np.array([80, 30, 30]), np.array([140, 255, 255]))
    blue_ratio = cv2.countNonZero(blue_mask) / (screen.shape[1] * screen.shape[0])
    result["blue_ratio"] = round(blue_ratio, 3)
    
    if blue_ratio < min_blue_ratio:
        return result   # ← 早期退出，跳过所有轮廓分析
    
    result["is_gameplay"] = True
    # ... 仅在确认 gameplay 后执行灰色转换、轮廓分析等
```

### [REC-6] `_deduplicate` 子串标签匹配导致假阳性去重

**文件**：`src/core/capability/element_recognition/recognizer.py:388-394`  
**严重程度**：Low

**根因**：

```python
label_match = (
    label_a == label_b
    or label_a in label_b
    or label_b in label_a
)
```

子串匹配 `in` 过于宽松。例如：
- `label_a = "button"`，`label_b = "button_ok"` → `"button" in "button_ok"` → True → 被去重
- `label_a = "close"`，`label_b = "close_info"` → `"close" in "close_info"` → True → 被去重

实际场景中，"button"（通用按钮模板）和 "button_ok"（确认按钮模板）是两个不同的元素，但去重逻辑将它们合并为同一个。

**影响面**：不同但名称包含公共子串的元素被错误合并，导致页面分类时缺少关键元素，分类置信度下降。

**修改建议**：使用精确匹配或前缀/后缀匹配替代子串匹配：

```python
label_match = (
    label_a == label_b
    or label_a.startswith(label_b + "_")
    or label_b.startswith(label_a + "_")
)
```

### [REC-7] `SceneGeometryAnalyzer.analyze` 每次调用分配 ~30MB 中间数组

**文件**：`src/core/capability/element_recognition/backends/scene_geometry.py:50-52,69`  
**严重程度**：Low

**根因**：

```python
gray = cv2.cvtColor(screen, cv2.COLOR_BGR2GRAY).astype(np.float32)     # ~8MB (1920x1080)
hsv = cv2.cvtColor(screen, cv2.COLOR_BGR2HSV).astype(np.float32)       # ~8MB
lab = cv2.cvtColor(screen, cv2.COLOR_BGR2LAB).astype(np.float32)       # ~8MB
# ...
rendered = screen.copy()                                                  # ~6MB
```

每次 `analyze()` 调用产生 4 个全尺寸数组副本（3 个 float32 + 1 个 uint8 拷贝）。对于 1920x1080 截图，总计约 30MB。如果调用方保留 `SceneAnalysis3D` 对象（包括 `rendered_image`），这些数组不会被立即释放。

此外，`.astype(np.float32)` 在 `cv2.cvtColor` 返回的 uint8 数组上再创建一个 float32 副本，产生两次分配（cvtColor 内部 + astype）。可以合并为一次：

```python
gray = cv2.cvtColor(screen, cv2.COLOR_BGR2GRAY)
gray = gray.astype(np.float32)  # 仍两次分配，但可改用 cv2.convertScaleAbs 或直接操作 uint8
```

但更好的优化是：`Canny` 和 `GaussianBlur` 可以直接接受 uint8 输入，无需提前转为 float32。`_compute_edge_map` 中 `blurred.astype(np.uint8)` 又转回了 uint8（行 136），说明 float32 是不必要的。

**影响面**：每次 3D 场景分析分配 30MB，在低内存设备或长时间运行中可能触发 GC 压力。

**修改建议**：
1. 仅在需要浮点精度的步骤中转换类型，尽量复用 uint8 中间结果
2. `rendered_image` 设为可选——调用方若不需可视化，传 `render=False` 跳过 `screen.copy()`

### [REC-8] `TemplateRegistry` 单例 `__new__` 无锁保护

**文件**：`src/core/capability/element_recognition/pipeline/template_registry.py:22-25`  
**严重程度**：Low

**根因**：

```python
def __new__(cls) -> TemplateRegistry:
    if cls._instance is None:
        cls._instance = super().__new__(cls)   # ← 无锁
    return cls._instance
```

与 234853.md R-1（`_android_clients` 不清理）、2315.md R3（logger 惰性初始化）同源。两个线程同时首次调用 `TemplateRegistry()` 可能创建两个实例。`__init__` 的 `_initialized` 守卫（行 28）防止重复初始化，但两个实例各自持有独立的 `_templates` dict。

**影响面**：低概率——模板通常在启动早期由单线程加载。但若 GUI 并行启动路径提前触发模板加载，可能产生双实例。

**修改建议**：使用双检锁：

```python
def __new__(cls) -> TemplateRegistry:
    if cls._instance is None:
        with cls._init_lock:
            if cls._instance is None:
                cls._instance = super().__new__(cls)
    return cls._instance
```

### [REC-10] `_run_maafw_ocr` 未校验 `best.box` 格式，[x1,y1,x2,y2] 与 [x,y,w,h] 混淆风险

**文件**：`src/core/capability/element_recognition/backends/ocr_backend.py:142-152`  
**严重程度**：Low

**根因**：

```python
bx1, by1, bw, bh = best.box
results.append(ElementInfo(
    bbox=(bx1 / w, by1 / h, (bx1 + bw) / w, (by1 + bh) / h),
    center=((bx1 + bw / 2) / w, (by1 + bh / 2) / h),
```

代码假设 `best.box` 为 `[x, y, w, h]` 格式，但 MaaFW 的 `JOCRResult.best.box` 实际格式取决于内部实现。若返回 `[x1, y1, x2, y2]`，则 `bw = x2 - x1` 被当作宽度，`bh = y2 - y1` 被当作高度，但 `(bx1 + bw) / w` 实际为 `(x1 + (y2-y1)) / w`——宽度和高度混用导致 bbox 严重错误。

`_run_ocr_manager`（行 291-292）也有相同问题：`x, y, bw, bh = box` 假设 [x,y,w,h] 格式。

`_parse_ocr_box`（行 237-273）已处理多种格式，但 `_run_maafw_ocr` 不使用该方法，直接解包 `best.box`。

**影响面**：若 MaaFW OCR 返回 [x1,y1,x2,y2] 格式，所有 OCR 元素的 bbox 和中心点错误，影响后续页面分类和点击坐标计算。

**修改建议**：使用 `_parse_ocr_box` 的格式推断逻辑统一处理：

```python
bx, by, bw, bh = self._parse_ocr_box(
    {"box": list(best.box)}, 0, 0, w, h, w, h
)
```

**文件**：`src/core/capability/element_recognition/backends/scene_geometry.py:134-137, 139-143, 145-151`  
**严重程度**：Info/Design

**根因**：三个 `_compute_*` 方法均对输入执行 `cv2.GaussianBlur`（uint8/float32 均可接受），但上游在行 50-52 已将所有输入转为 float32。后续 `_compute_edge_map` 在行 136 又将 float32 转回 uint8（`blurred.astype(np.uint8)`）供 Canny 使用，Canny 输出再除以 255 转回 float32。这种往返转换产生不必要的内存分配和类型转换开销。

**影响面**：单次分析中约 6 次多余的类型转换（3 个上游 astype + 3 个下游 roundtrip），在 1920x1080 下每转换约 8ms。

**修改建议**：`_compute_edge_map` 直接接受 uint8 输入，避免 float32 roundtrip。或将所有中间计算统一为 float32 但移除不必要的 astype 往返。
