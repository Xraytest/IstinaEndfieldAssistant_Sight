# 数据流完整性审计报告 — IstinaEndfieldAssistant Sight

> **批次定位**：本批次对 CLI → Runtime → Device 的完整数据流进行端到端追踪审计，识别**数据契约破损**——即调用方期望的返回值类型与实际返回值不一致的问题。不重复提交历史已记录的独立 bug，而是分析**跨层数据传递**中的系统性缺陷。
>
> **审查方式**：纯静态代码逻辑分析，未执行测试；所有结论均经当前 `main` 分支源文件逐行核对。

---

## 一、审计范围与方法

### 1.1 追踪的数据流路径

| 路径 | 入口 | 出口 | 追踪目的 |
|------|------|------|---------|
| **CLI → Runtime → Device** | `istina.py` `_interactive_loop` → `CLIDispatch.dispatch` → `execute` → `_nav3_walk` → `VlmWalkNavigator` → `_vlm_keyevent` → `android.keyevent` | 设备 | nav3 walk 完整链路（已审计 W1/C10） |
| **CLI → Runtime → GUI** | `handlers.py` `_handle_nav3` → `execute` → `CLIBridge._on_stdout` → `commandFinished` → `main_window._on_bridge_command_finished` | GUI 信号 | **本次审计重点** |
| **GUI → Runtime → Device** | `main_window` → `CLIBridge.execute` → `istina.py` → `runtime` → `_screenshot` → `android.screenshot` | 设备 | GUI 截图链路 |
| **LLM 初始化链** | `runtime._llm_client_instance` property → `_get_llm_client` → `LlmClient` → `llama-server` | 子进程 | C10 条件触发路径 |

### 1.2 数据契约定义

数据契约指：**函数签名/文档约定的返回值类型与实际返回类型的一致性**。以下契约在代码中被隐式约定但未显式 enforce：

| 契约 | 约定方 | 期望类型 | 实际类型（部分路径） |
|------|--------|---------|-------------------|
| `execute()` 返回值 | `handlers.py` 调用方 | `Dict[str, Any]`（含 "status" 键） | `None`（未知命令路径） |
| `_handle_nav3` 返回值 | `CLIDispatch.dispatch` | `Dict[str, Any]` | `None`（未知 action 路径） |
| `commandFinished` signal payload | `CLIBridge._on_stdout` | `dict` | `None`（execute 返回 None 时） |
| `_on_bridge_command_finished` 参数 | Qt signal/slot | `dict` | `None`（同上层） |

## 二、核心发现：execute() 数据契约破损

### DF-1（Medium）`execute()` 对未知命令返回 `None` 而非错误 dict

**位置**：`src/core/service/runtime.py:348-349`

```python
# 当前代码：
if domain == "llm":
    if action in ("chat", "prompt", "run"):
        return self._llm_run(params)
    if action == "status":
        return self._llm_status()
    return {"status": "error", "message": "unknown llm action"}  # ← 仅 llm 有兜底
self._logger.warning(LogCategory.MAIN, "未知命令", command=command)
return None  # ← 其他所有域的未知命令返回 None
```

**问题**：`execute()` 对大多数命令域（nav3、nav2、scene、task 等）的未知 action 返回 `None`，而非与 llm 域一致的错误 dict。这违反了函数签名 `-> Any` 的隐式约定——调用方期望 dict 类型。

**影响链**：

```
execute("nav3.invalid_action", {})
  → 无匹配分支 → return None
    → _handle_nav3 透传 None
      → CLIDispatch.dispatch 透传 None
        → _interactive_loop: result.get("status") → AttributeError
        → _on_bridge_command_finished: result.get("status") → AttributeError
```

**触发场景**：
1. CLI 交互模式：用户输入 `nav3 walk` 但 action 不存在（`nav3 invalid`）
2. GUI 桥接：任何通过 `CLIBridge` 发送的未知命令
3. 测试/脚本：调用 `execute("unknown.command", {})` 时

**与历史发现的关联**：
- 与 **XC-1**（execute 返回 None）相关，但 XC-1 被降级为"非终端用户问题"（因 `CLIDispatch.dispatch` 在 line 91 拦截未知命令）。本次审计发现 XC-1 的降级结论**不完整**——`execute()` 返回 `None` 不仅影响 CLI 交互模式，还影响 GUI 桥接链路。

### DF-2（Medium）`_on_bridge_command_finished` 缺少 None 保护

**位置**：`src/gui/pyqt6/main_window.py:279-287`

```python
def _on_bridge_command_finished(self, command: str, result: dict) -> None:
    """同步设备页与标准推理页的连接状态。"""
    if command.startswith("system connect"):
        if result.get("status") == "success":  # ← 若 result 为 None → AttributeError
            ...
```

**问题**：该 slot 直接调用 `result.get("status")` 而不检查 `result` 是否为 `None`。当 `execute()` 返回 `None` 时，此方法抛 `AttributeError`。Qt 的 signal/slot 机制会捕获异常并打印到 stderr，但 GUI 状态更新被中断。

**对比**：`CLIBridge._on_stdout`（`cli_bridge.py:176-179`）有正确的保护：
```python
try:
    result = json.loads(line.strip())
    self.commandFinished.emit(command, result)
except json.JSONDecodeError:
    ...
```

但 `_on_stdout` 只保护 JSON 解析失败的情况。如果 `execute()` 在 CLI 子进程中成功返回 `None`（即 CLI 子进程正常退出，stdout 输出空行），`json.loads("\n")` 会抛 `JSONDecodeError`，该行被跳过，`commandFinished` 不会被发出。这意味着 `_sync_execute` 会一直等到超时。

**修正**：DF-1 的修复（`execute()` 返回错误 dict 而非 `None`）会自动解决 DF-2。在修复前，DF-2 应添加防御性检查。

### DF-3（Medium）`_interactive_loop` 对 `None` 结果的错误消息误导

**位置**：`src/cli/istina.py:356-362`

```python
result = CLIDispatch(runtime).dispatch(args)
# result 可能为 None
self._logger.info("CLI 交互循环: 执行完成", status=result.get("status") if isinstance(result, dict) else None)
# ← isinstance 保护了日志，但...

except Exception as exc:
    result = {"status": "error", "message": str(exc)}
    # ← 用户看到: "VLM keyevent 'w' failed: invalid keyevent: 'w'"
    #    而非: "未知命令: nav3.invalid"
```

**问题**：交互模式对 `None` 结果的处理方式是捕获 `AttributeError` 并返回 `str(exc)`。用户看到的错误消息是**异常类型+消息**（如 `"VLM keyevent 'w' failed: invalid keyevent: 'w'"`），而非**业务语义**（如 `"未知命令: nav3.invalid"`）。

**影响**：用户无法从错误消息中理解真正的错误原因（命令不合法），而是看到内部实现细节（VLM keyevent 失败）。

## 三、数据契约破损的完整影响矩阵

### 3.1 `execute()` 返回 `None` 的所有下游影响

| 下游接收方 | 位置 | 对 `None` 的处理 | 结果 |
|-----------|------|-----------------|------|
| `_handle_nav3` | `handlers.py:253-254` | 直接 `return` 透传 | 上级收到 `None` |
| `CLIDispatch.dispatch` | `handlers.py:70-91` | `return None` 透传 | 上级收到 `None` |
| `_interactive_loop` | `istina.py:355-356` | `result.get("status")` → `AttributeError` | 用户看到误导性错误消息 |
| `_on_bridge_command_finished` | `main_window.py:282` | `result.get("status")` → `AttributeError` | GUI 状态更新中断 |
| `_sync_execute` | `maaend_control_page.py:326-336` | `_on_finished` 将 `None` 存入 `result`，`timed_out = True` | 将"命令错误"误判为"超时" |

### 3.2 `_sync_execute` 的误判问题

**位置**：`src/gui/pyqt6/pages/maaend_control_page.py:326-338`

```python
def _on_finished(cmd: str, res: dict):
    nonlocal result
    if cmd.split()[:len(expected)] == expected:
        result = res          # ← res 可能为 None
        loop.quit()

# ...
timed_out = result is None  # ← None 被判定为超时，而非命令错误
```

**问题**：当 `execute()` 返回 `None` 时，`_sync_execute` 将结果解读为"超时"（`timed_out = True`），而非"命令错误"。调用方可能基于"超时"判断触发重试或超时处理，而非针对"命令错误"的处理。

**触发场景**：通过 `_sync_execute` 执行的任何未知命令。

## 四、修复建议

### 4.1 根本修复：`execute()` 添加默认错误返回

**位置**：`src/core/service/runtime.py:348-349`

```python
# 修改前：
self._logger.warning(LogCategory.MAIN, "未知命令", command=command)
return None

# 修改后：
self._logger.warning(LogCategory.MAIN, "未知命令", command=command)
return {"status": "error", "message": f"unknown command: {command}"}
```

**修改量：1 行。**

**影响**：修复所有下游 `None` 传播问题。所有 `execute()` 调用方现在收到一致的 error dict。

### 4.2 防御性修复：`_on_bridge_command_finished` 增加 None 保护

**位置**：`src/gui/pyqt6/main_window.py:279-287`

```python
# 修改前：
def _on_bridge_command_finished(self, command: str, result: dict) -> None:
    if command.startswith("system connect"):
        if result.get("status") == "success":

# 修改后：
def _on_bridge_command_finished(self, command: str, result: Optional[dict]) -> None:
    if not isinstance(result, dict):
        self._logger.warning("bridge command 返回非 dict", command=command, result_type=type(result).__name__)
        return
    if command.startswith("system connect"):
        if result.get("status") == "success":
```

### 4.3 防御性修复：`_interactive_loop` 改进错误消息

**位置**：`src/cli/istina.py:355-362`

```python
# 修改前：
result = CLIDispatch(runtime).dispatch(args)
self._logger.info("...", status=result.get("status") if isinstance(result, dict) else None)

# 修改后：
result = CLIDispatch(runtime).dispatch(args)
if result is None:
    result = {"status": "error", "message": f"dispatch returned None for: {line}"}
```
