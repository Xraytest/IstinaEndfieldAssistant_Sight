### [S1] `scripts/verify_llm.py` 的 "TPS" 实为字符速率，性能择优结论失真
**优先级：Low（边界脚本，不影响正式功能）**
**位置**：`scripts/verify_llm.py:96-99`（`measure` 函数）

```python
output = client.chat("Hello, reply with one short Chinese greeting.", max_tokens=32)
t1 = time.time()
tokens = max(len(output), 1)          # ← output 是回复文本字符串，len() 统计字符数
return tokens / max(t1 - t0, 1e-6)    # ← 返回单位为 chars/sec，却被命名为/判定为 TPS
```

**根因分析**：`LlmClient.chat()` 返回的是模型回复的**文本字符串**（`tests/test_llm_client_embeds_image_as_data_uri.py` 中 `fake_post` 返回 `{"choices":[{"message":{"content":"ok"}}]}`，`chat()` 返回 `"ok"` 可佐证）。`len(output)` 统计的是**字符数**，而非 token 数。但同仓 `scripts/verify_llm_simple.py:131` 已正确实现为 `completion_tokens = int(usage.get("completion_tokens") or max_tokens)`，二者对 "TPS" 的定义不一致。

**影响面**：
- `main()` 中以 `tps > 100` 作为"找到可用配置"的判据（line 130 `return 0 if best > 100 else 3`）。由于实际是字符速率，中文回复下几乎必然 >100，判定恒真，"最优配置"择优结果不可信，且退出码恒为 0，掩盖了真实吞吐不足的情况。
- 与 `verify_llm_simple.py` 的口径冲突，两份基准脚本结论不可横向比较。

**修改建议**（最小可行）：
```python
client = LlmClient(base_url=f"http://127.0.0.1:{port}/v1")
t0 = time.time()
resp = client.chat("Hello, reply with one short Chinese greeting.", max_tokens=32)
t1 = time.time()
# 从 chat 返回中提取 usage；若 chat 不返回 usage，则改用 /v1/chat/completions 的原始响应
completion_tokens = getattr(resp, "usage", {}).get("completion_tokens") or max(len(str(resp)), 1)
return completion_tokens / max(t1 - t0, 1e-6)
```
或直接复用 `verify_llm_simple.py` 中基于 `usage.completion_tokens` 的算法，统一两份脚本口径。

**非期待变化**：若 chat 接口确实不回传 `usage`，需下沉到 `LlmClient` 原始响应；建议先确认 `LlmClient.chat` 返回结构（dict 还是 str）再定方案，避免引入新耦合。

### [S2] `src/cli/handlers.py` `_handle_gpu_recommend` 的 4GB 阈值分支为死代码
**优先级：Low（逻辑冗余，不影响主流程）**
**位置**：`src/cli/handlers.py:672-684`

```python
gpus = _handle_gpu_status(runtime, args).get("gpus", [])
recommendation = "CPU"
if gpus:
    mem = gpus[0].get("free_memory_bytes", 0)
    if mem and mem >= 4 * 1024 * 1024 * 1024 or mem and mem >= 2 * 1024 * 1024 * 1024:
        recommendation = "GPU"
```

**根因分析**：Python 运算符优先级中 `and` 高于 `or`，该行等价于 `(mem and mem>=4GB) or (mem and mem>=2GB)`。当 `mem>=2GB` 时第二个子式已为真，整式为真并置 `"GPU"`；`mem>=4GB` 这个"更高档位"分支永远不会改变结果——它只是 `>=2GB` 的真子集。作者意图显然是区分两档（如 4GB→高配 GPU、2GB→低配 GPU，或给出不同文案），但当前写法把两档坍缩为一档。

**影响面**：功能上仅 "GPU" 推荐文案，不会崩溃；但 4GB 阈值纯属误导，未来维护者若想基于该阈值做分级（如返回 `recommendation` 细节或日志），会发现分支无效。

**修改建议**（最小可行，明确分级意图）：
```python
if mem and mem >= 4 * 1024 * 1024 * 1024:
    recommendation = "GPU"          # 高显存
elif mem and mem >= 2 * 1024 * 1024 * 1024:
    recommendation = "GPU"          # 低显存（两档若最终文案相同，可直接合并为一行）
```
若两档本就等价，则应删去 4GB 比较，仅保留 `mem >= 2GB` 并加注释说明意图，消除歧义。

**非期待变化**：无副作用；纯逻辑澄清。

## 3. 对既往报告的审计（纠错与必要性复核）

### [A1] `20260711_0200_nav.md` 中 NAV-05 的异常类型标注不准确（TypeError 应为 ValueError）
**审计结论：既往报告存在事实性错误，予以纠正。**

NAV-05 原文称："`raw_location` 为字符串或 dict 时……`float(rl[0])` 全部失败并抛出 `TypeError`"。

实际推导（`entity_db.py:Entity.from_raw`，结合 Python 语义）：
- `raw_location = {"x":1,"y":2}`（dict）：`len(rl)` 为键数，`rl[0]` 为首个**键名字符串**（如 `"x"`），`float("x")` 抛 **`ValueError`**，非 `TypeError`。
- `raw_location = "1,2,3"`（字符串）：`rl[0]=="1"`、`rl[1]==","`，`float(",")` 抛 **`ValueError`**。
- `raw_location = [1, 2, 3]`（合法列表）：正常。
- 仅当元素本身为非数值且不可转为 float 时抛 `ValueError`；`TypeError` 仅在 `rl` 为不可下标/不可 `len` 的类型（如 `None`）时出现，而 `.get()` 默认 `[0,0,0]` 已保护缺失键，且 `rl[0]` 对 `int` 等会抛 `TypeError`——但合法 JSON 中 `raw_location` 缺失时走默认值，不会到这一步。

**结论**：NAV-05 的崩溃确实存在，但异常类型是 **`ValueError`**（数值转换失败），而非其标注的 `TypeError`。`memory.md` 中"NAV-05 审计子论断 dict 走默认分支不崩、真实崩溃来自字符串/非数字列表的 ValueError"已部分纠正，但仍在措辞上保留"float(rl[0]) 抛 TypeError"的误差。修复建议（类型校验 + 跳过畸形条目）**仍然成立且方向正确**，仅异常类型描述需更正，以免维护者在 `except` 中误捕 `TypeError` 而漏掉真实的 `ValueError`。

### [A2] XC-1（execute 未知命令返回 None）的"用户可感知影响"被高估
**审计结论：严重程度应为 Info/Design，既往报告对其负面影响的表述不可经实际调用链复现。**

既往记录称 XC-1 会导致"CLI 退出码 1 但无错误文案"。但对照调用链：
- CLI 入口 `istina.py:main` → `CLIDispatch(runtime).dispatch(args)`。
- `CLIDispatch.dispatch`（`handlers.py:48-91`）在所有已知顶层命令（`system/daily/.../llm/nav*/...`）都不匹配时，**先行返回 `{"status":"error","message":"unknown command"}`**（line 91），根本不会把请求转发到 `IstinaRuntime.execute`。
- GUI 侧 `MaaEndControlPage._sync_execute` 永远经 `CLIBridge` → CLI 进程 → `dispatch`，拿到的都是 dict 错误体；`runtime.execute("unknown.command")` 直接返回 `None` 的路径仅出现在**内部直调或单元测试**（如 `tests/test_istina_runtime.py::test_execute_returns_none_for_unknown_command` 显式断言 `result is None`，属预期行为）。

**结论**：XC-1 在"经 CLI/GUI 正常使用"路径上**不可达**，其"退出码 1 无文案"的影响描述缺乏调用链支撑。建议将 XC-1 降级为 Info/Design（契约清晰度问题），并明确"仅内部 API 误用场景存在，不影响终端用户"。既往报告将其列为 Low/Design 基本合理，但附带的影响描述需收窄。

### [A3] `20260711_0200_nav.md` 中 NAV-11 属于"不必要建议"
**审计结论：该建议无功能影响，不应作为待办项执行。**

NAV-11 建议删除 `navigator.py:164` 的 `self._entities.load()` 调用，理由是"每次调用无条件触发已加载的 load()，语义上误导维护者"。

但 `EntityDatabase.load()` 在已加载时仅做 `__loaded` 布尔判断即返回（开销可忽略），且 `list_entities` 作为对外查询入口，显式确保数据库已就绪是**防御性良构**——它使方法在"Navigator 构造后数据库被外部清空/热更新"等边界下仍健壮。删除该调用：
1. 无任何功能/性能收益（开销为一次布尔判断）；
2. 反而削弱健壮性，与项目"明确惰性加载意图"的其它写法（`_ensure_loaded`）方向相反。

**结论**：NAV-11 是"可维护性洁癖"型建议，非缺陷修复，属于**不必要建议**。建议从待办清单移除，或至多改为注释说明"load() 幂等，已加载时立即返回"，而非删代码。
