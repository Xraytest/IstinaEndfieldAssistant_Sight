### 2.1 🔴 高危问题

#### L01 - `MinimapLocator._find_minimap_bbox` 完全依赖硬编码 bbox，无真实小地图检测

**文件**: `src/core/service/navigation/minimap_locator.py:27-29, 105-114`
**风险等级**: 🔴 高危
**影响**: 分辨率变化或游戏UI更新后，小地图裁剪区域错误，定位完全失效

**问题代码**:
```python
# Known minimap regions in scrcpy 1280x720 coordinate space
_FALLBACK_MINIMAP_BBOX = (925, 15, 1250, 340)  # ❌ 硬编码 1280x720 坐标

def _find_minimap_bbox(self, frame, fw, fh):
    scale_x = fw / 1280.0
    scale_y = fh / 720.0
    fx1 = int(_FALLBACK_MINIMAP_BBOX[0] * scale_x)
    fy1 = int(_FALLBACK_MINIMAP_BBOX[1] * scale_y)
    fx2 = int(_FALLBACK_MINIMAP_BBOX[2] * scale_x)
    fy2 = int(_FALLBACK_MINIMAP_BBOX[3] * scale_y)
    return (fx1, fy1, fx2, fy2)  # ❌ 仅做线性缩放，不做实际检测
```

**问题分析**:
- `_FALLBACK_MINIMAP_BBOX` 基于 1280x720 的 scrcpy 分辨率定义
- `_find_minimap_bbox` 仅做线性缩放（`fw/1280`, `fh/720`），不做任何视觉检测
- 如果游戏UI更新导致小地图位置/大小变化，裁剪区域将不包含小地图
- 如果用户使用非标准分辨率（如 1920x1080、自定义窗口大小），缩放后的 bbox 可能裁剪到无关区域
- 缩放使用 `int()` 截断（非四舍五入），在低分辨率下累积误差更大
- `crop_minimap` 裁剪后传递给 `classify`，如果裁剪区域不包含小地图，ONNX 模型会对纯色/噪声区域分类，返回完全错误的位置
- 这直接导致批次34 J01 的伪造坐标问题更易触发——因为小地图裁剪失败时 `locate` 返回 None，触发 J01 的伪造 (0,0) 逻辑
- 注释承认是 "fallback" 和 "approximate"，但代码中完全没有 fallback 检测逻辑——这是唯一的小地图定位方式

**修复建议**:
```python
def _find_minimap_bbox(self, frame, fw, fh):
    # 优先使用基于视觉特征的检测（颜色/形状匹配）
    bbox = self._detect_minimap_region(frame)
    if bbox is not None:
        return bbox
    # fallback: 基于 1280x720 缩放
    scale_x = fw / 1280.0
    scale_y = fh / 720.0
    fx1 = int(_FALLBACK_MINIMAP_BBOX[0] * scale_x)
    fy1 = int(_FALLBACK_MINIMAP_BBOX[1] * scale_y)
    fx2 = int(_FALLBACK_MINIMAP_BBOX[2] * scale_x)
    fy2 = int(_FALLBACK_MINIMAP_BBOX[3] * scale_y)
    return (fx1, fy1, fx2, fy2)
```

### 2.2 🟡 中危问题

#### L02 - `_parse_tile` 固定宽度切片解析 map_id/level_id，格式变化时截取错误

**文件**: `src/core/service/navigation/minimap_locator.py:180-181`
**风险等级**: 🟡 中危
**影响**: 新的地图 tile_class 格式导致 level_id 截取错误

**问题代码**:
```python
if tile_class.startswith(("Map01Lv", "Map02Lv")):
    level_id = tile_class[5:11].lower()  # ❌ 假设 "Map01LvXXX" 格式，固定切片
```

**问题分析**:
- `tile_class[5:11]` 假设 `tile_class` 格式为 `"Map01LvXXXXXX"`（Map01/Map02 共5字符 + Lv + 6字符ID = 11字符）
- 如果 tile_class 为 `"Map01Lv01_01"`（下划线分隔），切片 `[5:11]` 截取 `"Lv01_0"` 而非预期的 `"01_01"`
- 对于 `"Map02Lv02_01"`，切片 `[5:11]` 截取 `"Lv02_0"` 而非 `"02_01"`
- 切片从索引5开始是因为 `"Map01"` 和 `"Map02"` 都是5字符——但如果未来有 `"Map10"` 之类的新地图，切片位置完全错误
- `_parse_tile` 中的 `startswith(("Map01Lv", "Map02Lv"))` 与切片 `[5:11]` 耦合，如果前缀与切片逻辑不一致会产生错误
- 对比: `Base01` 类的 `__r` 解析使用正则（`__r(\d+)_c(\d+)`），更健壮

**修复建议**:
```python
if tile_class.startswith(("Map01Lv", "Map02Lv")):
    # 提取 Lv 后的部分作为 level_id
    lv_part = tile_class.split("Lv", 1)
    if len(lv_part) == 2:
        level_id = lv_part[1].lower()
```

#### L03 - `_parse_tile` 置信度阈值 0.3 过低，ONNX 分类噪声导致大量误定位

**文件**: `src/core/service/navigation/minimap_locator.py:147`
**风险等级**: 🟡 中危
**影响**: 低置信度分类结果通过阈值，产生错误的位置估计

**问题代码**:
```python
def _parse_tile(self, tile_class, class_id, confidence):
    if tile_class == "None" or confidence < 0.3:
        return None  # ❌ 0.3 阈值过低
```

**问题分析**:
- 0.3 阈值意味着 ONNX 模型输出概率高于 30% 的分类就会被接受
- 对于多分类问题（如 50+ 类 tile），top-1 概率为 0.3 意味着模型对分类结果不确定
- `np.argmax(probs)` 选择概率最高的类，即使最高概率只有 0.3
- 如果 `cls.json` 定义了 50+ 个类别，随机猜测的期望最高概率约为 1/50 = 0.02，0.3 阈值意味着接受模型仅比随机好 15 倍的结果
- 对于小地图这种关键输入，误分类会导致 VLM 收到错误的位置信息
- 对比: 批次34 J01 中，伪造坐标的问题；L03 产生的误分类坐标同样会误导 VLM
- 没有阈值配置入口——值硬编码在方法体中

**修复建议**:
```python
# 在 __init__ 或 VlmWalkConfig 中配置
_MINIMAP_CLASSIFY_CONFIDENCE = 0.5  # 提高到 0.5

def _parse_tile(self, tile_class, class_id, confidence):
    if tile_class == "None" or confidence < self._MINIMAP_CLASSIFY_CONFIDENCE:
        return None
```

#### L06 - `VlmWalkNavigator._parse_action` 解析失败静默返回 None，无日志记录

**文件**: `src/core/service/navigation/vlm_walk_navigator.py:313-326`
**风险等级**: 🟡 中危
**影响**: VLM 输出格式异常时，导航步骤静默跳过，调试困难

**问题代码**:
```python
def _parse_action(self, reply):
    cleaned = reply.strip()
    if cleaned.startswith("```") and cleaned.endswith("```"):
        cleaned = cleaned.strip("`").strip()
        if cleaned.lower().startswith("json"):
            cleaned = cleaned[4:].strip()
    try:
        obj = json.loads(cleaned)
    except json.JSONDecodeError:
        return None  # ❌ 静默失败，无日志
    if not isinstance(obj, dict) or "action" not in obj:
        return None  # ❌ 静默失败，无日志
    return obj
```

**问题分析**:
- `_parse_action` 在解析失败时返回 None，调用方 `walk_to` 记录 `parse_error` 但只保存原始回复的前 120 字符
- 没有记录 WHY 解析失败——是 JSON 格式错误？缺少 "action" 字段？还是 VLM 返回了非 JSON 文本？
- VLM 可能输出多种格式：纯 JSON、markdown 代码块、带解释文本的 JSON、纯文本指令
- 当前解析器只处理 ` ```json...``` ` 格式，对于 ` ```json`（无闭合）或 ` ```JSON`（大写）等变体无效
- 对于 VLM 返回的 `{"action": "forward", "duration": 1.5} extra text`，`json.loads` 会失败
- 无解析失败的统计——如果 VLM 持续输出格式错误的回复，无法从日志中识别模式

**修复建议**:
```python
def _parse_action(self, reply):
    cleaned = reply.strip()
    if cleaned.startswith("```") and cleaned.endswith("```"):
        cleaned = cleaned.strip("`").strip()
        if cleaned.lower().startswith("json"):
            cleaned = cleaned[4:].strip()
    try:
        obj = json.loads(cleaned)
    except json.JSONDecodeError:
        self._logger.warning("VLM reply JSON parse failed: %s", reply[:200])
        return None
    if not isinstance(obj, dict) or "action" not in obj:
        self._logger.warning("VLM reply missing 'action' field: %s", reply[:200])
        return None
    return obj
```

#### L04 - `MinimapLocator` 直接访问 `MapDataLoader._maaend_root` 私有属性

**文件**: `src/core/service/navigation/minimap_locator.py:59`
**风险等级**: 🟢 低危
**影响**: MapDataLoader 内部路径结构变更时，MinimapLocator 隐式崩溃

model_path = (

#### L07 - `_execute_action` 对 duration 无范围校验，VLM 可能返回极端值

**文件**: `src/core/service/navigation/vlm_walk_navigator.py:264-282`
**风险等级**: 🟢 低危
**影响**: VLM 返回异常 duration（如 999 秒）导致角色长时间移动

**问题代码**:
```python
def _execute_action(self, action):
    act = action["action"]
    duration = action.get("duration")
    if act == "forward":
        self._input_fn("w", duration)  # ❌ duration 无范围校验
```

**问题分析**:
- `VlmWalkConfig` 中 `move_key_duration_s: float = 1.5` 是默认值，但 `_execute_action` 直接使用 VLM 返回的 duration
- VLM system prompt 限制 duration 为 1-5 秒，但 LLM 可能违反约束返回极端值
- 如果 VLM 返回 `{"action": "forward", "duration": 999}`，角色会持续前进 999 秒
- `_input_fn` 最终调用 `TouchManager.long_press` 或等效操作，duration 直接传递给 ADB swipe 命令
- ADB swipe 的 duration 限制取决于设备实现，但通常支持任意长按

**修复建议**:
```python
_MIN_DURATION = 0.5
_MAX_DURATION = 5.0

def _execute_action(self, action):
    act = action["action"]
    duration = action.get("duration")
    if duration is not None:
        duration = max(self._MIN_DURATION, min(self._MAX_DURATION, float(duration)))
    ...
```

#### L09 - `EntityDatabase.load()` 无 JSON 顶层结构校验

**文件**: `src/core/service/navigation/entity_db.py:78-83`
**风险等级**: 🟢 低危
**影响**: JSON 顶层为 dict 而非 list 时，遍历产生错误键名

**问题代码**:
```python
with open(path, "r", encoding="utf-8") as f:
    raw_maps = json.load(f)  # ❌ 无结构校验
for map_entry in raw_maps:  # 如果 raw_maps 是 dict，遍历的是键名
    map_id = map_entry.get("map_id", "")  # str.get() 不存在！
```

**问题分析**:
- `json.load()` 返回顶层 JSON 结构——如果文件是 `{}`（空 dict），for 循环不执行，静默返回空数据库
- 如果文件是 `{"error": "corrupt"}`，for 循环遍历键名 `"error"`，`"error".get("map_id", "")` 抛出 AttributeError
- 与批次2 NAV-04 发现类似但不同：NAV-04 关注的是 dict 顶层触发 TypeError，此处分析的是 AttributeError
- 已有外层 try/except 捕获 AttributeError，但会中断加载并返回 False，调用方无法区分"文件不存在"和"格式错误"

**修复建议**:
```python
with open(path, "r", encoding="utf-8") as f:
    raw_maps = json.load(f)
if not isinstance(raw_maps, list):
    self._logger.error("entity db: expected list, got %s", type(raw_maps).__name__)
    return False
```

#### L10 - `MapDataLoader.get_grid_cell` 惰性加载时加载全部数据

**文件**: `src/core/service/navigation/map_data_loader.py:157-160`
**风险等级**: 🟢 低危
**影响**: 单次查询触发全量 JSON 解析，增加不必要的 I/O 和内存

**问题代码**:
```python
def get_grid_cell(self, key: str) -> Optional[GridCell]:
    if not self._grid_tiers:
        self.load_grid_tiers()  # ❌ 加载全部 grid_tiers 而非单个 key
    return self._grid_tiers.get(key)
```

**问题分析**:
- `get_grid_cell("tier_3_2")` 仅需要一个 cell，但触发加载全部 `grid_tiers.json`
- 如果 `grid_tiers.json` 很大（大量 grid cell），每次调用都全量解析
- 多次调用 `get_grid_cell` 不同 key 时，每次都检查缓存（已加载后高效）
- 首次调用时全量加载是合理的设计（惰性加载），但接口语义暗示单 key 查询
- 对比: `_parse_tile` 在 minimap_locator.py 中的循环调用 `load_grid_tiers()` + 遍历搜索，效率更低

**修复建议**: 此问题属于设计选择，影响有限。建议在文档中注明 `get_grid_cell` 会触发全量加载，或在 `MapDataLoader` 中添加单 key 查询接口。

#### L05 - `MinimapLocator._load_onnx` ONNX Runtime providers 硬编码 CUDA 优先

**文件**: `src/core/service/navigation/minimap_locator.py:79-82`
**风险等级**: 信息
**影响**: CPU-only 环境尝试 CUDA 导致额外延迟

**问题代码**:
```python
self._session = ort.InferenceSession(
    str(model_path),
    providers=["CUDAExecutionProvider", "CPUExecutionProvider"],
)
```

**问题分析**:
- `providers=["CUDAExecutionProvider", "CPUExecutionProvider"]` 硬编码 CUDA 优先
- ONNX Runtime 在初始化 CUDA provider 时可能需要加载 CUDA 库，在 CPU-only 环境产生额外延迟
- ONNX Runtime 会自动回退到可用 provider，所以功能上不会失败
- 但初始化时间会增加（CUDA 探测通常需要 1-3 秒）
- 应使用 `ort.get_available_providers()` 动态检测

**修复建议**:
```python
import onnxruntime as ort
available = ort.get_available_providers()
providers = [p for p in ["CUDAExecutionProvider", "CPUExecutionProvider"] if p in available]
self._session = ort.InferenceSession(str(model_path), providers=providers)
```

#### L08 - `VlmWalkNavigator._frame_to_base64` 不校验 `cv2.imencode` 返回值

**文件**: `src/core/service/navigation/vlm_walk_navigator.py:299-301`
**风险等级**: 信息
**影响**: 编码失败时抛出异常

**问题代码**:
```python
def _frame_to_base64(self, frame):
    _, buf = cv2.imencode(".png", frame)  # ❌ 不校验返回值
    return base64.b64encode(buf).decode("ascii")
```

**问题分析**:
- `cv2.imencode` 返回 `(success: bool, buf: ndarray)` 元组
- 如果 `frame` 是空数组、内存不足或格式不支持，`success` 为 False，`buf` 为 None
- `base64.b64encode(None)` 抛出 TypeError
- 调用方 `walk_to` 的 try/except 会捕获异常并记录 `vlm_error`，但会消耗一步 VLM 配额
- 实际触发概率极低（frame 来自 `_grab_frame` 的 cv2.imdecode 成功结果）

**修复建议**:
```python
def _frame_to_base64(self, frame):
    success, buf = cv2.imencode(".png", frame)
    if not success:
        raise ValueError("cv2.imencode failed for frame")
    return base64.b64encode(buf).decode("ascii")
```

#### L11 - `MapDataLoader.load_layout` 无 JSON 结构类型校验

**文件**: `src/core/service/navigation/map_data_loader.py:98-103`
**风险等级**: 信息
**影响**: 文件内容为 dict 而非 list 时，错误被 try/except 吞掉

**问题代码**:
```python
try:
    with open(path, "r", encoding="utf-8") as f:
        raw = json.load(f)
except Exception as exc:
    self._logger.warning("failed to load layout %s: %s", map_name, exc)
    return None  # ❌ JSON 语法错误和类型错误返回相同的 None
```

**问题分析**:
- `json.load(f)` 成功但返回 dict 时，`raw.get("levels", {})` 正常执行（dict.get 存在）
- 但后续 `for lid, lv in raw.get("levels", {}).items()` 遍历的是 dict 的 items（key-value 对），而非 list
- 遍历 dict items 时 `lv` 是 value（通常是 dict），`lv["x"]` 在 value 为 dict 时正常
- 但如果 "levels" 的值是 list 而非 dict，遍历 list 时 `lv` 是元素（dict），`lv["x"]` 仍正常
- 实际上：如果 "levels" 缺失，使用 `{}`（dict），items() 遍历 key-value 对
- 真正的问题：如果 raw 本身是 list（而非 dict），`raw.get("levels", {})` 抛出 AttributeError
- 但 try/except 捕获了此异常，返回 None——调用方无法区分"文件不存在"和"格式错误"
- 此问题与 L09 类似，属于数据加载层的通用模式

**修复建议**: 在 try/except 后添加结构类型校验（同 L09 修复方案）。

#### 批次34 J01 — VlmWalkNavigator 伪造坐标（确认准确）

**原报告**: 082654.md

## 6. 结论

本次审计发现11个新问题，其中1个高危、3个中危、4个低危、3个信息。历史报告零修正。

**关键发现**:
1. **L01（高危）**: `MinimapLocator._find_minimap_bbox` 完全依赖硬编码 bbox 缩放，无真实小地图检测——分辨率变化或UI更新后裁剪区域错误，定位完全失效
2. **L02（中危）**: `_parse_tile` 使用固定宽度切片 `tile_class[5:11]` 解析 level_id——新地图格式导致截取错误
3. **L03（中危）**: ONNX 分类置信度阈值 0.3 过低——大量低置信度误分类通过
4. **L06（中危）**: `_parse_action` 解析失败静默返回 None，无日志记录——VLM 输出格式异常时难以调试
5. **L04/L07/L09/L10**: 低危问题涵盖跨模块耦合、duration 无校验、JSON 结构无校验、惰性加载语义不清晰
6. **L05/L08/L11**: 信息级问题涵盖 ONNX provider 硬编码、imencode 返回值不校验、结构类型校验缺失

**建议修复顺序**:
1. **L01**: 添加小地图视觉特征检测（高危，导航正确性基础）
2. **L06**: 添加解析失败日志（中危，调试可观测性）
3. **L02**: 修复 level_id 切片（中危，数据正确性）
4. **L03**: 提高置信度阈值（中危，定位准确率）
5. **L04**: 添加公共 getter 替代私有属性访问
6. **L07**: 添加 duration 范围校验
7. **L09/L11**: 添加 JSON 结构校验
8. **L05/L08/L10**: 代码质量改进

**关联分析**: L01（硬编码 bbox）与批次34 J01（伪造坐标）形成叠加效应——小地图裁剪失败时 `locate` 返回 None，触发 J01 的伪造 (0,0) 坐标逻辑。修复 L01 可减少 J01 的触发频率。
