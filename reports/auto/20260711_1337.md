### [NEW-1 Medium] `handlers.py` `config get/set` 对白名单中的点分键写入「扁平键」，与嵌套配置契约不一致 —— 设置项静默失效

**位置**：`src/cli/handlers.py`
- `:511` `_handle_config_get`：`value = runtime.config.get(args.key)`
- `:614` `_handle_config_set`：`runtime.config[key] = _coerce_config_value(key, args.value)`（其中 `key` 为白名单中的点分键，如 `device.serial`、`llm.port`、`logging.level`、`system.minimize_to_tray` 等）

**根因分析**：
本轮为 `config set` 新增了 `_ALLOWED_CONFIG_KEYS` 白名单（P02，正确且必要），但白名单里的键都是**点分字符串**（`"device.serial"`、`"llm.port"`、`"logging.level"` …），而 `runtime.config` 返回的是**嵌套 dict**。于是：

- `config set device.serial 127.0.0.1:5555` 实际执行的是 `self._config["device.serial"] = "127.0.0.1:5555"`，写入的是**顶层扁平键** `"device.serial"`；
- 全仓所有消费方读取的是嵌套键 `self._config["device"]["serial"]`（如 `runtime.py:156-157` `device_cfg.get("last_connected") / device_cfg.get("serial")`），**永远读不到**这个扁平键；
- `config get device.serial` 又因为同样用 `runtime.config.get("device.serial")`（扁平键）去读，而加载时嵌套键并不会生成扁平键，所以**首次未设置时返回 None**。即便本次 set 过（扁平键已持久化），get 能读回扁平键，但**运行时完全不生效**。

**影响面**：中（功能性缺陷）。P02 白名单营造出「已可安全配置」的假象，但白名单内的所有点分键（`device.serial`、`device.auto_reconnect`、`llm.port`、`llm.model_path`、`llm.temperature`、`logging.level`、`system.minimize_to_tray`、`preview_interval_ms` …）经 CLI 设置后**对运行时无任何影响**，用户误以为配置生效。这是本轮新增安全代码引入的契约错误。

**修改方案**（二选一）：
```python
# 方案 A：set/get 落库时按点分路径写入嵌套结构
def _set_nested(cfg: dict, key: str, value):
    node = cfg
    parts = key.split(".")
    for p in parts[:-1]:
        node = node.setdefault(p, {})
        if not isinstance(node, dict):
            raise ValueError(f"config 键冲突: {key}")
    node[parts[-1]] = value

def _get_nested(cfg: dict, key: str):
    node = cfg
    for p in key.split("."):
        if not isinstance(node, dict) or p not in node:
            return None
        node = node[p]
    return node
```
或在 `IstinaRuntime` 上暴露 `get/set_config(key, value)` 统一处理点分键（与 `save_config()` 配合）。`runtime.py:142` 的 `config` property 不应直接暴露扁平点分写入。

### [NEW-2 Medium] `llm/runtime.py:_resolve_model_path` 仅约束绝对路径，相对 `..` 路径可逃逸项目根（CFG-10 防护不完整）

**位置**：`src/core/capability/llm/runtime.py:241-260`

```python
def _resolve_model_path(self, model_path: str) -> str:
    root = get_project_root().resolve()
    p = Path(model_path)
    if p.is_absolute():
        resolved = p.resolve()
        if root not in resolved.parents and resolved != root:
            self._logger.warning("model_path 绝对路径越界，已忽略: %s", model_path)
            p = Path(p.name)  # 退化为仅文件名
    in_root = (root / str(p)).resolve()
    if in_root.exists():
        return str(in_root)
    under_models = (root / "models" / str(p)).resolve()
    if under_models.exists():
        return str(under_models)
    return str(p)
```

**根因分析**：
防护只在 `p.is_absolute()` 分支生效；当用户/CLI 传入**相对路径含 `..`**（如 `../../windows/system32/foo.gguf` 或 `../secret.gguf`）时：
1. `p.is_absolute()` 为 `False` → 跳过越界校验；
2. `in_root = (root / str(p)).resolve()` 解析后为**项目根之外**的路径；
3. 若该文件存在，直接 `return str(in_root)` —— 路径遍历逃逸成功，llama-server 将加载项目目录之外的任意文件（信息泄露/加载非预期模型）。

`model_path` 经 `istina config set llm.model_path <value>`（NEW-1 的扁平键问题另论）进入配置，并在启动时被 `_resolve_model_path` 解析，故该入口可达。CFG-10 的本意是「禁止路径遍历逃逸」，但相对 `..` 这一主路径未被覆盖。

**影响面**：中（纵深防御缺口）。本地场景危害有限，但与 CFG-09/CFG-10 的「禁止越界」承诺相矛盾，应在合并前补齐。

**修改方案**：
```python
def _resolve_model_path(self, model_path: str) -> str:
    root = get_project_root().resolve()
    p = Path(model_path)
    if p.is_absolute():
        resolved = p.resolve()
        if root not in resolved.parents and resolved != root:
            self._logger.warning("model_path 绝对路径越界，已忽略: %s", model_path)
            p = Path(p.name)
    # 统一对解析结果做越界校验（覆盖相对 ../ 情形）
    candidate = (root / str(p)).resolve()
    if root not in candidate.parents and candidate != root:
        self._logger.warning("model_path 越界，已忽略: %s", model_path)
        return str(p)  # 或返回 "" 交由上层报错
    if candidate.exists():
        return str(candidate)
    under_models = (root / "models" / str(p)).resolve()
    if root not in under_models.parents and under_models != root:
        self._logger.warning("model_path 越界（models），已忽略: %s", model_path)
        return str(p)
    if under_models.exists():
        return str(under_models)
    return str(p)
```

## 2. 既往报告审计纠错（重点：第五十一批次 1530 报告）

### [A1 推翻] 1530 报告 `REC01`（High）「`am force-stop` 参数拼接错误导致强制停止完全无效」是**错误论断（假阳性）**

**1530 报告原论断**：`recovery.py:72` 将 `"am force-stop"` 作为单个参数传递，`adb shell` 经 mksh 执行时「将整个字符串解释为单一命令名（尝试执行名为 `am force-stop` 的命令）」，故强制停止**完全无效**，评级 High、列为 P0。

**逐行核验当前代码**（`recovery.py:66-79`，本轮未改动）：
```python
def _run(self, args: list[str], serial: Optional[str]) -> None:
    cmd = self._resolve_adb(serial) + args          # ["adb.exe"] + ["shell","am force-stop",pkg]
    subprocess.check_output(cmd, text=True, timeout=30)

def _force_stop(self, serial: Optional[str]) -> None:
    self._run(["shell", "am force-stop", self._package], serial)
```
最终进程参数为：`["adb.exe", "shell", "am force-stop", "com.hypergryph.endfield"]`。

**机理纠正（关键）**：
`adb shell` 在收到**多个** `shell` 之后的参数时，会将其**用空格拼接**成一条命令字符串再发给设备端 shell（mksh）。因此设备端实际执行的命令行是：
```
am force-stop com.hypergryph.endfield
```
mksh 将其解析为：命令 `am`，参数 `force-stop`，参数 `com.hypergryph.endfield` —— **这正是 `am force-stop` 的正确语法，强制停止实际生效**。

1530 报告的根本错误在于：它假设列表元素 `"am force-stop"` 会被「原样当成命令名传给 mksh」。这只有在 **单个** 参数 `adb shell "am force-stop com.hypergryph.endfield"` 时才会「命令名含空格」——而即便那种写法，adb 仍是把整串作为一条命令行发给 mksh，mksh 照样解析为 `am force-stop pkg`，**同样正确**。换言之，无论用两个列表元素还是单个元素，`am force-stop <pkg>` 都正确执行，**不存在 1530 描述的中断**。

**与既有审计的一致性**：本结论与批次 32（0659，A1）及批次 47（1017，推翻 D1/FINAL C01）的早期权威审计**完全一致**——二者早已确认 `_force_stop` 实际生效、注入面不可达。1530 报告在已有推翻结论后**重新将其列为 High 新发现**，属于重复计数 + 错误定级，会虚高最终风险台账。

**结论**：`REC01` 应**从风险台账移除**（或降为 Info 级「风格建议：可拆分为 `["shell","am","force-stop",pkg]` 提升可读性」）。原本针对它的「拆分三参数」修改**不改变任何运行行为**，纯风格调整。建议修订第五十一批次报告与 `FINAL.md`。
