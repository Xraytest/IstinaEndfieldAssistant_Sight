### 2.1 🔴 高危问题

#### J01 - `VlmWalkNavigator` 定位失败时伪造 (0,0) 坐标传递给 VLM

**文件**: `src/core/service/navigation/vlm_walk_navigator.py:161-170`
**风险等级**: 🔴 高危
**影响**: 小地图定位失败时，VLM 收到错误的 (0,0) 坐标作为"当前位置"，做出完全错误的导航决策

**问题代码**:
```python
current_pos = self._locator.locate(frame)
if current_pos is None:
    self._logger.warning("cannot locate position at step %d", step_idx)
    # Still send frame — VLM may orient without minimap
    current_context = MapPosition(
        map_id=map_name, level_id=level_id or "unknown",
        tile_class="unknown", grid_cell=None,
        center_x=0, center_y=0,           # ❌ 伪造 (0,0) 坐标
        confidence=0.0, raw_class_id=-1,
    )
```

**问题分析**:
- 当 `_locator.locate(frame)` 返回 `None` 时（小地图遮挡、画面模糊、角色静止等场景），代码构造了一个伪造的 `MapPosition`，其中 `center_x=0, center_y=0`
- 这个伪造坐标直接参与后续计算（第175-177行）:
  ```python
  dx = target_x - current_context.center_x  # target_x - 0 = target_x
  dy = target_y - current_context.center_y  # target_y - 0 = target_y
  dist = (dx * dx + dy * dy) ** 0.5       # 距离为目标点的欧几里得距离
  ```
- VLM 收到的上下文提示为: `Target Position: (500.0, 300.0) distance=583.1`，意味着角色被认为在地图原点而目标在远处
- VLM 将据此做出错误的导航决策——可能认为角色需要朝完全错误的方向移动
- 此外，伪造的 `confidence=0.0` 信号并未阻止后续流程——代码在定位失败后仍继续执行 VLM 查询和动作执行
- 与 `_is_stuck` 检测的交互也存在问题：连续步骤使用相同的伪造 (0,0) 坐标时，`spread` 为 0，可能触发误判的 "stuck" 状态

**修复建议**:
```python
current_pos = self._locator.locate(frame)
if current_pos is None:
    self._logger.warning("cannot locate position at step %d", step_idx)
    if self._config.fallback_to_navmesh:
        return {
            "status": "partial",
            "action": "vlm_walk",
            "steps_taken": step_idx,
            "message": "position unknown, fallback to navmesh required",
            "history": history,
        }
    # 使用上一帧位置（如果有）而非伪造坐标
    if history:
        last_pos = history[-1].get("position")
        if last_pos:
            current_context = last_pos
            current_context.confidence = 0.0
        else:
            # 无可参考位置，停止而非提供错误坐标
            return {
                "status": "partial",
                "action": "vlm_walk",
                "steps_taken": step_idx,
                "message": "position unknown, cannot proceed",
                "history": history,
            }
    else:
        return {
            "status": "error",
            "message": "no position available at start, cannot navigate",
        }
else:
    current_context = current_pos
```

### 2.2 🟡 中危问题

#### J02 - `PipelineRunner.run_pipeline` 重试时不重置命中计数

**文件**: `src/core/capability/element_recognition/pipeline/pipeline_runner.py:101-119`
**风险等级**: 🟡 中危
**影响**: Pipeline 重试时，上一轮的命中次数/限流状态残留，导致第二轮过早触发限流或命中上限

**问题代码**:
```python
def run_pipeline(self, screen, graph, entry, ...):
    result = self.run(screen, graph, entry, max_steps)
    retries = 0
    while result["status"] != "matched" and ... and retries < max_retries:
        time.sleep(retry_backoff_s)
        retries += 1
        result = self.run(screen, graph, entry, max_steps)  # ❌ 未重置 _hit_counts/_last_run
```

**问题分析**:
- `self.run()` 内部会调用 `self._hit_counts.clear()`（第51行），所以第一轮后 hit_counts 被清空
- 但 `_last_run` 不会被 `run()` 清空（只在 `reset()` 中清空）
- 重试时，第二轮循环中 `_is_rate_limited` 检查 `self._last_run`，仍包含第一轮的时间戳
- 如果第二轮在 rate_limit 间隔内执行，节点会被误判为"限流中"而跳过
- 例如: 节点 A 的 `rate_limit=5000ms`，第一轮在 T=0 执行，第二轮在 T=100ms 重试时 `(100-0)*1000 < 5000` → 节点 A 被跳过
- 这导致重试逻辑实际上只执行了部分节点，而非完整重跑

**修复建议**:
```python
def run_pipeline(self, screen, graph, entry, ...):
    result = self.run(screen, graph, entry, max_steps)
    retries = 0
    while result["status"] != "matched" and ... and retries < max_retries:
        time.sleep(retry_backoff_s)
        retries += 1
        self.reset()  # 重置命中计数和限流状态
        result = self.run(screen, graph, entry, max_steps)
```

#### J04 - `SceneGeometryAnalyzer._extract_candidates` saliency 累积乘法过度抑制

**文件**: `src/core/capability/element_recognition/backends/scene_geometry.py:266-270`
**风险等级**: 🟡 中危
**影响**: 画面边缘区域的候选目标被过度抑制，可能导致边缘实体丢失

**问题代码**:
```python
saliency = saliency_map.copy()
saliency[: int(h_img * 0.12), :] *= 0.45    # 顶部12% × 0.45
saliency[: int(h_img * 0.20), :] *= 0.55    # 顶部20% × 0.55（与上行有重叠区域）
saliency[:, : max(1, int(w_img * 0.04))] *= 0.70  # 左侧4% × 0.70
saliency[:, max(0, w_img - int(w_img * 0.04)) :] *= 0.70  # 右侧4% × 0.70
```

**问题分析**:
- 第267行和第268行有区域重叠（`h_img * 0.12` 区域同时被两次乘法处理）
- 重叠区域的实际抑制系数为 `0.45 × 0.55 = 0.2475`（约75%抑制），而非设计意图中的0.45
- 左上角区域的累积抑制为 `0.45 × 0.55 × 0.70 ≈ 0.173`（约83%抑制）
- 这导致画面四个角的候选目标显著性被压至极低值
- 阈值计算公式: `max(np.percentile(saliency, 89), saliency.mean() + saliency.std() * 0.8)`
- 过度抑制会导致边缘的实体（如角色侧翼、远处建筑）无法通过阈值，被遗漏
- 设计意图可能是"分别独立抑制"，但代码实现是"累积抑制"

**修复建议**:
```python
saliency = saliency_map.copy()
# 使用最大抑制而非累积乘法，避免重叠区域过度抑制
top_region = saliency[: int(h_img * 0.20), :]
top_region[:] = top_region * 0.45  # 单次应用最强抑制

edge_margin = max(1, int(w_img * 0.04))
saliency[:, :edge_margin] *= 0.70
saliency[:, w_img - edge_margin:] *= 0.70
```

#### J06 - `EntityDatabase.find_by_name` 正则回溯风险

**文件**: `src/core/service/navigation/entity_db.py:129-136`
**风险等级**: 🟢 低危
**影响**: 长名称查询可能导致 ReDoS（正则拒绝服务）

**问题代码**:
```python
def find_by_name(self, name: str, exact: bool = False, limit: int = 50) -> List[Entity]:
    ...
    pattern = re.compile(re.escape(name), re.IGNORECASE)
    # ❌ 对 _by_name 的每个 key 执行正则搜索，无超时保护
    results = []
    for key, entities in self._by_name.items():
        if pattern.search(key):
            results.extend(entities)
```

**问题分析**:
- `re.escape(name)` 本身是安全的（转义所有元字符），不会产生回溯
- 但 `re.IGNORECASE` 在 Unicode 场景下可能产生额外的匹配分支
- 实际风险较低：`re.escape` 后正则变为纯字符串匹配，回溯风险极低
- 真正的性能问题是 O(n×m) 复杂度（n=实体数, m=名称长度），对大型数据库可能变慢
- 标注为低危因为 `re.escape` 消除了经典 ReDoS，但代码意图上仍有改进空间

**修复建议**:
```python
def find_by_name(self, name: str, exact: bool = False, limit: int = 50) -> List[Entity]:
    self._ensure_loaded()
    if exact:
        return list(self._by_name.get(name, []))[:limit]
    name_lower = name.lower()
    results = []
    for key, entities in self._by_name.items():
        if name_lower in key.lower():  # 用 in 替代正则，消除回溯风险
            results.extend(entities)
            if len(results) >= limit:
                break
    return results[:limit]
```

#### J07 - `PipelineLoader.extract_module_nodes` 共享 PipelineNode 引用

**文件**: `src/core/capability/element_recognition/pipeline/pipeline_loader.py:61-68`
**风险等级**: 🟢 低危
**影响**: 提取的子图与原始图共享节点对象，修改子图节点影响原始图

**问题代码**:
```python
def extract_module_nodes(self, graph, module_prefix):
    sub = PipelineGraph()
    for name, node in graph.nodes.items():
        if name.startswith(module_prefix):
            sub.add_node(node)  # ❌ 共享同一个 PipelineNode 对象引用
    return sub
```

**问题分析**:
- `sub.add_node(node)` 将原始 `PipelineNode` 对象的引用直接存入子图
- 如果后续修改子图中的节点（如修改 `action`、`threshold`），会同步修改原始图中的同名节点
- `PipelineNode` 是 `@dataclass`，字段可变，任何修改都会影响所有引用方
- `extract_module_nodes` 目前可能仅用于只读查询，但 API 没有保护措施
- 对比: `PipelineGraph.merge()` 同样使用 `self.nodes.update(other.nodes)` 合并，也共享引用

**修复建议**:
```python
from copy import deepcopy

def extract_module_nodes(self, graph, module_prefix):
    sub = PipelineGraph()
    for name, node in graph.nodes.items():
        if name.startswith(module_prefix):
            sub.add_node(deepcopy(node))  # 深拷贝避免引用共享
    return sub
```

#### J08 - `YOLOBackend._is_available` 只尝试一次导入，失败后永不恢复

**文件**: `src/core/capability/element_recognition/backends/yolo_backend.py:102-112`
**风险等级**: 🟢 低危
**影响**: ultralytics 包后安装/升级后无法自动恢复

**问题代码**:
```python
def _is_available(self):
    if self._model is not None:
        return True
    try:
        from ultralytics import YOLO
        self._model = YOLO(self._model_path)
        return True
    except Exception as e:
        logger.debug(f"YOLO not available: {e}")
        return False  # ❌ 第一次失败后 _model 保持 None，后续调用永不重试
```

**问题分析**:
- 第一次导入失败后，`self._model` 保持 `None`
- 后续调用 `_is_available` 时，`self._model is not None` 为 False，重新尝试导入
- 等等——实际上 `_is_available` 每次调用都会重新尝试导入（因为 `self._model is None` 会进入 try 块）
- 所以这个标记为低危可能不准确... 让我再看一次
- 实际上如果 `from ultralytics import YOLO` 成功但 `YOLO(self._model_path)` 失败（模型文件不存在），则 `self._model` 保持 None，但后续调用会重新尝试。这是正确行为。
- 但如果 `from ultralytics import YOLO` 失败（包未安装），每次调用都会尝试重新导入，这是浪费但无害
- **结论**: 此代码实际行为正确，不构成 bug。从报告中移除此发现。
