## 1. 代码错误（功能失效类）

### [TST-02] 两个文件测试同一 QueueState 类，覆盖率严重重叠且执行翻倍

**优先级：Medium**
**位置**：`tests/test_queue_state.py` vs `tests/gui/pyqt6/test_queue_state.py`

**根因**：
两个文件均 `from gui.pyqt6.queue_state import QueueState`，测试同一类。

重叠矩阵：

| `tests/test_queue_state.py` | `tests/gui/pyqt6/test_queue_state.py` |
|---|---|
| test_empty_initialization | 被 test_full_lifecycle setup 覆盖 |
| test_set_queue_items | 被 test_full_lifecycle 覆盖 |
| test_save_and_load_options | 被 test_round_trip_preserves_options 覆盖 |
| test_selected_task_and_preset_persistence | 被 test_full_lifecycle 覆盖 |
| test_clear_queue | 两文件均有（重复） |
| test_clear_saved_options | 两文件均有（重复） |
| test_update_queue_item_options | 被 mutations 覆盖 |
| test_full_roundtrip_persistence | 被 test_full_lifecycle 覆盖 |

根目录文件的 3 个独有测试：
- `test_persist_creates_parent_dirs`（125-129 行）
- `test_persist_failure_returns_false`（170-176 行，使用 `monkeypatch.setattr(Path.mkdir)`）
- `test_set_queue_items_shallow_copies`（178-185 行）

gui/pyqt6 文件的独有测试：
- TestQueueStatePerInstance（3 项：重复任务选项隔离、round-trip 保持隔离、共享快照不泄漏）
- TestQueueStateMutations（5 项：clear_queue、clear_saved_options、clear_saved_options_nonexistent、load_options_returns_copy、selected_task_preset_persistence）
- test_illegal_index_returns_none、test_update_queue_item_out_of_range、test_corrupted_json_is_ignored、test_missing_file_is_ignored、test_empty_queue_items_skipped_on_load、test_non_dict_queue_entries_skipped、test_load_options_returns_copy、test_selected_task_preserved_when_missing_in_file、test_empty_options_dict

**影响面**：
- `pyproject.toml` 设置 `testpaths = ["tests"]`，pytest 同时发现两个文件，TestQueueStateLifecycle 被执行两次，测试时间翻倍。
- 若只更新一个文件，覆盖率会悄然分化。
- `test_persist_failure_returns_false` 使用 `monkeypatch.setattr(Path.mkdir)`，若根目录测试先于 gui/pyqt6 测试执行，Path.mkdir 仍被 patch，造成污染。

**修改建议**：
1. 保留 `tests/gui/pyqt6/test_queue_state.py` 为唯一权威测试（覆盖更全）。
2. 将根目录文件的 3 个独有测试迁移到 gui/pyqt6 版本。
3. 删除 `tests/test_queue_state.py`。

## 2. 测试基础设施问题

### [TST-01] 两个测试文件使用 autouse fixture 全局 monkeypatch `logging.Logger`，测试内无法验证日志行为

**优先级：Medium（非 High，影响范围限于两个文件）**
**位置**：`tests/test_istina_runtime.py:15-18`、`tests/test_error_paths.py:17-20`

**根因**：
```python
@pytest.fixture(autouse=True)
def _disable_broken_project_logging() -> None:
    for method in ("debug", "info", "warning", "error", "critical", "log", "exception"):
        setattr(logging.Logger, method, lambda self, *args, **kwargs: None)
```

两个测试文件各自定义了 `autouse=True` 的 fixture，在模块级别将 `logging.Logger` 的所有日志方法替换为无操作 lambda。影响仅限于这两个文件内的测试（pytest fixture 作用域为定义它的模块），**非全局影响**——`conftest.py` 中不存在此 fixture，`test_gui_scripting.py` 的 caplog 正常生效。

但在这两个文件内部：
- 任何 future caplog 测试会静默失效（`caplog.records` 永远为空）。
- 无法通过日志验证代码行为——生产代码中的 `logger.warning("xxx")` 被静默吞掉。

**影响面**：
- 当前两文件均无 caplog 测试，因此**当前无运行时影响**。
- 但 fixture 名称 `_disable_broken_project_logging` 暗示"已知 workaround 未恢复"——说明项目历史上存在 logging 初始化问题（可能是 `init_logger()` 未在测试环境正确配置 handler 导致重复输出），该问题可能已解决，但 fixture 未清理。

**修改建议**：
1. 先确认原始问题是否仍存在：移除 fixture 后运行两个文件的测试，观察是否有 logging 相关失败。
2. 若问题已解决，删除 fixture。
3. 若问题仍存在，改用更精确的修复：在 `init_logger()` 的测试路径中配置 NullHandler，或在该 fixture 内仅 patch 项目自定义 logger（通过 name 前缀过滤），而非整个 `logging.Logger` 类。
4. 若确需保留，将 fixture 从 `autouse` 改为显式参数，避免 future 测试编写者误以为 caplog 可用。

> **注意**：batch 12 agent 将此发现标为 High 并声称影响全测试套件。经源码验证，该结论不准确——fixture 为模块级 autouse，仅影响各自文件。但测试维护隐患本身（无法在文件内验证日志）确实存在。

### [SCR-01] `scripts/debug/` 下 46 个脚本含硬编码开发者特定路径与乱码 docstring

**优先级：Low**
**位置**：`scripts/debug/*.py`（46 个文件）

**根因**：
- 38+ 文件含 `C:\Users\xray\Documents\ArkStudio\IstinaAI\IstinaEndfieldAssistant\cache\...`（另一开发者的绝对路径）
- 至少 1 文件含 `c:\Users\cheng\...\config\endfield_world_model.json`（当前开发者绝对路径）

- `.gitignore` 第 32 行排除 `debug/` 目录，这些路径从未进入版本控制。
- 文件不参与生产构建或分发。

**影响面**：无生产影响。但开发者复制这些脚本到 debug/ 外部时会遇到 FileNotFoundError。

**修改建议**：
1. 在 `scripts/debug/` 下添加 README 说明这些是一次性调试脚本、路径硬编码、不可直接复用。
2. 对可复用的脚本（如 verify_llm.py、verify_ocr_integration.py、check_llm_cuda.py）将硬编码路径改为项目相对路径。
3. 统一使用 `ensure_src_path()` 初始化 sys.path。
4. 修复乱码 docstring（保存为 UTF-8）。
