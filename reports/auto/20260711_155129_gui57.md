### [GUI57-01 Medium] `scene_service.py:104-112` — `analyze_elements` 硬编码 `"yolo": False`

```python
# scene_service.py:97-112
def analyze_elements(
    self,
    screen: np.ndarray,
    enable_template: bool = True,
    enable_ocr: bool = True,
    enable_color: bool = True,
) -> PageInfo:
    return self.recognizer.recognize(
        screen,
        enable={
            "template": enable_template,
            "ocr": enable_ocr,
            "color": enable_color,
            "yolo": False,       # ← 始终关闭
        },
    )
```

**问题**：`SceneUnderstandingService.__init__` 接受 `enable_yolo` 参数并传递给 `EndfieldElementRecognizer`，但 `analyze_elements` 在 `enable` 字典中将 `"yolo"` **硬编码为 `False`**。这意味着：

- 即使构造时 `enable_yolo=True`，调用 `analyze_elements` 时 YOLO 检测永远不会被启用
- `analyze_elements` 是高层 API，调用方无法感知此隐藏开关
- 与批次 54 SCENE-02 报告的问题一致，但该报告定位在 `SceneUnderstandingService` 层面未识别 `analyze_elements` 的具体硬编码行

**影响面**：YOLO 检测能力在 `analyze_elements` 调用路径中完全不可用。

**建议**：将 `enable_yolo` 参数暴露给 `analyze_elements`：

```python
def analyze_elements(
    self,
    screen: np.ndarray,
    enable_template: bool = True,
    enable_ocr: bool = True,
    enable_color: bool = True,
    enable_yolo: bool = False,
) -> PageInfo:
    return self.recognizer.recognize(
        screen,
        enable={
            "template": enable_template,
            "ocr": enable_ocr,
            "color": enable_color,
            "yolo": enable_yolo,
        },
    )
```

### [GUI57-02 Medium] `recorder.py:176-182` — 防抖 timer 闭包捕获循环变量导致数据竞争

```python
# recorder.py:176-182
if obj_name not in self._text_timers:
    timer = QTimer(self)
    timer.setSingleShot(True)
    timer.timeout.connect(lambda: self._emit_text_action(obj_name, widget_type, action_type, text))
    self._text_timers[obj_name] = timer
self._pending_text[obj_name] = text
self._text_timers[obj_name].start(800)
```

**问题**：lambda 闭包捕获的 `text` 是局部变量引用，而非值快照。`_record_text_change` 在同一事件处理中可能被同一 widget 连续调用（如组合框展开选择时 `currentText` 多次变更）。每次调用都覆写 `text` 局部变量，但所有 timer 的 lambda 共享同一变量引用。当 timer 触发时，`text` 可能已被后续调用覆写为不同值。

**场景重现**：
1. 用户快速输入 "ABC"（3 次 textChanged 事件）
2. 每次创建 timer（前 2 个被第 3 个覆写，但 `start(800)` 重置计时器）
3. 800ms 后 timer 触发，`text` 已为最后一次输入的值
4. 但由于 `_emit_text_action` 中的去重检查，如果最终值 == 初始值则不会产生多余记录
5. **真正的风险**：用户输入 "AB" → 暂停 → 删除 → 输入 "CD"，timer 触发时 `text` 为 "CD"，记录了用户放弃的值

**建议**：使用 `functools.partial` 或 lambda 默认参数捕获当前值：

```python
from functools import partial
timer.timeout.connect(partial(self._emit_text_action, obj_name, widget_type, action_type, text))
```

或使用 lambda 默认参数：
```python
timer.timeout.connect(lambda ot=obj_name, wt=widget_type, at=action_type, t=text:
    self._emit_text_action(ot, wt, at, t))
```

### [GUI57-03 Medium] `task_loader.py:31-39` — `load_task` 吞掉异常返回 `None`，调用方无法区分"不存在"与"加载失败"

```python
# task_loader.py:31-39
def load_task(self, task_name: str) -> Optional[Dict[str, Any]]:
    candidates = [...]
    for path in candidates:
        if path.is_file():
            try:
                with open(path, "r", encoding="utf-8") as f:
                    data = json.load(f)
                return data
            except Exception as e:
                logger.error(f"Failed to load task '{task_name}' from {path}: {e}")
                return None  # ← 立即返回，不尝试其他候选路径
    logger.debug(f"Task not found: {task_name}")
    return None
```

**问题**：
1. **候选路径短路**：第一个存在的文件（即使损坏）导致立即 `return None`，后续候选路径永远不会被尝试
2. **无区分度**：调用方收到 `None` 后无法知道是"文件不存在"还是"JSON 解析失败"，无法提供有意义的错误信息
3. **`load_all_tasks` 同样问题**：line 74 `logger.debug` 吞掉错误后静默跳过

**影响面**：任务文件损坏时，系统静默跳过而非报告错误，导致任务列表不完整。

**建议**：
```python
def load_task(self, task_name: str) -> Optional[Dict[str, Any]]:
    last_error = None
    for path in candidates:
        if not path.is_file():
            continue
        try:
            with open(path, "r", encoding="utf-8") as f:
                data = json.load(f)
            return data
        except Exception as e:
            last_error = e
            logger.debug(f"Failed to load task '{task_name}' from {path}: {e}")
    if last_error:
        logger.warning(f"Task '{task_name}' not loadable: {last_error}")
    else:
        logger.debug(f"Task not found: {task_name}")
    return None
```

### [GUI57-04 Low] `pipeline_node.py:69-80` — `from_dict` action 类型未经验证直接存入

```python
# pipeline_node.py:69-80
if isinstance(action_raw, str):
    action = action_raw
    if action_raw == "Custom":
        custom_action = data.get("custom_action")
        custom_action_param = data.get("custom_action_param")
elif isinstance(action_raw, dict):
    action = action_raw.get("type", "DoNothing")
    if action == "Custom":
        custom_action = action_raw.get("param", {}).get("custom_action") if isinstance(action_raw.get("param"), dict) else None
        custom_action_param = action_raw.get("param", {}).get("custom_action_param") if isinstance(action_raw.get("param"), dict) else None
    elif action in ("Swipe",):
        custom_action_param = action_raw.get("param")
```

**问题**：
1. **字符串 action 无验证**：`action_raw` 为字符串时，直接赋值给 `action`，不检查是否为 `NodeAction` 的有效值。JSON 中的笔误（如 `"Clik"` 而非 `"Click"`）会被静默接受，pipeline 运行时 `_evaluate` 不处理未知 action 类型

**建议**：添加 action 类型校验，将未知类型回退到 `DoNothing` 或记录警告。

### [GUI57-05 Low] `player.py:94-101` — `_schedule_next` 在停止/完成时不触发完成信号

```python
# player.py:92-101
def _schedule_next(self, delay_ms: int) -> None:
    if self._stopped:
        self._on_finished()    # ← 会触发 playback_finished
        return
    if self._paused:
        return                # ← 静默返回，不触发任何信号
    if self._current_index >= len(self._script.actions):
        self._on_finished()   # ← 会触发 playback_finished
        return
```

**问题**：`_paused=True` 时 `_schedule_next` 静默返回。如果用户在暂停状态下调用 `stop()` → `play()`，`_schedule_next(self._default_delay_ms)` 被调用，此时 `_paused` 在 `play()` 中被设为 `False`，所以不会触发此分支。

但如果在 `_execute_action` 执行期间（timer 触发后、下一轮 `_schedule_next` 前）用户调用 `pause()`，然后调用 `stop()`，`stop()` 停止 timer 并设置 `_stopped=True`。下一次 `_schedule_next` 不会被调用。但 `playback_stopped` 信号在 `stop()` 中已发射，所以这是正确的。

实际风险场景：`_schedule_next` 在 timer 回调中被调用，如果 `_current_index` 恰好在 `len(self._script.actions)` 时（最后一条动作执行后），`_on_finished()` 被调用。但如果在 `_execute_action` 执行过程中 `_script` 被置为 `None`（如 `load(None)`），`_current_index` 未重置，`_schedule_next` 在下一轮调用时 `_current_index >= len(self._script.actions)` 会抛出 `TypeError`。

**建议**：在 `_schedule_next` 中添加 `_script` 空值检查。

### [GUI57-06 Info] `tray_icon.py:71-73` — 仅响应双击恢复窗口，单点击无动作

```python
# tray_icon.py:71-73
def _on_tray_activated(self, reason: QSystemTrayIcon.ActivationReason) -> None:
    if reason == QSystemTrayIcon.ActivationReason.DoubleClick:
        self._show_window()
```

**问题**：托盘图标激活仅响应 `DoubleClick`。多数平台（Windows）用户期望**单击**托盘图标即可恢复/最小化窗口。双击要求增加了操作成本，且 Windows 系统托盘的双击行为通常被解释为"打开属性"而非"恢复应用"。

**建议**：同时响应 `Trigger`（单击）和 `DoubleClick`。
