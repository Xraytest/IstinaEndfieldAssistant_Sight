#### S01 - `scripts/verify_llm.py` 绕过 `ensure_src_path()`，使用手动 `sys.path.insert`

**文件**: `scripts/verify_llm.py:12`
**风险等级**: 🟡 中危
**影响**: 违反项目路径管理规范，路径解析不一致

**问题代码**:
```python
# scripts/verify_llm.py:12
sys.path.insert(0, str(Path(__file__).resolve().parent.parent / "src"))

# scripts/verify_llm_simple.py:12 — 正确使用
from core.foundation.paths import ensure_src_path
ensure_src_path()

# scripts/check_llm_cuda.py:10 — 与 verify_llm.py 相同
sys.path.insert(0, str(Path(__file__).resolve().parent.parent / "src"))
```

**分析**:
- 项目规范（CLAUDE.md）明确要求使用 `core.foundation.paths` 模块进行路径解析
- `verify_llm.py` 和 `check_llm_cuda.py` 使用手动 `sys.path.insert`，而 `verify_llm_simple.py` 正确使用 `ensure_src_path()`
- 同一目录下的脚本路径管理方式不一致，维护者可能复制模式但不了解差异
- 手动路径插入在路径变更时需要同步修改三处（verify_llm.py, check_llm_cuda.py, verify_ocr_integration.py）
- `ensure_src_path()` 还处理了 `PYTHONPATH` 初始化和路径规范化，手动插入缺少这些保护

**修复建议**:
```python
# 统一使用 ensure_src_path()
from core.foundation.paths import ensure_src_path
ensure_src_path()
# 移除手动 sys.path.insert
```

#### QF01 - `qt_log_filter.py` 安装失败后 `_INSTALLED = True`，阻止后续重试

**文件**: `src/gui/pyqt6/qt_log_filter.py:64-66, 83-85`
**风险等级**: 🟡 中危
**影响**: 首次安装失败后，后续所有调用直接跳过，Qt 日志无法被接管

**问题代码**:
```python
def install_qt_message_filter() -> None:
    global _INSTALLED
    if _INSTALLED:
        return  # ← 幂等保护，但连失败状态也保护了

    project_root = Path(__file__).resolve().parent.parent.parent.parent
    log_dir = project_root / "logs"
    try:
        log_dir.mkdir(parents=True, exist_ok=True)
    except OSError:
        _INSTALLED = True  # ← ❌ 标记为"已安装"，但实际未安装
        return  # 后续调用全部跳过
    ...
    try:
        handler = logging.FileHandler(log_dir / "qt.log", encoding="utf-8")
        ...
    except OSError:
        _INSTALLED = True  # ← ❌ 同上
        return
```

**分析**:
- 幂等保护（`_INSTALLED` 标志）设计正确——防止重复安装
- 但在失败路径上，`_INSTALLED` 被设置为 `True`，导致后续调用直接 `return`
- 如果 `logs/` 目录在首次启动时不存在（`mkdir` 失败），Qt 日志过滤器永久失效
- 如果 `qt.log` 文件不可写（权限问题），过滤器同样永久失效
- 后续 GUI 重启不会重试——Qt 字体警告噪声将持续污染 stderr
- 对比：成功路径中 `_INSTALLED = True` 是正确的（实际已安装）

**修复建议**:
```python
_INSTALLED = False
_INSTALL_FAILED = False  # ← 新增：区分"未安装"和"安装失败"

def install_qt_message_filter() -> None:
    global _INSTALLED, _INSTALL_FAILED
    if _INSTALLED:
        return
    if _INSTALL_FAILED:
        return  # 已知失败，不重复尝试（可选：可以改为重试）
    ...
    except OSError:
        _INSTALL_FAILED = True  # ← 标记为"安装失败"，允许后续重试
        return
```

#### R01 - `responsive.py` DPI 阈值顺序错误，150% 缩放被当作 125%

**文件**: `src/gui/pyqt6/responsive.py:23-42`
**风险等级**: 🟡 中危
**影响**: 150% DPI 缩放下字体大小使用 125% 缩放系数，UI 元素偏小

**问题代码**:
```python
_DPI_THRESHOLDS = [
    (144.0, 1.5),   # 150%
    (120.0, 1.25),  # 125%
    (96.0, 1.0),    # 100%
]

def get_dpi_scale(widget: QWidget) -> float:
    dpi = screen.logicalDotsPerInch()
    for threshold, scale in _DPI_THRESHOLDS:
        if dpi >= threshold:
            return scale
    return 1.0
```

**分析**:
- 逻辑：`if dpi >= threshold: return scale`——从高阈值开始遍历
- 144 DPI（150%缩放）时：`144 >= 144` → 返回 `1.5` ✓ **正确**
- 120 DPI（125%缩放）时：`120 < 144`，继续；`120 >= 120` → 返回 `1.25` ✓ **正确**
- 实际上...这个逻辑是正确的！

**等等，让我再检查一遍**:
- 阈值列表：`[(144.0, 1.5), (120.0, 1.25), (96.0, 1.0)]`
- 函数：`for threshold, scale in _DPI_THRESHOLDS: if dpi >= threshold: return scale`
- 144 DPI：144 >= 144 → True → return 1.5 ✓
- 120 DPI：120 >= 144 → False → 120 >= 120 → True → return 1.25 ✓
- 96 DPI：96 >= 144 → False → 96 >= 120 → False → 96 >= 96 → True → return 1.0 ✓
- 72 DPI（75%）：72 >= 144 → False → 72 >= 120 → False → 72 >= 96 → False → return 1.0

**结论**: 逻辑是正确的。但是有一个边界问题——96 DPI 是 100% 缩放的参考值，低于此值时返回 1.0，这是合理的。

**等等，再仔细看**: 阈值列表是从高到低排列的（144→120→96），匹配逻辑是 `>=`。这意味着：
- 当 dpi 为 144 时，匹配第一个阈值 → 1.5 ✓
- 当 dpi 为 120 时，不匹配第一个，匹配第二个 → 1.25 ✓
- 当 dpi 为 96 时，不匹配前两个，匹配第三个 → 1.0 ✓

**逻辑完全正确**。

---

**经过仔细分析，`responsive.py` 的 DPI 阈值逻辑是正确的**。高阈值优先匹配的设计是正确的——当 DPI 为 144 时，144 >= 144 为 True，返回 1.5。低阈值作为 fallback 也不会被高 DPI 错误匹配。

**因此，R01 不成立**——代码逻辑正确，无需修复。

#### S03 - `check_llm_cuda.py` 使用 `subprocess.check_output` + `shell=True`

**文件**: `scripts/check_llm_cuda.py:20`
**风险等级**: 🟢 低危
**影响**: shell 注入风险（本地脚本场景下实际风险极低）

**问题代码**:
```python
out = subprocess.check_output(
    ["nvidia-smi", "--query-gpu=...", "--format=csv,noheader,nounits"],
    stderr=subprocess.STDOUT, text=True, shell=True
)
```

**分析**:
- `shell=True` 在 Windows 上允许 shell 元字符解析
- 参数列表中所有值都是硬编码字符串，无用户输入
- 但 `subprocess.check_output` 在 `shell=True` 时，如果第一个参数是列表，Python 会用 `/c` + 列表[0] 执行，其他参数被忽略——这可能导致意外的行为
- 正确的做法是传入字符串：`subprocess.check_output("nvidia-smi ...", shell=True)` 或移除 `shell=True`

**修复建议**:
```python
# 方案1: 移除 shell=True（推荐）
out = subprocess.check_output(
    ["nvidia-smi", "--query-gpu=index,name,memory.used,memory.total,utilization.gpu",
     "--format=csv,noheader,nounits"],
    stderr=subprocess.STDOUT, text=True
)

# 方案2: 使用字符串参数
out = subprocess.check_output(
    "nvidia-smi --query-gpu=index,name,memory.used,memory.total,utilization.gpu --format=csv,noheader,nounits",
    shell=True, text=True, stderr=subprocess.STDOUT
)
```

#### S06 - `scripts/verify_llm.py` 中 `measure_completion` 的 `subprocess.Popen` 无超时保护

**文件**: `scripts/verify_llm.py:84-148`
**风险等级**: 信息
**影响**: llama-server 启动失败或推理卡住时，进程可能长时间运行

**问题代码**:
```python
proc = subprocess.Popen(args, ...)
try:
    for _ in range(240):  # 240 秒健康检查
        if proc.poll() is not None:
            ...
    # 推理请求超时 120 秒
    with urllib.request.urlopen(req, timeout=120) as resp:
        ...
finally:
    proc.terminate()
    proc.wait(timeout=5)
```

**分析**:
- 健康检查循环有上限（240次 × 1秒 = 240秒）
- 推理请求有超时（120秒）
- `finally` 块正确执行 `terminate()` + `wait()`
- 但如果 `urllib.request.urlopen` 超时（120秒），进程仍在运行，`finally` 会终止它
- 实际保护机制是完整的，仅缺少对 `proc.wait()` 本身的超时保护（如果进程忽略 SIGTERM）
