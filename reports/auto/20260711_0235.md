## 1. 配置加载安全

### [SEC-01] `_write_or_base64` + `--config`/`--out` 路径未验证，可任意写入/覆盖文件

**优先级：Medium**
**位置**：`src/cli/handlers.py:30-38`（`_write_or_base64`）+ `src/cli/handlers.py:513-515`（`_handle_config_set`）+ `src/cli/istina.py:42-46`（`--config` 参数）

```python
# _write_or_base64: --out 路径未验证
def _write_or_base64(data: bytes, out_path: Optional[str]) -> Dict[str, Any]:
    out = Path(out_path) if out_path else None
    if out:
        out.parent.mkdir(parents=True, exist_ok=True)  # 创建任意目录
        out.write_bytes(data)  # 写入任意路径

# _handle_config_set: 写回 --config 指定的路径
def _handle_config_set(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    runtime.config[args.key] = args.value
    runtime.save_config()  # 写回 _resolve_config_path() 返回的路径
```

**根因分析**：
三条路径均接受用户提供的任意路径字符串，无任何验证：

1. **`--out` 路径**（用于 `screenshot`、`scene capture` 等命令）：
   - `_write_or_base64` 接受任意 `out_path`，创建父目录并写入 bytes
   - 可写绝对路径（如 `--out C:\Windows\System32\malicious.dll`）
   - 可写相对路径（如 `--out ../../etc/cron.d/malicious`）
   - 无路径白名单、无扩展名校验、无写入前确认

2. **`--config` 路径**（用于 `istina --config <path>`）：
   - `_resolve_config_path` 直接返回用户提供的路径
   - `_load_config` 读取该路径的 JSON
   - `_handle_config_set` 调用 `save_config` 将修改后的配置**写回同一路径**
   - 若用户传入 `--config /etc/passwd`，`config set` 命令将覆盖 `/etc/passwd`

3. **`_load_config` 的 bare except**：
   ```python
   try:
       with open(path, "r", encoding="utf-8") as f:
           return json.load(f)
   except Exception as e:
       self._logger.warning(...)
   return {}
   ```
   路径不存在、权限不足、JSON 格式错误全部静默返回 `{}`，用户无法诊断配置加载失败的原因。

**影响面**：
- **实际风险低**：IEA 是本地 GUI 工具，用户已拥有本地文件系统权限，任意写入的威胁模型等价于用户已具备的权限。
- **但存在逻辑风险**：用户可能因拼写错误传入错误的 `--config` 路径，配置静默加载失败后以默认值运行，导致设备地址、LLM 参数等关键设置丢失。
- **用户体验风险**：`--out` 写入错误路径时无任何确认或警告。

**修改建议**：
1. `_write_or_base64` 增加路径验证：
   ```python
   def _write_or_base64(data: bytes, out_path: Optional[str]) -> Dict[str, Any]:
       if out_path:
           out = Path(out_path).resolve()
           # 禁止写入项目根目录之外
           project_root = get_project_root().resolve()
           if not str(out).startswith(str(project_root)):
               return {"status": "error", "message": f"output path must be within project directory: {out}"}
           out.parent.mkdir(parents=True, exist_ok=True)
           out.write_bytes(data)
           return {"status": "success", "path": str(out), "size": len(data)}
   ```
2. `--config` 路径增加校验：
   ```python
   def _resolve_config_path(self) -> Path:
       path = self._config_path or get_project_root() / "config" / "client_config.json"
       # 禁止指向项目根目录之外的 config
       project_root = get_project_root().resolve()
       try:
           path.resolve().relative_to(project_root)
       except ValueError:
           raise ValueError(f"config path must be within project directory: {path}")
       return path
   ```
3. `_load_config` 区分错误类型：
   ```python
   except FileNotFoundError:
       self._logger.info("配置文件不存在，使用默认值")
       return {}
   except json.JSONDecodeError as e:
       self._logger.error("配置文件 JSON 格式错误", error=str(e))
       return {}
   except PermissionError as e:
       self._logger.error("配置文件无读取权限", error=str(e))
       return {}
   except Exception as e:
       self._logger.error("加载配置时未知错误", error=str(e), type=type(e).__name__)
       return {}
   ```

### [CFG-15] `_load_config` 使用 bare `except Exception`  swallows 所有错误类型

**优先级：Medium**
**位置**：`src/core/service/runtime.py:447-455`

```python
def _load_config(self) -> Dict[str, Any]:
    path = self._resolve_config_path()
    if path.exists():
        try:
            with open(path, "r", encoding="utf-8") as f:
                return json.load(f)
        except Exception as e:  # ← 捕获所有 Exception 子类
            self._logger.warning(LogCategory.MAIN, "加载配置失败，使用默认值", error=str(e))
    return {}
```

**根因分析**：
`except Exception` 捕获以下子类：
- `json.JSONDecodeError` — 配置文件 JSON 格式错误（预期内）
- `PermissionError` — 配置文件无读取权限（预期内）
- `FileNotFoundError` — 文件在 `exists()` 检查后被删除（边缘情况）
- `MemoryError` — 内存不足（不应被捕获，应向上传播）
- `KeyboardInterrupt` — 不继承 Exception，不受影响
- `SystemExit` — 不继承 Exception，不受影响

虽然 `MemoryError` 在配置加载场景中极罕见，但 bare except 的模式本身是反模式——它掩盖了诊断信息，使调试配置问题变得困难。

**修改建议**：如 SEC-01 所述，拆分 `except` 为具体异常类型。

## 2. 依赖声明问题

### [CFG-17] `requirements.txt` 同时声明 `ultralytics>=8.4` 与独立 `torch`/`torchvision`，存在版本冲突风险

**优先级：Low**
**位置**：`requirements.txt`

```
ultralytics>=8.4
torch>=2.12
torchvision>=0.27
onnxruntime>=1.27
```

**根因分析**：
`ultralytics` 包将 `torch` 和 `torchvision` 作为其核心依赖（通过 `install_requires` 声明）。同时显式声明 `torch>=2.12` 和 `torchvision>=0.27` 可能导致：

1. **版本冲突**：`ultralytics` 可能要求 `torch>=2.0,<2.5`（具体范围取决于版本），与项目要求的 `torch>=2.12` 重叠但不完全一致。pip  resolver 可能选择满足所有约束的版本，但也可能产生冲突。
2. **冗余安装**：如果 `ultralytics` 的 torch 版本约束与项目约束兼容，pip 只安装一次 torch；若不兼容，pip 可能拒绝安装或安装两个版本。
3. **未使用的依赖**：项目使用 `onnxruntime` 进行 YOLO 推理（`yolo_backend.py`），**不使用 ultralytics 的训练或推理接口**。`ultralytics` 的引入可能仅用于某个辅助脚本或已被遗忘。

**影响面**：
- `pip install -r requirements.txt` 在新环境上可能因版本冲突失败。
- `ultralytics` 包体积约 500MB+（包含 torch 模型权重），大幅增加安装时间和磁盘占用。

**修改建议**：
1. 确认 `ultralytics` 是否被实际使用：
   ```bash
   grep -rn "ultralytics\|from ultralytics\|import ultralytics" src/ scripts/
   ```
2. 若未使用，从 `requirements.txt` 中移除 `ultralytics>=8.4`。
3. 若仅用于某个脚本，将该脚本的依赖移至独立 `requirements-dev.txt` 或脚本级注释。

## 3. 配置文件内容问题

### [CFG-16] `client_config.json`（本地）包含硬编码设备 IP 和 LLM 路径，无模板文件供参考

**优先级：Low**
**位置**：`config/client_config.json`（gitignored 本地文件）

```json
"device": {
    "serial": "192.168.1.12:16512",
    "last_connected": "192.168.1.12:16512"
},
"llm": {
    "model_path": "/models/test.gguf",
    "mmproj_path": "models/LLM/mmproj-F16.gguf"
}
```

**根因分析**：
1. **硬编码设备 IP**：`192.168.1.12:16512` 是开发者本地 MuMu/模拟器 ADB 地址。新用户克隆仓库后，`config/client_config.json` 不存在（gitignored），需自行创建。但项目中**无 `client_config.example.json` 模板文件**提供参考结构，新用户需从 CLAUDE.md 或代码推断配置格式。

2. **硬编码模型路径**：`model_path: "/models/test.gguf"` 使用 Unix 风格绝对路径（`/models/`），但项目为 Windows-only。在 Windows 上该路径解析为 `C:\models\test.gguf`，通常不存在。

3. **`n_gpu_layers: 999`**：CFG-11 已知问题，本地配置中同样存在此不安全默认值。

**影响面**：
- 新用户 onboarding 困难：无模板文件，需自行摸索配置格式。
- 模型路径在 Windows 上无效，LLM 功能首次使用时才暴露问题。

**修改建议**：
1. 创建 `config/client_config.example.json` 模板文件，提供注释化的配置结构参考：
   ```json
   {
     "device": {
       "serial": "localhost:16384",
       "last_connected": "",
       "auto_connect_last": true
     },
     "llm": {
       "enabled": false,
       "model_path": "models/LLM/Qwen3.5-4B-UD-Q4_K_XL.gguf",
       "n_gpu_layers": -1
     }
   }
   ```
2. `_load_config` 在返回 `{}` 时使用代码中的安全默认值，而非依赖配置文件的存在。

### [PLAN-01] `.plan/execution_plan.md` 引用过时的行号

**优先级：Info**
**位置**：`.plan/execution_plan.md`（698KB）

```markdown
### P0-1：PipelineRunner._evaluate_and/_evaluate_or 复合条件恒真

- **位置**：L281–L315
```

**根因分析**：
实施计划中引用的行号（如 `L281–L315`）是基于**编写计划时的代码版本**。当前代码经过多次重构（批次 2-6 审查后已有修改），行号可能已偏移。若开发者按计划中的行号查找代码位置，可能找不到目标代码或定位到错误的代码段。

**影响面**：
- 实施计划作为修改指南，行号不准确会降低执行效率。
- 若计划中描述的"问题"在当前代码中已被修复（如 P0-3 `final_dist = -1.0` → `float('inf')` 已在某批次修复），按计划执行可能导致冗余修改。

**修改建议**：
1. 在实施计划中引用代码位置时，使用函数名+特征描述替代行号：
   ```markdown

   ```
2. 或在执行每批修改前，先用 `grep` 验证目标代码是否仍存在于计划描述的状态。
