## 1. 新发现（历史未覆盖）

### [NEW-1] `minimap_locator.py:181` — `level_id` 提取偏移 2 字符（`tile_class[5:11]` 应为 Map01Lv/Map02Lv 前缀之后）

**优先级：Medium（导航定位错误）**
**位置**：`src/core/service/navigation/minimap_locator.py:167-181`（`_parse_tile`）

```python
if "Tier" in tile_class:
    parts = tile_class.split("Tier")
    ...
    if tile_class.startswith(("Map01Lv", "Map02Lv")):
        level_id = tile_class[5:11].lower()   # ← BUG
```

**根因分析**：
- `tile_class` 形如 `"Map01Lv02Tier3"`（小地图 ONNX 分类输出）。前缀 `"Map01Lv"` 长度为 7，字符索引 `M0 a1 p2 0 3 1 4 L 5 v 6 ...`。
- `tile_class[5:11]` 取的是索引 5–10，即 `"Lv"` 开头再加后续 4 个字符（例如 `"Map01Lv02Tier3"` → `"Lv02Ti"`），**把 "Lv" 字面量也纳入了 level_id**。
- 正确的 level_id 应当是 `"Map01Lv"` 之后的等级编号段，即 `tile_class[7:]`（或更稳健地按长度截取 2–3 位等级码）。当前切片起点错误且长度 `[5:11]`（固定 6 字符）是臆造的，导致 level_id 恒为垃圾串 `"lv02ti"` 之类。

**影响面**：
- `level_id` 经 `MapPosition` → `Navigator.where_am_i()`（navigator.py:128 用 `pos.level_id` 调 `find_nearby`）、`to_coords()`（level_to_nav 推导）、`nav3` 命令回传。level_id 错误 → 周边实体检索 `find_nearby(map_id, level_id, ...)`（entity_db.py:146）按 `f"{map_id}_{level_id}"` 查表**全部落空**（key 不匹配），`where_am_i` 的 `nearby_entities` 恒为空，按名/按等级导航的目标 level 解析错误。
- 该分支仅在 `tile_class` 含 `"Tier"` 且以 `"Map01Lv"/"Map02Lv"` 开头时进入，对应大世界层级小地图瓦片，属于实际会命中的主路径。

**修改方案**（最小可行）：
```python
# 在 _parse_tile 内，"Tier" 分支中：
if tile_class.startswith(("Map01Lv", "Map02Lv")):
    # 去掉固定前缀 "Map01Lv"(7)/"Map02Lv"(7)，取等级编号段
    prefix_len = 7
    rest = tile_class[prefix_len:]
    # 取 "Lv" 之后到 "Tier" 之前的等级码（如 "02"）
    m = re.search(r"Lv(\d+)", rest)
    level_id = ("lv" + m.group(1)) if m else rest.lower()
```
或直接 `level_id = tile_class[7:11].lower()`（前提等级码固定 4 字符，需与 `grid_tiers.json` 约定对齐）。

**非期待变化 / 回退**：若后续 ONNX `cls.json` 的 tile_class 命名规范变更（如去掉 "Lv"），`[7:11]` 需同步调整；建议以正则 `Lv(\d+)` 提取以解耦前缀长度，避免再次硬编码偏移。影响范围仅 `MinimapLocator._parse_tile` 一处，回退容易。

### [NEW-2] `recognizer.py:298-301` — `yolo_classes` 与 `element_types` 类型错配，required YOLO 元素评分分支恒为假（死代码）

**优先级：Low / Medium（页面分类健壮性）**
**位置**：`src/core/service/../capability/element_recognition/recognizer.py:298-301`（`_score_page`）

```python
yolo_cls = catalog_entry.get("yolo_classes", [])
if any(c in element_types for c in yolo_cls):   # ← BUG
    score += 1.5
    required_matched += 1
```

**根因分析**：
- `element_types = {e.element_type for e in elements}`（recognizer.py:277），其取值来自各后端的 `element_type` 字段：模板→`"icon"`、OCR→`"text"`、颜色→`"character"/"region"`、YOLO→`"yolo_object"`（见 `yolo_backend.py:83`，**硬编码为 `"yolo_object"`**）。
- 而目录 `yolo_classes` 配置的是 YOLO **类别名**（如 `"person"/"car"`，即 `YOLOBackend` 输出的 `cls_name`/`label`）。类别名 `"person"` 永远不可能出现在 `{"yolo_object","text",...}` 集合中。
- 因此 `any(c in element_types ...)` 对任何 catalog 配置都恒为 `False`：**页面签名里声明的「required YOLO 元素」永远无法命中 +1.5 评分**，且 `required_matched` 不递增。该 Tier-1 分支是死代码。

**影响面**：
- 依赖 `yolo_classes` 作为 required 元素的页面签名（如某些「场景含人物/载具」判定）无法获得应有的 +1.5 分，分类器对这些页面的置信度偏低，可能退化为 `unknown` 或被错误页面抢分，间接影响 `recognize()` 的 `page_type` 输出与下游导航/任务决策。
- 仅影响**分类**阶段；不影响元素检测本身（YOLO 检测仍正常返回 `ElementInfo`）。

**修改方案**（最小可行，二选一）：
```python
# 方案 A（推荐）：用 label 集合而非 element_type 匹配 YOLO 类别
element_labels = {e.label.lower() for e in elements}   # 已存在
yolo_cls = catalog_entry.get("yolo_classes", [])
if any(str(c).lower() in element_labels for c in yolo_cls):
    score += 1.5
    required_matched += 1
```
```python
# 方案 B：YOLOBackend 输出 element_type 设为类别名（影响面更大，不推荐）
```

**非期待变化 / 回退**：方案 A 改为基于 `label` 匹配，与同函数 OCR 关键词匹配（`ocr_kws in element_labels`）逻辑一致，语义正确；回退只需还原该行。注意 `yolo_cls` 配置若本就为空，则改动无影响。

## 2. 对既往报告的审计纠正（基于当前代码核验）

### [A1] 纠正：W1（长期标记 Critical「VLM 行走字母键被拒 → 完全失效」）已实质缓解，应降级为 Low/UX

**既往论断**（memory 多批次，如 `20260711_1122.md` §3 仍列 W1=Critical「字母键被拒、三重静默失败」）：
W1 称 VLM 行走发送小写字母键（w/a/s/d）被守护进程 `_is_valid_keyevent` 拒绝，导致 VLM 导航完全失效，且 `keyevent()`/`shell()` 吞错不抛，构成三重静默。

**当前代码核验（证据）**：
1. `_KNOWN_KEYEVENT_NAMES`（`android_runtime.py:90-96`）**已包含** `KEYCODE_W/A/S/D/Q/E/F`，`_is_valid_keyevent`（:101-110）对 `"KEYCODE_W"` 返回 `True` → 守护进程 `keyevent` 方法（:632-638）**不再拒绝**移动键。
2. `VlmWalkNavigator._ACTION_KEYCODE_MAP`（vlm_walk_navigator.py:285-293）已将 `forward/backward/left/right/turn_left/turn_right/interact` 映射为 `KEYCODE_W/S/A/D/Q/E/F`——导航层**根本不再发送裸字母**，而是发送被 allow-list 接受的 `KEYCODE_*` 常量名。
3. `runtime.py:_vlm_keyevent`（:712-729）向 `android.keyevent("KEYCODE_W")` 透传，整条链路可贯通。

**结论**：
- W1 的**核心失败模式（字母键被拒 → VLM 行走完全失效）在当前代码中已不存在**——allow-list 扩展 + 导航层 KEYCODE 映射共同修复了根因。原 "Critical / VLM 行走完全失效" 评级与描述**不再成立**。
- **残留问题（低危）**：`AndroidRuntime.keyevent()`（android_runtime.py:810-812）与 `shell()`（:814-816）在守护进程返回 `{"error":...}` 时执行 `return response.get("result", "")`，**吞掉错误返回空串**，与 `tap()`/`swipe()`（:799-800、:807-808 显式 `raise AndroidRuntimeError`）行为不一致。这仅在传输/守护进程异常时导致静默失败，属 **Low / UX 健壮性**缺陷，而非导航阻断。
- **处置建议**：将 W1 由 Critical **降级为 Low（错误吞没一致性）**，并从「VLM 行走失效」风险台账移除；修复仅需在 `keyevent()`/`shell()` 增加 `if response.get("error"): raise AndroidRuntimeError(...)` 与 `tap`/`swipe` 对齐。
