## 2. 新发现漏洞与错误

### 2.1 🔴 严重问题

#### MCP01 - `_sync_execute` 从 TaskRunWorker 线程直接访问 `self._bridge`（Qt 线程违规）

**文件**: `src/gui/pyqt6/pages/maaend_control_page.py:332-333, 824`
**风险等级**: 🔴 严重
**影响**: Qt GUI 对象跨线程访问，可能导致崩溃或堆损坏

**问题代码**:
```python
# _sync_execute (line 332-333)
self._bridge.commandFinished.connect(_on_finished)
self._bridge.execute(expected, params or {})

# _runtime_queue_runner (line 824)
result = self._sync_execute(f"task run {clean_name} --options {payload}")
```

**分析**:
- `_runtime_queue_runner` 在 `TaskRunWorker`（QThread）内执行
- `self._bridge` 是 `CLIBridge` 实例，其 `commandFinished` 信号在 GUI 主线程发射
- Qt 要求 GUI 对象只能从创建它的线程访问——`self._bridge` 在主线程创建
- 从 Worker 线程调用 `self._bridge.connect()` 和 `self._bridge.execute()` 违反 Qt 线程亲缘规则
- 虽然在实践中 CPython GIL 可能掩盖部分问题，但在信号发射/接收时仍存在堆损坏风险
- 批次37 M05 报告过类似问题（`_sync_execute` 线程安全），但仅关注了 QEventLoop 嵌套

**修复建议**:
```python
# 方案1: 通过 QMetaObject.invokeMethod 将 bridge 调用路由到主线程
QMetaObject.invokeMethod(
    self._bridge, "execute",
    Qt.ConnectionType.BlockingQueuedConnection,
    Q_ARG(str, expected),
    Q_ARG(dict, params or {})
)

# 方案2: 将 bridge 操作封装为信号，由主线程槽执行
self._bridge_command_requested.emit(command, params)
```

#### MCP02 - `_sync_execute` 在 Worker 线程中嵌套 QEventLoop 导致死锁

**文件**: `src/gui/pyqt6/pages/maaend_control_page.py:321-325, 824`
**风险等级**: 🔴 严重
**影响**: Worker 线程嵌套事件循环阻止自身 queued 信号投递，任务执行永久挂起

**问题代码**:
```python
# _sync_execute (line 321-325)
loop = QEventLoop()
...
self._bridge.commandFinished.connect(_on_finished)
self._bridge.execute(expected, params or {})
QTimer.singleShot(timeout_ms, loop.quit)
loop.exec()  # ← Worker 线程阻塞在这里

# _runtime_queue_runner (line 824)
result = self._sync_execute(...)  # ← 在 TaskRunWorker 中调用
```

**分析**:
- `QThread` 的事件循环负责处理 queued connections
- 当 `_sync_execute` 在 Worker 线程中调用 `loop.exec()` 时，Worker 的事件循环被嵌套的 QEventLoop 阻塞
- `commandFinished` 信号通过 `Qt.QueuedConnection` 从主线程投递到 Worker 线程
- 但 Worker 线程的事件循环被嵌套的 `loop.exec()` 阻塞——queued 信号无法被处理
- 结果：信号永远不会到达 `_on_finished`，任务永久挂起直到超时
- 这是 MCP01 的具体表现形式——线程违规导致死锁

**修复建议**:
```python
# 方案1: 主线程执行 _sync_execute，Worker 只触发信号
def _runtime_queue_runner(self):
    for item in self._queue_items:
        self._request_sync_execute.emit(item)  # 信号 → 主线程
        self._wait_for_completion()  # QWaitCondition

# 方案2: 使用 Future/Promise 模式
def _sync_execute_async(self, command, params):
    future = concurrent.futures.Future()
    def _on_finished(cmd, res):
        if cmd == expected:
            future.set_result(res)
    self._bridge.commandFinished.connect(_on_finished)
    self._bridge.execute(expected, params)
    return future.result(timeout=timeout_ms / 1000)
```

### 2.2 🟠 高危问题

#### MCP03 - `_failed_indices` 跨线程读写无同步

**文件**: `src/gui/pyqt6/pages/maaend_control_page.py:832, 1449, 1478`
**风险等级**: 🟠 高危
**影响**: 自动重试逻辑可能读取到过期或部分更新的失败索引列表

**问题代码**:
```python
# _runtime_queue_runner (line 832) — Worker 线程写入
self._failed_indices = failed

# _on_execution_finished (line 1478) — 主线程读取
if self._failed_indices:
    self._retry_failed(self._failed_indices)
```

**分析**:
- CPython GIL 保证列表赋值操作的原子性，但不保证内存可见性
- Worker 线程写入 `self._failed_indices` 后，主线程可能读取到旧值
- 如果主线程先读取到空列表，自动重试被跳过
- 如果主线程读取到部分更新的列表，重试可能包含错误的索引
- 批次31 G04 已报告同类线程安全问题（`_queue_state` 无同步访问）

**修复建议**:
```python
# 使用 threading.Lock 保护
self._failed_indices_lock = threading.Lock()

# Worker 线程写入
with self._failed_indices_lock:
    self._failed_indices = failed

# 主线程读取
with self._failed_indices_lock:
    indices = list(self._failed_indices)
```

#### MCP04 - `_on_execution_finished` 不清理 `self._worker`，QThread 句柄泄漏

**文件**: `src/gui/pyqt6/pages/maaend_control_page.py:1471-1481`
**风险等级**: 🟠 高危
**影响**: 每次任务执行后 Worker QThread 实例泄漏，长期运行累积原生线程资源

**问题代码**:
```python
def _on_execution_finished(self, success: bool) -> None:
    ...
    self._update_execution_ui()
    self._is_executing = False
    # ❌ self._worker 未被清理
```

**分析**:
- `_start_execution` 创建 `self._worker = TaskRunWorker(...)` 并启动
- 任务完成后 `_on_execution_finished` 更新 UI 但不清理 worker
- `TaskRunWorker` 继承 QThread，其底层 OS 线程资源未被释放
- 每次运行任务都创建新 QThread 实例，旧实例仍然被 `self._worker` 引用
- 用户运行 100 次任务后，100 个已完成的 QThread 实例仍占用内存
- 批次37 M11 已报告 AgentClient 异常后仍标记 `_connected=True`，但未覆盖 QThread 泄漏

**修复建议**:
```python
def _on_execution_finished(self, success: bool) -> None:
    ...
    self._is_executing = False
    if self._worker is not None:
        self._worker.wait(5000)  # 等待线程结束
        self._worker = None
```

#### CLI01 - `llm stop` 先触发 warmup 再执行停止

**文件**: `src/cli/istina.py:263` + `src/cli/handlers.py:222-225`
**风险等级**: 🟠 高危
**影响**: 用户执行 `llm stop` 时，CLI 先花 60 秒启动 llama-server，再立即停止

**问题代码**:
```python
# istina.py:263
if args.command == "llm" and getattr(args, "action", None) not in ("status",):
    self._runtime.warmup_llm()  # ← llm stop 也触发 warmup

# handlers.py:222-225
def _handle_llm(self, args):
    action = getattr(args, "action", "")
    if action == "start":
        return {"status": "success" if self._runtime.warmup_llm() else "error"}
    elif action == "stop":
        return self._runtime.cooldown_llm()
```

**分析**:
- `_auto_warmup` 的排除列表仅排除 `"status"`，未排除 `"stop"`
- `llm stop` 语义上不需要 warmup——停止操作与 warmup 无关
- 实际行为：`llm stop` → warmup_llm()（阻塞 ~60s）→ cooldown_llm()（停止）
- 用户等待 60 秒看到"启动"日志，然后服务被停止——体验矛盾
- 同时存在 warmup 副作用：启动后立即停止可能留下僵尸进程

**修复建议**:
```python
# istina.py:263
if args.command == "llm" and getattr(args, "action", None) not in ("status", "stop"):
    self._runtime.warmup_llm()
```

#### CLI02 - `_handle_llm_prompt` 未验证 `float()`/`int()` 转换

**文件**: `src/cli/handlers.py:854-856`
**风险等级**: 🟠 高危
**影响**: 传入非法数值参数时崩溃，返回模糊错误消息

**问题代码**:
```python
# handlers.py:854-856
if temperature is not None:
    params["temperature"] = float(temperature)  # ValueError if temperature="abc"
if max_tokens is not None:
    params["max_tokens"] = int(max_tokens)  # ValueError if max_tokens="xyz"
```

**分析**:
- `istina.py:193-194` 定义 `--temperature` 和 `--max-tokens` 时**未设置 `type=`**，所以 argparse 不验证类型
- 参数作为原始字符串到达 handler
- `float("abc")` 或 `int("xyz")` 引发 `ValueError`
- 异常传播到 `main()` 的外层 catch，返回通用错误消息
- 用户看到类似 `"执行出错: invalid literal for float()"` 而非 `"温度参数必须为数字"`

**修复建议**:
```python
# 方案1: 在 argparse 中设置 type
parser.add_argument("--temperature", type=float, default=0.3)
parser.add_argument("--max-tokens", type=int, default=512)

# 方案2: 在 handler 中 try/except
try:
    params["temperature"] = float(temperature)
except (ValueError, TypeError):
    return {"status": "error", "message": f"temperature must be a number, got: {temperature}"}
```

### 2.3 🟡 中危问题

#### MCP05 - 队列执行后索引漂移：`_runtime_queue_runner` 使用快照索引更新实时状态

**文件**: `src/gui/pyqt6/pages/maaend_control_page.py:798-832`
**风险等级**: 🟡 中危
**影响**: 用户在任务执行期间修改队列（重排/删除），后续状态更新到错误的队列项

**问题代码**:
```python
# line 798: 快照
items = list(self._queue_state.queue_items)
for idx, item in enumerate(items):
    ...
    result = self._sync_execute(...)  # ← 阻塞 5 分钟
    # line 827-828: 用原始 idx 更新
    self._queue_state.update_queue_item_status(idx, "running")
    self._queue_state.update_queue_item_status(idx, "success" if ok else "failed")
```

**分析**:
- `items = list(self._queue_state.queue_items)` 在执行前创建快照
- `_sync_execute` 阻塞最长 5 分钟
- 在这 5 分钟内，用户可能在 GUI 中重排/删除队列项
- `update_queue_item_status(idx, ...)` 使用快照中的原始 idx
- 如果队列已被修改，idx 可能指向不同的任务
- `QueueState.update_queue_item_status` 有边界检查（`0 <= index < len`），但不检查索引对应的是否是同一个任务
- 这会导致 UI 上错误的任务显示为"运行中"或"已完成"

**修复建议**:
```python
# 使用任务名称而非索引更新状态
for idx, item in enumerate(items):
    name = item["name"]
    ...
    # 用名称查找当前索引
    current_idx = next((i for i, q in enumerate(self._queue_state.queue_items) if q["name"] == name), None)
    if current_idx is not None:
        self._queue_state.update_queue_item_status(current_idx, "running")
```

#### CLI04 - `_handle_task_list` 不一致的错误处理

**文件**: `src/cli/handlers.py:316-324`
**风险等级**: 🟡 中危
**影响**: 相似调用有不同错误处理策略，用户体验不一致

**问题代码**:
```python
# line 316 — 无 try/except
tasks = runtime.execute("task.list", {"serial": getattr(args, "serial", None)})

# line 321 — 有 try/except
try:
    task_option_defs = runtime.maaend(getattr(args, "serial", None)).task_option_defs()
except Exception:
    task_option_defs = {}
```

**分析**:
- `runtime.execute("task.list")` 失败时，异常传播到 `main()` 的外层 catch
- `runtime.maaend(...).task_option_defs()` 失败时，异常被静默吞掉
- 用户看到的行为取决于哪个调用失败：
  - task.list 失败 → 通用错误消息 + 退出码 1
  - task_option_defs 失败 → 部分数据返回（任务列表 + 空选项定义）
- 两处代码在同一个 handler 中，但错误处理策略完全不同

**修复建议**: 统一错误处理策略——要么都在 handler 层 try/except，要么都依赖外层 catch。
