## 1. 代码错误（功能/逻辑类）

### [I18N-1] `install_qt_translator` 中 `QTranslator` 为局部变量，方法返回后即被 GC；且该方法是全仓库零调用的死代码
**优先级：Low（潜在缺陷 / 无效代码）**
**位置**：`src/gui/pyqt6/i18n/__init__.py:70-80`

```python
def install_qt_translator(self, app: Optional[QApplication] = None) -> None:
    if app is None:
        app = QApplication.instance()
    if app is None:
        return
    translator = QTranslator(app)          # 局部变量
    locale = QLocale(self._current_locale)
    qm_name = f"qt_{locale.name()}"
    if translator.load(qm_name):
        app.installTranslator(translator)  # 方法结束 → translator 引用消失 → 被 GC
```

**根因**：
1. Qt 的 `QCoreApplication::installTranslator()` **不接管 translator 的所有权**，调用方必须自行保活。此处 `translator` 是方法内的局部变量，函数返回后无任何引用，Python 即可回收该对象。一旦回收，已安装的 Qt 翻译随即失效（极端情况下 Qt 内部仍持有裸指针，存在悬垂引用风险）。
2. 全仓库 grep `install_qt_translator` 命中 **0 处调用**；应用实际的国际化完全依赖 `LocaleManager.tr()` 的手动字典查询（grep `get_locale_manager` / `.tr(` 命中 254 处，均为该手动路径）。因此本方法是未被接线的死代码。

**影响面**：当前无运行时影响（未被调用）。但若未来有开发者试图通过 Qt 原生翻译体系做 i18n，本方法会因 translator 生命周期错误而“看似安装成功、实际翻译无效”，形成难以排查的隐性缺陷。

**修改建议**（二选一）：
- 若确认不需要 Qt 原生翻译：直接删除 `install_qt_translator` 方法及 `QTranslator` / `QLocale` / `QApplication` 的导入，消除死代码与潜在陷阱。
- 若确需保留：将 translator 存为实例属性以保活：
  ```python
  self._qt_translator = QTranslator(app)
  if self._qt_translator.load(qm_name):
      app.installTranslator(self._qt_translator)
  ```

> 注：与此相对，`LocaleManager` 的路径解析 `Path(__file__).parent.parent / "locales"` 经验证正确指向 `src/gui/pyqt6/locales/`（该目录确含 `zh_CN.json` / `en_US.json`），不构成缺陷。

### [M1] `TemplateMatcher.match` 与 `match_all_instances` 去重策略不一致，小尺寸模板密集图标会被漏检
**优先级：Low**
**位置**：`src/core/capability/element_recognition/pipeline/matcher.py:60-69`（网格去重） vs `136-149`（IoU-NMS 去重）

`match()` 的去除重复逻辑为固定 5px 网格分桶：
```python
key = (r["x"] // 5, r["y"] // 5)
if key not in seen:
    seen.add(key); deduped.append(r)
```
而 `match_all_instances()` 用的是基于包围盒交并比（IoU）的非极大抑制（NMS）：
```python
if ix < ix2 and iy < iy2:
    overlap = True; break
```

**问题**：两者对同一输入的产出不一致。固定 5px 网格去重**完全无视模板本身尺寸**——当模板宽度 `< 5px`（极小图标、像素级标记）或同屏存在间距 `< 5px` 的重复图标时，本应作为独立实例返回的多个匹配会被合并为 1 个，造成**漏检（false negative）**。例如两个 8×8 图标分别位于 (10,10) 与 (12,10)，网格键同为 `(2,2)`，被错误合并。固定网格也不同于 IoU 语义，调用方无法预期哪条路径会生效。

**修改建议**：统一两种匹配模式的去重口径。建议两者都采用基于模板尺寸的自适应 NMS（IoU 或像素位移阈值取 `max(t_w, t_h) // 2`），避免与模板大小脱钩的硬编码 5px。

### [M2] `match` / `match_all_instances` 对 ROI 负坐标越界未做边界校验，越界时静默取错区域
**优先级：Low**
**位置**：`src/core/capability/element_recognition/pipeline/matcher.py:33-42`

```python
if roi:
    rx, ry, rw, rh = roi
    if rx < 0:
        rx = w_img + rx
    if ry < 0:
        ry = h_img + ry
    screen_roi = screen_gray[ry:ry + rh, rx:rx + rw]
```

当调用方传入的 `rx` 绝对值超过图像宽度（如 `rx = -150`、`w_img = 100`），`rx` 经 `w_img + rx` 仍为负（此处为 `-50`）。随后 `screen_gray[..., -50:...]` 在 Python/NumPy 中被解释为**从尾部回绕的索引**，返回的是图像另一端的错误区域，而非空结果或报错。`t_h > screen_roi.shape[0]` 的早退检查仅覆盖“模板大于 ROI”的情形，无法拦截“ROI 来自回绕错误区域且尺寸恰巧足够”的情况，最终会在错误像素上产生错误匹配。

**修改建议**：对归一化后的 `rx/ry` 增加边界钳制与合法性校验：
```python
rx = max(0, w_img + rx) if rx < 0 else rx
ry = max(0, h_img + ry) if ry < 0 else ry
if rx < 0 or ry < 0 or rx + rw > w_img or ry + rh > h_img:
    return []
```

### [M3] 4 通道（BGRA/RGBA）输入在灰度转换分支会崩溃
**优先级：Low**
**位置**：`src/core/capability/element_recognition/pipeline/matcher.py:27`（及 `102` 重复）

```python
screen_gray = cv2.cvtColor(screen, cv2.COLOR_BGR2GRAY) if len(screen.shape) == 3 else screen
```

判定条件只看维度数 `len(screen.shape) == 3`，但 BGRA/RGBA 截图同样满足 `ndim == 3`（shape 为 `(H, W, 4)`）。此时进入 `cvtColor(..., COLOR_BGR2GRAY)`，而该转换要求 3 通道输入，对 4 通道会抛出 `cv2.error`。`else` 分支（保留原图）只覆盖真正的 2D 灰度情形。

**修改建议**：按通道数显式分支，4 通道先剥离 alpha 或转 BGR：
```python
if screen.ndim == 2:
    screen_gray = screen
elif screen.shape[2] == 4:
    screen_gray = cv2.cvtColor(screen, cv2.COLOR_BGRA2GRAY)
else:
    screen_gray = cv2.cvtColor(screen, cv2.COLOR_BGR2GRAY)
```

## 2. 一致性 / 可维护性问题

### [A1] `Annotation.points` 与 `AnnotationShape.pts` 字段命名不一致，易引发静默错误
**优先级：Info**
**位置**：`src/core/capability/element_recognition/annotation.py:11-14`（`points`） vs `22-24`（`pts`）

两个表达“同一概念（标注多边形顶点）”的并行 dataclass，字段名分别为 `points`（`Annotation`）和 `pts`（`AnnotationShape`）。任何在两者间转换的代码都必须记住改名，若误用 `ann.points` 访问 `AnnotationShape` 会得到 `AttributeError`，而反向误用 `shape.pts` 访问 `Annotation` 同样报错；更危险的是若某处用 `getattr`/字典式访问则可能静默拿到 `None`。建议统一命名字段（例如统一为 `points`），或在转换处显式映射并加单测守护。

### 审计 `reports/auto/20260711_0200_nav.md` 的 [NAV-05]

| 原报告论断 | 源码核对结论 |
|------------|--------------|
| “若 `raw_location` 为字符串 `"1,2,3"` 或 dict `{"x":1,"y":2}`…`float(rl[0])` 全部失败并抛出 `TypeError`” | **该论断对 dict 子情况是错误的。** `Entity.from_raw`（`entity_db.py:27-34`）中 `rl = raw.get("raw_location", [0,0,0])`，随后 `raw_location=(float(rl[0]),...) if len(rl) >= 3 else (0.0,0.0,0.0)`。对 dict `{"x":1,"y":2}`，`len(rl)` 返回键数 `2`，`>=3` 为 False，直接走 `else` 默认分支 `(0.0,0.0,0.0)`——**不会崩溃**，仅静默得到 (0,0,0) 坐标。 |
| 崩溃路径 | **真实崩溃来自字符串 / 含非数字元素的列表**：如 `"1,2,3"` → `len==5>=3` → `float(rl[1]) = float(",")` 抛 `ValueError`；或 `["a","b","c"]` → `float("a")` 抛 `ValueError`。原报告将异常类型写作 `TypeError` 亦不准确，应为 `ValueError`。 |
| 影响范围 | 原报告称异常“传播至 `load()` 循环，导致整个数据库加载中断”。**实际更严重**：`load()` 的 `try/except`（`entity_db.py:78-83`）**只包裹 `json.load`**，而 `for map_entry in raw_maps:` 实体构建循环（85-99 行）**不在** try 内。`from_raw` 抛出的 `ValueError` 会直接穿透 `load()`、再穿透 `Navigator.__init__` 的 `self._entities.load()`（41 行，亦无保护），**最终作为未捕获异常令整个 `Navigator` 构造失败**，而非“加载中断、返回空库”。 |

**修正结论**：NAV-05 的**修复建议（在 `from_raw` 中校验类型并跳过畸形条目）仍然成立且必要**，但其对触发机制的说明（dict→TypeError、被吞没）有误。正确机制是：字符串/非数字列表值触发 `ValueError`，且该异常**未被 `load()` 捕获**，会向上中断 `Navigator` 初始化。建议将 NAV-05 补注为“未捕获异常导致 `Navigator` 构造崩溃”，并优先在 `from_raw` 内对 `raw_location`/`pixel_location`/`map_location` 做类型与长度校验（非 list / 元素非数值时记录 warning 并使用默认值，而非放任 `float()` 抛错）。
