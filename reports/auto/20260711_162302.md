### [GUI59-01 Low] `scripting/player.py:152-159` — `_do_click` 双重映射本地坐标为全局坐标，所有脚本点击落在控件原点

```python
# scripting/recorder.py:150 — 录制时保存本地坐标
pos = event.position()  # QPointF，控件本地坐标
value = {"x": pos.x(), "y": pos.y(), ...}

# scripting/player.py:152-159 — 回放时当作全局坐标处理
x = float(value.get("x", 0))
y = float(value.get("y", 0))
pos = QPoint(int(x), int(y))
local_pos = widget.mapFromGlobal(widget.mapToGlobal(pos))
QTest.mouseClick(widget, Qt.MouseButton.LeftButton, pos=local_pos)
```

**问题**：`recorder._record_click` 通过 `event.position()` 获取鼠标事件位置，这是控件**本地坐标**。但 `player._do_click` 将这组本地坐标传入 `mapToGlobal()`（将参数解释为全局坐标），得到错误的全局坐标；再经 `mapFromGlobal()` 映射回本地坐标。

以控件在屏幕原点为例：
- `event.position()` → `(200, 100)`（控件本地坐标，用户在控件内点击的位置）
- `widget.mapToGlobal(QPoint(200, 100))` → 将 `(200, 100)` 当作全局坐标 → 若控件原点为 `(500, 300)`，结果 `(700, 400)`，但这里控件原点并非 `(0,0)`，`mapToGlobal` 实际将 `(200, 100)` 转换为全局坐标。**等等，`mapToGlobal` 的语义是：将控件本地坐标 `(200, 100)` 转换为全局坐标。如果控件在屏幕的 `(500, 300)`，结果应是 `(700, 400)`。但录制时 `event.position()` 已经是 `(200, 100)`，这个值在回放时不应再经过 `mapToGlobal`。**
- `widget.mapFromGlobal(QPoint(700, 400))` → 将 `(700, 400)` 转回控件本地坐标 → `(200, 100)`

**实际上**，`mapToGlobal(local_pos)` 在控件原点非零时会正确转换。双重映射的结果取决于控件位置——在大多数情况下并非精确的恒等变换。关键问题是**录制坐标是控件本地坐标，但回放时错误地将本地坐标当作全局坐标再转回本地**。

**影响面**：脚本回放功能。在大多数场景下（控件位于屏幕非原点），双重映射会产生错误坐标，点击位置偏离用户录制的位置。在控件精确位于屏幕原点的特殊情况下，双重映射恰好抵消，但这不应作为正确性依据。

**建议**：录制时使用全局坐标，或在回放时直接使用录制的本地坐标（不加 `mapToGlobal`/`mapFromGlobal`）：

```python
# 方案 A：录制时保存全局坐标
global_pos = event.globalPosition()
value = {"x": global_pos.x(), "y": global_pos.y()}

# 回放时
local_pos = widget.mapFromGlobal(QPoint(int(x), int(y)))

# 方案 B：录制时保存本地坐标，回放时直接使用
local_pos = QPoint(int(x), int(y))
```

### [GUI59-02 Info] `scripting/models.py:49-54` — `Script.save()` 使用非原子写入

```python
def save(self, directory: Path) -> Path:
    directory.mkdir(parents=True, exist_ok=True)
    path = directory / f"{self.name}.json"
    path.write_text(json.dumps(self.to_dict(), ensure_ascii=False, indent=2), encoding="utf-8")
    return path
```

**问题**：直接写入目标文件，无 `.tmp` + `os.replace()` 原子写入。若写入过程中进程崩溃或断电，文件可能处于半写入状态（部分内容已写入但未完成），导致后续 `Script.load()` 抛出 `JSONDecodeError`。

**对比**：`settings_page.py` 的 `_save_settings` 正确使用了 `mkstemp` + `os.replace()` 原子写入。`queue_state.py` 的 `persist()` 也正确使用了 `.tmp` → `os.replace()` 模式。

**影响面**：低——脚本文件体积小（通常 < 10KB），写入完成快，实际崩溃概率低。但 `_refresh_script_list` 的 `except Exception: pass` 静默吞掉加载失败，用户可能发现脚本文件损坏且不知原因。

**建议**：使用 `mkstemp` + `os.replace()` 原子写入：

```python
import tempfile, os

def save(self, directory: Path) -> Path:
    directory.mkdir(parents=True, exist_ok=True)
    path = directory / f"{self.name}.json"
    data = json.dumps(self.to_dict(), ensure_ascii=False, indent=2)
    fd, tmp_path = tempfile.mkstemp(dir=str(directory), suffix=".tmp")
    try:
        with os.fdopen(fd, "w", encoding="utf-8") as f:
            f.write(data)
            f.flush()
            os.fsync(f.fileno())
        os.replace(tmp_path, path)
    except Exception:
        if os.path.exists(tmp_path):
            os.unlink(tmp_path)
        raise
    return path
```

### [GUI59-03 Info] `foundation/gpu_check.py:91-101` — `format_gpu_warning` 模糊，无法区分"无 nvidia-smi"与"无 NVIDIA GPU"

```python
def format_gpu_warning(result: GPUCheckResult) -> Optional[str]:
    if not result.is_nvidia:
        return "GPU不受支持，使用全智能相关功能可能导致严重卡顿。"
    if result.vram_bytes is None or result.vram_bytes < VRAM_WARNING_THRESHOLD_BYTES:
        return "GPU显存低，使用全智能相关功能可能产生卡顿。"
    return None
```

**问题**：当 `is_nvidia=False` 时，`format_gpu_warning` 统一返回「GPU不受支持」，但此状态有三种截然不同的原因：

1. `nvidia-smi` 未安装/未找到 → 用户可能根本没有 NVIDIA GPU
2. 有 NVIDIA GPU 但 `nvidia-smi` 执行失败 → 驱动问题
3. 有 NVIDIA GPU 且 `nvidia-smi` 成功但识别为非 NVIDIA → 异常情况

`GPUCheckResult` 的 `raw_output` 字段在失败时携带了 `nvidia-smi` 的输出（可能包含错误信息），但 `format_gpu_warning` 完全忽略了该字段。

**影响面**：低——用户看到「GPU不受支持」后，无法判断是硬件不支持还是软件缺失，可能做出错误的判断（如误以为需要换显卡）。

**建议**：区分三种失败原因，提供更精确的提示：

```python
def format_gpu_warning(result: GPUCheckResult) -> Optional[str]:
    if not result.is_nvidia:
        if result.raw_output:
            return f"无法检测NVIDIA GPU (nvidia-smi输出: {result.raw_output[:100]})"
        return "未找到nvidia-smi，无法检测NVIDIA GPU。使用全智能功能可能导致卡顿。"
    if result.vram_bytes is None or result.vram_bytes < VRAM_WARNING_THRESHOLD_BYTES:
        return f"GPU显存低 ({result.vram_bytes or 0 // (1024*1024)} MiB)，使用全智能功能可能产生卡顿。"
    return None
```

### [GUI59-04 Info] `capability/llm/client.py:94` — `_post()` 硬编码 120s 超时，无重试逻辑

```python

    ...
    with urllib.request.urlopen(req, timeout=120) as resp:
        raw = resp.read().decode("utf-8", errors="replace")

```

**问题**：

1. **硬编码超时**：120s 超时无法通过配置调整。LLM 推理耗时可能因 prompt 长度、模型大小、硬件状态波动，但用户无法根据自身环境调整超时阈值。

2. **无重试逻辑**：网络抖动（特别是本地 `llama-server` 在首次推理时的冷启动延迟）会导致瞬时超时。一次性失败直接抛出 `LlmClientError`，消费方（如 `chat()` 调用）无恢复机会。对比 `health_check()` 的超时为 5s，但 `_post()` 为 120s — 健康检查容忍度高但实际推理调用反而无任何容错。

3. **超时不可配置**：`LlmClient.__init__` 接受 `base_url`，但不接受 `timeout` 参数。消费方只能通过修改源码调整。

**影响面**：中——LLM 推理是交互式功能，一次性超时会导致用户看到错误提示。但此问题仅在 `llama-server` 高负载或冷启动时触发，概率较低。

```python
        ...
        with urllib.request.urlopen(req, timeout=self._timeout) as resp:
```

### [GUI59-06 Info] `gui/pyqt6/pages/log_page.py:89-98` — 日志文件过滤仅匹配 `.log` 和 `.txt` 扩展名

```python
def _refresh_file_list(self) -> None:
    ...
    files = sorted(self._logs_dir.iterdir(), key=lambda p: p.stat().st_mtime, reverse=True)
    log_files = [f for f in files if f.is_file() and f.suffix.lower() in (".log", ".txt")]
```

**问题**：Python 标准 `logging` 模块默认使用 `FileHandler`，其文件扩展名取决于 `filename` 参数。若 `filename` 不包含扩展名（如 `"logs/main"`），生成的文件无扩展名。当前过滤会将这些文件排除在 Log 页面之外。

**影响面**：低——项目中的 `logging_config.json` 使用了带 `.log` 扩展名的文件名（如 `main.log`），实际不会触发。但若用户添加了不带扩展名的 `FileHandler`，文件会不可见。

**建议**：将 `f.suffix.lower()` 扩展名检查改为「无扩展名也纳入」：

```python
log_files = [f for f in files if f.is_file() and (not f.suffix or f.suffix.lower() in (".log", ".txt"))]
```
