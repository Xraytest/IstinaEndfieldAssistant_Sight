### LOCALE-NO-SIGNAL（LocaleManager 无变更通知信号，全项目语言切换静默失效）— `i18n/__init__.py`

**等级**: 代码错误 / 中
**位置**: `src/gui/pyqt6/i18n/__init__.py:51-55`, `src/gui/pyqt6/pages/settings_page.py:136-149`

**问题代码**:

```python
# i18n/__init__.py:51-55 — 语言切换无信号机制
def set_locale(self, locale: str) -> None:
    if locale not in self._translations:
        return
    self._current_locale = locale
    self._settings.setValue("gui/locale", locale)
    # ❌ 无 pyqtSignal 发射 — 没有任何页面能感知到语言变更

# settings_page.py:136-149 — 语言切换仅更新窗口标题，不刷新任何 widget
def _on_language_changed(self, index: int) -> None:
    new_locale = self._language_combo.currentData()
    if not new_locale:
        return
    locale.set_locale(new_locale)
    main_window = self.window()
    if isinstance(main_window, QMainWindow):
        main_window.setWindowTitle(locale.tr("app_title", "IstinaEndfieldAssistant Sight"))
        main_window.statusBar().showMessage(locale.tr("status_ready", "Ready"), 2000)
    QMessageBox.information(
        self,
        locale.tr("language_changed", "Language changed"),
        locale.tr("restart_for_changes", "Some changes will take effect after restart."),
    )
```

**根因分析**:

`LocaleManager.set_locale()` 仅修改内部状态 `self._current_locale` 并持久化到 QSettings，但**不发射任何 pyqtSignal**。`LocaleManager` 类中无任何 `pyqtSignal` 声明。

导致的结果：

1. **设置页面自身不刷新**：`_on_language_changed` 只更新了 `main_window.setWindowTitle()` 和 statusBar，但**不刷新 settings_page 自身的任何 widget**。页面上所有 `QLabel`、`QGroupBox` 标题、按钮文本等仍显示旧语言。
2. **其他页面永不更新**：其他已初始化的页面（maaend_control_page、device_settings_page、prts_full_intelligence_page 等）在 `__init__` 中通过 `locale.tr()` 获取文本后，**永远不会收到语言变更通知**。
3. **QMessageBox 自相矛盾**：提示框显示 "Some changes will take effect after restart"，但因为设置页面本身也不刷新，用户无法看到当前语言是否正确应用。

**与正确模式的对比**:

| 对比点 | 正确实现（main_window title） | 错误实现（page labels） |
|--------|-------------------------------|------------------------|
| 语言切换响应 | `setWindowTitle(locale.tr(...))` 显式更新 | widget 仅在 `__init__` 中调用一次 `locale.tr()` |
| 信号机制 | 无（硬编码更新） | 无（无更新路径） |

**影响面**:
- **中**：语言切换功能在 UI 层面完全失效——只有重启应用才能看到新语言。所有已初始化的页面持续显示旧语言文本。
- 这是 O-14（settings_page 语言切换）的根因。O-14 描述的是症状（settings page 不刷新），而 **LOCALE-NO-SIGNAL 揭示了架构性缺陷**：整个项目所有页面都受影响，不是仅 settings_page。

**修复建议**:

```python
# 方案 1（最小修改）：在 LocaleManager 中添加 locale_changed 信号
from PyQt6.QtCore import QObject, pyqtSignal

class LocaleManager(QObject):
    locale_changed = pyqtSignal(str)  # 新增

    def set_locale(self, locale: str) -> None:
        if locale not in self._translations:
            return
        self._current_locale = locale
        self._settings.setValue("gui/locale", locale)
        self.locale_changed.emit(locale)  # 新增

# 方案 2：各页面连接 locale_changed 信号，实现 _refresh_ui() 方法
# settings_page.py 中：
class SettingsPage(QWidget):
    def __init__(self, ...):
        ...
        locale.locale_changed.connect(self._refresh_ui)

    def _refresh_ui(self, new_locale: str) -> None:
        # 重新设置所有 widget 的文本
        self._llm_group.setTitle(locale.tr("llm_settings", "LLM Settings"))
        self._preview_interval_spin.setPrefix(locale.tr("interval", "Interval: "))
        # ... 其他 widget
```

**推荐方案 1 + 方案 2**（添加信号 + 各页面适配 `_refresh_ui()`）。作为过渡方案，可先添加信号，各页面逐步适配。

### I18N-HARDCODED-LABEL（硬编码英文字符串，未经过 i18n 框架）— `main_window.py`, `maaend_control_page.py`

**等级**: 用户体验 / 低
**位置**: `src/gui/pyqt6/main_window.py:208`, `src/gui/pyqt6/pages/maaend_control_page.py:1321`

**问题代码**:

```python
# main_window.py:208 — 标题栏内的标签硬编码
# 同一文件已正确使用 locale.tr("app_title", ...) 设置窗口标题，但此标签未 i18n
title_label = QLabel("Istina Endfield Assistant")

# maaend_control_page.py:1321 — 选项编辑器错误提示硬编码
return QLabel("Unsupported option type")
```

**根因分析**:

1. **main_window.py:208**：主窗口标题栏的 `QLabel` 使用硬编码字符串，而同一窗口的 `setWindowTitle` 已正确使用 `locale.tr("app_title", ...)`。这是一个遗漏——当用户切换语言后，窗口标题更新为翻译文本，但标题栏内的标签仍显示英文。与 LOCALE-NO-SIGNAL 叠加后，该标签**永远无法自动更新**。

2. **maaend_control_page.py:1321**：`_create_option_widget` 方法中对未知选项类型的 fallback 使用了硬编码英文字符串。这是一个内部错误状态，用户通常不会触发（只有任务定义错误时才会显示），但作为代码规范应使用 `locale.tr()`。

**影响面**:
- **低**：仅影响个别 UI 元素的翻译覆盖。main_window.py 的标签在中文用户界面中显示英文，视觉不一致。

**修复建议**:

```python
# main_window.py:208
title_label = QLabel(locale.tr("app_title", "Istina Endfield Assistant"))

# maaend_control_page.py:1321
return QLabel(locale.tr("unsupported_option_type", "Unsupported option type"))
```

### DUP-B-NEW（硬编码 parent 链新实例，3 处）— `queue_state.py`, `qt_log_filter.py`, `maaend_control_page.py`

**等级**: 代码质量 / 低
**位置**: `src/gui/pyqt6/queue_state.py:29`, `src/gui/pyqt6/qt_log_filter.py:59`, `src/gui/pyqt6/pages/maaend_control_page.py:1541`

**问题代码**:

```python
# queue_state.py:29 — 应使用 get_project_root()
base = Path(__file__).resolve().parent.parent.parent.parent / "config"

# qt_log_filter.py:59 — 应使用 get_project_root()
project_root = Path(__file__).resolve().parent.parent.parent.parent

# maaend_control_page.py:1541 — 应使用 get_project_root()
base = Path(__file__).resolve().parent.parent.parent.parent / "config"
```

**根因分析**:

DUP-B 模式的新实例。项目规范要求使用 `core.foundation.paths.get_project_root()` 而非 `__file__` 硬编码 parent 链。三个文件均**未导入** `get_project_root`：

- `queue_state.py` — 仅用于 `Path(__file__)...` 路径构造，`get_project_root` 可用但未使用
- `qt_log_filter.py` — 用于日志路径构造，`get_project_root` 可用但未使用
- `maaend_control_page.py:1541` — 第 1556 行已 `from core.foundation.paths import get_project_root`，但第 1541 行仍使用硬编码路径（不一致使用）

**影响面**:
- **低**：4 层 parent 链在当前项目结构下等同于 `get_project_root()`，功能正确。但当项目目录结构变更时，硬编码路径将 silently break，而 `get_project_root()` 有单点维护优势。

**修复建议**:

```python
from core.foundation.paths import get_project_root

# queue_state.py
base = get_project_root() / "config"

# qt_log_filter.py
project_root = get_project_root()

# maaend_control_page.py:1541
base = get_project_root() / "config"
```
