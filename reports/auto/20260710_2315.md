### [N1] `_wait_for_freeze` 完全空实现，冻结检测功能完全失效
**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:348-351`
```python
def _wait_for_freeze(self, screen: np.ndarray, freeze_spec: Any) -> None:
    pass
```
**根因**：该方法在 `_evaluate()` 中被调用（第 71-72 行）：
```python
if current.pre_wait_freezes is not None:
    self._wait_for_freeze(screen, current.pre_wait_freezes)
```
当 pipeline 节点配置了 `pre_wait_freezes` 参数时，调用了一个什么都不做的方法。节点期望等待屏幕画面稳定（连续 N 帧无变化），但实际上执行立即继续。
**影响面**：所有配置了 `pre_wait_freezes` 的 pipeline 节点（如等待加载画面稳定）会立即执行而非等待，导致模板匹配/OCR 在画面仍变化时触发，产生误识别。
**修改建议**：实现帧差比较逻辑——连续捕获 N 帧，逐帧比较像素差异，全部低于阈值时才认为画面稳定。
```python
def _wait_for_freeze(self, screen, freeze_spec):
    if not freeze_spec or freeze_spec <= 0:
        return
    threshold = getattr(freeze_spec, 'pixel_threshold', 5)
    max_frames = getattr(freeze_spec, 'max_frames', 30)
    interval = getattr(freeze_spec, 'interval_ms', 100) / 1000.0
    prev = screen
    for _ in range(max_frames):
        time.sleep(interval)
        # 需要外部提供帧获取能力；当前 PipelineRunner 不持有 screenshot 能力
        # 可通过回调注入或扩展参数
        break  # 当前无法实现，需架构调整
```
**优先级**：High — 功能完全失效，但需要架构调整（PipelineRunner 需要 screenshot 能力注入）。

### [N2] `_hit_counts` 在 `run_pipeline` 重试循环中被每轮 `run()` 清空，速率限制和命中上限在重试间失效
**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:38/51` + `run_pipeline:111-118`
**根因**：
- `_hit_counts` 是实例级属性（`self._hit_counts: Dict[str, int] = {}`）
- `run()` 方法在开头调用 `self._hit_counts.clear()`（第 51 行）
- `run_pipeline()` 在重试循环中反复调用 `run()`，每轮调用都清空上一轮的命中计数
- `_is_rate_limited` 和 `_hit_limit_reached` 依赖 `_hit_counts`，因此在重试间永远为 0
**影响面**：配置了 `rate_limit` 或 `max_hit` 的节点，在重试场景下会绕过速率限制和命中上限，可能触发过度操作。
**修改建议**：
1. 将 `_hit_counts` 和 `_last_run` 改为 `run_pipeline` 的局部变量，按调用栈传递
2. 或在 `run()` 方法中移除 `self._hit_counts.clear()`，改为在 `PipelineRunner.reset()` 或外部显式清理
**优先级**：Medium — 重试场景下的速率控制失效，但重试本身是低频操作。

### [N3] `_evaluate_or` 日志消息与行为矛盾
**位置**：`src/core/capability/element_recognition/pipeline/pipeline_runner.py:308-309`
```python
for sub_name in sub_names:
    sub_node = graph.get_node(sub_name) if graph is not None else None
    if sub_node is None:
        logger.warning(f"Missing sub-node '{sub_name}' for OR evaluation, treating as non-match")
```

**影响面**：低 — 不影响功能，但误导调试。如果开发者根据日志认为缺失节点会导致 OR 失败，可能错误地修复 pipeline JSON。

**优先级**：Low — 纯误导性日志，不影响功能。

### [N4] `_KNOWN_KEYEVENT_NAMES` 白名单仅 20 个常量，缺少大量合法 keycode
**位置**：`src/core/capability/device/android_runtime.py:52-71`
**根因**：第二波修复引入的 keyevent 白名单仅包含 20 个常用常量（BACK/HOME/POWER/VOLUME_UP/DOWN 等），但 Android 定义了数百个 `KEYCODE_*` 常量。以下合法 keycode 被白名单拒绝：
- `KEYCODE_WAKEUP`（唤醒设备）
- `KEYCODE_SLEEP`（休眠）
- `KEYCODE_CAMERA`（相机）
- `KEYCODE_CALL`/`KEYCODE_ENDCALL`（通话）
- `KEYCODE_MEDIA_PLAY_PAUSE`/`KEYCODE_MEDIA_NEXT`/`KEYCODE_MEDIA_PREVIOUS`（媒体控制）
- `KEYCODE_APP_SWITCH` 已在白名单中，但 `KEYCODE_ASSIST` 不在
**影响面**：VLM navigator 或其他自动化流程如果使用非白名单 keycode，会收到 `{"error": "invalid keyevent"}` 响应，操作静默失败。
**修改建议**：
1. 将白名单扩展为允许所有 `KEYCODE_*` 开头的字符串（而非枚举列表）
2. 或在白名单末尾加一条兜底规则：`s.startswith("KEYCODE_")` 即放行
**优先级**：High — 第二波修复的过度限制直接导致合法 keycode 操作失败。

### [N5] `_is_allowed_shell_cmd` 中的 `$` 黑名单误杀合法命令
**位置**：`src/core/capability/device/android_runtime.py:95`
```python
if any(ch in stripped for ch in (";", "|", "&", "`", "$", "(", ")", "{", "}", "<", ">", "\n", "\r")):
    return False
```
**根因**：`$` 在 shell 中用于变量替换（如 `$PID`），是合法的命令片段。黑名单拒绝包含 `$` 的命令，导致以下合法诊断命令被阻止：
- `dumpsys meminfo $PID` — 内存诊断
- `cat /proc/$PID/cmdline` — 进程诊断
- `ls /data/data/$PKG` — 包管理诊断
**影响面**：中等 — 目前守护进程的 shell 调用主要由 VLM navigator 和诊断工具发起，但 `$` 变量替换在自动化脚本中很常见。
**修改建议**：
1. 将 `$` 从黑名单中移除，改为只拒绝 `$(`（命令替换）和 `$` 开头的 `$VAR` 在引号外的情况
2. 或改为更精确的正则匹配：拒绝 `$(`（命令替换）而非所有 `$`
**优先级**：Medium — 合法命令被误拒，但当前调用方可能不使用 `$`。

### [N6] `AndroidRuntimeProxy._client_for` 无锁创建实例，与上层 DCL 形成锁缺口
**位置**：`src/core/service/runtime.py:94-100`
```python
def _client_for(self, serial: Optional[str]) -> Any:
    resolved = serial or self._device_address or "default"
    client = self._clients.get(resolved)
    if client is None:
        client = _get_android_runtime()(serial=resolved, adb_path=self._adb_path)
        self._clients[resolved] = client
    return client
```
**根因**：`IstinaRuntime.android()` 使用 DCL 保护 `_android_clients` 字典的并发创建，但 `AndroidRuntimeProxy` 的 `_client_for()` 方法内部无任何锁保护。如果两个线程同时通过同一个 proxy 实例调用 `_client_for("same_serial")`，可能创建两个 `AndroidRuntime` 实例。
**影响面**：低概率但严重 — 同一 serial 的两个独立 `AndroidRuntime` 实例各自维护独立的 `_Daemon` 和 scrcpy 会话，产生资源泄漏（端口冲突、进程累积）。
**修改建议**：在 `_client_for` 中使用 `threading.Lock` 保护字典读写，或在 `AndroidRuntimeProxy` 层面也使用 DCL。
**优先级**：Medium — 并发路径下的资源泄漏风险。

### [N7] `_replace_tokens` 使用 `str.replace` 顺序替换，可能损坏重叠占位符
**位置**：`src/core/service/maa_end/runtime.py:520-526`
```python
def _replace_tokens(self, text: str, values: Dict[str, Any]) -> str:
    result = text
    for token, replacement in values.items():
        placeholder = "{" + token + "}"
        if placeholder in result:
            result = result.replace(placeholder, str(replacement))
    return result
```
**根因**：`str.replace()` 替换所有匹配子串，包括嵌套/重叠占位符。例如：
- 模板：`"{a_b}"`
- values：`{"a": "X", "a_b": "Y"}`
- 如果 dict 迭代顺序是 `a` 先于 `a_b`：
  - 第 1 轮：`"{a_b}".replace("{a}", "X")` → `"X_b}"`（`{a}` 是 `{a_b}` 的子串）
  - 第 2 轮：`"X_b}"` 中不含 `{a_b}`，替换失败 → 结果 `"X_b}"` 而非期望的 `"Y"`
**影响面**：当 pipeline override 的 `input` 类型选项中有重叠占位符名时，token 解析产生错误值，可能导致自动化执行错误任务或操作。
**修改建议**：
```python
def _replace_tokens(self, text: str, values: Dict[str, Any]) -> str:
    import re
    # 按占位符长度降序排列，先替换长占位符避免子串冲突
    sorted_tokens = sorted(values.items(), key=lambda kv: len(kv[0]), reverse=True)
    result = text
    for token, replacement in sorted_tokens:
        placeholder = re.escape("{" + token + "}")
        result = re.sub(placeholder, str(replacement), result)
    return result
```
**优先级**：High — 可能导致 pipeline override 数据损坏，触发错误的自动化行为。

### [N8] `_tasks_loaded` 在 for 循环中异常被静默吞掉后仍设为 True
**位置**：`src/core/service/maa_end/runtime.py:155-157`
```python
for json_path in tasks_root.rglob("*.json"):
    ...
    except Exception as e:  # pragma: no cover
        self.logger.debug(LogCategory.MAIN, "加载任务定义失败", path=str(json_path), error=str(e))
        # 静默继续，不中断循环
self._tasks_loaded = True  # ← 即使有 JSON 加载失败也设为 True
```
**根因**：异常在循环内部被 `except Exception` 捕获并记录为 `debug` 级别，循环继续。但循环结束后立即设置 `_tasks_loaded = True`，无论是否有加载失败。如果某 JSON 文件格式损坏导致部分任务丢失，调用方无法区分"完整加载"和"部分加载"。
**影响面**：任务列表不完整，GUI 的任务列表或 CLI 的 `metadata list` 缺少某些任务。
**修改建议**：
1. 在循环结束后检查是否有异常发生（记录异常计数），有异常则标记为部分加载
2. 或将 `_tasks_loaded = True` 改为 `_tasks_loaded = all_ok`，并在 `tasks()` 方法中如果未完全加载则重试
**优先级**：Medium — 静默数据丢失。

### [N11] `_apply_preset_to_queue` 和 `_add_to_queue`（预设分支）选项合并逻辑的微妙差异

### [R1] `get_cache_subdir()` 路径遍历漏洞

**文件**：`src/core/foundation/paths.py:34-45`  
**严重程度**：High

`get_cache_subdir(name)` 使用 `get_cache_dir() / name` 构造路径并调用 `mkdir(parents=True)`。Path `__truediv__` 不消除 `..` 组件，因此 `get_cache_subdir("../../etc")` 解析到项目根目录上两层之外的目录并尝试创建。

当前 3 个调用点（`android_runtime.py:668-669`、`scene_service.py:33`）均传入字面量，不直接暴露攻击面，但 `get_cache_subdir` 是公开 API，任何未来传入用户/配置派生字符串的调用点都会触发。

**修复建议**：在函数内验证最终路径在 `get_cache_dir()` 下，拒绝 traversal：
```python
def get_cache_subdir(name: str) -> Path:
    p = get_cache_dir() / name
    try:
        p.relative_to(get_cache_dir())
    except ValueError:
        raise ValueError(f"Path traversal detected: {name}")
    p.mkdir(parents=True, exist_ok=True)
    return p
```

### [R3] `get_logger` 惰性初始化存在线程竞争

**文件**：`src/core/foundation/logger.py:166-171`  
**严重程度**：Low

```python
def get_logger(name=None):
    global _logger_initialized
    if not _logger_initialized:
        init_logger()          # ← check-then-act, 无锁
    return ProjectLogger(logging.getLogger(name))
```

当 GUI 并行启动（设计文档 `superpowers/specs/2026-07-09-gui-parallel-startup-design.md` 明确提到多线程初始化）时，多个线程可能同时看到 `_logger_initialized == False`，同时调用 `init_logger()`。`init_logger()` 调用 `root_logger.handlers.clear()` 再重新添加 handler，导致日志 handler 被重复清空/重建，出现日志丢失或写入竞争。

**修复建议**：添加线程锁保护初始化：
```python
_init_lock = threading.Lock()
def get_logger(name=None):
    if not _logger_initialized:
        with _init_lock:
            if not _logger_initialized:
                init_logger()
    return ...
```

### [R4] `logger.py` `_format` 对 `kwargs` 做 `pop` 但方法签名接收的是 `**kwargs`

**文件**：`src/core/foundation/logger.py:37-43`  
**严重程度**：Info/Design

```python
def _format(self, args, kwargs):
    logging_kwargs = {}
    for key in ("exc_info", "stack_info", "stacklevel", "extra"):
        if key in kwargs:
            logging_kwargs[key] = kwargs.pop(key)   # ← 修改了调用方的 kwargs dict
```

实际上每个 wrapper 方法调用 `_format(args, kwargs)` 时，`kwargs` 是 `**kwargs` 解包后**新构造的局部 dict**，因此 `pop` 不会影响调用方的字面量。但此行为依赖 Python 的 `**kwargs` 解包语义，代码读者可能误以为 `pop` 影响了外部状态。当前无 bug，但可读性差，且 `ProjectLogger.__getattr__` 透传未经过 `_format` 的 logger 方法（如 `logger.addHandler()`）不会触发 pop。

**修复建议**：用 `kwargs.get(key)` 替代 `kwargs.pop(key)`，或注释说明 `kwargs` 是局部副本。

### [R14] `task_loader.py` `tasks()`/`presets()` 惰性加载存在竞态

**文件**：`src/core/capability/element_recognition/tasks/task_loader.py:106-114`

```python
def tasks(self):
```

多线程首次调用 `tasks()` 时，两个线程可能同时进入 `load_all_tasks()`，重复进行磁盘 I/O，且最后一个写入的 `self._tasks` 胜出。虽然 Python GIL 防止 dict 损坏，但加载结果可能丢失其中一个线程加载的部分数据（如果目录在两次调用之间被修改）。

```python
def tasks(self):

                return self.load_all_tasks()
    return self._tasks
```

### [R16] `recovery.py` `_run` 硬编码 30 秒超时

```python
def _run(self, args: list[str], serial: Optional[str]) -> None:
```
