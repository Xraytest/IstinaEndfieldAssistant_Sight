### [SRV-01 Info] `service/runtime.py:473-477` — `save_config` 使用非原子写入

```python
def save_config(self) -> None:
    path = self._resolve_config_path()
    path.parent.mkdir(parents=True, exist_ok=True)
    with open(path, "w", encoding="utf-8") as f:
        json.dump(self._config, f, ensure_ascii=False, indent=2)
```

**问题**：直接写入目标文件，无 `.tmp` + `os.replace()` 原子写入。若写入过程中进程崩溃、断电或磁盘满，文件可能处于半写入状态（部分 JSON 内容已写入但未完成），导致后续 `_load_config` 抛出 `JSONDecodeError` 并回退到空配置 `{}`。

**对比**：项目内已有两处正确实现了原子写入：
- `settings_page.py` `_save_settings` 使用 `mkstemp` + `os.replace()`
- `queue_state.py` `persist()` 使用 `.tmp` → `os.replace()`

`runtime.py` 的 `save_config` 是唯一仍使用非原子写入的配置持久化入口。

**影响面**：低——配置写入频率低（用户修改设置时触发），且文件体积小。但一旦损坏，`_load_config` 回退到 `{}` 意味着所有配置（设备地址、LLM 参数等）丢失，用户体验严重下降。

**建议**：使用 `mkstemp` + `os.replace()` 原子写入：

```python
import tempfile, os

def save_config(self) -> None:
    path = self._resolve_config_path()
    path.parent.mkdir(parents=True, exist_ok=True)
    data = json.dumps(self._config, ensure_ascii=False, indent=2)
    fd, tmp_path = tempfile.mkstemp(dir=str(path.parent), suffix=".tmp")
    try:
        with os.fdopen(fd, "w", encoding="utf-8") as f:
            f.write(data)
            f.flush()
            os.fsync(f.fileno())
        os.replace(tmp_path, path)
    except Exception:
        if os.path.exists(tmp_path):
            os.unlink(tmp_path)
        raise
```

### [TASK-01 Low] `task_runner.py:89-90` — switch 类型 falsy 值强制转换 bug

```python
# task_runner.py:87-100
if opt_type == "switch":
    case_name = "Yes" if value else "No"
    for case in cases:
        if case.get("name") == case_name:
            result.update(case.get("pipeline_override") or {})
            return result
    default_case = opt_def.get("default_case")
    if default_case:
        for case in cases:
            if case.get("name") == default_case:
                result.update(case.get("pipeline_override") or {})
                return result
```

**问题**：`case_name = "Yes" if value else "No"` 使用 Python 的 falsy 值隐式转换。当 `value=0`（整型）时，`bool(0)` 为 `False`，`case_name` 被设为 `"No"`。但当 `value=""`（空字符串）时，同样被设为 `"No"`。当 `value=False`（布尔型）时，正确设为 `"No"`。

**关键 bug**：当 `value=0` 且任务选项期望整型值（如数量、层级）时，`0` 被当作 falsy 值强制转换为 `"No"`，导致选项被错误地关闭。例如：`{"option": {"repeat_count": 0}}` 会被当作 "关闭重复" 处理，而非 "重复 0 次"。

更严重的情况：当 `value=[]`（空列表）或 `value={}`（空字典）时，同样被转换为 `"No"`，可能错误地关闭期望集合类型的选项。

**影响面**：中——预设配置中若包含值为 `0` 的 switch 选项（如"使用 0 个加速道具"），会被错误关闭。但 MaaEnd 任务配置通常使用布尔值作为 switch 选项，实际触发概率较低。

**对比**：批次 36 S08 报告了 `if value is None: continue` 导致选项被跳过的问题，但未涉及 falsy 值的强制转换。本批次为此场景的补充发现。

**建议**：显式检查 `None` 而非依赖 falsy 转换：

```python
if opt_type == "switch":
    case_name = "Yes" if value is not None and value else "No"
    # 或更严格：
    case_name = "Yes" if isinstance(value, bool) and value else "No"
```

### [DEVICE-02 Info] `touch_manager.py:53-56` — `back()` 无异常处理

```python
def back(self, serial: Optional[str] = None) -> None:
    import subprocess
    args = self._adb_base_args(serial) + ["shell", "input", "keyevent", "KEYCODE_BACK"]
    subprocess.check_output(args, timeout=10)
```

**问题**：`subprocess.check_output()` 在以下情况会抛出异常且无捕获：
1. ADB 未连接 → `FileNotFoundError` 或 `subprocess.CalledProcessError`
2. 设备无响应 → `subprocess.TimeoutExpired`
3. ADB 返回非零退出码 → `subprocess.CalledProcessError`

对比同一文件中的 `tap`、`swipe`、`long_press` 方法，它们都使用了 `try/except` 包裹并记录了日志：

```python
def tap(self, x: int, y: int, serial: Optional[str] = None) -> None:
    try:
        self._tap_adb(x, y, serial)
    except Exception as e:
        self._logger.error(LogCategory.EXECUTION, "点击失败", ...)
        raise
```

但 `back()` 缺少同样的保护，是文件内唯一裸调 `subprocess` 的方法。

**影响面**：低——`KEYCODE_BACK` 通常在正常流程中成功。但当 ADB 连接中断时，异常会直接传播到调用方，可能导致任务执行流程崩溃。

**建议**：添加与 `tap`/`swipe` 一致的异常处理：

```python
def back(self, serial: Optional[str] = None) -> None:
    try:
        import subprocess
        args = self._adb_base_args(serial) + ["shell", "input", "keyevent", "KEYCODE_BACK"]
        subprocess.check_output(args, timeout=10)
    except Exception as e:
        self._logger.error(LogCategory.EXECUTION, "返回键失败", error=str(e))
        raise
```
