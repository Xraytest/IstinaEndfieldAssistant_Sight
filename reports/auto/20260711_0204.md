### [AGT-01] `_next_pending` 依赖循环无检测，循环依赖任务永远不可执行

**优先级：Medium**
**位置**：`orchestrator.py:90-94`

```python
def _next_pending(tasks: List[Task]) -> Optional[Task]:
    ready = [t for t in tasks if t.status == "pending" and all(d in [x.id for x in tasks if x.status == "done"] for d in t.dependencies)]
    if not ready:
        return None
    return sorted(ready, key=lambda t: t.priority)[0]
```

**根因**：
1. 依赖完成判断为 `d in [x.id for x in tasks if x.status == "done"]`——若 A 依赖 B、B 依赖 A，两者均为 pending，列表为空，`ready` 为空，返回 `None`。调用方 `cmd_reserve` 静默返回"无待处理任务"，**不提示循环依赖**。
2. 内嵌列表推导 `[x.id for x in tasks if x.status == "done"]` 对每个 pending 任务的每个依赖重复执行，O(n²) 复杂度；队列增大时性能退化。

**影响面**：
- `implement-001` 依赖 `audit-001`（当前队列已体现），若后续新增 `audit-003` 依赖 `implement-002`、`implement-002` 依赖 `audit-003` 即形成循环，两者永远卡在 pending。
- 开发者看到"无待处理任务"但实际有卡住的依赖链，难以诊断。

**修改建议**：
1. 增加循环依赖检测：
   ```python
   def _detect_cycle(tasks: List[Task]) -> Optional[List[str]]:
       done_ids = {x.id for x in tasks if x.status == "done"}
       pending = [t for t in tasks if t.status == "pending"]
       dep_map = {t.id: set(t.dependencies) - done_ids for t in pending}
       # DFS cycle detection
       visited, path = set(), []
       def visit(node):
           if node in path:
               return path[path.index(node):]
           if node in visited:
               return None
           visited.add(node)
           path.append(node)
           for dep in dep_map.get(node, []):
               if dep in dep_map:
                   result = visit(dep)
                   if result:
                       return result
           path.pop()
           return None
       for t in pending:
           cycle = visit(t.id)
           if cycle:
               return cycle
       return None
   ```
2. 或简化：预计算 `done_ids = {x.id for x in tasks if x.status == "done"}` 一次，避免内嵌推导。

### [AGT-02] `_parse_queue` 正则对描述中含冒号的任务解析失败

**优先级：Medium**
**位置**：`orchestrator.py:40-46`

```python
m = re.match(r"^- `([^`]+)`:", line)
if not m:
    continue
task_id = m.group(1)
desc_m = re.search(r":\s*(.*?)\s*\[priority=", line, re.DOTALL)
desc = desc_m.group(1).strip().strip('"') if desc_m else ""
```

**根因**：
`desc_m = re.search(r":\s*(.*?)\s*\[priority=", line, re.DOTALL)` 使用 `(.*?)` 非贪婪匹配从**第一个** `:` 到 `[priority=` 之间的内容。当前队列格式为：
```
- `research-001`: "搜集 Arknights 终末地官网/宣传图 UI 截图与设计分析" [priority=1, status=pending]
```

描述中的冒号在引号内，正则从第一个 `:`（即 `research-001` 后的冒号）开始匹配到 `[priority=`，**截断描述**。例如描述为 `"分析颜色规范: 主色与辅助色"` 的任务，解析后 `desc` 变为 `"分析颜色规范`（丢失冒号后内容）。

**影响面**：
- 任何描述含冒号的 enqueue 任务在 `_parse_queue` 后截断，`_write_queue` 再写回时丢失原始描述内容。
- 当前 7 行队列中无此情况，但 `cmd_enqueue` 允许任意描述文本，后续任务可能触发。

**修改建议**：
改为从**最后一个** `:`（即 task_id 后的冒号）开始匹配：

```python
desc_m = re.search(r":\s*(.*?)\s*\[priority=", line, re.DOTALL)
# 改为匹配最后一个冒号之后的内容
last_colon = line.rfind(":", 0, line.find("[priority="))
if last_colon >= 0:
    desc = line[last_colon + 1:line.find("[priority=")].strip().strip('"')
```

或更稳健的方式：将队列改为 JSON 格式，避免正则 fragility。

### [AGT-03] `task_id.split("-", 1)[0]` 对非标准 task_id 解析错误

**优先级：Low**
**位置**：`orchestrator.py:61`

```python
type = task_id.split("-", 1)[0]
```

`cmd_enqueue` 生成 `task_id = f"{task_type}-{idx:03d}"`（如 `research-001`），`split("-", 1)[0]` 取 `research` 匹配 `roles/research.md`。但：
1. 若 `task_type` 本身含连字符（如 `gui-audit`），`split` 取 `gui`，`roles/gui-audit.md` 不存在，回退到默认 prompt。
2. 若 `task_id` 被手动编辑为不含连字符（如 `audit001`），`split` 返回 `audit001`，同样回退。

当前队列和 enqueue 逻辑保证格式，但 `_parse_queue` 从文件读取的 `task_id` 可能被手动修改。

**修改建议**：增加 task_id 格式校验，或改为从 `[priority=...]` 前的描述推断类型（已有 `task.type` 字段）。

```python
def _write_queue(tasks: List[Task]) -> None:
```

### [AGT-05] `_parse_queue`/`_write_queue` 无锁并发写入风险

**优先级：Low**
**位置**：`orchestrator.py:30-68, 71-78`

`_write_queue` 直接 `write_text` 覆盖文件，无任何文件锁。若两个进程同时 `cmd_enqueue`，后写入者覆盖前者，丢失一个任务条目。

**影响面**：低——当前 orchestrator 为单进程 CLI 工具，多进程并发场景罕见。

**修改建议**：增加 `fcntl`/`msvcrt` 文件锁，或至少写临时文件后 `os.replace` 实现原子写入。

### [AGT-06] `_build_prompt` 硬编码自动 git commit/push 指令

**优先级：Low**
**位置**：`orchestrator.py:120-122`

```python
constraints = [
    "After changes: git add -A && git commit -m "feat(gui): ..." && git push",
    "Update docs/TASK_LOG.md with the change",
]
```

prompt 中直接嵌入 `git add -A && git commit && git push` 命令序列。子 agent 执行时：
1. `git add -A` 可能意外暂存非 GUI 文件的修改（如 `sitecustomize.py` 改动）。
2. commit message 为硬编码模板，可能不反映实际变更内容。
3. 子 agent 若对项目不熟悉，可能误修改 `src/core/` 文件后仍按模板提交。

**修改建议**：改为更精确的指令：
```
After changes: git add <only the modified file paths> && git commit -m "feat(gui): <describe the specific change>"
```
并在 constraints 中明确列出禁止触碰的目录。

## 2. SKILL.md 问题

### [AGT-07] SKILL.md 列出的必读文档包含不存在的文件

**优先级：Low**
**位置**：`.agents/skills/read-and-log-docs/SKILL.md:15-21`

```yaml
MANDATORY: Conversation Start Rule

每次与该项目的用户开始对话时，必须先读取以下全部文件：

- `CLAUDE.md`
- `docs/ARCHITECTURE.md`
- `docs/RUNTIME_DEVICE_AND_MAAEND.md`
- `docs/GUI_CLI_AND_AUTOMATION.md`
- `docs/LLM_AND_NAVIGATION.md`
- `docs/RECOGNITION_PIPELINE_AND_TASKS.md`
- `docs/CODE_QUALITY_AND_CLEANUP.md`
- `docs/WORKFLOW.md`
```

**根因分析**：
逐项检查各文件存在性（经 `ls` 验证）：
- `CLAUDE.md` ✓ 存在（项目根目录）
- `docs/ARCHITECTURE.md` ✓ 存在（12.5K）
- `docs/RUNTIME_DEVICE_AND_MAAEND.md` ✗ **不存在**
- `docs/GUI_CLI_AND_AUTOMATION.md` ✗ **不存在**
- `docs/LLM_AND_NAVIGATION.md` ✗ **不存在**
- `docs/RECOGNITION_PIPELINE_AND_TASKS.md` ✗ **不存在**
- `docs/CODE_QUALITY_AND_CLEANUP.md` ✗ **不存在**
- `docs/WORKFLOW.md` ✓ 存在（902B）

8 个必读文件中 **5 个不存在**。SKILL.md 要求在每次对话开始时读取这些文件，但文件缺失会导致 agent 报错或跳过读取，违反 skill 自身的强制规则。CLAUDE.md 存在但未被 SKILL.md 列为"项目文档"（列为独立条目），造成文档覆盖范围的语义混淆——CLAUDE.md 实际包含了项目架构的全部关键信息，而 SKILL.md 试图通过 6 个分拆文档覆盖相同范围但仅实现了 2 个。

**修改建议**：
1. 将 SKILL.md 的必读文件列表更新为实际存在的文件。
2. 或将 `docs/*.md` 改为 glob 模式自动发现：`ls docs/*.md` 然后逐一读取。
3. 如相关文档尚未编写，优先补写缺失的架构文档。
