### [DATA01 Medium] `element_info.py:67-68` `__post_init__` 静默篡改 `action` 字段，掩盖调用方数据错误

**位置**：`src/core/capability/element_recognition/element_info.py:64-73`

```python
def __post_init__(self) -> None:
    if self.element_type not in ELEMENT_TYPES:
        self.element_type = "unknown"
    if self.action not in ("tap", "swipe", "none", "unknown"):
        self.action = "unknown"           # ← 静默篡改
    self.confidence = max(0.0, min(1.0, float(self.confidence)))
    x1, y1, x2, y2 = self.bbox
    self.bbox = (float(x1), float(y1), float(x2), float(y2))
    cx, cy = self.center
    self.center = (float(cx), float(cy))
```

**根因分析**：
`__post_init__` 对 `action` 字段执行**静默截断**——任何不在白名单 `("tap", "swipe", "none", "unknown")` 中的值（包括合法的 `"click"`、`"long_press"`、`"keyevent"` 等）被强制替换为 `"unknown"`，且无任何日志或警告。

**影响面**：
- 高——所有 5 种后端（TemplateBackend、OCRBackend、ColorBackend、YOLOBackend、SceneGeometryAnalyzer）在构造 `ElementInfo` 时，若 `action` 字段填写了白名单外的合法值（如 `"click"`、`"input"`、`"wait"`），会被静默替换为 `"unknown"`。
- 下游消费方（`pipeline_runner.py`、`task_runner.py`）根据 `action` 字段决定执行策略——`"unknown"` 可能导致操作被跳过或降级。
- 调用方无法区分"原始值为 unknown"和"原始值被截断"，数据溯源链断裂。

**对比**：`element_type` 的截断（line 65-66）是合理的（`ELEMENT_TYPES` 是封闭集合），但 `action` 的白名单过于狭窄——项目实际使用的 action 值远不止 4 个。

**修改方案**：
```python
# 方案1：扩大白名单或移除截断
ALLOWED_ACTIONS = ("tap", "swipe", "none", "unknown", "click", "long_press", "keyevent", "input", "wait")
if self.action not in ALLOWED_ACTIONS:
    self._logger.warning("Unknown action=%r, falling back to 'unknown'", self.action)
    self.action = "unknown"

# 方案2：完全移除截断，允许任意 action 值
# （下游消费方自行验证）
```

### [DATA02 Low] `element_info.py:92-104` `SceneAnalysis3D` 无 `__post_init__` 类型强制，反序列化后字段类型不可控

**位置**：`src/core/capability/element_recognition/element_info.py:91-104`

```python
@dataclass
class SceneAnalysis3D:
    annotations: Any = None
    rendered_image: Optional[NDArray[Any]] = None
    rendered_image: Optional[NDArray[Any]] = None
    raw_text: str = ""
    raw_tool_calls: List[Dict[str, Any]] = field(default_factory=list)
    usage: Dict[str, Any] = field(default_factory=dict)
    ground: Dict[str, Any] = field(default_factory=dict)
    entities: List[Dict[str, Any]] = field(default_factory=list)
    camera: Dict[str, Any] = field(default_factory=dict)
    summary: Dict[str, Any] = field(default_factory=dict)
    metadata: Dict[str, Any] = field(default_factory=dict)
```

**根因分析**：
`SceneAnalysis3D` 未定义 `__post_init__`，与 `ElementInfo` 和 `PageInfo` 形成对比。这意味着：

1. **从 JSON 反序列化时**（如 `SceneAnalysis3D(**json.loads(data))`），所有字段保持 JSON 原始类型——`bbox` 为 `list` 而非 `tuple`，`confidence` 为 `int` 而非 `float`。后续 OpenCV 操作（如 `cv2.rectangle`）期望 tuple/numpy 类型，可能引发 `TypeError`。
2. **`annotations: Any = None`** 类型声明过于宽松——实际应接收 `AnnotationSet` 实例（`scene_geometry.py:110-114` 构造时传入 `AnnotationSet(annotations=..., raw_text=..., raw_tool_calls=[])`）。
3. **`entities: List[Dict[str, Any]]`** 与 `scene_geometry.py` 的 `_build_entity` 返回类型一致，但无强制校验。

**影响面**：低——当前 `SceneAnalysis3D` 仅在 `scene_geometry.py` 和 `scene_service.py:144` 中使用，且均由代码构造（非 JSON 反序列化）。但若未来接入持久化或网络传输，类型不一致会导致隐藏 bug。

**修改方案**：
```python
def __post_init__(self) -> None:
    # 强制类型转换，与 ElementInfo/PageInfo 对齐
    if self.annotations is None:
        self.annotations = AnnotationSet()
    if not isinstance(self.entities, list):
        self.entities = []
    if not isinstance(self.ground, dict):
        self.ground = {}
    # ... 其余字段
```

### [DATA03 Low] `annotation.py:14/25` `Annotation.points` 与 `AnnotationShape.pts` 字段命名不一致

**位置**：`src/core/capability/element_recognition/annotation.py:14` 和 `:25`

```python
# Annotation (batch 45 报告中标注为 A1 Info)
class Annotation:
    label: str
    shape_type: str
    points: list[tuple[int, int]]    # ← "points"
    confidence: float = 0.0
    metadata: dict[str, Any] = field(default_factory=dict)

# AnnotationShape (scene_geometry.py 构造)
class AnnotationShape:
    label: str
    shape_type: str
    pts: list[tuple[int, int]]       # ← "pts"（缩写）
    color: tuple[int, int, int]
    confidence: float = 0.0
    metadata: dict[str, Any] = field(default_factory=dict)
```

**根因分析**：
两个数据类表示同一概念（标注点集合），但使用不同字段名：`Annotation.points` vs `AnnotationShape.pts`。这导致：
1. **代码可读性差**：读者需要记忆两种命名约定。
2. **转换代码冗余**：`scene_geometry.py` 构造 `Annotation` 时使用 `points=self._ground_polygon_points(...)`，但若需转换为 `AnnotationShape`，需手动映射 `pts=annotation.points`。
3. **潜在 bug**：若有人误写 `annotation.pts`（实际为 `AnnotationShape` 的字段），对 `Annotation` 实例会抛 `AttributeError`。

**影响面**：低——两个类目前无直接转换代码，但命名不一致是结构性隐患。

**修改方案**：
```python
# 统一为 "points"
class AnnotationShape:
    label: str
    shape_type: str
    points: list[tuple[int, int]]    # ← 与 Annotation 对齐
    ...
```

### [DATA04 Info] `element_info.py:25` `PAGE_TYPES` 包含 `"credit_shop"` 但项目实际使用 `"CreditShopping"` 命名

**位置**：`src/core/capability/element_recognition/element_info.py:28-48`

```python
PAGE_TYPES = (
    "world_map",
    "base_hub",
    ...
    "credit_shop",      # ← snake_case
    ...
)
```

**根因分析**：
`PAGE_TYPES` 中 `"credit_shop"` 使用 snake_case，但其他页面类型（如 `"world_map"`、`"base_hub"`、`"quest_panel"`）也使用 snake_case，仅 `PageInfo.__post_init__` 的校验逻辑基于此列表。实际页面识别结果可能来自 `recognizer.py` 的 `_score_page`，其输出为 `catalog_entry` 的 key（可能是 camelCase 或 PascalCase，取决于 catalog JSON 配置）。

**影响面**：Info——若 catalog JSON 中页面类型为 `"CreditShopping"`（PascalCase），`PageInfo.__post_init__` 会将其截断为 `"unknown"`，导致页面识别结果丢失。

### [DATA05 Info] `element_info.py:92-104` `SceneAnalysis3D` 字段顺序与 `scene_geometry.py:109-132` 构造调用不匹配

**位置**：
- `element_info.py:92-104` 字段声明顺序
- `scene_geometry.py:109-132` 构造调用

```python
# element_info.py 声明顺序：
annotations, rendered_image, raw_text, raw_tool_calls, usage, ground, entities, camera, summary, metadata

# scene_geometry.py:109-132 构造顺序：
SceneAnalysis3D(
    annotations=AnnotationSet(...),    # 第1个
    rendered_image=rendered,            # 第2个
    raw_text=raw_text,                  # 第3个
    raw_tool_calls=[],                  # 第4个
    usage={},                           # 第5个
    ground=ground,                      # 第6个
    entities=entities,                  # 第7个
    camera={...},                       # 第8个
    summary=summary,                    # 第9个
    metadata={...},                     # 第10个
)
```

**根因分析**：
`SceneAnalysis3D` 使用 `@dataclass` 且所有字段都有默认值，构造时使用关键字参数。字段声明顺序与实际构造顺序一致（均为 10 个字段），**无 bug**。但 dataclass 的字段顺序对 `asdict()` 序列化有影响——若未来需要 JSON 序列化，字段顺序与预期不一致可能导致解析错误。

**影响面**：Info——当前代码正确，但属于 API 设计隐患。若新增字段时未注意顺序，可能破坏序列化兼容性。
