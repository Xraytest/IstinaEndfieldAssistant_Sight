## 二、对既往报告的审计纠正（深入分析）

### A1 纠错 — 批次 2350 `N4`「反斜杠绕过 shell 黑名单」论断不成立

**原论断**：`android_runtime.py` `_is_allowed_shell_cmd` 的 `$` 字符级黑名单可被 `dumpsys meminfo\$(id)` 这类反斜杠转义绕过，实现命令注入。

**深入分析（经源码核对 `android_runtime.py:87-97`）**：
黑名单检查为：
```python
if any(ch in stripped for ch in (";","|","&","`","$","(",")","{","}","<",">","\n","\r")):
    return False
```
该检查作用于**传入的字符串字面量**。`dumpsys meminfo\$(id)` 字面量中**仍包含 `$` 字符**（`\` 并不删除 `$`，它只是多了一个 `\` 字符；而 `\` 本身不在黑名单元组中）。因此 `any(...)` 对 `$` 命中 → 返回 `False` → 命令被拒绝。反斜杠**无法**移除 `$`，也就**无法**绕过字符级黑名单。

故 N4 的"利用性"描述**不成立**：守护进程执行层对 `$` 的字符级拦截在 Python 侧即生效，命令替换根本不会下达到设备 shell。

**结论**：将 N4 的利用性表述降级/删除，仅保留其**另一半合法关切**——`$` 字符被无差别拦截会误杀 `dumpsys meminfo $PID`、`echo $PATH` 等合法诊断命令（即 2350 批次 `K2`，属 UX/可用性，非安全漏洞）。

### A2 纠错 — 批次 0020 `D2`「CLI shell 命令注入」严重度需下调

**原论断**：`_handle_shell` 绕过 `android_runtime.py` 白名单，CLI `istina shell <cmd>` 存在设备端命令注入（High）。

**深入分析**：批次 0020 报告撰写时，白名单校验器 `_is_allowed_shell_cmd` 尚未在调用链上接线。但当前代码中（见本批 §1.2 D2 行号 `android_runtime.py:613-619`），守护进程 `shell` RPC 分支统一调用 `_is_allowed_shell_cmd`，即**无论调用方是谁，所有经 RPC 的 shell 命令都过白名单**。`_handle_shell` 虽"直接传递"，但其下游执行点已强制校验，用户态 `istina shell` 输入无法逃逸白名单。

**结论**：D2 作为"用户可注入"的安全漏洞**已不成立**，下调为"已缓解/设计受约束"。若仍要记录，应改为 Low 级别的"白名单过于严格导致合法诊断命令不可用（K2）"，而非命令注入高危。

### A3 纠错 — 批次 234853/2350 `_evaluate_or`「误导日志」论断已过时

**原论断**：`_evaluate_or`（`pipeline_runner.py`）日志称 "treating as non-match" 但实际执行 `continue` 而非 `return None`，日志与行为不符。

**深入分析（核对 `pipeline_runner.py:301-313`）**：当前 `_evaluate_or` 实现为：
```python
for sub_name in sub_names:
```

## 三、补充观察（已知高危项的调用链深化，非新增独立严重项）

### O1 — W1 的"三重静默"失败链（深化 W1 机理）
W1 之所以危害极大，不仅在于字母键被拒，更在于失败**完全不可观测**：
1. 守护进程 `keyevent` 分支返回 `{"error": "invalid keyevent: 'w'"}`，但 `AndroidRuntime.keyevent`（`android_runtime.py:784-786`）只 `return response.get("result", "")`，**丢弃 error 字段、不抛异常、返回空串**。
2. `runtime._vlm_keyevent`（`runtime.py:712-730`）的 `try/except` 永远捕获不到异常，仅对真正抛异常的情况记 warning，**对"返回空串"无记录**。
3. `VlmWalkNavigator._execute_action`（`vlm_walk_navigator.py:264-280`）调用 `input_fn` 后不检查返回值，直接 `history.append({**action, ...})`，把"未执行的动作"记为"已执行"。

→ 结果：VLM 认为自己在移动、日志显示动作已发出，但设备上零位移，直到 stuck 检测触发才 fallback。建议修复时**至少**让 `keyevent()` 在守护进程返回 error 时抛异常或返回错误码，并在 `_vlm_keyevent` 记录告警，使失败可见。

### O2 — C10 的触发条件澄清（深化 2345-C10）
C10 并非"永远传 None"：门面 `_llm_client_instance` 属性（`runtime.py:129-132`）在首次访问时会创建 `LlmClient` 并回填 `self._llm_client`。由于 CLI/GUI 的 `llm chat` 走 `_llm_client_instance`（`runtime.py:878`），**只要用户先发过任意 LLM 请求，`_llm_client` 即被填充**，随后 `nav3 walk` 才能用到真实 VLM。因此 C10 的实战触发面是"启动后直接 `nav3 walk` 而未先 `llm chat`"这一顺序——属初始化顺序缺陷，修复成本极低（将 `runtime.py:697` 的 `self._llm_client` 改为 `self._llm_client_instance` 即可），建议优先于 W1 的架构级修复先行落地。
