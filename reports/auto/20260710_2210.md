## 0. 边界与需求简述（用于界定"错误"）

## 1. 新发现的代码错误（优先）

### [H1] scrcpy 首帧超时阈值过紧 + 失败后永不重试 + 误报成功
**位置**：
- `src/core/capability/device/android_runtime.py:145`（`_ScrcpySession.start` 的 `deadline = time.time() + 8.0`）
- `src/core/capability/device/android_runtime.py:265`（`_decode_loop` 内同样 `deadline = time.time() + 8.0`）
- `src/core/service/runtime.py:223-229`（`IstinaRuntime.connect` 中 `start_scrcpy` 后无条件记 success）

**现象与根因**：
`start()` 在 `_run` 线程内部要依次完成 jar 推送（→设备网络写入，约 1–2s）、`app_process` 启动 server（约 1–3s）、forward 端口等待、socket 握手、codec 协商，**之后**才开始出首帧。两层独立 8s 超时（`start()` 轮询 + `_decode_loop` 握手）都从"线程启动"起算，且被前面的阻塞 ADB 步骤大量消耗。首次连接时首帧几乎必然 >8s，于是 `start()` 抛 `TimeoutError` → 被 `_dispatch` 的 `except Exception` 吞掉并返回 `{"error": ...}`。

更严重的是：
1. `IstinaRuntime.connect()` 调用 `self.android(serial).start_scrcpy(serial=serial)` 后**不检查返回值**，直接 `self._logger.info(... "scrcpy 预览通道启动成功")`，造成"假成功"日志。
2. `_Daemon._dispatch` 的 `startScrcpy` 分支：若 `self._scrcpy_session is not None` 则**直接跳过重新创建**。一旦首次启动失败，该非空引用会永久阻止后续 `connect` 重新拉起 scrcpy；`_run` 线程也已 `finally` 自清理，于是 `get_latest_frame()` 永远返回 `None`，整个会话静默退化为慢速 ADB 截图，且用户无任何提示。

**影响**：预览帧率被压制回 ADB 截图水平（文档 4.x 已分析的 2–4s/帧问题在 scrcpy 路径上"形同虚设"），且连接日志误导。

**修复建议**：
- 将"握手超时"与"等待首帧超时"解耦：`_decode_loop` 的外层 8s 仅用于 socket/握手，首帧等待放到独立、更长的超时（如 20–30s）或放到 `start()` 的轮询里。
- `IstinaRuntime.connect()` 检查 `start_scrcpy` 返回的 dict 是否含 `"error"`，失败则记 `warning` 并**显式置空/清理** daemon 中的 `_scrcpy_session`，允许下次连接重试；同时把日志改为"scrcpy 预览通道启动{成功/失败}"。
- 在 `startScrcpy` 中，若旧 session 的 `_run` 线程已非 alive，应先 `stop()` 再重建，避免死引用阻止重试。

**优先级**：High（影响核心预览体验，且行为静默错误）

### [H2] `MaaEndRuntime.screenshot()` 把只读截图失败误判为"连接断开"
**位置**：`src/core/service/maa_end/runtime.py:811-829`（`screenshot` 方法：screencap 失败与异常两处均 `self._connected = False`）

**现象与根因**：
`screenshot()` 是只读操作，但在 `job.succeeded is False` 与 `except` 两处都会把实例级 `_connected` 置为 `False`。在交互模式长会话中，`IstinaRuntime._screenshot` 优先走 scrcpy，scrcpy 无帧时回退到 `MaaEndRuntime.screenshot()`；而设备瞬时繁忙、MaaFW 偶发 screencap 失败都会令 `_connected=False`。随后任意 `task run` 经 `_ensure_maaend_ready` 看到 `not connected`，触发**整轮 `_connect()` 重连**——一次无害的瞬时截图失败演变成重连风暴或意外重连失败。

**影响**：长会话中偶发截图抖动 → 静默触发全量重连；若重连因设备忙失败，则后续任务被误判失败。

**修复建议**：
- 只读截图不应污染连接态。建议引入独立的 `_preview_available`/最后成功时间戳，或直接返回 `None` 而不改 `_connected`；真正的连接断开只由 `connect()/_connect_once()` 中 `post_connection()` 失败等权威信号决定。
- 若确需"截图连续失败即认为断连"，应由上层（GUI 的 `_preview_fail_count` 阈值机制，见 `main_window.py:336`）统一裁决，而非在底层 runtime 直接翻转。

**优先级**：Medium

## 2. 新发现的逻辑/健壮性问题

### [M1] `_handle_gpu_recommend` 运算符优先级使 4GB 阈值分支被完全吸收
**位置**：`src/cli/handlers.py:677`
```python
if mem and mem >= 4 * 1024 * 1024 * 1024 or mem and mem >= 2 * 1024 * 1024 * 1024:
    recommendation = "GPU"
```
**根因**：`and` 优先级高于 `or`，表达式等价于 `(mem and mem>=4GB) or (mem and mem>=2GB)`。由于 `mem>=4GB` 必然蕴含 `mem>=2GB`，第一个析取项被第二个完全覆盖，4GB 阈值形同虚设；结论实际上只取决于"空闲显存 ≥ 2GB"。若作者本意是分级（≥4GB 强推 / ≥2GB 弱推），则分级逻辑缺失。

**影响**：低（不影响主流程），但属于逻辑冗余/潜在误推荐（显存 2–4GB 的设备也可能被建议开 GPU，而 4B Q4 模型在 2GB 余量下极易 OOM）。

**修复建议**：
```python
if mem and mem >= 4 * 1024**3:
    recommendation = "GPU"
elif mem and mem >= 2 * 1024**3:
    recommendation = "GPU"  # 或 "GPU (low)" 以示区分
else:
    recommendation = "CPU"
```
或直接 `recommendation = "GPU" if mem and mem >= 2 * 1024**3 else "CPU"`。

**优先级**：Low

### [M2] `metadata list` 在启动阶段被重复执行多次
**位置**：
- `maaend_control_page.py:604-616`（`_refresh_task_list`）
- `maaend_control_page.py:631-643`（`_refresh_preset_list`）
- `maaend_control_page.py:1326-1328`（`_do_metadata_load`）

**现象**：缓存为空时，`_refresh_task_list` 与 `_refresh_preset_list` 各自独立调用一次 `_sync_execute("metadata list")`，随后 `_delayed_init → _do_metadata_load` 再调用一次。每次都是一次完整 CLI 子进程启动 + MaaFW `connect()` + `load_tasks/load_presets`（含全量 JSON 扫描）。启动早期最多执行 3 次相同重命令。

**影响**：启动卡顿（每次数百毫秒到数秒，取决于设备/资源规模）；CLI 子进程并发连接同一设备可能产生竞争。

**修复建议**：在 `MaaEndControlPage` 中合并为单次 `metadata list` 拉取后同时刷新任务列表与预设列表；或利用 `_load_metadata_cache()` 的本地缓存，仅在缓存缺失且未在进行中时发起一次。

**优先级**：Low（效率/UX）

## 4. 可提升用户体验的细节点（非错误）

1. **scrcpy 启动失败的可见性**：见 [H1]——失败时 `_connected` 仍被 GUI 标记为 True、连接日志无告警，预览静默退化为 ADB 截图。建议在 `IstinaRuntime.connect()` 依据 `start_scrcpy` 返回值，于设备连接日志明确输出"scrcpy 预览通道启动失败，已回退 ADB 截图"。

2. **预览首帧前的抖动**：`main_window.py:284` 在 `system connect` 成功后立即 `QTimer.singleShot(0, self._refresh_preview)`。此时 scrcpy 常驻通道大概率尚未拿到首帧（见 [H1] 的 8s 窗口），前若干次预览必然走 ADB 回退，造成启动后短暂卡顿。可在 `_refresh_preview` 中区分"scrcpy 尚未就绪"与"设备断开"，前者不计入失败阈值、不告警。

3. **`gpu recommend` 文案**：[M1] 中 2GB 即建议 GPU 可能误导低显存用户，建议在返回结构里附带"建议显存下限"提示或分级文案。
