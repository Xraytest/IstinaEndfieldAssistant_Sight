## 一、跨模块一致性新发现（XC 系列，均为历史未覆盖）

### XC-2 [Medium] `Navigator.list_entities` 在传入 `name_filter` 时静默丢弃 `map_name` / `category`

- **位置**：`src/core/service/navigation/navigator.py:162-199`（`list_entities`）+ `src/core/service/runtime.py:672-679`（`_nav2_list_entities`）。
- **根因分析**：参数优先级为 `if name_filter: ... elif category and map_name: ... elif category: ... elif map_name: ...`。当调用方同时提供 `name`（→`name_filter`）与 `map_name` 时，`name_filter` 分支短路，直接调用 `self._entities.find_by_name(name_filter, limit=limit)`，该查询**完全不按地图过滤**（见 `entity_db.py:125-136`，仅做 `key` 正则子串匹配）。`map_name` 与 `category` 被忽略。
- **影响面**：CLI `nav2 list_entities --name <x> --map_name map01`、GUI 对应调用会返回**所有地图**中名为 `<x>` 的实体，用户预期的"某地图内按名筛选"语义失效；可能返回跨地图同名实体导致后续 `to_entity` 误导航。
- **修改方案**：在 `find_by_name` 之上增加地图/分类后过滤，或拆分查询：
  ```python
  if name_filter:
      matches = self._entities.find_by_name(name_filter, limit=limit * 4)
      if map_name:
          matches = [e for e in matches if e.map_id == map_name]
      if category:
          matches = [e for e in matches if e.category == category]
      matches = matches[:limit]
  ```
- **非期待变化**：`find_by_name` 的 `limit` 语义变为"预取上限"，需在内部适当放大后过滤；若外部依赖"按名返回全地图前 N 条"的既有行为，需评估。

### XC-3 [Medium/Low] minimap 分类器输出词表与 `MapDataLoader._ZONE_MAP` 不一致

- **位置**：`src/core/service/navigation/minimap_locator.py:156-165`（`_parse_tile` 可输出 `map01/map02/dung01/base01`）+ `src/core/service/navigation/map_data_loader.py:58-62`（`_ZONE_MAP` 仅含 `map01/map02/base01`）。
- **根因分析**：`MinimapLocator` 在 `tile_class` 命中 `Dung01*` 时会将 `map_id` 设为 `"dung01"`（第 160-161 行），但 `_ZONE_MAP` 没有 `dung01` 条目，于是 `get_zone_id("dung01")` 回退为 `"dung01_Base"`（第 224-225 行）。若 MaaEnd 资源中不存在名为 `dung01_Base` 的 zone，`Navigator._navigate_navmesh` 中的 `run_pipeline` 将返回 `False`，调用方得到 `status:"error"` 且**无任何关于"未知地图 zone"的明确提示**。
- **影响面**：任何以 `dung01` 为目标的导航（拷贝/副本地下城）将静默失败；`get_map_label("dung01")` 同样回退为原字符串，UI 显示不友好。
- **修改方案**：在 `_ZONE_MAP` 补齐 `dung01`（及未来新增地图），并对 `get_zone_id` 的回退路径增加 `logger.warning("unknown map, fallback zone: %s", ...)`；或在 `to_coords` 入口对 `map_name` 做白名单校验，提前返回"未知地图"错误而非依赖下游 pipeline 失败。
- **非期待变化**：需与 `3rd-part/maaend` 内真实的 zone 命名对齐；若 MaaEnd 侧 zone 命名本就为 `dung01_Base`，则本项为误报，需以运行时数据为准复核（建议加入注释说明契约来源）。

### XC-4 [Low] `IstinaRuntime.navigator()` 缓存的 `screenshot_fn` 在设备切换后变为陈旧引用

- **位置**：`src/core/service/runtime.py:203-209`（`navigator()` 单例构造时 `screenshot_fn=self.android().screenshot`）。
- **根因分析**：`Navigator` 是进程级单例，`_screenshot_fn` 在首次构造时绑定为 `AndroidRuntimeProxy.screenshot`（经由 `__getattr__` 解析到底层默认设备 client 的 bound method）。`AndroidRuntimeProxy.__getattr__` 始终 `_client_for(None)` → 以 proxy 的 `_device_address` 解析 client。若用户在同一 `IstinaRuntime` 生命周期内切换了"当前设备"（如 `last_connected` 变化或显式连接另一 serial），已缓存的 `Navigator` 仍向**旧设备**取帧，`to_coords`/`to_coords_vlm`/`where_am_i` 将基于错误设备的画面做定位与导航。
- **影响面**：多设备场景下的导航子系统；GUI 切换设备后导航结果错乱且难以定位。
- **修改方案**：将 `screenshot_fn` 改为"每次调用时重新解析当前活动设备"的 thunk：
  ```python
  def _nav_screenshot():
      return self.android().screenshot()
  self._nav = _get_navulator()(maaend=self.maaend(), screenshot_fn=_nav_screenshot)
  ```
  或使 `Navigator` 每次 `set_screenshot_fn` 刷新。
- **非期待变化**：thunk 增加一次 proxy 属性解析开销（可忽略）；语义上更正确。

### XC-1 [Low/Design] `IstinaRuntime.execute()` 返回类型弱契约（dict / None / bytes 混合）

- **位置**：`src/core/service/runtime.py:349`（`return None` 未知命令）、`:424-445`（`_screenshot` 返回 `Optional[bytes]`）、`src/cli/handlers.py` 多处 `return runtime.execute(...)`。
- **根因分析**：`execute()` 多数分支返回 `dict(status=...)`，但 (a) 未知命令返回裸 `None`（第 349 行）；(b) `screenshot` 分支返回 `bytes`。下游 `_handle_*` 直接 `return runtime.execute(...)`，最终到达 `istina.py:278` 的 `0 if isinstance(result, dict) and result.get("status")=="success" else 1`。当 `result is None` 时，退出码为 1 但**没有任何错误文案**，把"未知命令"掩盖成"命令执行失败"。
- **影响面**：CLI 诊断用户体验；任何新增 handler 若未自行归一化返回 dict，便继承该脆弱契约。
- **修改方案**：在 `execute()` 顶层统一包一层状态字典——未知命令返回 `{"status":"error","message":"unknown command: <cmd>"}`；`screenshot` 等二进制分支由专属 handler 处理并归一化。降低调用方对返回类型的隐式假设。
- **非期待变化**：需同步检查 GUI `cli_bridge` 对 `execute` 返回值的处理（若存在非 dict 假设）。

### 2.2 复核修正：`memory.md` 中 NAV-05 审计子论断存在不准确表述

`memory.md` 对 0200_nav NAV-05 的修正写道：**"dict 类型 `raw_location` 实际走默认分支不崩；真实崩溃来自字符串/非数字列表的 `ValueError`"**。经复核 `entity_db.py:26-40`（`from_raw`）与 `load()`（第 78-107 行），该论断的"dict 走默认分支不崩"子句**不准确**：

- `rl = raw.get("raw_location", [0, 0, 0])`：当 `raw_location` **键缺失**时返回默认 `[0,0,0]`（安全，不崩）——这是正确的部分；
- 但当 `raw_location` **键存在且为 dict** 时，`rl = dict`，`len(rl) >= 3` 成立，进入 `float(rl[0])` → **`TypeError: 'dict' object is not subscriptable`**，并非"走默认分支"；
- 字符串 / 列表含非数字元素时确实抛 `ValueError`——此部分正确。

此外，`from_raw` 的调用位于 `load()` 的 `try` **之外**（第 78-83 行的 `try` 仅包裹 `json.load(f)`），因此 dict/字符串类异常**不会被 `load()` 捕获**，直接向上传播至 `Navigator.__init__`（第 41 行未包裹 `try`），导致 `IstinaRuntime.navigator()` 构造失败。

> 修正后的准确表述：**缺失** `raw_location` 才走安全默认；**存在但类型错误**（dict→TypeError，str/非数字 list→ValueError）的 `raw_location` 会**未经捕获地中断 Navigator 构造**。原 NAV-05 的"崩溃未被吞"结论正确，但"dict 走默认分支"机制描述错误，应予纠正。

### 2.3 复核确认：VLM 行走导航失效（2320-W1 / 0020-W1）仍未修复

当前 `vlm_walk_navigator.py:264-280`（`_execute_action`）仍向 `input_fn` 下发字母键 `"w"/"a"/"s"/"d"/"q"/"e"/"f"`，经 `IstinaRuntime._vlm_keyevent`（`runtime.py:712-730`）→ `android.keyevent(key)` → 守护进程 `input keyevent <letter>`。Android `input keyevent` 仅接受 KEYCODE 数字或 `KEYCODE_*` 名称，裸字母不是合法 keycode，故该链路在设备端仍静默失效；加之 `runtime.py:697` `_nav3_walk` 传入的 `llm_client=self._llm_client`（原始属性，初始为 `None`，见 2345-C10）在 LLM 实例未被提前触发时恒为 `None`，进一步使 `to_coords_vlm` 直接回退 navmesh。

> 结论：2320-W1 / 0020-W1 诊断准确且**截至当前代码未被修复**；2345-C10（`_nav3_walk` 传 `None` LLM client）同样仍有效，根因在于门面读取了原始属性 `_llm_client` 而非惰性属性 `_llm_client_instance`（跨模块契约错配），修复点应在 `runtime.py:697` 改为 `self._llm_client_instance`。
