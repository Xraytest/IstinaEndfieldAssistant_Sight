### [W1] VLM 行走导航的所有移动指令完全失效（整条 nav3 walk 特性无效）
**优先级：Critical**
**链路位置**：
- `src/core/service/navigation/vlm_walk_navigator.py:264-282`（`_execute_action`）
- `src/core/service/runtime.py:712-730`（`_vlm_keyevent`）
- `src/core/capability/device/android_runtime.py:75-84 / 606-612`（`_is_valid_keyevent` + `keyevent` dispatch）

**根因**：`_execute_action` 将动作映射为**键盘字母**下发：
```python
if act == "forward":   self._input_fn("w", duration)
elif act == "backward":self._input_fn("s", duration)
elif act == "left":    self._input_fn("a", duration)
elif act == "right":   self._input_fn("d", duration)
elif act == "turn_left":  self._input_fn("q", None)
elif act == "turn_right": self._input_fn("e", None)
elif act == "interact":   self._input_fn("f", None)
```
`_input_fn` 实际是 `IstinaRuntime._vlm_keyevent`，其内部 `android.keyevent("w")`。而守护进程 `_dispatch` 的 keyevent 分支强制校验：
```python
def _is_valid_keyevent(key):
    s = str(key).strip()
    if s.isdigit(): return True
    return s in _KNOWN_KEYEVENT_NAMES   # 仅 17 个 KEYCODE_* 常量
```
`"w"/"a"/"s"/"d"/"q"/"e"/"f"` 既非纯数字，也不在白名单内 → 一律返回 `{"error": "invalid keyevent: 'w'"}`。即使忽略白名单，`input keyevent w` 也不是合法 ADB 语法（Android 需要数字 keycode 或 `KEYCODE_W` 形式）。因此每一步移动都被丢弃。

**为何完全静默**：`AndroidRuntimeProxy.keyevent`（`android_runtime.py:784-786`）在收到 error 时**不抛异常**，仅 `return response.get("result", "")`（返回空串）；`_vlm_keyevent` 只有 `try/except`、不检查返回值，故 except 永不触发、无任何告警。结果：`walk_to` 循环空转到 `stuck`/`max_steps`，最终回退 navmesh —— 用户以为 VLM 在走路，实际角色从未移动。

**影响面**：`nav3 walk`、`nav3 to_entity` 两条 CLI 命令、`Navigator.to_coords_vlm/to_entity_vlm` 全部失效；VLM 决策、LLM 调用、截图开销全部白费；仅靠 fallback navmesh 兜底（若 fallback 也失败则彻底无效）。

**修改建议**：在 navigator 层建立「语义动作 → Android keycode」映射，而非透传字母：
```python
_KEY_TO_KEYCODE = {"w":"51","s":"47","a":"29","d":"32","q":"45","e":"33","f":"34"}
# 或使用常量名 KEYCODE_W(51)/KEYCODE_S(47)/KEYCODE_A(29)/KEYCODE_D(32)...
self._input_fn(_KEY_TO_KEYCODE[act], duration)
```
并同步在 `_KNOWN_KEYEVENT_NAMES` 中放行对应常量（若用常量名形式）。**注意**：仅按前次报告建议「放行所有 KEYCODE_*」并不能修复本问题，因为病根在 navigator 发的是字母而非 keycode。

**非期待变化**：修复后按键将真实下发到设备；需评估游戏是否响应 ADB 单次 keyevent（移动类操作通常需持续按压/摇杆，单击 keyevent 可能仍不足以驱动角色，属更深层设计问题，本报告仅修复「指令被丢弃」这一确定错误）。

### [N-1] `ensure_src_path(__file__)` 计算错误项目根路径
**优先级：High**
**位置**：`src/cli/istina.py:37` + `src/core/foundation/paths.py:58-61`

**修改建议**：将 `istina.py:37` 改为 `ensure_src_path()`（无参数），使其使用 `get_project_root()`（从 `paths.py` 自身位置计算，结果正确）。

### [R1] `get_cache_subdir()` 路径遍历漏洞
**优先级：High**
**位置**：`src/core/foundation/paths.py:34-45`

```python
def get_cache_subdir(name: str) -> Path:
    p = get_cache_dir() / name
    p.mkdir(parents=True, exist_ok=True)
    return p
```

`Path.__truediv__` 不消除 `..` 组件，因此 `get_cache_subdir("../../etc")` 解析到项目根目录上两层之外的目录并尝试创建。当前 3 个调用点均传入字面量，不直接暴露攻击面，但 `get_cache_subdir` 是公开 API，任何未来传入用户/配置派生字符串的调用点都会触发。

**修改建议**：在函数内验证最终路径在 `get_cache_dir()` 下：
```python
def get_cache_subdir(name: str) -> Path:
    p = get_cache_dir() / name
    try:
        p.relative_to(get_cache_dir())
    except ValueError:
        raise ValueError(f"Path traversal detected: {name}")
    p.mkdir(parents=True, exist_ok=True)
    return p
```

### [N-7] `ThemeManager` 线程不安全的单例
**优先级：High**
**位置**：`src/gui/pyqt6/theme/theme_manager.py:393-402`

```python
class ThemeManager:
    _instance: Optional["ThemeManager"] = None

    def __new__(cls) -> "ThemeManager":
        if cls._instance is None:
            cls._instance = super().__new__(cls)
        return cls._instance

    @classmethod
    def get_instance(cls) -> "ThemeManager":
        if cls._instance is None:
            cls._instance = cls()
        return cls._instance
```

`__new__` 和 `get_instance` 均检查 `cls._instance is None` 但无锁保护。若两线程同时调用，可能创建两个独立实例，导致主题闪烁或颜色不一致。

**修改建议**：使用 DCL 保护单例创建：
```python
    def __new__(cls) -> "ThemeManager":
        if cls._instance is None:
            with threading.Lock():
                if cls._instance is None:
                    cls._instance = super().__new__(cls)
        return cls._instance
```

### [W2] `_vlm_keyevent` 丢弃 keyevent 的错误返回，失败不可见
**优先级：Medium**
