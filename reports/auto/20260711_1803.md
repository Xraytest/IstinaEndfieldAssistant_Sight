## §1 对既往报告的审计（核心）

### [A1 · 推翻] 批次 64 NEW-MEDIUM（`_attempt_reconnect` 无限重连 / ADB 风暴）—— **假阳性，前提错误**

**批次 64 原论断**：`_attempt_reconnect` 在固定 5 秒定时器下**无限重连无退避**，造成 ADB 风暴、日志刷屏。

**代码复核（实际行为）**：

`device_settings_page.py:198-206`（断开分支）启动定时器 → 定时器触发 `_attempt_reconnect`（:144-152）发出 `system connect` → 该命令完成时进入 `_on_command_finished` 的 `system connect` 分支：

```python
# device_settings_page.py:181-197
elif command.startswith("system connect"):
    ok = bool(result.get("status") == "success")
    ...
    self._connected = ok
    ...
    if ok and serial:
        self._remember_device(serial)
        if self._reconnect_enabled:
            self._reconnect_timer.start()   # 仅成功时重启（连上后 _attempt_reconnect 早退，无害）
        self._pulse_connection_status()
    else:
        self._reconnect_timer.stop()        # ← 失败即停止定时器
```

关键点：

1. **`_attempt_reconnect` 自身从不重启定时器**（它只发命令、写日志、return），定时器的续命完全依赖 `_on_command_finished` 的某个分支 `start()`。
2. **失败分支（:197）调用了 `self._reconnect_timer.stop()`**。设备离线时 `system connect` 通常返回 `status != "success"`（正常结束、非异常），于是 `commandFinished` 触发失败分支 → 定时器被停。
3. 因此「断开 → 重连定时器启动 → 首次重连失败 → 定时器停止」，**自动重连在第一次失败后彻底终止**，根本不存在"每 5 秒无限重连"。

**结论**：批次 64 的"无限重连 / ADB 风暴"前提与代码直接矛盾，属**假阳性**，应从活跃台账移除。其修复补丁（加 max attempts + 指数退避）的**方向可用但理由错误**——真实动机应是"让重连持续进行"，而非"抑制本不存在的风暴"。该论断所暴露的**真正缺陷**见下方 NEW-1（自动重连首次失败后终止，功能失效）。

> 旁证：`_on_command_error`（:217-223）对 connect/disconnect 异常仅重启用按钮、不停止定时器；而 `commandFinished` 的失败分支却停止定时器——两条错误路径行为不一致，进一步说明"失败即停"是人为写入的逻辑错误，而非设计。

### [A2 · 确认] 批次 63 NEW-MEDIUM（`_do_auto_connect` 阻塞主线程 25 秒）—— 严重度偏高，已降 Low

## §2 新发现

### [NEW-1 · Medium · 功能缺陷] 自动重连在首次失败后终止，功能实质失效

**位置**：`device_settings_page.py:197`（`_on_command_finished` 失败分支的 `self._reconnect_timer.stop()`），配合 `:144-152`（`_attempt_reconnect` 不续命定时器）。

**问题**：如 A1 所述，断开后定时器只驱动**一次**重连尝试；一旦该尝试失败（`status != "success"`），定时器被停止，后续不再重试。后果：

- 设备瞬时抖动 / ADB 服务短暂崩溃后恢复（> 5 秒）的用户，连接**永久无法自动恢复**；
- "Auto-reconnect on disconnect" 勾选项名不副实，仅等于"断线后尝试连接一次"。

这与该勾选框的产品语义（持续保活）直接冲突，属功能级缺陷，故评 **Medium**（比被推翻的批次 64 假阳性更值得修复）。

**修复建议**：失败分支**不要停止**定时器，让重连持续；成功分支重置计数。为防真正死循环（如 `commandError` 路径本就不停定时器），引入"最大尝试次数 + 指数退避"护栏：

```python
def __init__(self, bridge, parent=None):
    ...
    self._reconnect_timer = QTimer(self)
    self._reconnect_timer.setInterval(5000)
    self._reconnect_timer.timeout.connect(self._attempt_reconnect)
    self._reconnect_enabled = True
    self._reconnect_attempts = 0
    self._max_reconnect_attempts = 0   # 0 = 无限（带退避）；可设为 12 等上限
    self._reconnect_backoff_base = 5000

def _attempt_reconnect(self) -> None:
    if self._connected or not self._reconnect_enabled:
        return
    serial = self._address_input.text().strip()
    if not serial:
        return
    if self._max_reconnect_attempts and self._reconnect_attempts >= self._max_reconnect_attempts:
        self._reconnect_timer.stop()
        self._append_log(locale.tr("auto_reconnect_give_up", "Auto-reconnect gave up after {n} attempts.").format(n=self._reconnect_attempts))
        return
    self._reconnect_attempts += 1
    interval = min(self._reconnect_backoff_base * (2 ** (self._reconnect_attempts - 1)), 60000)
    self._reconnect_timer.setInterval(interval)
    self._set_connecting_state()
    self._append_log(locale.tr("auto_reconnect_attempt", "Auto-reconnect attempt {n}: {serial}").format(n=self._reconnect_attempts, serial=serial))
    self._bridge.execute("system connect", {"serial": serial})

def _on_command_finished(self, command: str, result: dict) -> None:
    ...
    elif command.startswith("system connect"):
        ok = bool(result.get("status") == "success")
        serial = self._address_input.text().strip()
        self._connected = ok
        self._connection_status.setText(...)  # 原样
        self._selected_device.setText(serial or "-")
        self._append_log(...)
        self._connect_btn.setEnabled(True)
        self._disconnect_btn.setEnabled(True)
        self._refresh_btn.setEnabled(True)
        if ok and serial:
            self._reconnect_attempts = 0
            self._reconnect_timer.setInterval(5000)   # 重置为默认间隔
            self._remember_device(serial)
            if self._reconnect_enabled:
                self._reconnect_timer.start()
            self._pulse_connection_status()
        else:
            # 失败：保持定时器继续重试（仅当重连被关闭时才停）
            if not self._reconnect_enabled:
                self._reconnect_timer.stop()
```

> 注：本修复同时消解了 `_on_command_error` 与 `commandFinished` 失败路径的不一致（失败都应继续重试），并保留批次 64 建议的退避护栏，但动机从"抑制风暴"纠正为"维持重连"。

### [NEW-2 · Low · 健壮性] `settings_page._load_settings` 对 None/非数字配置值 `int()` 崩溃

**位置**：`settings_page.py:164-165`

```python
self._port_input.setValue(int(llm.get("port", 9998)))
self._threads_input.setValue(int(llm.get("threads", 12)))
```

**问题**：`config/llm.port` / `config/llm.threads` 由本页 `_save_settings`（:190-191）写为 `int`，但 `client_config.json` 可能被 `config set`（`handlers.py` 白名单点分键问题见历史 NEW-1）、手动编辑或其他模块写入为字符串/`null`：

- 若值为 `null` → `int(None)` 抛 `TypeError`；
- 若值为非数字字符串（如 `"auto"`）→ `int("auto")` 抛 `ValueError`；
- 二者均**未被捕获**，`_load_settings` 在打开设置页时崩溃，导致整个 `SettingsPage` 无法渲染。

**修复建议**（与"缺失即回退默认"的既有风格一致，参考 `int(llm.get("port", 9998))` 本意）：

```python
def _coerce_int(value, default: int) -> int:
    try:
        return int(value)
    except (TypeError, ValueError):
        return default

self._port_input.setValue(_coerce_int(llm.get("port"), 9998))
self._threads_input.setValue(_coerce_int(llm.get("threads"), 12))
self._preview_interval_spin.setValue(_coerce_int(config.get("preview_interval_ms"), 1500))
```

> 同页 `_read_config`（:221-228）已对 `JSONDecodeError` 弹窗并回 `{}`，此处补齐值级容错即可闭环。属边界健壮性，评 Low。

## §4 用户体验优化点（非缺陷，仅供参考）

- **冗余定时器自启**：`device_settings_page.py:194` 在连接**成功**后仍 `self._reconnect_timer.start()`；因 `_attempt_reconnect`（:145）在 `_connected=True` 时早退，这仅造成每 5 秒一次空 `system connect` 入队（被 CLI 桥接单进程序列化）。建议连接成功后停止重连定时器（`if self._reconnect_enabled and not self._connected: self._reconnect_timer.start()`），减少无效 CLI 调用。低优先。
