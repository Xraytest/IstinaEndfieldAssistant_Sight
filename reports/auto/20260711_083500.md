### 2.1 🟡 中危问题

#### G01 - `android_runtime.py:_connect_once` Agent 部分启动后仍标记连接成功

**文件**: `src/core/capability/device/android_runtime.py:251-265`
**风险等级**: 🟡 中危
**影响**: Agent 客户端创建失败但进程已启动时，连接仍标记为成功，后续任务执行缺少 Agent 功能支撑

**问题代码**:
```python
def _connect_once(self) -> bool:
    # ... Tasker bind 成功...
    self._start_agent()
    if self._agent_client is None and self._agent_process is None:
        self.logger.error("Agent 未启动，连接中止")
        self._cleanup_partial()
        return False
    if self._agent_client is not None:
        try:
            self._agent_client.bind(self._resource)
            self._agent_client.register_sink(self._resource, self._controller, self._tasker)
            self._agent_client.connect()
        except Exception as e:
            self.logger.warning("AgentClient 初始化异常", error=str(e))
    self._connected = True  # ❌ agent_client=None 时仍执行
    return True
```

**问题分析**:
- `_start_agent()` 内部有两步：启动 `go-service.exe` 子进程，创建 `AgentClient` 实例
- 如果进程启动成功但 `AgentClient` 创建失败（line 400-401），`_agent_process` 不为 None，`_agent_client` 为 None
- 第一个检查 `if self._agent_client is None and self._agent_process is None` 仅当两者都为 None 时才返回 False
- 如果 `_agent_process` 不为 None 但 `_agent_client` 为 None，检查通过，跳过 agent_client.bind()，直接设置 `self._connected = True`
- 后果：连接状态显示为已连接，但 Agent 的 sink 注册失败，任务执行时可能缺少 Agent 提供的功能（如 MaaEnd 回调、状态推送）
- 更隐蔽的是：`_agent_process` 已在后台运行但未绑定，占用端口和进程资源

**修复建议**:
```python
self._start_agent()
if self._agent_client is None and self._agent_process is None:
    self.logger.error("Agent 未启动，连接中止")
    self._cleanup_partial()
    return False
if self._agent_client is None:
    self.logger.error("AgentClient 创建失败，连接中止")
    self._cleanup_partial()
    return False
# AgentClient 可用，继续绑定
self._agent_client.bind(self._resource)
self._agent_client.register_sink(self._resource, self._controller, self._tasker)
self._agent_client.connect()
self._connected = True
return True
```

#### G02 - `main_window.py:_refresh_preview` 截图失败阈值固定为5，硬编码不可配置

**文件**: `src/gui/pyqt6/main_window.py:336-341`
**风险等级**: 🟢 低危
**影响**: 连续5次截图失败后标记连接断开，但阈值不可配置

**问题代码**:
```python
self._preview_fail_count = getattr(self, "_preview_fail_count", 0) + 1
if self._preview_fail_count >= 5:
    self._maaend_page.set_connected(False)
```

**问题分析**:
- 批次23/STATEMACHINE.md 已记录"截图瞬态失败反写连接态"的原始问题
- 当前代码通过5次失败阈值来缓解，但阈值硬编码
- 在低性能设备或弱ADB连接场景下，5次可能过于激进（正常截图偶尔超时不应断连）
- 在高延迟网络（远程ADB）场景下，5次可能不足（需要更高容限）
- 阈值应可配置，与设备性能/连接质量匹配

**修复建议**:

```python
config = self._maaend_page._config if hasattr(self._maaend_page, '_config') else {}
threshold = config.get("preview_fail_threshold", 5)
self._preview_fail_count = getattr(self, "_preview_fail_count", 0) + 1
if self._preview_fail_count >= threshold:
    ...
```

#### G04 - `maa_end/runtime.py:_connect_with_timeout` 超时后线程残留

**文件**: `src/core/service/maa_end/runtime.py:267-286`
**风险等级**: 🟢 低危
**影响**: 连接超时后后台线程继续运行，可能与后续连接冲突

**问题代码**:
```python
def _connect_with_timeout(self, timeout: int) -> bool:
    result = {"success": False}
    def target() -> None:
        try:
            result["success"] = self._connect_once()
        except Exception as exc:
            result["success"] = False
    thread = threading.Thread(target=target, daemon=True)
    thread.start()
    thread.join(timeout=timeout)
    if thread.is_alive():
        self.logger.error("ADB 连接超时")
        self._cleanup_partial()
        return False
    return result["success"]
```

**问题分析**:
- 当 `_connect_once` 在超时时间内未完成时，`thread.is_alive()` 为 True
- `_cleanup_partial()` 被调用清理当前实例资源，但后台线程仍在运行
- daemon线程不会被强制终止，它会继续尝试连接并可能创建新的 `_resource`、`_controller` 等对象
- 如果用户立即重试连接，新线程创建的资源可能与旧线程残留资源冲突
- 旧线程最终完成的连接会覆盖新线程的结果（通过共享的实例属性）

**修复建议**:
```python
def _connect_with_timeout(self, timeout: int) -> bool:
    result = {"success": False}
    done = threading.Event()
    def target() -> None:
        try:
            result["success"] = self._connect_once()
        except Exception as exc:
            result["success"] = False
        finally:
            done.set()
    thread = threading.Thread(target=target, daemon=True)
    thread.start()
    done.wait(timeout=timeout)
    if thread.is_alive():
        self.logger.error("ADB 连接超时，后台线程仍在运行")
        self._cleanup_partial()
        return False
    return result["success"]
```

#### G05 - `main_window.py:closeEvent` 托盘可用时跳过状态持久化

**文件**: `src/gui/pyqt6/main_window.py:108-122`
**风险等级**: 🟢 低危
**影响**: 通过托盘关闭窗口时队列状态未持久化

**问题代码**:
```python
def closeEvent(self, event: QCloseEvent) -> None:
    settings = QSettings("ArkStudio", "IstinaEndfieldAssistant")
    settings.setValue("mainWindow/geometry", self.saveGeometry())
    if self._tray_icon is not None and self._tray_icon.is_available():
        event.ignore()
        self.hide()
        # ❌ 未调用 _persist_state()
    else:
        maaend_page = getattr(self, "_maaend_page", None)
        if maaend_page is not None:
            try:
                maaend_page._persist_state()  # ✅ 仅在非托盘路径持久化
            except Exception as exc:
                ...
        super().closeEvent(event)
```

**问题分析**:
- 当托盘图标可用时，`closeEvent` 被忽略（窗口隐藏），状态持久化被跳过
- 队列状态（任务列表、选项、选中任务等）仅在真正退出时保存
- 如果应用在托盘状态下崩溃或被任务管理器终止，队列状态丢失
- 队列状态持久化应独立于窗口关闭事件，不应区分托盘/非托盘路径

**修复建议**:
```python
def closeEvent(self, event: QCloseEvent) -> None:
    settings = QSettings("ArkStudio", "IstinaEndfieldAssistant")
    settings.setValue("mainWindow/geometry", self.saveGeometry())
    # 无论托盘是否可用，始终持久化状态
    maaend_page = getattr(self, "_maaend_page", None)
    if maaend_page is not None:
        try:
            maaend_page._persist_state()
        except Exception as exc:
            self._logger.warning("closeEvent 持久化队列状态失败", error=str(exc))
    if self._tray_icon is not None and self._tray_icon.is_available():
        event.ignore()
        self.hide()
    else:
        super().closeEvent(event)
```

#### G06 - `client.py:98` 日志格式字符串与 ProjectLogger 约定不一致

**文件**: `src/core/capability/llm/client.py:98`
**风险等级**: 🟢 低危
**影响**: 日志输出格式不一致

**问题代码**:
```python
self._logger.error("[%s] LLM request failed url=%s error=%s", LogCategory.MAIN, url, str(exc))
```

**问题分析**:
- 项目中其他日志调用使用约定：`self._logger.error(LogCategory.MAIN, "消息", key=value)`
- ProjectLogger._format 检测第一个参数是否为 LogCategory 常量，自动添加 `[CATEGORY]` 前缀
- 此调用将 LogCategory 嵌入格式字符串 `[%s]`，与 _format 的约定冲突
- 实际效果：`[%s]` 被替换为 `MAIN`，产生 `[MAIN] LLM request failed url=... error=...`
- 结果与约定输出相同，但实现方式不一致——如果 ProjectLogger 更新格式逻辑，此调用可能失效

**修复建议**:
```python
self._logger.error(LogCategory.MAIN, "LLM request failed", url=url, error=str(exc))
```

#### G07 - `pipeline_node.py:61` `from_dict` 冗余条件判断

**文件**: `src/core/capability/element_recognition/pipeline/pipeline_node.py:59-62`
**风险等级**: 信息
**影响**: 死代码

**问题代码**:
```python
raw = data.get("recognition", "DirectHit")
recognition = raw if isinstance(raw, RecognitionType) else RecognitionType(raw) if isinstance(raw, str) and raw in RecognitionType._value2member_map_ else RecognitionType.DirectHit
if recognition == RecognitionType.DirectHit and not raw:  # ❌ 不可达
    recognition = RecognitionType.DirectHit
```

**问题分析**:
- `recognition` 在 line 60 已经被赋值为 `RecognitionType(raw)` 或 `RecognitionType.DirectHit`
- 如果 `raw` 是空字符串 `""`，line 60 的 else 分支会赋 `RecognitionType.DirectHit`
- 此时 `recognition == RecognitionType.DirectHit` 且 `not raw` 为 True
- 但 line 62 的赋值 `recognition = RecognitionType.DirectHit` 与 line 60 的结果相同
- 如果 `raw` 是有效字符串（如 "TemplateMatch"），`recognition` 不是 DirectHit，条件不满足
- 如果 `raw` 是 RecognitionType 实例，`not raw` 为 False
- 结论：此条件永远触发的是与 line 60 相同的赋值，属于死代码

**修复建议**:
```python
raw = data.get("recognition", "DirectHit")
if isinstance(raw, RecognitionType):
    recognition = raw
elif isinstance(raw, str) and raw in RecognitionType._value2member_map_:
    recognition = RecognitionType(raw)
else:
    recognition = RecognitionType.DirectHit
```
