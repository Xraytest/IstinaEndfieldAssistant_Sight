### [C1] Shell 黑名单缺失反斜杠，允许命令注入
**位置**：`src/core/capability/device/android_runtime.py:95`
```python
if any(ch in stripped for ch in self._SHELL_BLACKLIST_CHARS):
```
`_SHELL_BLACKLIST_CHARS` 包含 `;`, `|`, `&`, `` ` ``, `$`, `(`, `)`, `{`, `}`, `<`, `>`, `\n`, `\r`，但**不包含 `\`（反斜杠）**。

**根因**：在 Android `mksh` shell 中，反斜杠是行续行和转义字符。攻击者可构造如下 payload 绕过黑名单：
```
dumpsys meminfo\$(id)
```
此处 `\` 转义了 `$`，使 `ch in stripped` 检测不到 `$` 字符，但 shell 解释时反斜杠被消费，`$(id)` 作为命令替换执行。同样可作用于所有白名单前缀命令：`getprop ro.product.manufacturer\$(input tap 500 500)`。

**影响面**：任何通过守护进程调用 shell 的路径均可被利用。当前白名单允许 `dumpsys`、`input`、`getprop`、`am` 等，意味着攻击者可执行任意 ADB shell 命令。

**修改建议**：
1. 将 `\` 加入黑名单
2. 更根本的修复：将字符黑名单改为 shell 引号保护 + 参数列表化，避免字符串拼接传递 shell

**优先级**：Critical

### [C2] 队列行编辑在焦点切换时被静默丢弃
**位置**：`src/gui/pyqt6/pages/maaend_control_page.py:1236-1240`
```python
if self._focused_queue_index is not None and self._selected_task and 0 <= self._focused_queue_index < len(self._queue_state.queue_items):
    old_row = self._focused_queue_index
    old_entry = self._queue_state.queue_items[old_row]
    if old_entry.get("name") == self._selected_task:
        self._queue_state.update_queue_item_options(old_row, self._collect_options())
```

**根因**：`_on_queue_focus_changed` 在焦点离开行时尝试保存编辑，但只保存 `old_entry.get("name") == self._selected_task` 的行。当用户在任务列表中点选了一个新任务（`_selected_task` 已变更），队列行的 `name` 字段仍对应当前队列项，与新的 `_selected_task` 不匹配，条件为 `False`，编辑被跳过。

**触发路径**：
1. 用户在队列行编辑选项（如修改执行次数）
2. 用户在任务列表中点击了另一个任务（`_selected_task` 变更）
3. 焦点离开队列行，`_on_queue_focus_changed` 被调用
4. `old_entry.get("name") != self._selected_task` → `update_queue_item_options` 不执行
5. 用户编辑被静默丢弃

**影响面**：用户在队列中的编辑内容在特定交互顺序下永久丢失。

**修改建议**：移除 `old_entry.get("name") == self._selected_task` 条件，始终将当前焦点行的编辑刷入队列状态。

```python
def _on_queue_focus_changed(self, old_entry, new_entry):
    if self._focused_queue_index is not None and 0 <= self._focused_queue_index < len(self._queue_state.queue_items):
        self._queue_state.update_queue_item_options(self._focused_queue_index, self._collect_options())
```

**优先级**：High（数据丢失）

### [C5] `_connect_with_timeout` 超时后守护线程仍翻转 `_connected` 标志
**位置**：`src/core/service/maa_end/runtime.py:267-286`

**修改建议**：在超时路径中，守护线程完成时应再次验证 `_connected` 标志未被其他线程修改，或使用事件/锁同步超时结果。

**优先级**：High（状态不一致）

### [C7] `_cleanup_partial` 不释放原生 MaaFW 资源
**位置**：`src/core/service/maa_end/runtime.py:309-353`

**根因**：`_cleanup_partial` 仅将 `Tasker`、`Resource`、`AdbController`、`AgentClient` 设为 `None`，不调用任何 `dispose()`/`release()`/`close()`。这些对象持有原生 DLL 引用、线程和文件句柄。清理完全依赖不确定的 `__del__` 执行。

**影响面**：反复连接/断开后，原生资源（DLL 句柄、线程、socket）泄漏累积，可能导致 MaaFW DLL 锁定、端口耗尽。

**修改建议**：为每个原生资源类型添加 `dispose()` 方法，在 `_cleanup_partial` 中显式调用：
```python
if self._tasker is not None:
    self._tasker.stop()  # 或等效清理
    self._tasker = None
# ... 其他资源
```

**优先级**：High（资源泄漏）

### [C8] `get_latest_frame` 返回内部可变数组引用（API 设计问题）
**位置**：`src/core/capability/device/android_runtime.py:155-159`

**验证结论**：经源码验证，`_decode_loop` 在行 358 为每帧创建新数组（`img = frame.to_ndarray(format="bgr24")`），行 360 在锁内赋值指针。因此**不存在实际的读/写数据竞争**——调用方持有旧数组引用，下次解码创建新数组并替换指针。

### [C9] `_ScrcpySession.stop` join 超时后遗留孤儿线程清理新会话端口
**位置**：`src/core/capability/device/android_runtime.py:147-153`

**根因**：`stop()` 调用 `self._thread.join(timeout=5)`，但如果 `_decode_loop` 在 30 秒 socket 读取上阻塞，join 超时。主线程继续执行 `_cleanup()` 移除旧端口 forward。随后 `_dispatch` 的 `startScrcpy` 创建新 session（新端口）。旧孤儿线程最终退出时，其 `finally` 调用 `_cleanup()` 移除**新** session 的端口 forward，新会话的 socket 变为不可达。

**影响面**：scrcpy 重连后画面静止或断开，且无法自动恢复。

**修改建议**：在 join 超时后不执行 `_cleanup()`（或仅移除 forward 但不设 `_scrcpy_session = None`），改为等待线程自然退出后统一清理。

**优先级**：Medium（状态不一致）

### [L1] `_replace_tokens` 链式替换
**位置**：`src/core/service/maa_end/runtime.py:520-526`

**根因**：`str.replace()` 替换所有匹配子串，包括替换值中出现的后续占位符。例如 `values={"A": "{B}", "B": "final"}` 应用于 `"{A}"` 时，结果为 `"final"` 而非期望的 `"{B}"`。

**修改建议**：按占位符长度降序排列后替换，避免子串冲突：
```python
sorted_tokens = sorted(values.items(), key=lambda kv: len(kv[0]), reverse=True)
for token, replacement in sorted_tokens:
    result = result.replace("{" + token + "}", str(replacement))
```

**优先级**：High（数据损坏）

## 3. 安全漏洞

### [S1] `_dispatch` 将原始异常字符串返回给 RPC 客户端
**位置**：`src/core/capability/device/android_runtime.py:622-624`
```python
except Exception as exc:
    self._logger.error("守护进程执行失败 [method=%s]: %s", method, str(exc))
    return {"error": str(exc)}
```

**根因**：`str(exc)` 对于 Python 异常经常包含绝对路径（如 `PermissionError: [WinError 32] ... C:\Users\cheng\...`）、内部类名或部分堆栈。直接返回给 RPC 调用方，任何本地进程可收集内部路径和实现信息。

**影响面**：信息泄露。本地进程可利用路径信息构造更精准的攻击。

**修改建议**：对错误消息脱敏，仅返回通用错误码和用户友好消息。将详细异常记录到日志，不在 RPC 响应中暴露。

**优先级**：High（信息泄露）

## 5. 用户体验提升点

### [U1] 预览定时器在非标准推理页面切换时仍启动全量截图子进程
**位置**：`src/gui/pyqt6/main_window.py:277`

**根因**：`_on_nav_changed` 在每个页面切换时无条件调用 `self._preview_timer.start()`，即使用户在设置页或其他页面。每次切换都触发一次完整的 `_sync_execute("screenshot")` 子进程。

**修改建议**：仅当当前页面为标准推理页时启动定时器。

**优先级**：Medium（UX/性能）

---

### [U2] `_refresh_preview` 阻塞 GUI 主线程 2-4 秒
**位置**：`src/gui/pyqt6/main_window.py:319`

**根因**：`_refresh_preview` 通过 `_sync_execute("screenshot")` 同步执行截图，阻塞 GUI 主线程 2-4 秒。用户界面完全冻结。

**修改建议**：改为异步执行（QThread + 信号回调），或使用 scrcpy 常驻帧通道。

**优先级**：Medium（UX）

---

### [U3] `_do_auto_connect` 阻塞主线程 15 秒
**位置**：`src/gui/pyqt6/pages/maaend_control_page.py:1309`

**修改建议**：移至后台 QThread。

**优先级**：Medium（UX）

---

### [U4] `LlmChatWorker.finished` 信号从未连接
**位置**：`src/gui/pyqt6/pages/prts_full_intelligence_page.py:33,46,249`

**优先级**：Medium（UX/稳定性）
