### [YOLO-01 High] `yolo_backend.py:85-86` — bbox 归一化缺少 clamp 保护

```python
# yolo_backend.py:82-86
elements.append(ElementInfo(
    element_type="yolo_object",
    label=cls_name,
    bbox=(x1 / w_img, y1 / h_img, x2 / w_img, y2 / h_img),
    center=(cx / w_img, cy / h_img),
    ...
))
```

**问题**：YOLO 模型可能输出超出图像边界的检测框（例如框边缘延伸到画布外），导致 `x1/w_img` 等归一化坐标超出 [0, 1] 范围。下游消费者（如导航、去重）假定 bbox 在 [0, 1] 内，越界值会导致逻辑错误或崩溃。

**建议**：在赋值前 clamp 坐标：
```python
bbox = (
    max(0.0, min(1.0, x1 / w_img)),
    max(0.0, min(1.0, y1 / h_img)),
    max(0.0, min(1.0, x2 / w_img)),
    max(0.0, min(1.0, y2 / h_img)),
)
```

### [YOLO-02 High] `yolo_backend.py:66-77` — box tensor 索引无防御性检查

```python
# yolo_backend.py:66-77
for box in boxes:
    conf = float(box.conf[0])      # 无长度检查
    if conf < self._conf_threshold:
        continue
    cls_id = int(box.cls[0])       # 无长度检查
    cls_name = names.get(cls_id, str(cls_id))
    ...
    xyxy = box.xyxy[0].tolist()    # 无长度检查
```

**问题**：`box.conf`、`box.cls`、`box.xyxy` 作为 tensor，在异常情况下可能为空（shape=(0,)），直接 `[0]` 索引会导致 `IndexError` 崩溃。空 tensor 常见于 YOLO 模型返回空 batch 或某些特殊帧。

**建议**：在索引前检查 tensor 长度：
```python
if box.conf is None or len(box.conf) == 0:
    continue
conf = float(box.conf[0])
```
同理处理 `box.cls` 和 `box.xyxy`。

### [REC-01 Medium] `recognizer.py:179-183` — gameplay override 只升不降必然性

```python
# recognizer.py:179-183
if (gameplay_info.get("is_gameplay") and
        gameplay_info.get("blue_ratio", 0) > 0.75 and
        page_info.page_type not in ("exit_dialog", "loading", "title_screen")):
    page_info.page_type = "gameplay"
    page_info.confidence = max(page_info.confidence, 0.7)
```

**问题**：`max(page_info.confidence, 0.7)` 只提升置信度，永不下调。如果初始分类器给出 0.3 的 "unknown"（低置信度），但游戏场景检测因蓝色比例高而触发，则置信度被强行拉到 0.7。这掩盖了初始分类的不确定性，可能导致下游（如导航）基于虚假的高置信度做出错误决策。

**建议**：区分场景：
- 如果 `page_info.page_type` 已是 "gameplay" 且 confidence >= 0.7，保持不变
- 如果 `page_info.page_type` 不是 "gameplay"（如 "unknown"），使用更保守的置信度，例如 `min(0.7, page_info.confidence + 0.3)` 或直接使用 `gameplay_info` 的 blue_ratio 作为置信度

### [REC-02 Medium] `recognizer.py:287 vs 316` — 匹配逻辑不一致（精确匹配 vs 子串匹配）

```python
# Tier 1: 精确匹配 (line 287)
if any(t.lower() in element_labels for t in tpl_names):
    score += 2.0

# Tier 3: 子串匹配 (line 316)
if any(kw.lower() in elem.label.lower() for kw in ocr_keywords):
    score += 0.5
```

**问题**：Tier 1 使用 `t.lower() in element_labels`（set 成员测试，必须精确匹配），Tier 3 使用 `kw.lower() in elem.label.lower()`（子串匹配）。这导致：
- 模板名 "btn_confirm" 不会匹配 OCR 识别的 "确认按钮"（对用户来说语义相同）
- OCR 关键词 "确认" 会匹配 "确认取消"（子串匹配，可能误判）

不一致的匹配策略使得页面分类结果依赖元素来源（模板 vs OCR），而非语义等价性。

**建议**：统一使用子串匹配或至少对 Tier 1 也支持模糊匹配，或者为 OCR 关键词增加边界匹配（如 `kw.lower() == elem.label.lower()` 或 `kw.lower() in elem.label.lower().split()`）。

### [YOLO-03 Medium] `yolo_backend.py:102-112` — 失败加载无冷却/重试限制

```python
def _is_available(self) -> bool:
    if self._model is not None:
        return True
    try:
        from ultralytics import YOLO
        self._model = YOLO(self._model_path)
        logger.info(f"YOLO model loaded: {self._model_path}")
        return True
    except Exception as e:
        logger.debug(f"YOLO not available: {e}")
        return False
```

**问题**：`_is_available()` 在每次 `recognize()` 调用时被调用（line 47）。如果模型文件缺失或加载失败，每帧都会重试导入和加载，产生不必要的异常开销（每帧一次 `ModuleNotFoundError` 或 `FileNotFoundError`）。高频调用场景下（如实时导航），这会导致持续的性能拖累。

**建议**：添加失败标记和冷却：
```python
def __init__(self, ...):
    ...
    self._load_failed = False
    self._last_load_attempt = 0.0
def _is_available(self) -> bool:
    if self._model is not None:
        return True
    if self._load_failed:
        # 冷却 30 秒后再重试
        if time.time() - self._last_load_attempt < 30:
            return False
    ...
    except Exception as e:
        self._load_failed = True
        self._last_load_attempt = time.time()
        logger.debug(...)
        return False
```

## 审计纠错

### [AUDIT-1] 批次 45 N04 误报 — `_is_available` 并非"永不重试"

批次 45 报告声称 YOLO 后端的 `_is_available()` "加载失败后永不重试，YOLO 功能永久失效"。**此判断错误**。

**实际行为**：`_is_available()` 在每次 `recognize()` 调用时执行（line 47），如果 `self._model is None` 且上次加载失败，它会再次尝试加载。这意味着失败后**每次识别帧都会重试**，而非"永不重试"。批次 45 将"频繁重试"误判为"永久失效"，导致错误的修复建议。
