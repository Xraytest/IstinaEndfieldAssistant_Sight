## 二、新发现：资源耗尽向量

### SEC-02（Medium）交互式 CLI 循环无输入长度限制，内存耗尽

**位置**：`src/cli/istina.py:281-371`（`_interactive_loop`）

```python
def _interactive_loop(parser):
    buffer = ""
    while True:
        chunk = sys.stdin.read(1)       # ← 逐字符读取
        buffer += chunk                  # ← 无长度限制追加
        if chunk == "\n":
            line = buffer.strip()
            buffer = ""
            ...
            args = parser.parse_args(shlex.split(line))
            result = CLIDispatch(runtime).dispatch(args)
```

**问题**：`buffer` 无最大长度限制。攻击者（或故障的 stdin 源）可以持续向 buffer 追加数据而不发送换行符，导致内存无限增长。

**攻击场景**：
1. 恶意脚本通过管道向 istina 的 stdin 发送 1GB 无换行数据
2. `buffer` 增长到 1GB
3. 进程被 OOM killer 终止或系统变慢

**触发条件**：
- 必须以 `--interactive` 模式运行
- 攻击者需要控制 stdin（如通过管道输入）
- 普通命令行使用（无 `--interactive`）不受影响

**影响**：Medium——需要交互模式且攻击者控制 stdin，但一旦触发无任何防护。

### SEC-03（Medium）CLIBridge stdout 解析器无缓冲区上限，内存耗尽

**位置**：`src/gui/pyqt6/cli_bridge.py:166-185`（`_on_stdout`）

```python
def _on_stdout(self):
    data = bytes(self._process.readAllStandardError()).decode("utf-8", errors="replace")
    for raw in data.splitlines():
        ...
    # 另外在 _on_ready_read_stdout 中：
    self._stdout_buffer += data  # ← 无长度限制
    while True:
        newline_index = self._stdout_buffer.find("\n")
        ...
```

**问题**：`_stdout_buffer` 无最大长度限制。如果 CLI 子进程输出大量无换行数据（如 C++ 调试日志、MaaFW 输出），缓冲区会持续增长。

**攻击场景**：
1. CLI 子进程因某种原因输出大量无换行调试信息
2. `_stdout_buffer` 持续增长
3. GUI 进程内存耗尽

**触发条件**：需要 CLI 子进程输出异常大量的无换行数据。正常运行时每命令输出一行 JSON，不会触发。

**影响**：Medium——正常使用不触发，但异常子进程输出可导致 GUI 内存耗尽。

## 三、输入边界检查缺失

### SEC-04（Low）`device tap`/`swipe` 无坐标边界检查

**位置**：`src/cli/handlers.py:408-434`

```python
def _handle_device_tap(runtime, args):
    android.tap(int(args.x), int(args.y))  # ← 无范围检查
    # int() 对任意字符串成功，包括 "99999999999999999999"

def _handle_device_swipe(runtime, args):
    android.swipe(
        int(args.x1), int(args.y1),     # ← 无范围检查
        int(args.x2), int(args.y2),
        duration_ms=int(args.duration),
    )
```

**问题**：坐标仅做 `int()` 类型转换，无范围检查。Android 屏幕通常为 1080×2400 或类似分辨率。传入 `999999999` 会导致：
1. ADB shell 命令接收超大坐标
2. 触摸注入层尝试在屏幕外点击
3. 可能触发触摸驱动异常行为

**与历史发现的关联**：D8（001647.md）已指出触控操作硬编码 10s 超时无重试，但未涉及坐标边界检查。

### SEC-05（Low）`nav3 walk` 坐标无范围检查

**位置**：`src/cli/handlers.py:826-833`

```python
return runtime.execute("nav3.walk", {
    "map_name": args.map_name,    # ← 未验证是否为已知地图
    "x": args.x,                  # ← float 转换，无范围检查
    "y": args.y,                  # ← float 转换，无范围检查
    ...
})
```

**问题**：`map_name` 未验证是否为已加载的地图；`x`/`y` 为 float 类型无范围约束。传入超大坐标值会传递到 `VlmWalkNavigator.walk_to`，可能导致：
1. VLM 路径规划计算异常（超大坐标导致浮点溢出）
2. 导航循环无法收敛（目标在"无限远"）

## 四、脚本系统攻击面

### SEC-06（Low）脚本回放可强制发射 `editingFinished` 信号，绕过 UI 验证流程

**位置**：`src/gui/pyqt6/scripting/player.py:156-165`

```python
def _do_text(self, widget, action):
    value = action.value or {}
    text = value.get("text", "")
    if isinstance(widget, QLineEdit):
        widget.setText(text)               # ← 不触发 editingFinished
        widget.editingFinished.emit()      # ← 强制发射信号
    elif isinstance(widget, QComboBox):
        widget.setCurrentText(text)
```

**问题**：`QLineEdit.setText()` 不触发 `editingFinished` 信号（正常用户操作才会触发）。但代码显式调用 `widget.editingFinished.emit()` 强制发射。这意味着：
1. 连接到 `editingFinished` 的验证槽会被触发
2. 但验证逻辑可能期望 `setText` 与 `editingFinished` 之间有用户交互（如焦点切换）
3. 某些槽可能依赖焦点状态，在脚本回放时焦点状态不正确

**攻击场景**：恶意脚本可以：
1. 设置敏感字段的值（如设备地址、LLM 路径）
2. 强制触发验证/保存逻辑
3. 绕过正常 UI 流程中的确认对话框

**影响**：Low——脚本系统本身是自动化工具，此行为是 by-design 的副作用。

## 六、修复建议

### 6.1 优先级修复

| 优先级 | 项 | 修改量 | 理由 |
|--------|----|--------|------|
| **P1** | SEC-02：交互式 CLI 增加输入长度限制 | 2 行 | 防止内存耗尽，修改成本低 |
| **P1** | SEC-03：CLIBridge stdout 缓冲区增加上限 | 3 行 | 防止 GUI 内存耗尽 |
| **P2** | SEC-04：tap/swipe 坐标增加范围检查 | 4 行 | 防止异常坐标值传递 |
| **P2** | SEC-05：nav3 walk 增加地图名/坐标校验 | 5 行 | 防止无效导航请求 |
| **P3** | SEC-06：脚本 editingFinished 发射加注释/开关 | 1 行 | by-design 行为，加注释说明 |

### 6.2 具体修改建议

**SEC-02 修复**：
```python
# _interactive_loop 中增加最大输入长度
MAX_INPUT_LENGTH = 1024 * 1024  # 1MB
buffer = ""
while True:
    chunk = sys.stdin.read(1)
    if len(buffer) >= MAX_INPUT_LENGTH:
        self_logger.error("CLI 交互循环: 输入超过最大长度", length=len(buffer))
        result = {"status": "error", "message": f"input exceeds {MAX_INPUT_LENGTH} bytes"}
        _write_result(result)
        buffer = ""
        continue
    buffer += chunk
```

**SEC-03 修复**：
```python
# CLIBridge._on_stdout 中增加缓冲区上限
MAX_STDOUT_BUFFER = 4 * 1024 * 1024  # 4MB
self._stdout_buffer += data
if len(self._stdout_buffer) > MAX_STDOUT_BUFFER:
    self._logger.error("CLI stdout 缓冲区超过上限，清空", size=len(self._stdout_buffer))
    self._stdout_buffer = ""
    self.commandError.emit(" ".join(self._last_command), "CLI output exceeded buffer limit")
    continue
```
