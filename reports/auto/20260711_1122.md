### [NEW-A] `maa_end/runtime.py:152` & `:175` — `relative_to(self._maaend_root)` 在 assets 布局下抛 ValueError，任务/预设被静默丢弃

**优先级：Low（潜伏性）**
**位置**：`src/core/service/maa_end/runtime.py:151-154` 与 `:174-175`

```python
# load_tasks()
task_copy["_source"] = str(json_path.relative_to(self._maaend_root))
# load_presets()
self._presets[name]["_source"] = str(json_path.relative_to(self._maaend_root))
```

**根因**：
`load_tasks` / `load_presets` 先用 `self._resolve_asset_path("tasks")` / `("tasks","preset")` 解析真实基目录，再遍历 `rglob("*.json")`。但计算 `_source` 时硬编码 `json_path.relative_to(self._maaend_root)`，而非相对**实际解析出的基目录**。

`_resolve_asset_path` 明确支持两种布局：
- 发布布局：`<root>/tasks`（直接用）
- 开发布局：`<root>/assets/tasks`（兜底）

当 `tasks` 实际解析到 `<root>/assets/tasks`（即走开发/资源布局）时，`json_path` 形如 `<root>/assets/tasks/xxx.json`，对其调用 `relative_to(<root>)` 必然抛 `ValueError`。该异常被外层 `try/except`（行 155、176）吞掉，仅打 `debug` 日志，**任务/预设条目被整条跳过、不进入 `self._tasks` / `self._presets`**。

**影响面**：
- 当前 `3rd-part/maaend` 部署为扁平布局（`<root>/tasks` 存在，含 47 个 JSON），`relative_to(root)` 成立，**当前不触发**。
- 一旦切换到代码所兼容的 `assets/` 布局（或 `tasks` 在 root 不存在、仅在 assets 下存在），**全部任务/预设静默消失**，且 CLI `tasks()` / `presets()` 返回空，自动化完全不可用且无任何错误提示。属于与 `_resolve_asset_path` 兼容性承诺自相矛盾的潜伏缺陷。

**修改建议**：
```python
# load_tasks 内：用 tasks_root 而非 _maaend_root
task_copy["_source"] = str(json_path.relative_to(tasks_root))
# load_presets 内：用 preset_root
self._presets[name]["_source"] = str(json_path.relative_to(preset_root))
```
或统一用 `json_path.relative_to(self._resolve_asset_path("tasks"))` 动态取值。

> 经 `git log --all -p -- reports/auto/` 全量核对，`relative_to(self._maaend_root)` 在 `maa_end/runtime.py` 中从未被既往报告提及 → 确属新发现。

### [NEW-C] `runtime.py:536` — `_wait_for_in_world` 调用 `run_pipeline` 不传超时，`job.wait()` 可永久挂死，外部 120s 截止线失效

**优先级：Medium**
**位置**：
- `src/core/service/runtime.py:531-543`（`_wait_for_in_world`）
- `src/core/service/maa_end/runtime.py:561-579`（`run_pipeline`）→ `:537-559`（`_wait_job`）

```python
# runtime.py _wait_for_in_world
deadline = time.time() + timeout
while time.time() < deadline:
    try:
        if runtime.run_pipeline("EnterGame", {}):   # ← 无 timeout
            return True
    except Exception as exc:
        self._logger.warning(...)
    time.sleep(interval)
return False
```

```python
# maa_end/runtime.py run_pipeline → _wait_job
def _wait_job(self, job, timeout):
    if timeout is None or timeout <= 0:
        return job.wait()          # ← 阻塞直到 MaaFW 返回，无上限
```

**根因**：
`_wait_for_in_world` 的 120s 循环依靠每次迭代**及时返回**才能推进 `while` 条件。但 `run_pipeline("EnterGame", {})` 未传 `timeout`，默认 `None`，最终走到 `job.wait()`（行 544）**无限阻塞**。若 `EnterGame` pipeline 因资源未完全加载、节点卡在 wait、或 Tasker 处于异常态而**永不返回**，`run_pipeline` 永不返回，`while` 循环体不结束，`time.time() < deadline` 永远得不到重新求值 → **整个 `daily.run` / `harvest.run` / `explore.run` 流程永久挂死**，外部 120s 截止线完全失效。

**与既有 M05 的区别（避免重复计数）**：
既往报告 M05（`_wait_for_in_world` 无连续失败计数，循环跑满 120s）描述的是 `EnterGame` **每次都返回 False** 时浪费 120s 的浪费问题；本 NEW-C 描述的是 `EnterGame` **永不返回** 时**永久挂死**的更严重故障模式——M05 的 120s 上限对"返回 False"有效，对"不返回"无效。二者互补，不重叠。

**影响面**：`daily` / `harvest` / `explore` 命令在游戏卡加载/Tasker 异常时可能无响应挂死，用户只能强杀进程。

**修改建议**：在 `_wait_for_in_world` 内为每次 `run_pipeline` 传入剩余时间作为超时上限：
```python
remaining = max(1, deadline - time.time())
if runtime.run_pipeline("EnterGame", {}, timeout=min(remaining, 30)):
    return True
```
或在 `run_pipeline` 入口强制 `timeout` 默认值（如 30s），杜绝 `job.wait()` 无限阻塞。

### [A2] 降级：NEW-1（批次 39 `0914.md`，Medium）**触发面被高估**，实为 Low/边界

**既往论断**：NEW-1 称 `runtime.py:868`（`_llm_run`）在 try 之外访问 `self._llm_runtime_instance.ready`，且 `get_instance` 内 `int(config["port"])` 异常时命令崩溃。

**核验**：
- `_llm_runtime_instance` 属性（`:122-126`）→ `_get_llama_runtime(self._config)` → `LlamaServerRuntime.get_instance(config)`（`llm/runtime.py:69-77）：`port = int(config.get("port", 9998))`。

**结论**：真实崩溃面仅为"顶层 port 配置非法"这一窄边界，正常部署不可达。将 NEW-1 由 **Medium 降为 Low（配置健壮性）**，并明确触发条件。原 "Medium + 命令崩溃" 评级高估。

### [A3] 修正深化：W1 根因确认为"守护进程拒绝字母键 + `keyevent()`/`shell()` 吞错不抛"双重缺陷

**既往论断**（记忆 W1）：VLM 行走字母键被拒、三重静默失败（`vlm_walk_navigator` → `runtime.py:712-729` `_vlm_keyevent` → `android_runtime.py` 守护进程拒绝）。

**补充核验（当前代码）**：
1. `_is_valid_keyevent`（`android_runtime.py:101-110`）仅接受纯数字或 `KEYCODE_*` 常量名；VLM 发送的小写字母键（如 `"w"/"a"/"s"/"d"`）被拒，返回 `{"error": ...}`（`:635-636`）。
2. 同文件 `keyevent()`（`:810-812`）与 `shell()`（`:814-816`）在守护进程返回 `{"error": ...}` 时**返回 `""`**（取 `response.get("result", "")`），**吞掉错误不抛异常**；而相邻的 `tap()`（`:797-800`）/`swipe()`（`:802-808`）在 `response.get("error")` 时**显式 raise `AndroidRuntimeError`**。
3. 错误处理**不一致**直接导致 VLM 行走的按键失败被完全静默——这是 W1"三重静默"中关键的一环，也是修复 W1 时**必须同步修正**的点（仅放开 `_is_valid_keyevent` 仍不够，还需让 `keyevent()`/`shell()` 在守护进程报错时 raise，与 `tap`/`swipe` 对齐）。

**结论**：W1 仍存活（见 §3），但修复建议应扩展为"放开字母键 + 统一错误抛出"，既往仅描述"字母键被拒"不完整。本批予以深化，**不新增编号**（属 W1 的一部分）。
