### 2.2 🟡 中危问题

#### D03 - 多个调试脚本硬编码其他用户的绝对路径

**文件**: `_test_adb_tap_claim.py`, `_diagnose_touch.py`, `_analyze_claim_screenshots.py`, `_find_claim_button.py`, `_maafw_quest_1280.py`
**风险等级**: 🟡 中危
**影响**: 脚本在当前用户环境下无法写入缓存文件

**问题代码**:
```python
# _test_adb_tap_claim.py:7, _diagnose_touch.py:7
CACHE = r'C:\Users\xray\Documents\ArkStudio\IstinaAI\IstinaEndfieldAssistant\cache'
```

**分析**:
- `C:\Users\xray\...` 是另一个用户（`xray`）的目录，当前用户是 `cheng`
- 至少5个脚本硬编码了此路径
- 这些脚本会尝试将调试截图写入 `C:\Users\xray\...\cache`，当前用户无写权限
- 即使有权限，写入的是另一个用户的目录，调试结果不可见
- 这是明显的 copy-paste 痕迹——脚本从 `xray` 的项目复制到 `cheng` 的项目时，路径未更新
- 项目目录名也不同：`IstinaEndfieldAssistant`（xray） vs `IstinaEndfieldAssistant_Sight`（cheng）
- `_nav_comparison.py` 和 `_scan_quest_y.py` 正确使用了相对路径计算：`CACHE = os.path.join(PROJECT_ROOT, 'cache')`

**修复建议**:
```python
# 统一使用项目根目录相对路径
from core.foundation.paths import get_project_root
CACHE = str(get_project_root() / "cache")
```

#### D05 - `verify_llm.py` / `check_llm_cuda.py` 使用魔法数字 999

**文件**: `scripts/verify_llm.py:37,39`, `scripts/check_llm_cuda.py:35,37`
**风险等级**: 🟢 低危
**影响**: 代码可读性差

**问题代码**:
```python
"-ngl", "999",
"--n-gpu-layers", "999",
```

**分析**:
- `-ngl 999` 传递给 llama-server，表示"使用全部 GPU 层"
- llama.cpp  silently caps 到实际层数，999 是约定俗成的"全部"值
- 但代码中没有注释说明 999 的含义，新开发者可能误以为是精确层数
- 推荐使用 `-ngl 999` 但添加注释，或使用文档化的值

**修复建议**:
```python
# 999 = "全部层"（llama.cpp 会 silently cap 到模型实际层数）
"-ngl", "999",
"--n-gpu-layers", "999",
```

## 3. 历史报告审计修正

### 3.1 批次39报告（091448.md）审计修正

#### 修正1：P01 分析不准确 — 条件非"恒真"，实际逻辑正确但阈值设计不合理

**原报告**: 091448.md P01
**修正内容**: P01 声称条件"恒真"（always true）。经源码核对，**此分析错误**。

**实际逻辑分析**:
```python
# handlers.py:677
if mem and mem >= 4GB or mem and mem >= 2GB:
```

Python 运算符优先级：`and` 高于 `or`，实际求值为：
```python
(mem and (mem >= 4GB)) or (mem and (mem >= 2GB))
```

真值表：
| mem 值 | 条件结果 | 推荐 |
|--------|----------|------|
| 0 | False | CPU ✓ |
| 1GB | False | CPU ✓ |
| 3GB | True | GPU |
| 5GB | True | GPU ✓ |

**结论**: 条件**非恒真**——当 `mem < 2GB` 时正确返回 CPU。代码逻辑本身正确，但有两个设计问题：
1. `mem >= 2GB` 条件被 `mem >= 4GB` 蕴含（冗余），`or` 连接无意义
2. 2GB 阈值过低——对于 LLM 推理（Q4_K_XL 4B 模型），4GB+ 显存才推荐 GPU 更合理

**修正后分类**: P01 从 Medium 降级为 **Info**——代码逻辑正确，阈值设计是设计选择而非代码错误。

**修正建议**:
```python
# 仅需一个条件
if mem >= _GPU_MEM_THRESHOLD:
    recommendation = "GPU"
```

#### 修正2：P02 风险高估 — 配置注入场景受限于本地访问

**原报告**: 091448.md P02
**修正内容**: P02 声称 `_handle_config_set` 可注入任意配置项影响设备连接安全。经深入分析，**实际风险低于评估**。

**风险修正分析**:
1. **攻击前提**: 攻击者需要本地 CLI 访问权限——这意味着攻击者已有本地 shell 访问
2. **config 文件被 gitignore**: `client_config.json` 不会被提交到仓库，注入不会传播到其他用户
3. **本地已有更强的攻击面**: 攻击者已有本地访问时，可直接修改 `client_config.json` 文件（无需通过 CLI）
4. **CLI 的 `--config` 参数**: 攻击者可直接通过 `--config /malicious/path` 加载恶意配置，无需先写入再读取
5. **实际影响**: 如果用户误操作（如 `config set device.serial attacker_device`），可能导致连接到错误设备——但这属于用户错误而非安全漏洞

**结论**: P02 是有效的代码质量问题（缺乏配置键校验），但风险等级应从 Medium 降级为 **Low**——攻击场景需要本地访问，且已有更强的攻击路径。

#### 修正3：P06 分析过度 — fd 双重关闭概率极低

**原报告**: 091448.md P06
**修正内容**: P06 声称 `_interactive_loop` 的 stdout fd 重定向与 `main()` 重复可能导致 fd 双重关闭。经深入分析，**实际不可能触发**。

**实际流程分析**:
```python
if __name__ == "__main__":
    if len(sys.argv) > 1 and sys.argv[1] == "--interactive":
        parser = build_parser()
        sys.exit(_interactive_loop(parser))  # ← 仅交互模式走这里
    sys.exit(main())  # ← 非交互模式走这里
```

1. `--interactive` 模式：`main()` 不执行，`_interactive_loop()` 独立执行——fd 重定向只执行一次
2. 非交互模式：`_interactive_loop()` 不执行，`main()` 独立执行——fd 重定向只执行一次
3. **两种模式互斥**，不会同时执行

**结论**: P06 的分析基于错误的假设（认为 `_interactive_loop` 在 `main` 内部调用），实际两个函数互斥执行。建议降级为纯代码质量观察（非 bug）。

## 6. 结论

### 新发现

本次审计转向 `scripts/` 目录，发现8项新问题：

1. **D01（严重）**: 7个调试脚本引用不存在的 `device.touch.maafw_touch_adapter` 模块——死代码，ImportError 崩溃
2. **D02（严重）**: 8个调试脚本引用不存在的 `_path_setup` 模块——死代码，ImportError 崩溃
3. **D03（中危）**: 5个调试脚本硬编码 `C:\Users\xray\Documents\...`——从其他用户项目复制时路径未更新
4. **D04（低危）**: 4个调试脚本中文注释乱码——UTF-8 编码问题
5. **D05-D08（信息）**: 魔法数字999、路径解析不一致、端口冲突、路径初始化模式不统一

### 历史报告审计修正

批次39报告存在3处修正：

1. **P01 分析错误**: `_handle_gpu_recommend` 条件**非恒真**——当 `mem < 2GB` 时正确返回 CPU。代码逻辑正确，阈值设计（2GB vs 4GB）是设计选择问题，降级为 Info。
2. **P02 风险高估**: `_handle_config_set` 攻击场景需要本地 CLI 访问，且已有更强的攻击路径（直接修改配置文件）。降级为 Low。
3. **P06 分析过度**: `_interactive_loop` 和 `main()` 互斥执行（通过 `--interactive` 参数选择），fd 重定向只执行一次，不可能双重关闭。

### 关键发现关联

D01 + D02 揭示了一个系统性问题：**项目包含从其他用户（`xray`）项目复制的调试脚本，但未完成适配**。这些脚本引用不存在的模块和不匹配的路径，是项目维护中的"技术债务"。建议清理 `scripts/debug/` 目录，删除或重写废弃脚本。
