## §1 新发现

### [I18N-01 Low] `i18n/__init__.py:29-36` — `_load_all()` 静默吞掉所有异常，翻译文件加载失败不可见

```python
# i18n/__init__.py:29-36
def _load_all(self) -> None:
    for path in self._locales_dir.glob("*.json"):
        locale = path.stem
        try:
            data = json.loads(path.read_text(encoding="utf-8"))
            self._translations[locale] = data
        except Exception:
            continue
```

**根因分析**：`except Exception: continue` 捕获了所有异常（`json.JSONDecodeError`、`PermissionError`、`UnicodeDecodeError`、`OSError` 等）均无任何日志输出。如果某个翻译 JSON 文件存在语法错误或文件系统问题，该语言会被静默跳过，程序正常运行但该语言完全不可用，开发者无法通过日志发现任何异常。

**影响面**：
- **可诊断性**：翻译文件损坏时，程序正常运行但该语言完全不可用，问题排查困难
- **用户体验**：用户切换语言时发现部分文本仍显示英文 key（回退到 `key` 字符串），但不知道原因
- **对比**：`queue_state.py` 的 `load()` 在解析失败时至少记录了 `logging.getLogger(__name__).warning(...)`，而 `_load_all()` 完全不记录

**建议**：至少记录一条 WARNING 日志：

```python
except Exception as exc:
    logging.getLogger(__name__).warning(
        "Failed to load locale file %s: %s", path, exc
    )
```

### [I18N-02 Low] `i18n/__init__.py:90-98` — `get_locale_manager()` 非线程安全单例

```python
# i18n/__init__.py:90-98
_instance: Optional[LocaleManager] = None

def get_locale_manager() -> LocaleManager:
    global _instance
    if _instance is None:
        _instance = LocaleManager()
        _instance.load_saved_locale()
    return _instance
```

**根因分析**：`_instance` 全局变量的 `if _instance is None` 检查与赋值之间没有锁保护。如果 GUI 主线程和后台线程同时首次调用 `get_locale_manager()`，两者都可能通过 `_instance is None` 检查，创建两个 `LocaleManager` 实例。第二个实例会覆盖 `_instance`，导致第一个实例持有的翻译数据被丢弃。

**对比**：
- `touch_manager.py:58-64` 的 `TouchManager.get_instance()` 使用了双重检查锁（DCL）
- `queue_state.py` 的 `QueueState` 内部使用 `threading.Lock` 保护所有读写操作
- `logger.py` 的 `_logger_initialized` 布尔标记也有线程安全问题（已在前序批次中报告为 R3）

`LocaleManager` 是系统中唯一同时存在两个问题的模块：既无锁保护全局单例，又在 `_load_all()` 中静默吞异常。

**建议**：参考 `TouchManager.get_instance()` 使用 `threading.Lock` + 双重检查：

```python
_lock = threading.Lock()

def get_locale_manager() -> LocaleManager:
    global _instance
    if _instance is None:
        with _lock:
            if _instance is None:
                _instance = LocaleManager()
                _instance.load_saved_locale()
    return _instance
```

### [I18N-03 Low] `i18n/__init__.py:62-70` — `tr()` 缺失翻译键无追踪，无法统计翻译完整度

```python
# i18n/__init__.py:62-70
def tr(self, key: str, default: str = "") -> str:
    data = self._translations.get(self._current_locale, {})
    if key in data:
        return data[key]
    # fallback to zh_CN
    fallback = self._translations.get("zh_CN", {})
    if key in fallback:
        return fallback[key]
    return default or key
```

**根因分析**：当 key 在当前语言和回退语言中均不存在时，`tr()` 静默返回 `default`（若为空则返回 `key` 字符串本身）。没有机制追踪哪些 key 缺失翻译，也没有日志记录。翻译维护者无法获知当前语言的翻译完整度，新增 UI 文本后忘记添加翻译时只能通过肉眼发现。

**影响面**：低——不影响功能，但会降低翻译质量和维护效率。如果新增了 UI 文本但忘记添加翻译，用户会看到英文 key 或空字符串，但无法批量发现。

**建议**：可选方案——在 DEBUG 模式下记录缺失 key 的日志：

```python
def tr(self, key: str, default: str = "") -> str:
    ...
    if key not in data and key not in fallback:
        if __debug__:
            logging.getLogger(__name__).debug("Missing translation key: %s", key)
    return default or key
```
