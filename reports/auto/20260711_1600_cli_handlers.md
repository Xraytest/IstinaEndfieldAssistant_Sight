### [CLI01 Medium] `handlers.py:410-444` 多个设备操作 handler 未检查 `android.default_client is None`

**位置**：`src/cli/handlers.py:410-444`

```python
def _handle_device_screenshot(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    data = android.screenshot()           # ← 未检查 default_client
    return _write_or_base64(data, getattr(args, "out", None))

def _handle_device_tap(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    try:
        android.tap(int(args.x), int(args.y))    # ← 未检查 default_client
        ...
```

**根因分析**：
`_handle_device_status`（line 392-407）正确检查了 `android.default_client is None` 并返回错误：
```python
if android.default_client is None:
    return {"status": "error", "message": "no device connected"}
```

但以下 handler **缺少相同检查**：
- `_handle_device_screenshot`（line 410-413）
- `_handle_device_tap`（line 416-422）
- `_handle_device_swipe`（line 425-444）
- `_handle_device_monitor`（line 462-472）— 部分检查（try/except），但未提前返回
- `_handle_shell`（line 475-484）
- `_handle_scene_capture`（line 487-490）

当用户未连接设备时调用这些命令，`android.screenshot()`/`android.tap()` 会访问 `self._default_client`（为 `None`），触发 `AttributeError: 'NoneType' object has no attribute '...'`。

**影响面**：
- 中——CLI 用户可能在不连接设备时执行 `istina screenshot`、`istina device tap` 等命令，导致异常堆栈输出而非友好的错误提示。
- 与 `_handle_device_status` 的不一致行为会困惑用户——`istina device status` 返回友好错误，但 `istina device screenshot` 崩溃。

**修改方案**：
```python
def _handle_device_screenshot(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    if android.default_client is None:
        return {"status": "error", "message": "no device connected"}
    data = android.screenshot()
    return _write_or_base64(data, getattr(args, "out", None))
```

### [CLI02 Low] `handlers.py:733` `_handle_gpu_recommend` 4GB 阈值导致 2GB 分支死代码

**位置**：`src/cli/handlers.py:726-744`

```python
def _handle_gpu_recommend(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    gpus = _handle_gpu_status(runtime, args).get("gpus", [])
    recommendation = "CPU"
    if gpus:
        mem = gpus[0].get("free_memory_bytes", 0)
        GB = 1024 * 1024 * 1024
        # P01: 分级推荐，避免运算符优先级歧义（>=4GB / >=2GB / CPU）
        if mem >= 4 * GB:          # ← 第一分支
            recommendation = "GPU (>=4GB)"
        elif mem >= 2 * GB:        # ← 永远不会执行（mem >= 4GB 已覆盖）
            recommendation = "GPU (>=2GB)"
        else:
            recommendation = "CPU"
```

**根因分析**：
`mem >= 4 * GB` 已经覆盖了 `mem >= 2 * GB` 的所有情况（4GB > 2GB）。因此 `elif mem >= 2 * GB` 分支是**死代码**——永远不会执行。

**影响面**：
- 低——逻辑上正确（4GB+ 推荐 GPU 是合理的），但代码意图不清晰。若未来需要调整阈值（如改为 6GB/3GB），开发者可能困惑于两个分支的关系。
- 注释 `P01: 分级推荐` 暗示三个分支都有效，但实际只有两个有效分支。

**修改方案**：
```python
if mem >= 4 * GB:
    recommendation = "GPU (>=4GB)"
elif mem >= 2 * GB:
    recommendation = "GPU (>=2GB)"
else:
    recommendation = "CPU"
```
或简化为两分支：
```python
if mem >= 4 * GB:
    recommendation = "GPU (>=4GB)"
elif mem >= 2 * GB:
    recommendation = "GPU (>=2GB)"
else:
    recommendation = "CPU"
```

### [CLI03 Low] `handlers.py:572-584` `_handle_auth_*` 返回 "ok" 而非 "not_implemented"

**位置**：`src/cli/handlers.py:572-584`

```python
def _handle_auth_status(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    # CLI03: 未实现的功能返回 ok 并退出码 0，而非 error
    return {"status": "ok", "message": "not implemented"}

def _handle_auth_login(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    # CLI03: 未实现的功能返回 ok 并退出码 0，而非 error
    return {"status": "ok", "message": "not implemented"}

def _handle_auth_logout(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    return {"status": "not_implemented"}
```

**根因分析**：
三个 auth handler 功能未实现，但返回格式不一致：
- `_handle_auth_status` 返回 `{"status": "ok", ...}` — 暗示成功
- `_handle_auth_login` 返回 `{"status": "ok", ...}` — 暗示成功
- `_handle_auth_logout` 返回 `{"status": "not_implemented"}` — 暗示未实现

CLI 调用方（如脚本或自动化工具）通常检查 `result["status"] == "success"` 判断执行结果。若返回 `"ok"`，调用方可能误认为认证成功，继续执行后续操作。

**影响面**：
- 低——当前 auth 功能未实现，用户手动调用时会看到 `"not implemented"` 消息。但自动化脚本可能将 `"ok"` 误判为成功。

**修改方案**：
```python
def _handle_auth_status(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    return {"status": "not_implemented", "message": "auth status not implemented"}

def _handle_auth_login(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    return {"status": "not_implemented", "message": "auth login not implemented"}

def _handle_auth_logout(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    return {"status": "not_implemented", "message": "auth logout not implemented"}
```

### [CLI04 Info] `handlers.py` 多函数内 `import` 语句（非模块级）

**位置**：`src/cli/handlers.py:557` 和 `:643` 等

```python
def _handle_config_set(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    key = args.key
    import re                          # ← 函数内 import
    if not re.fullmatch(...):
        ...

def _handle_gpu_status(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    try:
        import pynvml                  # ← 函数内 import
        pynvml.nvmlInit()
        ...
```

**根因分析**：
多个 handler 函数在函数体内使用 `import re`、`import pynvml`、`import GPUtil`、`import psutil`、`import torch` 等。这是**惰性导入**模式——仅在函数被调用时才导入依赖。

**影响面**：
- Info——惰性导入是合理的（避免在 CLI 启动时加载重型库如 `pynvml`/`torch`），且 `import` 语句有缓存机制（首次导入后后续调用无开销）。
- 但 `import re`（line 557）属于标准库，应在模块级导入。函数内导入 `re` 是多余的。

**修改方案**：
```python
# 模块级导入
import re

# 函数内仅保留惰性导入
def _handle_gpu_status(...):
    try:
        import pynvml
        ...
```
