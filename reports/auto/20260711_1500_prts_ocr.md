### [PRTS01 Medium] `prts_full_intelligence_page.py:267-271` `_append_chat` 未转义 HTML，LLM 输出可注入渲染

**位置**：`src/gui/pyqt6/pages/prts_full_intelligence_page.py:267-271`

```python
def _append_chat(self, source: str, text: str) -> None:
    self._chat_output.append(f"<b>[{source}]</b> {text}")
    cursor = self._chat_output.textCursor()
    cursor.movePosition(cursor.MoveOperation.End)
    self._chat_output.setTextCursor(cursor)
```

**根因分析**：
`_append_chat` 直接将 `text` 拼入 HTML 字符串并调用 `QTextEdit.append()`。`QTextEdit` 默认以富文本模式解析 HTML。如果 LLM 返回的内容包含 HTML 标签（如 `<script>alert('xss')</script>`、`<img onerror=...>`、`<a href="javascript:...">`），这些标签会被 Qt 的富文本渲染器执行或解析。

**对比**：`log_page.py`（batch 48 修复）的 `_highlight_log` 正确使用 `html.escape(line, quote=False)` 后再插入 `<span>` 标签。`prts_full_intelligence_page.py` 缺少同样的转义步骤。

**影响面**：
- LLM 聊天输出是用户可控输入（LLM 生成内容可能包含 HTML 标签）。
- 在当前项目中，LLM 模型是本地 llama-server，输出相对可控。但：
  1. 若未来接入外部 LLM API，输出内容不可预测。
  2. 即使本地模型，prompt 中若包含 HTML 片段（如网页截图分析），模型可能原样输出 HTML。
  3. Qt 的富文本渲染器支持有限但确实支持 `<script>`（在 Qt 5 中）和 `<img>` 标签。

**修改方案**：
```python
def _append_chat(self, source: str, text: str) -> None:
    escaped = html.escape(text, quote=False)
    self._chat_output.append(f"<b>[{source}]</b> {escaped}")
    cursor = self._chat_output.textCursor()
    cursor.movePosition(cursor.MoveOperation.End)
    self._chat_output.setTextCursor(cursor)
```

### [PRTS02 Low] `prts_full_intelligence_page.py:96-100` 硬编码横幅样式

**位置**：`src/gui/pyqt6/pages/prts_full_intelligence_page.py:96-100`

```python
banner = QLabel(locale.tr("prts_under_construction", "⚠ 此页面正在施工中，功能尚未完全实现。"))
banner.setProperty("variant", "warning")
banner.setWordWrap(True)
banner.setStyleSheet("background-color: #fff3cd; color: #856404; padding: 8px 12px; border-radius: 6px; font-weight: 500;")
```

**根因分析**：横幅使用硬编码的 hex 颜色（`#fff3cd`、`#856404`），而非从 `COLORS` 字典或主题系统读取。其他页面（如 `device_settings_page.py`）使用 `locale.tr()` + 主题样式。硬编码颜色在暗色主题下会显示异常（浅色背景在暗色模式中刺眼）。

**影响面**：低——仅在"施工中"横幅上。但暗色主题用户会看到不一致的视觉风格。

**修改方案**：
```python
banner.setStyleSheet(f"background-color: {COLORS['warning_bg']}; color: {COLORS['warning_text']}; padding: 8px 12px; border-radius: 6px; font-weight: 500;")
```

### [PRTS03 Low] `prts_full_intelligence_page.py:251-265` `_attach_image` 无文件大小限制

**位置**：`src/gui/pyqt6/pages/prts_full_intelligence_page.py:261-262`

```python
data = Path(path).read_bytes()
self._pending_image_b64 = base64.b64encode(data).decode("ascii")
```

**根因分析**：用户可通过文件对话框选择任意大小的图片。`read_bytes()` 将整个文件读入内存，`b64encode` 再生成大小约为原始文件 137% 的字符串。对于 4K 截图（~8MB），base64 编码后约 11MB，全部存储在 `_pending_image_b64` 中。多次选择大图而不发送会导致内存累积。

**影响面**：低——用户通常选择游戏截图（< 2MB）。但无上限保护属于健壮性缺陷。

**修改方案**：
```python
MAX_IMAGE_BYTES = 5 * 1024 * 1024  # 5MB
data = Path(path).read_bytes()
if len(data) > MAX_IMAGE_BYTES:
    self._append_chat("系统", locale.tr("image_too_large", "Image too large (max 5MB)"))
    return
self._pending_image_b64 = base64.b64encode(data).decode("ascii")
```

### [PRTS04 Info] `prts_full_intelligence_page.py:165` `commandFinished` 信号连接后未断开

**位置**：`src/gui/pyqt6/pages/prts_full_intelligence_page.py:165`

```python
self._bridge.commandFinished.connect(self._on_command_finished)
```

**根因分析**：`_setup_ui` 中连接 `commandFinished` 到 `_on_command_finished`，但页面销毁时未断开连接。PyQt6 在 `QObject` 销毁时会自动断开信号连接，但：
1. 如果页面被 `hide()` 而非 `deleteLater()`，连接保持活跃。
2. `_on_command_finished` 会修改页面 UI 状态（如 `_send_btn.setEnabled(True)`），即使页面不可见。

**影响面**：低——页面通常在应用生命周期内保持存活。但若未来实现页面动态创建/销毁，此连接会导致悬空引用。

### [OCR01b Medium] `ocr_backend.py:142` `_run_maafw_ocr` box 解包无长度校验

**位置**：`src/core/capability/element_recognition/backends/ocr_backend.py:142`

```python
bx1, by1, bw, bh = best.box
```

**根因分析**：与 batch 47 OCR-01（`_run_ocr_manager` 的 `x, y, bw, bh = box`）相同模式——直接解包 `box` 假设长度为 4。若 `best.box` 长度≠4，抛 `ValueError`。但 `_run_maafw_ocr` 外层没有 try/except 保护（batch 47 OCR-01 的 try/except 仅覆盖遍历循环），异常会传播到 `recognize()` 调用方。

**影响面**：中——`_run_maafw_ocr` 是 Route 1（优先路径），若 maafw OCR 返回异常 box 格式，整个 OCR 识别崩溃。

**修改方案**：
```python
box = getattr(best, "box", (0, 0, 0, 0))
if len(box) < 4:
    box = (0, 0, 0, 0)
bx1, by1, bw, bh = box[:4]
```

### [OCR02 Low] `ocr_backend.py:264` 角点格式检测逻辑可误分类小框

**位置**：`src/core/capability/element_recognition/backends/ocr_backend.py:264`

```python
vals = [int(v) for v in box[:4]]
if vals[2] > vals[0] and vals[3] > vals[1]:
    # Looks like [x1, y1, x2, y2]
    return (vals[0], vals[1], vals[2] - vals[0], vals[3] - vals[1])
# [x, y, w, h]
return (vals[0], vals[1], vals[2], vals[3])
```

**根因分析**：`_parse_ocr_box` 用 `vals[2] > vals[0] and vals[3] > vals[1]` 判断是否为角点格式 `[x1,y1,x2,y2]`。但对 `[x,y,w,h]` 格式，若 `w > x` 且 `h > y`（如小坐标 + 大宽高），会被误分类为角点格式。例如 `[10, 10, 100, 100]`（x=10, y=10, w=100, h=100）→ `100 > 10 and 100 > 10` → true → 误分类为 `[10, 10, 90, 90]`（实际应为 `[10, 10, 100, 100]`）。

**影响面**：低——OCR box 通常坐标值较大（>100），宽高小于坐标值的情况罕见。但代码逻辑缺陷存在。

**修改方案**：
```python
# 更稳健的格式判断：角点格式的 x2/y2 通常远大于 x1/y1
if vals[2] - vals[0] > 50 and vals[3] - vals[1] > 50:
    # 差值较大 → 可能是角点格式
    return (vals[0], vals[1], vals[2] - vals[0], vals[3] - vals[1])
return (vals[0], vals[1], vals[2], vals[3])
```
