## 新增发现（2 项）

### PREVIEW-TIMER-NORESTART（启动自动连接失败后 preview timer 永久停止）— `maaend_control_page.py`

**等级**: 代码错误 / 中
**位置**: `src/gui/pyqt6/pages/maaend_control_page.py:1675, 1698-1700`

**问题代码**:

```python
# maaend_control_page.py:1675 — _delayed_init 停止 preview timer
def _delayed_init(self) -> None:
    main_window = self.window()
    preview_timer = getattr(main_window, "_preview_timer", None)
    if preview_timer is not None:
        preview_timer.stop()  # ← 停止预览定时器

    self.refresh()
    QTimer.singleShot(50, self._do_auto_connect)  # → 调度自动连接

# maaend_control_page.py:1698-1700 — 仅在连接成功时重启
def _on_auto_connect_finished(self, success: bool) -> None:
    if success:
        self._connected = True
        self._auto_connect_attempted = False
    else:
        self._auto_connect_attempted = True  # ← 失败路径

    preview_timer = getattr(self.window(), "_preview_timer", None)
    if preview_timer is not None and self._connected:  # ← 仅连接成功时启动！
        preview_timer.start()
```

**根因分析**:

`_delayed_init` 在页面初始化时主动停止 `main_window._preview_timer`，目的是避免预览请求与自动连接的嵌套 `QEventLoop` 冲突。但 `_on_auto_connect_finished` 仅在 `self._connected == True` 时重启 timer——如果启动自动连接失败：

1. `_connected` 保持 `False`
2. `_auto_connect_attempted` 设为 `True`（阻止后续重试）
3. preview_timer **永不重启**
4. 用户看到空白预览区域
5. 即使用户后续从"设备"页面手动连接成功，preview_timer 也不恢复

**恢复路径缺失**: 手动连接成功后，`main_window._on_bridge_command_finished` 处理 `system connect` 成功时，调用 `QTimer.singleShot(0, self._refresh_preview)` 触发一次刷新，但**不重启周期性 preview_timer**。只有用户切换到其他标签页再切回时，`_on_nav_changed` → `_preview_timer.start()` 才恢复。

**调用链（错误场景）**:
```
应用启动 → MaaEnd 页面初始化
  → QTimer.singleShot(0, _delayed_init)
  → _delayed_init: preview_timer.stop() ← 停止
  → QTimer.singleShot(50, _do_auto_connect)
  → _do_auto_connect: _sync_execute("system connect") 超时或失败
  → _on_auto_connect_finished(False)
  → self._connected = False, preview_timer 不启动 ← 永久停止
  → 用户看到空白预览
  → 用户手动连接（设备页面）→ 连接成功
  → _on_bridge_command_finished: _maaend_page.set_connected(True)
  → 但 preview_timer 仍未启动！预览仍空白
  → 用户切换标签页 → _on_nav_changed → _preview_timer.start() ← 仅此恢复
```

**与 O-08 的模式对比**:

| 对比点 | O-08 (device_settings_page) | PREVIEW-TIMER-NORESTART |
|--------|---------------------------|----------------------|
| 停止条件 | 连接失败 | 自动连接失败 |
| 恢复条件 | 手动切换重连开关 | 手动连接 + 切换标签页 |
| 恢复路径 | 用户手动 toggle | 用户手动连接 + 导航切换 |
| 影响面 | 重连逻辑 | 预览功能 |

两者是同一模式的变体：**失败路径停止关键功能，恢复路径不可达或需额外操作**。

**影响面**:
- **中**：启动时自动连接失败（设备未开机/ADB 未就绪），用户手动连接后预览仍空白。用户必须切换标签页才能恢复预览，体验困惑。
- 概率：设备未就绪时启动应用（常见场景）。

**修复建议**:

```python
# 方案 1（最小修改）：_on_auto_connect_finished 中始终尝试重启 timer
def _on_auto_connect_finished(self, success: bool) -> None:
    if success:
        self._connected = True
        self._auto_connect_attempted = False
    else:
        self._auto_connect_attempted = True

    preview_timer = getattr(self.window(), "_preview_timer", None)
    if preview_timer is not None:
        preview_timer.start()  # ← 移除 self._connected 条件
```

```python
# 方案 2（更安全）：手动连接成功后也重启 timer
# 在 main_window._on_bridge_command_finished 的 system connect 成功分支：
if result.get("status") == "success":
    self._maaend_page.set_connected(True)
    preview_timer = getattr(self, "_preview_timer", None)
    if preview_timer is not None and not preview_timer.isActive():
        preview_timer.start()  # ← 新增：确保 timer 活跃
    QTimer.singleShot(0, self._refresh_preview)
```

**推荐方案 1**（`_on_auto_connect_finished` 中移除 `self._connected` 条件）。如果连接失败时不该显示预览，可在 `_refresh_preview` 中检查连接状态并显示占位提示，而非依赖 timer 停止来隐藏预览。

### LLM-WORKER-FINISHED-DEAD（LlmChatWorker.finished 信号无连接者）— `prts_full_intelligence_page.py`

**等级**: 代码质量 / 低
**位置**: `src/gui/pyqt6/pages/prts_full_intelligence_page.py:30, 224`

**问题代码**:

```python
# prts_full_intelligence_page.py:30 — 定义 finished 信号
class LlmChatWorker(QThread):
    finished = pyqtSignal(dict)  # ← 定义信号

    def run(self):
        ...
        self.finished.emit(result or {"status": "error", "message": "empty"})  # ← 发射信号

# prts_full_intelligence_page.py:224 — worker 置 None，finished 信号无人接收
self._worker = None  # ← QThread 对象变为孤立，finished 信号无人连接
```

**根因分析**:

`LlmChatWorker.finished` 信号在 `run()` 方法末尾被发射（传递 LLM 聊天结果），但代码库中**从未将此信号连接到任何槽函数**。实际结果处理通过 `_bridge.commandFinished` 信号 + `_on_command_finished` 完成，`finished` 信号的发射是冗余的。

具体影响：
1. `run()` 结束时发射 `finished(result_dict)`，但没有接收者
2. `_send_chat` 中 `self._worker = None` 清除引用，QThread 对象变为孤立
3. 由于 worker 是 daemon 线程（QThread 默认），线程退出后对象最终被 GC
4. `finished` 信号的发射是纯无意义操作——创建 dict 开销 + 信号投递开销

**与 PRTS-WORKER-LINGER 的关系**: batch 104 已报告 LlmChatWorker 缺少 `stop()/wait()/deleteLater()` 生命周期管理。本发现是另一个角度：即使 worker 生命周期得到正确管理，`finished` 信号本身也是死代码。

**影响面**:
- **低**：每次 LLM 聊天请求产生一次无意义的信号投递。不影响功能正确性，但浪费资源。

**修复建议**:

```python
# 方案 1（最小修改）：移除 finished 信号定义和发射
class LlmChatWorker(QThread):
    # 移除: finished = pyqtSignal(dict)
    def run(self):
        ...
        result = self._bridge.execute("llm chat", params)
        # 移除: self.finished.emit(result or {"status": "error", "message": "empty"})

# 方案 2（保留信号但正确使用）：连接 finished 到 _on_command_finished
# 在 _send_chat 中：
self._worker = LlmChatWorker(self._bridge, text, image_b64, self)
self._worker.finished.connect(self._on_command_finished)
self._worker.start()
# 同时移除 _on_command_finished 中 self._worker = None 前的清理逻辑
```

**推荐方案 1**（`finished` 信号未被设计为使用，移除即可。结果处理已通过 `commandFinished` 信号完成，无需重复）。
