### [S03 Medium] `models.py:53` 脚本保存无 IO 异常处理

**位置**：`src/gui/pyqt6/scripting/models.py:53`

```python
def save(self, directory: Path) -> Path:
    directory.mkdir(parents=True, exist_ok=True)
    path = directory / f"{self.name}.json"
    path.write_text(json.dumps(self.to_dict(), ensure_ascii=False, indent=2), encoding="utf-8")
    return path
```

**根因分析**：`write_text()` 在磁盘满、权限不足、路径过长、网络驱动器断开等场景下会抛出 `OSError`/`PermissionError`。当前无任何捕获，异常直接向上传播到 `scripting_page.py` 的 `_save_script` 方法（那里有 QMessageBox 兜底），但 `Script.save()` 作为独立 API 被其他潜在调用方使用时（如自动保存、批量导出）会导致崩溃。

**修改方案**：
```python
def save(self, directory: Path) -> Path:
    directory.mkdir(parents=True, exist_ok=True)
    path = directory / f"{self.name}.json"
    try:
        path.write_text(json.dumps(self.to_dict(), ensure_ascii=False, indent=2), encoding="utf-8")
    except OSError as e:
        raise IOError(f"Failed to save script to {path}: {e}") from e
    return path
```

**影响面**：`Script.save()` 的所有调用方（当前仅 `scripting_page.py` 和 `recorder.py:100`）。修改后调用方收到更明确的异常类型。

### [S04 Medium] `models.py:59` 脚本加载无 JSON 解析异常处理

**位置**：`src/gui/pyqt6/scripting/models.py:59`

```python
@classmethod
def load(cls, path: Path) -> Script:
    data = json.loads(path.read_text(encoding="utf-8"))
    return cls.from_dict(data)
```

**根因分析**：`json.loads()` 对损坏的 JSON 文件抛出 `json.JSONDecodeError`，`path.read_text()` 可能抛出 `OSError`。当前无任何捕获。当 `_refresh_script_list` 调用 `Script.load()` 时，异常被 `except Exception: pass` 吞掉（S02）；但 `_load_script`（用户手动加载）和 `player.py:46`（`load_from_file`）无兜底，会向用户展示原始 Python traceback。

**修改方案**：
```python
@classmethod
def load(cls, path: Path) -> Script:
    try:
        data = json.loads(path.read_text(encoding="utf-8"))
    except (OSError, json.JSONDecodeError) as e:
        raise ValueError(f"Failed to load script from {path}: {e}") from e
    return cls.from_dict(data)
```

**影响面**：所有调用 `Script.load()` 的位置。

## 2. 既往报告审计（4 项纠正）

### 审计纠正 A1：批次 26 C02「链式替换」描述需精确化

**原始报告**：`20260711_FINAL.md` 批次 #26 C02 — `maa_end/runtime.py:520-525` `_replace_tokens` 顺序替换导致二次替换。

**审计结论**：**描述准确，但触发条件需精确化**。经当前源码逐行核对，`_replace_tokens` 的实现为：

```python
def _replace_tokens(self, text: str, values: Dict[str, str]) -> str:
    for key, val in values.items():
        text = text.replace(f"{{{key}}}", val)
    return text
```

当 `values={"A": "{B}", "B": "Z"}` 且模板文本含 `{A}` 时，第一轮迭代将 `{A}` 替换为 `{B}`，第二轮迭代将 `{B}` 替换为 `Z}`。最终结果 `{A}` → `Z`，而非预期的 `{B}`。

**精确化**：该 bug 仅在 `values` 字典的 value 本身含 `{...}` 占位符语法时触发。若所有 value 均为纯文本（无花括号），则无二次替换问题。实际触发条件为：pipeline JSON 模板中的某个 token 值引用了另一个 token 的占位符——这在当前资产文件中未出现，但在未来扩展自定义 token 时可能触发。

**结论**：原始报告结论正确，无需修正。补充精确化说明供修复参考。
