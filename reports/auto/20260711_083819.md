#### K01 - `AnimatedButton._bg_opacity` Python property 无法被 `QPropertyAnimation` 驱动

**文件**: `src/gui/pyqt6/theme/animations.py:25-27, 36`
**风险等级**: 🟡 中危
**影响**: 按钮悬停动画完全不生效，`_bg_opacity` 始终为 0.0

**问题代码**:
```python
class AnimatedButton(QPushButton):
    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._bg_opacity = 0.0
        self._target_opacity = 0.0
        self._animation = QPropertyAnimation(self, b"_bg_opacity", self)
        # ❌ QPropertyAnimation 需要 Q_PROPERTY，但 _bg_opacity 是 Python property

    _bg_opacity = property(_get_bg_opacity, _set_bg_opacity)
    # ❌ 用 property() 覆盖了实例属性，动画 target 的 b"_bg_opacity" 找不到
```

**问题分析**:
- `QPropertyAnimation(self, b"_bg_opacity", self)` 要求目标对象有 `Q_PROPERTY` 声明的属性，或至少有 `setter`/`getter` 方法
- 代码试图用 Python 内置的 `property()` 装饰器模拟 Qt 属性，但 `QPropertyAnimation` 无法通过 `property` 对象读写值
- 此外，`_bg_opacity = property(...)` 在类体中的赋值会覆盖 `__init__` 中设置的实例属性 `self._bg_opacity = 0.0`
- `paintEvent` 中检查 `self._bg_opacity > 0.01` 始终为 False（因为 property getter 返回 0.0 且动画从未生效）
- 最终效果：按钮悬停/按下时背景色完全不变化，动画完全失效
- `mousePressEvent` 中手动设置 `self._bg_opacity = 0.5` 也会被 property 拦截，实际调用 setter 设置值，但由于动画未驱动，release 后回退到 0.0

**修复建议**:
```python
class AnimatedButton(QPushButton):
    def __init__(self, *args, **kwargs):
        super().__init__(*args, **kwargs)
        self._bg_opacity = 0.0
        self._target_opacity = 0.0
        self._animation = QPropertyAnimation(self, b"_bg_opacity", self)
        self._animation.setDuration(150)
        self._animation.setEasingCurve(QEasingCurve.Type.OutCubic)

    # 移除 class-level property，直接在 __init__ 中通过 setter 操作
    # 或者使用 Q_PROPERTY 宏（需继承 QObject 并使用 pyqtProperty）
```

#### K02 - `Player._schedule_next` 创建 QTimer 但不持有引用，无法 stop()

**文件**: `src/gui/pyqt6/scripting/player.py:101-104`
**风险等级**: 🟡 中危
**影响**: 脚本停止后，pending 的 QTimer 仍然触发，继续执行已停止的脚本

**问题代码**:
```python
def _schedule_next(self, delay_ms: int) -> None:
    if self._stopped:
        self._on_finished()
        return
    # ...
    self._action_timer = QTimer(self)       # 覆盖旧 timer
    self._action_timer.setSingleShot(True)
    self._action_timer.timeout.connect(self._execute_action)
    self._action_timer.start(delay_ms)
    # ❌ 旧 timer（如果有）未 stop()，可能仍在 pending
```

**问题分析**:
- 每次调用 `_schedule_next` 都会创建新的 `QTimer(self)` 并赋值给 `self._action_timer`
- 旧 timer 对象仍然存在（Qt parent-child 机制下由 `self` 持有），且其 `timeout` 信号连接到 `_execute_action`
- `Player.stop()` 只 stop 当前 `self._action_timer`（第82行），但如果在 stop 之前有多个 timer 被创建（理论上不会，但 Qt event loop 延迟可能导致意外），旧 timer 不会被清理
- 更实际的场景：`_schedule_next` 被递归调用（每次 `_execute_action` 结束后调度下一个），如果 `stop()` 在两次调度之间被调用，旧 timer 被 stop，但新 timer 在 `stop()` 之前已经创建
- `Player.stop()` 使用 `self._action_timer.isActive()` 检查，但如果 timer 已经触发（正在执行 `_execute_action` 中），`isActive()` 返回 False，stop 无效
- 这导致：调用 `stop()` 后，脚本可能在延迟后继续执行若干动作

**修复建议**:
```python
def _schedule_next(self, delay_ms: int) -> None:
    if self._stopped:
        self._on_finished()
        return
    if self._paused:
        return
    if self._current_index >= len(self._script.actions):
        self._on_finished()
        return

    # 先停止旧 timer 再创建新 timer
    if self._action_timer and self._action_timer.isActive():
        self._action_timer.stop()
    self._action_timer = QTimer(self)
    self._action_timer.setSingleShot(True)
    self._action_timer.timeout.connect(self._execute_action)
    self._action_timer.start(delay_ms)
```

#### K03 - `Recorder._should_skip` 对 QLineEdit/QComboBox 的跳过逻辑矛盾

**文件**: `src/gui/pyqt6/scripting/recorder.py:128-139`
**风险等级**: 🟡 中危
**影响**: QLineEdit 和 QComboBox 在小部件跳过列表中，但又单独处理文本变更录制，行为矛盾

**问题代码**:
```python
_SKIPPED_TYPES = (
    QAbstractScrollArea, QScrollBar, QAbstractSlider,
    QMenu, QDialog, QHeaderView, QStatusBar,
)

def _should_skip(self, widget):
    for skipped_type in _SKIPPED_TYPES:
        if isinstance(widget, skipped_type):
            return True           # QLineEdit/QComboBox 不在此列
    obj_name = widget.objectName()
    if not obj_name and not isinstance(widget, QLineEdit | QComboBox):
        return True               # 有 objectName 的非输入框不过滤
    if not obj_name:
        return True               # 无 objectName 的输入框过滤
    return False
```

**问题分析**:
- QLineEdit 和 QComboBox 不在 `_SKIPPED_TYPES` 中（合理，因为需要录制它们的文本变更）
- 但 `_record_click` 第147行额外检查 `if not isinstance(widget, QLineEdit | QComboBox)`，意味着点击输入框不会被录制
- 这导致：用户点击 QLineEdit 输入框本身（而非其中文本）的操作不被记录，但文本变更事件会被记录
- 实际场景：用户点击输入框聚焦→输入文字，录制结果缺少"点击聚焦"步骤
- 回放时 `Player._do_text` 直接设置文本，不模拟点击聚焦，可能导致 setText 在某些控件上不生效（如未聚焦的 QLineEdit 在某些平台可能不响应 setText）
- 更严重的问题：`_should_skip` 中有两个条件分支都可能导致跳过（无 objectName 的输入框被跳过，但 `_record_text_change` 也检查 `if not obj_name: return`，造成双重跳过逻辑）

**修复建议**:
```python
def _should_skip(self, widget):
    for skipped_type in _SKIPPED_TYPES:
        if isinstance(widget, skipped_type):
            return True
    obj_name = widget.objectName() if hasattr(widget, "objectName") else ""
    # 允许录制无 objectName 的输入框（仍需录制其文本变更）
    if not obj_name and not isinstance(widget, QLineEdit | QComboBox):
        return True
    return False
```

#### K04 - `Player._do_click` 坐标映射双重抵消

**文件**: `src/gui/pyqt6/scripting/player.py:152`
**风险等级**: 🟢 低危
**影响**: 录制时记录的全局坐标在回放时被错误地双重转换

**问题代码**:
```python
def _do_click(self, widget, action):
    value = action.value or {}
    x = float(value.get("x", 0))
    y = float(value.get("y", 0))
    pos = QPoint(int(x), int(y))
    # mapFromGlobal(mapToGlobal(pos)) 双重映射相互抵消
    local_pos = widget.mapFromGlobal(widget.mapToGlobal(pos))
    QTest.mouseClick(widget, ..., pos=local_pos)
```

**问题分析**:
- 录制时 `_record_click` 存储的是 `event.position()`（相对于 widget 的局部坐标）
- 回放时 `widget.mapToGlobal(pos)` 将局部坐标转为全局坐标，然后 `widget.mapFromGlobal(...)` 将全局坐标转回局部坐标
- 数学上 `mapFromGlobal(mapToGlobal(p)) == p`，双重转换相互抵消
- 但录制时 `event.position()` 返回的是 Qt6 的 `QPointF`（相对于 widget 的局部坐标），而回放时 `QTest.mouseClick` 的 `pos` 参数也期望局部坐标
- 所以实际上：如果录制值已经是局部坐标，回放时应该直接使用，不需要任何转换
- 如果录制值因某种原因（如事件过滤器拦截的是 QApplication 级别事件）是全局坐标，双重转换恰好能正确处理
- **结论**: 当前代码"碰巧正确"——双重转换抵消后等于直接使用原始坐标。但代码意图不清晰，且依赖于录制时坐标类型的假设。应显式注明或统一坐标语义

**修复建议**:
```python
def _do_click(self, widget, action):
    value = action.value or {}
    x = float(value.get("x", 0))
    y = float(value.get("y", 0))
    pos = QPoint(int(x), int(y))
    # event.position() 在 Qt6 中返回 widget 局部坐标，直接使用
    QTest.mouseClick(widget, Qt.MouseButton.LeftButton, pos=pos)
```

#### K05 - `get_status_icon` 忽略 size 参数，图标尺寸不一致

**文件**: `src/gui/pyqt6/theme/icons.py:259-333`
**风险等级**: 🟢 低危
**影响**: 调用方传入的 size 参数被忽略，图标渲染尺寸不一致

**问题代码**:
```python
def get_status_icon(status: str, size: int = 14) -> QIcon:
    factory = _STATUS_ICONS.get(status)
    if not factory:
        return QIcon()
    key = f"status:{status}:{size}"
    if key not in _cache:
        path = factory()          # ❌ 调用工厂时不传 size
        # ...
        pixmap = _pixmap_from_path(path, size=16, ...)  # ❌ 硬编码 16
        _cache[key] = QIcon(pixmap.scaled(size, size, ...))
```

**问题分析**:
- `_STATUS_ICONS` 中的工厂函数 `lambda: _icon_dot(4)` 和 `lambda: _icon_check(4)` 使用硬编码的 size=4
- `_pixmap_from_path` 也使用硬编码的 `size=16`
- 虽然最终通过 `pixmap.scaled(size, size, ...)` 缩放到目标尺寸，但初始 pixmap 始终是 16×16
- 当 size > 16 时（如高 DPI 环境），缩放会导致模糊
- 缓存 key 包含 size，所以不同 size 调用会缓存不同条目，但每次都是先渲染 16×16 再缩放
- 对比 `get_nav_icon` 和 `get_action_icon` 也使用硬编码 16，全图标系统存在一致性问题

**修复建议**:
```python
_STATUS_ICONS: Dict[str, callable] = {
    "pending": lambda size=4: _icon_dot(size),
    "running": lambda size=4: _icon_dot(size),
    "success": lambda size=4: _icon_check(size),
    "failed": lambda size=4: _icon_cross_small(size),
}

def get_status_icon(status: str, size: int = 14) -> QIcon:
    factory = _STATUS_ICONS.get(status)
    if not factory:
        return QIcon()
    key = f"status:{status}:{size}"
    if key not in _cache:
        path = factory(size)                          # 传递 size
        pixmap = _pixmap_from_path(path, size=size, ...)  # 使用 size
        _cache[key] = QIcon(pixmap)
```

#### K07 - `models.py:Script.from_dict` 无字段验证，缺失字段导致 AttributeError

**文件**: `src/gui/pyqt6/scripting/models.py:40-47`
**风险等级**: 🟢 低危
**影响**: 加载损坏的脚本文件时崩溃

**问题代码**:
```python
@classmethod
def from_dict(cls, data: dict[str, Any]) -> Script:
    actions = [ActionRecord.from_dict(a) for a in data.get("actions", [])]
    return cls(
        name=data.get("name", ""),        # 默认空字符串，可接受
        actions=actions,
        description=data.get("description", ""),
        created_at=data.get("created_at", ""),
    )
```

**问题分析**:
- 如果 JSON 文件缺少 `name` 字段，Script 的 `name` 变为空字符串
- `Script.save()` 使用 `self.name` 作为文件名（`directory / f"{self.name}.json"`），空名称导致 `".json"` 文件
- `ActionRecord.from_dict` 同理，缺失 `widget_type` 或 `object_name` 会导致 ActionRecord 创建时字段为空
- Player 回放时 `_find_widget` 使用 `action.object_name` 查找控件，空名称匹配到错误控件
- 没有 schema 验证，损坏文件可以静默加载

**修复建议**:
```python
@classmethod
def from_dict(cls, data: dict[str, Any]) -> Script:
    name = data.get("name", "").strip()
    if not name:
        raise ValueError("Script requires non-empty 'name' field")
    actions = []
    for a in data.get("actions", []):
        if isinstance(a, dict):
            actions.append(ActionRecord.from_dict(a))
    return cls(
        name=name,
        actions=actions,
        description=data.get("description", ""),
        created_at=data.get("created_at", ""),
    )
```

#### K08 - `icons.py` 图标缓存无限增长，无容量上限

**文件**: `src/gui/pyqt6/theme/icons.py:265`
**风险等级**: 🟢 低危
**影响**: 长期运行中内存中缓存所有图标变体

**问题代码**:
```python
_cache: Dict[str, QIcon] = {}

def get_nav_icon(label, size=18):
    key = f"nav:{label}:{size}"
    if key not in _cache:
        path = factory()
        pixmap = _pixmap_from_path(path, size=16, ...)
        _cache[key] = QIcon(pixmap.scaled(size, size, ...))
    return _cache[key]
```

**问题分析**:
- `_cache` 是全局字典，所有图标（nav + action + status）和所有尺寸变体都缓存其中
- 不同的 `size` 值会产生不同的 cache key，理论上可以无限增长
- 实际使用中 size 只有少数几个值（14, 16, 18），增长有限
- 但如果在主题切换时 `_cache` 不被清除，旧主题颜色的图标仍然占用内存
- `_pixmap_from_path` 使用 `color=COLORS["primary"]`，如果主题切换后 COLORS 更新，缓存中的旧颜色图标不会被刷新

**修复建议**:
```python
_cache: Dict[str, QIcon] = {}
_MAX_CACHE_SIZE = 200

def _get_cached_icon(key):
    if len(_cache) >= _MAX_CACHE_SIZE:
        _cache.clear()  # 主题切换时或缓存过大时清空
    return _cache.get(key)
```

### 2.3 信息级问题

#### K09 - `_pixmap_from_path` 使用透明 QPixmap 未检查 painter 状态

**文件**: `src/gui/pyqt6/theme/icons.py:20-29`
**风险等级**: 信息
**影响**: painter.end() 失败时 QPixmap 可能为空

**问题代码**:
```python
def _pixmap_from_path(path, size=16, color=COLORS["primary"], stroke_width=1.5):
    pixmap = QPixmap(size, size)
    pixmap.fill(QColor("transparent"))
    painter = QPainter(pixmap)
    painter.setRenderHint(QPainter.RenderHint.Antialiasing)
    painter.setPen(QPen(QColor(color), stroke_width, ...))
    painter.setBrush(Qt.BrushStyle.NoBrush)
    painter.setWindow(0, 0, 16, 16)
    painter.drawPath(path)
    painter.end()
    return pixmap
    # ❌ 无 painter.isActive() 检查，若 painter 初始化失败返回空白 pixmap
```

**问题分析**:
- `QPainter(pixmap)` 在 pixmap 未初始化或已损坏时可能返回非 active 状态
- `painter.end()` 返回 bool 表示是否成功结束，代码未检查
- 实际影响极低（QPixmap 在函数内创建，通常 active），但缺少防御性检查
- 如果 `QColor(color)` 中 color 格式错误（如 COLORS["primary"] 返回非标准颜色字符串），painter 可能处于异常状态

#### K10 - `Player.is_playing` 在 `_current_index == len(actions)` 时返回 False

**文件**: `src/gui/pyqt6/scripting/player.py:88`
**风险等级**: 信息
**影响**: 播放完成瞬间与暂停状态的边界不一致

**问题代码**:
```python
def is_playing(self):
    return self._script is not None and not self._stopped and self._current_index < len(self._script.actions)
```

**问题分析**:
- 当 `_current_index == len(self._script.actions)`（最后一个动作刚执行完，但 `_on_finished` 尚未被调用）时，`is_playing()` 返回 False
- 但此时 `_stopped` 仍为 False，播放实际上仍在进行（正在等待 timer 触发 `_on_finished`）

- 修复：增加 `_paused` 检查，更精确地反映播放状态

#### K11 - `LocaleManager` 单例非线程安全（同类问题第五次出现）

**文件**: `src/gui/pyqt6/i18n/__init__.py:87-95`
**风险等级**: 信息
**影响**: 多线程同时调用 `get_locale_manager()` 可能创建多个实例

**问题代码**:
```python
_instance: Optional[LocaleManager] = None

def get_locale_manager() -> LocaleManager:
    global _instance
    if _instance is None:            # ❌ 非原子操作，多线程下可能创建多个实例
        _instance = LocaleManager()
        _instance.load_saved_locale()
    return _instance
```

**问题分析**:
- 这是项目中第五个非线程安全单例（ThemeManager, TemplateRegistry, TouchManager, LlamaServerRuntime 后第五个）
- Qt GUI 应用通常在主线程运行，实际多线程触发的可能性较低
- 但 `_load_all()` 在构造时读取所有 JSON 文件，如果多线程触发重复构造，可能产生 I/O 竞争
- 与批次32 H05、批次31 G05 属同一模式

**修复建议**:
```python
import threading
_instance = None
_lock = threading.Lock()

def get_locale_manager() -> LocaleManager:
    global _instance
    if _instance is None:
        with _lock:
            if _instance is None:
                _instance = LocaleManager()
                _instance.load_saved_locale()
    return _instance
```

#### K12 - `Recorder` 停止后 pending 的 QTimer 回调仍可能执行

**文件**: `src/gui/pyqt6/scripting/recorder.py:176-183`
**风险等级**: 信息
**影响**: 停止录制后，debounce timer 仍可能触发并将文本变更追加到已停止的脚本

**问题代码**:
```python
def _record_text_change(self, widget):
    # ...
    timer = QTimer(self)
    timer.setSingleShot(True)
    timer.timeout.connect(lambda: self._emit_text_action(obj_name, widget_type, action_type, text))
    self._text_timers[obj_name] = timer
    self._pending_text[obj_name] = text
    self._text_timers[obj_name].start(800)
    # ❌ stop() 时 _text_timers.clear() 但不 stop 正在 pending 的 timer
```

**问题分析**:
- `Recorder.stop()` 调用 `self._text_timers.clear()` 和 `self._pending_text.clear()`
- 但 `clear()` 只清空字典，不停止已启动的 QTimer
- 已启动的 QTimer 在 800ms 后触发 `timeout`，调用 lambda 回调
- lambda 回调调用 `_emit_text_action`，其中检查 `if not self._active: return`（第187行）
- 所以当前代码通过 `_active` 标志提供了基本保护
- 但 `_emit_text_action` 中的 lambda 捕获了 `text` 变量，如果录制已停止，回调仍会执行（只是被 `_active` 检查拦截）
- 修复：stop() 时显式 stop 所有 pending timer

## 6. 结论

本次审计发现12个新问题，其中3个中危、5个低危、4个信息。历史报告零修正。

**关键发现**:
1. **K01（中危）**: `AnimatedButton` 使用 Python `property()` 模拟 Qt 属性，`QPropertyAnimation` 无法驱动——按钮悬停动画完全失效
2. **K02（中危）**: `Player._schedule_next` 创建 QTimer 但不清理旧 timer，`stop()` 可能无法阻止 pending 的回调
3. **K03（中危）**: `Recorder._should_skip` 对输入框的跳过逻辑矛盾——点击输入框不录制但文本变更录制，回放可能丢失聚焦步骤
4. **K04-K08**: 低危问题涵盖坐标映射冗余、图标尺寸不一致、变量重复定义、模型验证缺失、缓存无上限
5. **K09-K12**: 信息级问题涵盖 painter 状态检查、播放状态边界、单例线程安全、timer 清理

**建议修复顺序**:
1. **K01**: 修复 AnimatedButton 动画系统（中危，UX 影响明显）
2. **K02**: 修复 Player QTimer 清理（中危，稳定性）
3. **K03**: 统一 Recorder 跳过逻辑（中危，录制完整性）
