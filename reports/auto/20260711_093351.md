### 2.1 🔴 严重问题

#### T01 - `test_error_paths.py` 与 `test_istina_runtime.py` 重复定义 `_FakeMaaEndRuntime`，测试隔离破坏

**文件**: `tests/test_error_paths.py:102-131` vs `tests/test_istina_runtime.py:191-242`
**风险等级**: 🔴 严重
**影响**: 两个测试文件在同一个 pytest 进程中运行时，后加载的文件覆盖先加载的类定义

**问题代码**:
```python
# test_istina_runtime.py:191-242
class _FakeMaaEndRuntime:
    def __init__(self, connect_result=False, load_resource_result=False, ...):
        self._connect_result = connect_result
        self._connected = True
    # ... 完整实现

# test_error_paths.py:102-131
class _FakeMaaEndRuntime:  # ❌ 同名类，不同实现
    def __init__(self):
        self._connected = False
    # ... 简化实现
```

**分析**:
- 两个测试文件在同一个 pytest 进程中，`_FakeMaaEndRuntime` 在模块级别定义
- Python 模块导入顺序不确定——如果 `test_error_paths.py` 先加载，其简化版覆盖 `test_istina_runtime.py` 的完整版
- `test_istina_runtime.py` 的测试（如 `test_execute_routes_task_run`）依赖 `_FakeMaaEndRuntime(run_task_result=True)` 的完整参数签名
- 如果被 `test_error_paths.py` 的简化版覆盖，`__init__` 不接受 `run_task_result` 参数，测试崩溃
- pytest 的默认测试顺序按文件名排序（`test_error_paths` < `test_istina_runtime`），所以 `test_error_paths` 先加载
- 这意味着 **`test_istina_runtime.py` 的所有测试实际使用的是简化版 fake**，测试通过但验证不完整
- 对比: `_FakeProcess` 在 `conftest.py` 和 `test_cli_bridge.py` 中也重复定义（T07），但作用域不同（conftest 的 `_FakeProcess` 是 fixture，不冲突）

**修复建议**:
```python
# 方案1: 合并到一个共享 test_utils.py
# tests/test_utils.py
class FakeMaaEndRuntime:
    def __init__(self, **kwargs):
        self._connect_result = kwargs.get("connect_result", False)
        self._connected = kwargs.get("connected", True)
        ...

# 方案2: 使用唯一的类名
class _FullFakeMaaEndRuntime:  # test_istina_runtime.py
class _SimpleFakeMaaEndRuntime:  # test_error_paths.py
```

### 2.2 🟡 中危问题

#### T02 - `conftest.py` 全局禁用项目日志系统，测试中所有日志静默丢失

**文件**: `tests/conftest.py:17-19`
**风险等级**: 🟡 中危
**影响**: 测试失败时无法通过日志诊断问题

**问题代码**:
```python
@pytest.fixture(autouse=True)
def _disable_broken_project_logging() -> None:
    for method in ("debug", "info", "warning", "error", "critical", "log", "exception"):
        setattr(logging.Logger, method, lambda self, *args, **kwargs: None)
```

**分析**:
- 此 fixture 是 `autouse=True`——**所有测试自动执行**
- 它 monkeypatch `logging.Logger` 的所有日志方法为 no-op  lambdas
- 这意味着：
  1. 测试失败时没有任何日志输出可供诊断
  2. 项目中的 `ProjectLogger` 包装器（`src/core/foundation/logger.py`）内部调用 `logging.Logger` 方法，全部被静默
  3. 如果测试触发了项目代码中的 warning/error，这些日志不会出现在 pytest 输出中
- 注释称"broken project logging"——暗示项目日志系统有 bug 导致测试失败
- 但更好的做法是修复日志系统本身（如确保 `init_logger()` 在测试中正确初始化），而非禁用所有日志
- 副作用：如果测试需要验证日志输出（如 `caplog` fixture），此 monkeypatch 会破坏验证

**修复建议**:
```python
# 方案1: 修复项目日志系统以兼容测试环境
@pytest.fixture(autouse=True)
def _init_project_logging():
    from core.foundation.logger import init_logger
    init_logger(log_dir=Path(".tmp/test-logs"), log_level=logging.DEBUG, console_level=logging.WARNING)

# 方案2: 只禁用文件日志处理器，保留控制台输出
@pytest.fixture(autouse=True)
def _disable_file_logging_only():
    root_logger = logging.getLogger()
    for handler in root_logger.handlers[:]:
        if isinstance(handler, logging.FileHandler):
            root_logger.removeHandler(handler)
```

#### T03 - `test_istina_cli_commands.py` 模块级 `_can_execute_tasks()` 在导入时执行子进程调用

**文件**: `tests/test_istina_cli_commands.py:19-48`
**风险等级**: 🟡 中危
**影响**: 测试收集阶段阻塞 20+ 秒，且可能触发副作用

**问题代码**:
```python
def _can_execute_tasks() -> bool:
    proc = subprocess.run(
        CLI + ["device", "info"],
        cwd=str(PROJECT_ROOT),
        capture_output=True,
        timeout=5,
    )
    ...
    proc2 = subprocess.run(
        CLI + ["system", "connect"],
        cwd=str(PROJECT_ROOT),
        capture_output=True,
        timeout=15,
    )
    ...

_CAN_EXECUTE_TASKS = _can_execute_tasks()  # ❌ 模块导入时执行
```

**分析**:
- `_can_execute_tasks()` 在模块级别调用（非函数内），意味着 **import test_istina_cli_commands 时立即执行**
- 两次 `subprocess.run` 调用：`device info`（5秒超时）+ `system connect`（15秒超时）
- 如果设备未连接，两次调用都会超时 → 测试收集阶段阻塞 20 秒
- 更严重的是：`system connect` 会尝试连接真实设备！——这不是纯单元测试行为
- `_CAN_EXECUTE_TASKS` 是模块级常量，其值在测试运行前就已确定
- 如果用户在运行测试时连接了设备，`_CAN_EXECUTE_TASKS = True`，所有 `@pytest.mark.skipif` 标记变为执行态
- 这可能导致非预期地执行真实设备操作（发送任务、执行导航等）

**修复建议**:
```python
# 方案1: 使用 pytest fixture 延迟执行
@pytest.fixture(scope="session")
def can_execute_tasks():
    return _can_execute_tasks()

# 方案2: 使用 pytest.mark.skipif 的条件函数
def _skip_if_no_device():
    try:
        return not _can_execute_tasks()
    except Exception:
        return True

# 方案3: 完全移除模块级调用，仅使用 skipif 标记
_CAN_EXECUTE_TASKS = None  # 延迟初始化

def _get_can_execute_tasks():
    global _CAN_EXECUTE_TASKS
    if _CAN_EXECUTE_TASKS is None:
        _CAN_EXECUTE_TASKS = _can_execute_tasks()
    return _CAN_EXECUTE_TASKS
```

#### T04 - `test_runtime_execute_with_none_params_does_not_crash` 测试验证的是已报告为 bug 的行为

**文件**: `tests/test_error_paths.py:76-83`
**风险等级**: 🟡 中危
**影响**: 测试将 bug 行为固化为"预期行为"

**问题代码**:
```python
def test_runtime_execute_with_none_params_does_not_crash() -> None:
    runtime = IstinaRuntime()
    runtime._maaend = _FakeMaaEndRuntime()
    result = runtime.execute("task.run", None)  # ❌ params=None
    assert isinstance(result, bool)  # ❌ 断言通过，但行为是 bug
```

**分析**:
- `runtime.execute("task.run", None)` 传入 `None` 作为 params
- `execute()` 内部做 `params = params or {}`（line 274），所以 None → {}
- 然后 `_run_task(params)` → `_run_task({})` → `name = params.get("name")` → `name = None`
- 然后 `runtime.run_task(None, {}, None)` → `_FakeMaaEndRuntime.run_task` 返回 False
- 测试断言 `isinstance(result, bool)` 通过（False 是 bool）
- **但此测试文件名是 `test_error_paths`**，暗示测试的是"错误路径"——传入 None 应该是错误场景
- 批次39 P02 分析了 `_handle_config_set` 的配置注入风险，批次37 M02 分析了 `execute()` 返回 None 的契约问题
- 此测试没有验证 `execute(None, None)` 应该返回错误 dict 而非 bool——它断言了当前 bug 行为为正确

**修复建议**:
```python
def test_runtime_execute_with_none_params_returns_error() -> None:
    runtime = IstinaRuntime()
    result = runtime.execute("task.run", None)
    assert isinstance(result, dict)  # 应返回 error dict，而非 bool
    assert result.get("status") == "error"
```

#### T05 - `test_template_pipeline.py` 单例 `TemplateRegistry` 状态在测试间泄漏

**文件**: `tests/test_template_pipeline.py`
**风险等级**: 🟢 低危
**影响**: 测试间状态泄漏导致虚假通过/失败

**问题代码**:
```python
def test_template_registry_singleton():
    r1 = TemplateRegistry()
    r2 = TemplateRegistry()
    assert r1 is r2  # ✓ 验证单例

def test_template_registry_load_module():
    registry = TemplateRegistry()
    registry.clear()  # ← 清理，但影响全局
    count = registry.load_module("Common")
    ...
```

**分析**:
- `TemplateRegistry` 是单例（批次31 G01, 批次32 A1 已分析）
- `test_template_registry_load_module` 调用 `registry.clear()` 清空全局单例
- 如果 `test_template_registry_singleton` 在 `test_template_registry_load_module` 之后运行，`r1 is r2` 仍为 True（同一单例），但单例已被清空
- 测试通过但验证不完整——singleton 属性正确，但内容已被前一个测试清空
- `load_all()` 在 `test_template_registry_load_all` 中加载所有模块，但不清除——后续测试看到的可能是上一个测试加载的内容

**修复建议**: 使用 `registry.clear()` 的测试应在 fixture 中清理，确保每个测试从干净状态开始。
