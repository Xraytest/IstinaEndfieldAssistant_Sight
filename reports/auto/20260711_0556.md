## 一、新发现（代码错误优先）

### B4 — `pipeline_runner._pick_next` 兜底返回带括号令牌，导致 `run()` 自旋（Medium）

**根因分析**

`src/core/capability/element_recognition/pipeline/pipeline_runner.py:315-325`：

```python
315:    def _pick_next(self, node, graph):
316:        if not node.next:
317:            return None
318:        for next_name in node.next:
319:            if next_name.startswith("["):
320:                continue
321:            if graph.get_node(next_name) is not None:
322:                return next_name
323:        return node.next[0] if node.next else None    # ← 残留缺陷点
```

0449 批次的 B2 描述的是"循环体只有 `continue`、从不 `return`"的缺陷形态。当前 `main` 的代码**已经修复了循环本体**（第 321-322 行已在循环内对第一个合法分支 `return`）。但 **fallback 第 323 行未被修复**：

- 当 `node.next` 的所有条目**要么以 `[` 开头（条件跳转令牌），要么在图中不存在**时，循环体不会 `return`，落到第 323 行。
- 第 323 行 `return node.next[0]` 直接返回 **`node.next` 的第一个条目**——若它就是 `"[Hit]X"` 这类条件令牌，则返回的仍是一个**带 `[` 前缀、图中不存在的死令牌**。

紧接着在 `run()` 中：

```python
90:            next_name = self._pick_next(current, graph)
91:            if next_name is None:
92:                break
93:            nxt = graph.get_node(next_name)
94:            if nxt is not None:
95:                current = nxt
```

`next_name` 为非 `None` 字符串 → 第 91 行 `break` 不触发；`graph.get_node("[Hit]X")` 返回 `None` → 第 94 行 `current` **保持不变**。下一轮 `while` 仍以同一 `current` 进入，再次调用 `_pick_next` 得到同一死令牌，`current` 依旧不变……形成**自旋**：直到 `step` 累加到 `max_steps`（默认 100，`run_pipeline` 为 200）才因 `while current and step < max_steps` 退出。

自旋期间每个循环都重复执行 `_evaluate`（重复识别）并累加 `_hit_counts[current.name]`，最终返回一个**状态为 `matched`/`no_match`（取决于最后一次识别结果）、同一节点被执行 `max_steps` 次的错误结果**，且永不触发正确的"流水线结束"。

**直接原因**：0449 B2 的部分修复只补全了"首个合法分支优先"的逻辑（循环内 `return`），却保留了会回退返回带括号死令牌的 `return node.next[0]`。
**根本原因**：条件跳转令牌（`[...]`）与"无有效默认分支"的语义未被正确折叠——无默认分支时应显式结束流水线（`return None`），而非返回第一个（可能是条件）令牌。

**修改方案（最小可行，与 0449 对 B2 的建议一致）**

```python
def _pick_next(self, node, graph):
    if not node.next:
        return None
    for next_name in node.next:
        if next_name.startswith("["):
            continue
        if graph.get_node(next_name) is not None:
            return next_name
    return None
```

要点：循环体内已能正确处理"首个合法分支优先"；当所有 `next` 均为条件令牌或图中不存在时，**返回 `None`** 让 `run()` 第 91 行 `break` 正常结束流水线，消除自旋。

**影响面**
- 调用点：`PipelineRunner.run()`（行 90）、`_transition()`（行 330）。
- 触发条件：当且仅当某节点的 `next` 仅含条件令牌（如 `["[Hit]X"]`）或仅含图中不存在的节点名。单条目普通 `next`、含普通分支的 `next` 不受影响。
- 表现：此类节点本应"条件不满足则结束流水线"，实际却让 `run()` 空转 `max_steps` 次并返回错误结果——识别/点击被无效重复，流水线结果失真。

**非期待变化（回退策略）**
- 修改后，纯条件跳转节点（无默认分支）将正确以 `None` 结束流水线，而非自旋。若确有节点依赖"条件令牌兜底"（罕见），应在 JSON 中显式补一个默认 `next`。
- 回退：若担心回归，可暂时保留第 323 行但改为 `return None`，优先保证不自旋、不崩溃。
