### 2.1 🟡 中危问题

#### F01 - `touch_manager.py:back()` 无异常处理，静默吞错

**文件**: `src/core/capability/device/touch_manager.py:53-56`
**风险等级**: 🟡 中危
**影响**: 返回键操作失败时无日志、无重试，用户无法感知

**问题代码**:
```python
def back(self, serial: Optional[str] = None) -> None:
    import subprocess
    args = self._adb_base_args(serial) + ["shell", "input", "keyevent", "KEYCODE_BACK"]
    subprocess.check_output(args, timeout=10)  # ❌ 无 try/except
```

**问题分析**:
- `tap()`（line 24-30）、`swipe()`（line 32-38）、`long_press()`（line 46-51）均有 try/except + `_logger.error` + `raise`
- `back()` 是唯一无异常处理的方法
- `subprocess.check_output` 在 ADB 断开、设备 busy、timeout 时抛出 `subprocess.TimeoutExpired` / `FileNotFoundError`
- 异常直接传播到调用方，调用方若未捕获则崩溃；若被上层通用 except 吞掉则完全静默
- 返回键是导航核心操作（退出弹窗、关闭面板），失败影响面广

**修复建议**:
```python
def back(self, serial: Optional[str] = None) -> None:
    import subprocess
    args = self._adb_base_args(serial) + ["shell", "input", "keyevent", "KEYCODE_BACK"]
    try:
        subprocess.check_output(args, timeout=10)
    except Exception as e:
        self._logger.error(LogCategory.EXECUTION, "返回键失败", serial=serial, error=str(e))
        raise
```

### 2.2 🟢 低危问题

#### F02 - `recovery.py:_clear_canvas` 三个 try/except pass 全部吞错

**文件**: `src/core/capability/device/recovery.py:81-94`
**风险等级**: 🟢 低危
**影响**: 画布清理/唤醒/回Home全部失败时静默跳过，应用启动状态不确定

**问题代码**:
```python
def _clear_canvas(self, serial: Optional[str]) -> None:
    try:
        self._run(["shell", "wm", "dismiss-keyguard"], serial)
    except Exception:    # ❌ pass - 锁屏解除失败无感知
        pass
    try:
        self._run(["shell", "input", "keyevent", "KEYCODE_WAKEUP"], serial)
    except Exception:    # ❌ pass - 唤醒失败无感知
        pass
    try:
        self._run(["shell", "input", "keyevent", "KEYCODE_HOME"], serial)
    except Exception:    # ❌ pass - Home键失败无感知
        pass
```

**问题分析**:
- 三个清理步骤全部用 `except Exception: pass` 吞掉所有异常
- `_force_stop`（line 70-79）至少会 `_logger.warning` 记录失败
- 但 `_clear_canvas` 在 `_force_stop` 之后执行，若清理失败后续 `_launch` 仍继续
- 可能的场景：设备处于锁屏/黑屏状态，清理步骤全部失败，应用启动后仍被锁屏遮挡
- 与 F01 同源：设备层操作缺乏错误可观测性

**修复建议**:
```python
def _clear_canvas(self, serial: Optional[str]) -> None:
    steps = [
        ("dismiss-keyguard", ["shell", "wm", "dismiss-keyguard"]),
        ("KEYCODE_WAKEUP", ["shell", "input", "keyevent", "KEYCODE_WAKEUP"]),
        ("KEYCODE_HOME", ["shell", "input", "keyevent", "KEYCODE_HOME"]),
    ]
    for label, args in steps:
        try:
            self._run(args, serial)
        except Exception as exc:
            self._logger.warning(
                LogCategory.MAIN,
                "清理画布步骤失败",
                step=label,
                serial=serial,
                error=str(exc),
            )
```

#### F03 - `template_registry.py` 单例非线程安全

**文件**: `src/core/capability/element_recognition/pipeline/template_registry.py:22-25`
**风险等级**: 🟢 低危
**影响**: 多线程同时首次访问时创建多个实例，共享类变量污染

**问题代码**:
```python
def __new__(cls) -> TemplateRegistry:
    if cls._instance is None:        # ❌ 非原子检查
        cls._instance = super().__new__(cls)  # ❌ 多线程可穿透
    return cls._instance
```

**问题分析**:
- 批次29 SECPEN.md 已记录 ThemeManager 同类问题（N-7/N-8/N-9）
- TemplateRegistry 使用相同模式：`if cls._instance is None` 非原子检查
- 当 GUI 主线程和后台识别线程同时首次调用 `get_instance()` 或 `TemplateRegistry()` 时
- 两个线程同时看到 `_instance is None`，各自创建实例
- 一个实例的 `__init__` 中设置的 `self._root` 等属性只存在于其中一个实例上
- 类变量 `_templates`、`_template_paths` 等虽共享，但实例属性 `_root` 不共享
- 实际影响取决于调用方式——如果所有代码都用 `TemplateRegistry()` 构造器，则可能创建多实例

**修复建议**:
```python
def __new__(cls) -> TemplateRegistry:
    if cls._instance is None:
        with threading.Lock():
            if cls._instance is None:
                cls._instance = super().__new__(cls)
    return cls._instance
```

#### F04 - `template_registry.py:resolve()` 模糊匹配误匹配

**文件**: `src/core/capability/element_recognition/pipeline/template_registry.py:106-119`
**风险等级**: 🟢 低危
**影响**: 模板查找返回错误的模板图像

**问题代码**:
```python
def resolve(self, template_ref: str) -> Optional[np.ndarray]:
    # ...精确匹配...
    for tpl_key, tpl_img in self._templates.items():
        if tpl_key.endswith((key, key.replace("/", "_"))):   # ⚠️ 模糊匹配
            return tpl_img
        if key.endswith(tpl_key.split("/")[-1]):              # ⚠️ 后缀匹配
            return tpl_img
```

**问题分析**:
- 精确匹配（line 111-113）是主要路径
- 模糊匹配（line 114-118）是回退路径
- `tpl_key.endswith(key)` — 若 `key="btn"`，可能匹配 `"menu/btn_confirm"` 而非 `"ui/btn"`
- `key.endswith(tpl_key.split("/")[-1])` — 若 `key="confirm"`，可能匹配 `"ui/btn_confirm"` 或 `"ui/dialog_confirm"`
- 两者结合产生意外交叉：搜索 `"btn"` 时可能先匹配到 `"ui/btn_confirm"` 而非 `"ui/btn"`
- 在密集模板场景（大量UI元素），误匹配率升高
- 精确匹配已足够覆盖正常用例，模糊匹配应在找不到精确匹配时才启用，且需更严格的条件

**修复建议**:
```python
def resolve(self, template_ref: str) -> Optional[np.ndarray]:
    if template_ref.endswith(".png"):
        key = template_ref[:-4]
    else:
        key = template_ref
    img = self._templates.get(key)
    if img is not None:
        return img
    # 模糊匹配仅在精确匹配失败时启用，且限定为完整组件名匹配
    for tpl_key, tpl_img in self._templates.items():
        component = tpl_key.split("/")[-1]
        if component == key or component == key.replace("/", "_"):
            return tpl_img
    return None
```

#### F05 - `task_runner.py:execute_preset` 首个任务失败即终止

**文件**: `src/core/capability/element_recognition/tasks/task_runner.py:40-57`
**风险等级**: 🟢 低危
**影响**: 预设任务流中前置步骤失败时，后续步骤被跳过

**问题代码**:
```python
def execute_preset(self, screen: np.ndarray, preset_name: str) -> List[Dict[str, Any]]:
    # ...
    for task_entry in task_list:
        name = task_entry.get("name")
        options = task_entry.get("option") or {}
        result = self.execute_task(screen, name, options)
        results.append(result)
        if result.get("status") == "error":   # ❌ 首个error即break
            break
    return results
```

**问题分析**:
- 预设（preset）是多个任务的顺序执行流
- 当前逻辑：任何任务返回 error 就立即 break，后续任务全部跳过
- 这破坏了"执行全部任务并报告结果"的预期语义
- 合理的预设语义应是：尽量执行全部任务，每个任务独立报告状态
- 例如：预设包含 5 个任务，第 2 个失败时，第 3-5 个应继续执行
- 调用方可以根据 `results` 列表判断整体执行状态

**修复建议**:
```python
def execute_preset(self, screen: np.ndarray, preset_name: str) -> List[Dict[str, Any]]:
    preset = self._task_loader.presets().get(preset_name)
    if not preset:
        return [{"status": "error", "message": f"Preset '{preset_name}' not found"}]
    results = []
    task_list = preset.get("task", [])
    for task_entry in task_list:
        name = task_entry.get("name")
        options = task_entry.get("option") or {}
        result = self.execute_task(screen, name, options)
        results.append(result)
        # 不中断，继续执行后续任务
    return results
```

#### F06 - `task_runner.py:_build_option_override` 仅处理 switch 类型

**文件**: `src/core/capability/element_recognition/tasks/task_runner.py:81-99`
**风险等级**: 🟢 低危
**影响**: 非 switch 类型的选项定义被忽略，预设参数覆盖不生效

**问题代码**:
```python
def _build_option_override(self, opt_def: Dict[str, Any], value: Any) -> Dict[str, Any]:
    result: Dict[str, Any] = {}
    opt_type = opt_def.get("type", "switch")
    cases = opt_def.get("cases", [])
    if opt_type == "switch":
        # ...仅处理switch
        return result
    return result  # ❌ 其他类型直接返回空dict
```

**问题分析**:
- `opt_type` 默认为 `"switch"`，但如果选项定义使用其他类型（如 `"select"`、`"input"`、`"range"` 等）
- `_build_option_override` 只处理 `switch` 类型，其他类型直接返回空 dict
- `_build_task_graph`（line 74）调用 `_build_option_override` 并将结果添加到 PipelineGraph
- 非 switch 选项的覆盖逻辑被完全忽略
- 这限制了选项系统的扩展性——未来添加新选项类型时，此函数需要同步更新
- 当前实际影响取决于是否存在非 switch 类型的选项定义

**修复建议**:
```python
def _build_option_override(self, opt_def: Dict[str, Any], value: Any) -> Dict[str, Any]:
    result: Dict[str, Any] = {}
    opt_type = opt_def.get("type", "switch")
    if opt_type == "switch":
        cases = opt_def.get("cases", [])
        case_name = "Yes" if value else "No"
        for case in cases:
            if case.get("name") == case_name:
                result.update(case.get("pipeline_override") or {})
                return result
        default_case = opt_def.get("default_case")
        if default_case:
            for case in cases:
                if case.get("name") == default_case:
                    result.update(case.get("pipeline_override") or {})
                    return result
    # 其他类型暂不支持，记录日志以便后续扩展
    elif opt_type:
        logger.debug("选项类型 '%s' 暂不支持覆盖，值=%s", opt_type, value)
    return result
```

#### F07 - `log_page.py:_read_config` 损坏配置无备份

**文件**: `src/gui/pyqt6/pages/log_page.py:146-152`
**风险等级**: 🟢 低危
**影响**: 客户端配置损坏时丢失设备历史等设置

**问题代码**:
```python
def _read_config(self) -> Dict[str, Any]:
    if not self._config_path.exists():
        return {}
    try:
        return json.loads(self._config_path.read_text(encoding="utf-8"))
    except json.JSONDecodeError:
        return {}  # ❌ 静默返回空dict，损坏文件未被备份
```

**问题分析**:
- 与批次28 D05（`settings_page.py`）和批次29 D05扩展（`device_settings_page.py`）完全相同的问题
- `client_config.json` 包含：设备历史、自动重连设置、LLM 配置等
- 损坏时用户丢失所有配置
- `QueueState` 类已有 `_backup_corrupt_state` 方法实现备份逻辑
- 三个 settings/log 页面均未复用此模式

**修复建议**:
```python
def _read_config(self) -> Dict[str, Any]:
    if not self._config_path.exists():
        return {}
    try:
        return json.loads(self._config_path.read_text(encoding="utf-8"))
    except json.JSONDecodeError:
        self._backup_corrupt_config()
        return {}

def _backup_corrupt_config(self) -> None:
    try:
        backup = self._config_path.with_suffix(".json.bak")
        counter = 0
        while backup.exists():
            backup = self._config_path.with_suffix(f".json.bak.{counter}")
            counter += 1
        import os
        os.replace(self._config_path, backup)
    except Exception:
        pass
```

3. **D05/F07**: 统一三处配置备份逻辑
