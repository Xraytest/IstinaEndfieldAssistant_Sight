## 2. 新发现漏洞与错误

### 2.1 🟡 高危问题

#### N01 - runtime.py _nav3_walk 使用未初始化的 _llm_client

**文件**: `src/core/service/runtime.py:697`  
**风险等级**: 🟡 高危  
**影响**: VLM导航调用时LLM客户端为None，导致导航功能静默失败

**问题代码**:
```python
def _nav3_walk(self, params: Dict[str, Any]) -> Dict[str, Any]:
    nav = self.navigator()
    return nav.to_coords_vlm(
        ...
        llm_client=self._llm_client,  # ❌ 直接使用_raw属性，可能为None
        ...
    )
```

**问题分析**:
- `self._llm_client`在`__init__`中被初始化为`None`
- `self._llm_client`只有在通过`_llm_client_instance` property访问时才会被懒加载
- `_nav3_walk`直接传递`self._llm_client`（可能为None），而非`self._llm_client_instance`
- 如果LLM尚未初始化，VLM导航会收到None客户端，导致运行时错误或静默失败

**修复建议**:
```python
def _nav3_walk(self, params: Dict[str, Any]) -> Dict[str, Any]:
    nav = self.navigator()
    return nav.to_coords_vlm(
        map_name=params.get("map_name", ""),
        x=float(params.get("x", 0)),
        y=float(params.get("y", 0)),
        level_id=params.get("level_id"),
        zone_override=params.get("zone"),
        llm_client=self._llm_client_instance,  # ✅ 使用property确保初始化
        max_steps=int(params.get("max_steps", 40)),
        keyevent_fn=self._vlm_keyevent,
    )
```

### 2.2 🟢 中危问题

#### N02 - llm/runtime.py health_check 误设 _ready=False

**文件**: `src/core/capability/llm/runtime.py:140-155`  
**风险等级**: 🟢 中危  
**影响**: 临时网络波动导致llama-server被误判为未运行

**问题代码**:
```python
def health_check(self) -> bool:
    try:
        url = f"http://127.0.0.1:{self._default_port}/health"
        with urllib.request.urlopen(url, timeout=2) as resp:
            data = resp.read().decode("utf-8", errors="replace")
            if "ready" in data.lower() or '"status"' in data.lower():
                self._ready = True
                return True
    except Exception:
        pass
    if self._process is not None and self._process.poll() is None:
        return False  # ❌ 进程在运行但HTTP检查失败，_ready保持True？不，下面会设为False
    self._ready = False
    return False
```

**问题分析**:
- 如果`self._process`正在运行（`poll() is None`），但HTTP请求超时或失败
- `health_check`返回False，并且`self._ready`被设置为False
- 这会导致`start()`方法认为服务未运行，可能尝试重新启动
- 但`start()`开头检查`self._process and self._process.poll() is None`，如果进程正在运行，会直接返回`self.health_check()`的结果（False）
- 结果：进程在运行但服务被认为不可用，所有LLM调用失败

**修复建议**:
```python
def health_check(self) -> bool:
    try:
        url = f"http://127.0.0.1:{self._default_port}/health"
        with urllib.request.urlopen(url, timeout=2) as resp:
            data = resp.read().decode("utf-8", errors="replace")
            if "ready" in data.lower() or '"status"' in data.lower():
                self._ready = True
                return True
    except Exception:
        pass
    # 如果进程正在运行，保持ready状态（可能是临时网络问题）
    if self._process is not None and self._process.poll() is None:
        return self._ready  # 保持之前的状态，不翻转为False
    self._ready = False
    return False
```

#### N03 - pipeline_node.py get_node_or_entry 返回意外节点

**文件**: `src/core/capability/element_recognition/pipeline/pipeline_node.py:126-130`  
**风险等级**: 🟢 中危  
**影响**: 节点查找失败时返回entry_points的第一个节点，而非None，导致静默使用错误的节点

**问题代码**:
```python
def get_node_or_entry(self, name: str) -> Optional[PipelineNode]:
    node = self.nodes.get(name)
    if node is None and self.entry_points:
        node = self.nodes.get(self.entry_points[0])  # ❌ 回退到第一个entry point
    return node
```

**问题分析**:
- 如果`name`不在`self.nodes`中，且`self.entry_points`非空
- 会返回`self.entry_points[0]`对应的节点，而非None
- 调用者可能期望返回None以处理"节点未找到"的情况
- 静默返回entry point节点可能导致执行错误的pipeline流程

**修复建议**:
```python
def get_node_or_entry(self, name: str) -> Optional[PipelineNode]:
    node = self.nodes.get(name)
    if node is None:
        # 返回None，让调用者处理未找到的情况
        # 如需回退到entry point，应在调用方明确处理
        return None
    return node
```

#### N04 - pipeline_node.py merge 导致重复 entry_points

**文件**: `src/core/capability/element_recognition/pipeline/pipeline_node.py:135-137`  
**风险等级**: 🟢 中危  
**影响**: 合并PipelineGraph时entry_points重复，可能导致pipeline执行重复入口

**问题代码**:
```python
def merge(self, other: PipelineGraph) -> None:
    self.nodes.update(other.nodes)
    self.entry_points.extend(other.entry_points)  # ❌ 直接追加，可能重复
```

**问题分析**:
- `self.entry_points.extend(other.entry_points)`直接追加所有entry_points
- 如果两个graph有相同的entry point，合并后会重复
- 后续遍历entry_points时会重复执行相同的入口节点

**修复建议**:
```python
def merge(self, other: PipelineGraph) -> None:
    self.nodes.update(other.nodes)
    existing = set(self.entry_points)
    for ep in other.entry_points:
        if ep not in existing:
            self.entry_points.append(ep)
            existing.add(ep)
```

### 2.3 🟢 低危问题

#### N05 - matcher.py ROI负数坐标回绕处理

**文件**: `src/core/capability/element_recognition/pipeline/matcher.py:33-39`  
**风险等级**: 🟢 低危  
**影响**: ROI负数坐标通过回绕处理可能导致意外的裁剪区域

**问题代码**:
```python
if roi:
    rx, ry, rw, rh = roi
    if rx < 0:
        rx = w_img + rx  # ❌ 回绕处理：rx=-10 → rx=w_img-10，可能超出预期
    if ry < 0:
        ry = h_img + ry
    screen_roi = screen_gray[ry:ry + rh, rx:rx + rw]
```

**问题分析**:
- 当`rx`为负数时，代码将其转换为`w_img + rx`
- 例如，如果`w_img=1920`, `rx=-10`，则`rx=1910`
- 如果`rw`是正值（如100），裁剪区域为`screen_gray[1910:2010, ...]`
- 但`2010 > 1920`，numpy切片会自动截断，导致实际裁剪区域比预期小
- 如果调用者期望`rx=-10`表示"从左边减去10像素"，这个回绕处理可能不符合预期

**修复建议**:
```python
if roi:
    rx, ry, rw, rh = roi
    rx = max(0, rx)  # 限制为非负数，避免意外的回绕
    ry = max(0, ry)
    rw = min(rw, w_img - rx)
    rh = min(rh, h_img - ry)
    if rw <= 0 or rh <= 0:
        return []
    screen_roi = screen_gray[ry:ry + rh, rx:rx + rw]
```

#### N06 - element_info.py action 验证不完整

**文件**: `src/core/capability/element_recognition/element_info.py:67-68`  
**风险等级**: 🟢 低危  
**影响**: 合法的NodeAction值（如"Click", "Swipe"）被强制转换为"unknown"

**问题代码**:
```python
def __post_init__(self) -> None:
    if self.element_type not in ELEMENT_TYPES:
        self.element_type = "unknown"
    if self.action not in ("tap", "swipe", "none", "unknown"):  # ❌ 只允许4个值
        self.action = "unknown"
```

**问题分析**:
- `NodeAction`枚举包含：Click, Swipe, KeyDown, KeyUp, DoNothing, StopTask, Custom, StartApp
- 但`ElementInfo.__post_init__`只允许`"tap", "swipe", "none", "unknown"`
- 从pipeline_node.py创建的ElementInfo（通过pipeline_runner）可能包含"Click"等值
- 这些合法值会被强制转换为"unknown"，导致下游无法正确执行动作

**修复建议**:
```python
def __post_init__(self) -> None:
    if self.element_type not in ELEMENT_TYPES:
        self.element_type = "unknown"
    # 允许所有已知动作类型，包括NodeAction枚举值
    valid_actions = ("tap", "swipe", "none", "unknown", "click", "keydown", "keyup", "donothing", "stoptask", "custom", "startapp")
    if self.action not in valid_actions:
        self.action = "unknown"
```

#### N07 - cli_bridge.py _build_args 命令分割问题

**文件**: `src/gui/pyqt6/cli_bridge.py:89-99`  
**风险等级**: 🟢 低危  
**影响**: 带空格的命令参数被错误分割，导致CLI接收到意外的参数序列

**问题代码**:
```python
@staticmethod
def _build_args(command: str, params: Dict[str, Any]) -> List[str]:
    args = command.split()  # ❌ 简单按空格分割，无法处理带空格的参数
    for key, value in params.items():
        ...
    return args
```

**问题分析**:
- `command.split()`按空格分割命令字符串
- 如果command是`"shell am force-stop com.example.app"`，分割为`["shell", "am", "force-stop", "com.example.app"]`
- 但在recovery.py中，`"am force-stop"`是一个整体（ADB shell会把它当作单个命令执行）
- 简单分割破坏了参数的原始语义

**修复建议**:
```python
@staticmethod
def _build_args(command: str, params: Dict[str, Any]) -> List[str]:
    import shlex
    args = shlex.split(command)  # ✅ 使用shlex.split正确处理带引号的参数
    for key, value in params.items():
        if value is None:
            continue
        args.append(f"--{key}")
        if isinstance(value, (dict, list)):
            args.append(json.dumps(value, ensure_ascii=False))
        else:
            args.append(str(value))
    return args
```
