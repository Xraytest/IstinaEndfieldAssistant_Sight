### A1 — `recovery.py._force_stop`「命令失效 / 命令注入」是**错误论断**（D1 与 C01 应降级/移除）

**被复核的既往论断**

- 0020 批次 **D1 (High)**：`recovery.py:72` `_force_stop` 将 `"am force-stop"` 作为单个参数传递，mksh 将其解释为单个命令名而非 `am` + `force-stop`，强制停止**从未生效**。
- FINAL 批次 #26 **C01 (Critical)**：同一行被升级为 Critical，称 `self._package` 未过滤，**存在命令注入风险**。

**复核（当前 `main` 的 `recovery.py:66-79`）**

```python
66:    def _run(self, args: list[str], serial: Optional[str]) -> None:
67:        cmd = self._resolve_adb(serial) + args
68:        subprocess.check_output(cmd, text=True, timeout=30)
70:    def _force_stop(self, serial: Optional[str]) -> None:
72:        self._run(["shell", "am force-stop", self._package], serial)
```

关键点：`_run` 通过 `subprocess.check_output` 以 **list 参数**调用 `adb`，且 `self._package` 是硬编码常量 `"com.hypergryph.endfield"`，**非用户可控输入**。

**逐项驳斥：**

1. **「强制停止从未生效」不成立**。`adb shell` 的语义是：把 `shell` 之后的所有参数用空格**拼接**成一条命令字符串，再交给设备端 `sh -c` 执行。因此 `cmd = ["adb", "-s", serial, "shell", "am force-stop", "com.hypergryph.endfield"]` 在设备上等价于执行 `am force-stop com.hypergryph.endfield`，**这正是 `am force-stop` 的正确用法**，强制停止正常生效。同一文件 `_launch` 用 `["shell", "am", "start", "-n", ...]`（拆分 `"am"`/`"start"`）与 `_force_stop` 用 `["shell", "am force-stop", ...]`（合并 `"am force-stop"`）两种写法**在设备上产生完全一致的命令**——若 D1 成立，则 `_launch` 也应「从未生效」，这显然与实际可启动游戏的事实矛盾。

2. **「命令注入」不成立**。`self._package` 来自构造默认值（`recovery.py:20`），无任何外部输入路径；且 `adb` 以 list 形式调用（`subprocess` 不经由宿主 shell），参数不会被二次解析。即使 package 含空格，也只是该条 `am force-stop` 参数异常，不构成注入。

**结论（纠正）**
- D1（High）与 C01（Critical）的**核心机制均不成立**：命令可正确执行、且无注入面。二者应**从活跃缺陷清单移除**（或降级为 Info 级「代码风格不一致」——`_force_stop` 与 `_launch` 对 `am` 子命令的拼接风格不统一，可统一为 `["shell", "am", "force-stop", pkg]` 提升可读性）。
- 特别提示：C01 将一条**实际正常工作的硬编码命令**标为 **Critical 命令注入**，会在风险台账中制造一个虚高的 Critical 项，误导后续修复优先级排序，建议立即更正。

**结论（纠正）**
