### B1 — `adb_manager._screencap_via_subprocess` 盲目替换 CRLF 破坏二进制 PNG（Medium）

**根因分析**
`src/core/capability/device/adb_manager.py:120-130`：

```python
def _screencap_via_subprocess(self, serial=None) -> bytes:
    ...
    data = subprocess.check_output(args, timeout=self._timeout)
    # ADB on Windows may translate LF to CRLF in screencap output.
    if data[:4] == b"\x89PNG" and b"\x0d\x0a" in data:
        data = data.replace(b"\x0d\x0a", b"\x0a")   # ← 缺陷
    return data
```

`adb shell screencap -p` 在 Windows 上经控制台 pty 时会把 LF（`\x0a`）翻译成 CRLF（`\x0d\x0a`），这是真实问题；但修复手段错误：
- `bytes.replace(b"\x0d\x0a", b"\x0a")` 是**全局、无差别**替换，**包括 PNG 压缩流中恰好出现的 `0D 0A` 字节序列**。
- PNG 的图像数据经 zlib 压缩，本质为近似随机字节，`0D 0A` 组合以约 1/65536 概率出现。一张 ~100–300KB 的截图平均包含数处此类序列；这些字节被改写为 `0A` 后，zlib 流被篡改，解码大概率失败或图像局部损坏。
- 即：该修复把"可能"的 pty 翻译问题，换成了"确定存在"的数据损坏风险。

**直接原因**：用 `shell` 通道截图且用字符串替换"近似"处理二进制。
**根本原因**：未理解 ADB pty 翻译的真实规避手段——`exec-out` 通道不经过 pty，根本不会产生 CRLF 翻译，无需事后修补。

**修改方案（最小可行）**
将 `adb_manager.py:120-130` 的 `_screencap_via_subprocess` 改为走 `exec-out`：

```python
def _screencap_via_subprocess(self, serial=None) -> bytes:
    adb = self._resolve_adb_path()
    args = [adb]
    if serial:
        args += ["-s", serial]
    args += ["exec-out", "screencap", "-p"]   # exec-out 不经 pty，无 CRLF 翻译
    return subprocess.check_output(args, timeout=self._timeout)
```

如必须兼容旧 ADB 版本无 `exec-out`，则至少应先校验 PNG 完整性（`cv2.imdecode` 是否返回非 None），失败再重试，而非盲替换。

**影响面**
- 调用点：`ADBDeviceManager.screencap()`（行 84-95）——仅当 `adbutils` 路径抛异常时走此分支（即 CLI/GUI 截图主路径 `screencap()` 的兜底）。
- 下游：`IstinaRuntime` / `AndroidRuntime` 截图、`main_window._refresh_preview`、导航 `Navigator._get_frame` / `VlmWalkNavigator._grab_frame` 的 PNG 解码。
- 表现：Windows 上偶发截图解码失败、预览空白、或导航定位因坏帧异常。

**非期待变化（回退策略）**
- `exec-out` 在极旧 ADB（< 1.0.36，约 2017 年前）不存在。当前项目依赖 `3rd-part/adb/adb.exe`（现代版本），可安全使用。若需兼容，保留原 `shell` 分支作为异常回退即可。
- 回退：若替换为 `exec-out` 后某环境报错，临时还原为 `shell screencap -p` 并仅在 `cv2.imdecode` 失败时重试一次（不要全局替换）。

### B2 — `pipeline_runner._pick_next` 循环体为空操作，条件跳转被破坏（Medium）

**根因分析**
`src/core/capability/element_recognition/pipeline/pipeline_runner.py:315-325`：

```python
def _pick_next(self, node, graph):
    if not node.next:
        return None
    for next_name in node.next:
        if next_name.startswith("["):
            continue          # ← 循环体内只有 continue，从不 return
    if graph.get_node(next_name) is not None:   # ← next_name 已是最后一个元素
        return next_name
    return node.next[0] if node.next else None   # ← 可能返回带 "[" 的占位令牌
```

缺陷：
1. `for` 循环体内仅 `continue`，**从不 `return`**。循环结束后 `next_name` 停在 `node.next` 的**最后一个**元素。
2. 因此该函数实际只判断"最后一个 next 是否存在"，与"选第一个有效分支"的语义相悖——多分支节点总是选最后一个而非第一个，pipeline 路由错误。
3. 兜底 `return node.next[0]` 会**原样返回以 `[` 开头的条件跳转令牌**（如 `"[Hit]NodeA"`、MaaFW 的 `"[Color]"`，行 321-322 本想跳过它们却未生效）。`graph.get_node("[Hit]NodeA")` 返回 `None` → `run()` 中 `current = None` → 流水线直接结束，条件跳转形同虚设。

**直接原因**：把 `if ...: continue` 写进了循环，却忘了在循环内对"非括号且存在的 next"执行 `return`。
**根本原因**：条件分支（`[` 前缀）与默认分支的路由逻辑被错误地拆成两段，且循环变量泄漏到循环外使用。

**修改方案（最小可行）**

```python
def _pick_next(self, node, graph):
    if not node.next:
        return None
    for next_name in node.next:
        if next_name.startswith("["):
            continue
        if graph.get_node(next_name) is not None:
            return next_name
    return None
```

要点：在循环内对**第一个**非括号、存在的 next 立即 `return`；无有效默认分支时返回 `None`（与 `run()` 的 `if next_name is None: break` 一致），不再回退返回带括号令牌。

**影响面**
- 调用点：`PipelineRunner.run()` 行 90 `_pick_next(current, graph)`、行 330 `_transition()`。
- 影响所有经本地 `PipelineRunner` 图遍历的 pipeline（模板/OCR/And/Or 节点的后继路由）。节点 `next` 含多个条目或含条件条目的图受影响最大；单条目 `next` 不受影响。
- 表现：pipeline 走错分支、条件跳转失效、识别/点击在被错误节点上执行或提前结束。

**非期待变化（回退策略）**
- 修改后，纯条件跳转节点（如 `next: ["[Hit]X"]`）无默认分支时将返回 `None` 结束流水线，而非原先"返回带括号死令牌"——行为更正确（显式结束 vs 静默死分支）。若确有节点依赖"条件条目兜底"，应在 JSON 中显式补充一个默认 next。
- 回退：若引入回归，可暂时保留 `return node.next[0]` 但改为 `return None`（消除死令牌），优先保证不崩溃。

### B3 — `scripts/check_llm_cuda.py` 对空 `choices` 取 `[0]` 触发 IndexError（Low）

**根因分析**
`scripts/check_llm_cuda.py:99-100`：

```python
data = json.loads(resp.read().decode("utf-8", errors="replace"))
print("响应:", data.get("choices", [{}])[0].get("text", "")[:200])
```

当模型返回 `{"choices": []}`（合法但空）时，`data.get("choices", [{}])` 返回 `[]`（键存在，默认值不生效），随后 `[0]` 触发 `IndexError`，脚本崩溃并丢失后续 stderr 诊断输出。

**修改方案**

```python
choices = data.get("choices") or [{}]
text = (choices[0].get("text", "") if choices else "")[:200]
print("响应:", text)
```

**影响面**：仅 `scripts/check_llm_cuda.py` 调试脚本，不影响正式功能；但崩溃会掩盖 CUDA 排查信息。
**非期待变化**：无。属边界脚本，纯健壮性修复。
