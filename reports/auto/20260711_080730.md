### 2.1 🟡 中危问题

#### I01 - `istina.py:_interactive_loop` 交互循环缓冲区无界增长

**文件**: `src/cli/istina.py:340-344`
**风险等级**: 🟡 中危
**影响**: 长时间交互运行中，内存持续增长直至OOM

**问题代码**:
```python
try:
    buffer += chunk
except Exception as exc:
    self_logger.error("CLI 交互循环: buffer 追加异常", error=str(exc))
    break
# ❌ buffer 仅在遇到换行符时清空，无换行输入持续增长
```

**问题分析**:
- 交互循环逐字符读取 stdin（`sys.stdin.read(1)`）
- 每次读到字符追加到 `buffer` 字符串
- buffer 仅在遇到 `\n` 时清空（line 345-347）
- 如果用户输入极长行（如粘贴大段JSON）或 stdin 产生无换行字符流，buffer 无上限增长
- 每轮循环都执行 `buffer += chunk`，每次追加创建新字符串对象（Python字符串不可变）
- 长期运行（数小时交互会话）后内存占用持续上升
- 对比：`main()` 函数使用一次性命令行参数，无此问题

**修复建议**:
```python
MAX_BUFFER_SIZE = 1024 * 1024  # 1MB
try:
    buffer += chunk
    if len(buffer) > MAX_BUFFER_SIZE:
        self_logger.error("CLI 交互循环: buffer 超限，丢弃旧数据")
        buffer = buffer[-MAX_BUFFER_SIZE:]
except Exception as exc:
    self_logger.error("CLI 交互循环: buffer 追加异常", error=str(exc))
    break
```

#### I02 - `handlers.py:_write_or_base64` 写入文件失败时崩溃而非返回错误

**文件**: `src/cli/handlers.py:30-38`
**风险等级**: 🟡 中危
**影响**: 截图保存失败时抛出异常，CLI返回非预期错误格式

**问题代码**:
```python
def _write_or_base64(data: bytes, out_path: Optional[str]) -> Dict[str, Any]:
    if data is None:
        return {"status": "error", "message": "screenshot returned None"}
    out = Path(out_path) if out_path else None
    if out:
        out.parent.mkdir(parents=True, exist_ok=True)
        out.write_bytes(data)  # ❌ 无 try/except，IOError/PermissionError 直接抛出
        return {"status": "success", "path": str(out), "size": len(data)}
    return {"status": "success", "size": len(data), "base64": base64.b64encode(data).decode("ascii")}
```

**问题分析**:
- `out.write_bytes(data)` 在磁盘满、权限不足、路径不存在父目录等场景下抛出异常
- 调用方（`_handle_screenshot`、`_handle_device_screenshot`、`_handle_scene_capture`）未包裹 try/except
- 异常传播到 `main()` 的通用 except 块，返回 `{"status": "error", "message": str(exc)}`
- 调用方无法区分"截图失败"和"保存失败"——两者都返回 error status
- 在 GUI 桥接场景中，CLI 崩溃可能导致 QProcess 进入错误状态，触发重启逻辑

**修复建议**:
```python
def _write_or_base64(data: bytes, out_path: Optional[str]) -> Dict[str, Any]:
    if data is None:
        return {"status": "error", "message": "screenshot returned None"}
    out = Path(out_path) if out_path else None
    if out:
        try:
            out.parent.mkdir(parents=True, exist_ok=True)
            out.write_bytes(data)
            return {"status": "success", "path": str(out), "size": len(data)}
        except Exception as exc:
            return {"status": "error", "message": f"write failed: {exc}", "path": str(out)}
    return {"status": "success", "size": len(data), "base64": base64.b64encode(data).decode("ascii")}
```

#### I03 - `istina.py` 手动 sys.path 操作绕过 `ensure_src_path()`

**文件**: `src/cli/istina.py:14`
**风险等级**: 🟢 低危
**影响**: 路径管理不一致，可能产生重复或冲突的 sys.path 条目

**问题代码**:
```python
sys.path.insert(0, str(Path(__file__).resolve().parent.parent))
# ❌ 手动计算项目根并插入 sys.path

ensure_src_path(__file__)  # line 37: 重复操作
```

**问题分析**:
- line 14 手动将 `src/cli/` 的 parent.parent（即 `src/`）插入 sys.path
- line 37 调用 `ensure_src_path(__file__)` 也会将 `src/` 插入 sys.path
- 两次操作都向 sys.path 插入相同路径，但使用不同的计算方式
- 如果 `ensure_src_path` 内部路径计算与 line 14 不一致（如软链接、符号路径），可能产生两个不同的路径字符串指向同一目录
- 项目 CLAUDE.md 明确要求"Always use `ensure_src_path()`"，line 14 违反此规范
- 实际影响：sys.path 中出现重复条目，可能导致模块加载顺序不确定

**修复建议**:
```python
# 移除 line 14 的手动 sys.path 操作
# ensure_src_path(__file__) 在 line 37 已正确处理
```

#### I04 - `_handle_device_status` 异常信息原样返回，泄露内部路径

**文件**: `src/cli/handlers.py:387-399`
**风险等级**: 🟢 低危
**影响**: 异常消息可能包含敏感路径或内部实现细节

**问题代码**:
```python
def _handle_device_status(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    try:
        client = android.default_client
        server_version = client.version()
        devices = android.get_devices()
        return {
            "status": "success",
            "adb_server": {"version": server_version},
            "devices": [{"serial": d.serial, "state": d.state} for d in devices],
        }
    except Exception as exc:
        return {"status": "error", "message": str(exc)}  # ❌ 原样返回异常消息
```

**问题分析**:
- `str(exc)` 可能包含文件路径（如 ADB 二进制路径）、内部错误信息
- 通过 GUI 桥接时，异常消息会显示在用户界面上
- 项目其他 handler（如 `_handle_device_tap`、`_handle_device_swipe`）也使用相同模式
- 这不是安全漏洞（路径信息非高度敏感），但属于信息泄露的低危模式

**修复建议**:
```python
except Exception as exc:
    self_logger = get_logger(__name__)
    self_logger.error("设备状态查询失败", error=str(exc))
    return {"status": "error", "message": "device status query failed"}
```

#### I05 - `_handle_device_keyevent` CLI 白名单与 `_is_valid_keyevent` 不一致

**文件**: `src/cli/handlers.py:439-451`
**风险等级**: 🟢 低危
**影响**: CLI keyevent 接受 KEYCODE_* 常量但拒绝字母键，与 android_runtime 白名单行为不同

**问题代码**:
```python
def _handle_device_keyevent(runtime: IstinaRuntime, args: argparse.Namespace) -> Dict[str, Any]:
    android = runtime.android()
    key = str(getattr(args, "key", "")).strip()
    if not key:
        return {"status": "error", "message": "empty keyevent"}
    if not (key.isdigit() or key.startswith("KEYCODE_")):
        return {"status": "error", "message": f"invalid keyevent: {key!r} (must be digits or KEYCODE_* constant)"}
    # ❌ 拒绝字母键，但 android_runtime._is_valid_keyevent 也只接受数字和 KEYCODE_*
```

**问题分析**:
- CLI keyevent handler 的白名单（数字 + KEYCODE_*）与 android_runtime 的 `_is_valid_keyevent` 白名单完全一致
- 两者都拒绝字母键（w/a/s/d/q/e/f），与 VLM 导航需求矛盾
- 批次7 W1、批次23 W1、批次29 N01扩展、批次32 H02 已四次记录此问题
- 此 handler 的验证是客户端侧预校验，`_is_valid_keyevent` 是服务端守护进程校验
- 两者一致意味着即使修复守护进程，CLI 仍会拒绝字母键

**修复建议**:（与 H02 一致）
```python
_ALLOWED_KEY_PREFIXES = ("KEYCODE_",)  # 保留 KEYCODE_* 支持
_ALLOWED_GAME_KEYS = {"w", "a", "s", "d", "q", "e", "f"}  # VLM 导航必需

if not (key.isdigit() or key.startswith(_ALLOWED_KEY_PREFIXES) or key in _ALLOWED_GAME_KEYS):
    return {"status": "error", "message": f"invalid keyevent: {key!r}"}
```

#### I06 - `_apply_log_filter` 占位符——UI 暴露了无功能的过滤控件

**文件**: `src/gui/pyqt6/pages/maaend_control_page.py:1542-1547`
**风险等级**: 🟢 低危
**影响**: 用户看到日志过滤下拉框但选择后无任何效果

**问题代码**:
```python
def _apply_log_filter(self, index: int) -> None:
    # Simple filter: hide/show log lines by checking source tags
    # For a full implementation, we'd need to store line data; here we just
    # clear and re-emit filtered logs from a buffer.
    # This is a lightweight placeholder that future-proofs the UI.
    pass  # ❌ 空实现，UI 控件存在但无功能
```

**问题分析**:
- 设置页面（settings_page）和推理页面（maaend_control_page）均暴露日志过滤下拉框
- 下拉框包含"All/Info/Warning/Error"选项，切换时触发 `_apply_log_filter`
- 但方法体为空（pass），选择任何选项都无效果
- 用户可能认为功能已损坏或未实现
- 注释中承认是占位符，但未标记为"施工中"或禁用控件

**修复建议**:
```python
def _apply_log_filter(self, index: int) -> None:
    levels = ["ALL", "INFO", "WARN", "ERROR"]
    target_level = levels[index] if index < len(levels) else "ALL"
    if target_level == "ALL":
        # 重新显示所有日志行
        self._log_text.setHtml(self._full_log_html)
        return
    # 按级别过滤显示
    filtered = []
    for line in self._full_log_html.split("<br>"):
        if target_level in line:
            filtered.append(line)
    self._log_text.setHtml("<br>".join(filtered))
```

#### I07 - `_handle_gpu_recommend` 冗余比较，第二个条件不可达

**文件**: `src/cli/handlers.py:676-678`
**风险等级**: 信息
**影响**: 死代码

**问题代码**:
```python
if mem and mem >= 4 * 1024 * 1024 * 1024 or mem and mem >= 2 * 1024 * 1024 * 1024:
    recommendation = "GPU"
```

**问题分析**:
- 条件为 `mem >= 4GiB or mem >= 2GiB`
- 由于 `>= 4GiB` 蕴含 `>= 2GiB`，第二个条件永远被第一个覆盖
- 等效于 `if mem >= 2GiB`，逻辑上正确但可读性差
- 可能是想写 `mem >= 4GiB or (mem >= 2GiB and mem < 4GiB)` 或类似的阈值分级

**修复建议**:
```python
if mem and mem >= 2 * 1024 * 1024 * 1024:
    recommendation = "GPU"
```

#### I08 - `_interactive_loop` 非换行符结尾的输入数据在EOF时丢失

**文件**: `src/cli/istina.py:340-370`
**风险等级**: 信息
**影响**: 最后一条不以换行符结尾的命令被静默丢弃

**问题代码**:
```python
if chunk == "\n":
    line = buffer.strip()
    buffer = ""
    ...
    _write_result(result)
# EOF 时（chunk 为空），buffer 中剩余数据被丢弃
```

**问题分析**:
- 交互循环仅在遇到 `\n` 时处理 buffer
- stdin EOF 时（chunk 为空），buffer 中可能残留未换行结尾的完整命令
- 该残留数据被静默丢弃，用户最后输入的不换行命令不会被执行
- 实际影响：交互式使用中极少遇到（用户通常按Enter提交命令）
- 但自动化脚本通过管道发送不带换行的最后一个命令时会丢失

**修复建议**:
```python
if not chunk:
    # EOF：处理 buffer 中剩余数据
    line = buffer.strip()
    buffer = ""
    if line:
        self_logger.info("CLI 交互循环: EOF 处理残留命令", command=line)
        # ... 处理 line ...
    break
if chunk == "\n":
    ...
```

## 6. 结论

本次审计发现8个新问题，其中2个中危、4个低危、2个信息。历史报告零修正（F01/F02状态确认准确）。

**关键发现**:
1. **I01（中危）**: `_interactive_loop` 无界缓冲区增长——长时间交互会话可能导致内存耗尽
2. **I02（中危）**: `_write_or_base64` 文件写入无异常处理——截图保存失败时崩溃而非优雅降级

6. **I06**: 日志过滤下拉框存在但无功能——UX 误导

**建议修复顺序**:
1. **I02**: 添加文件写入异常处理（中危，稳定性）
2. **I01**: 添加缓冲区大小限制（中危，内存安全）
3. **I05**: 统一 keyevent 白名单（低危但 VLM 导航阻塞）
4. **I03**: 移除手动 sys.path 操作
5. **I04**: 异常信息脱敏
6. **I06**: 实现或禁用日志过滤控件
7. **I07/I08**: 代码清理
