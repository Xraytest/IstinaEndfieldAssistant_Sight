## 1. 新发现代码错误（历史未覆盖）

### [OCR-1] `OcrBackend._run_ocr_manager` 对 OCR 结果 `box` 字段做无长度校验的解包，畸形 box 直接抛未捕获 `ValueError`

**优先级：Medium**
**位置**：`src/core/capability/element_recognition/backends/ocr_backend.py:275-305`（关键行 291-292）

```python
def _run_ocr_manager(self, screen, roi, expected):
    results = []
    try:
        ocr_results = self._ocr_manager.run_ocr(roi=roi, expected=expected)
    except Exception as e:
        logger.debug(f"OCRManager failed: {e}")
        return results
    h, w = screen.shape[:2]
    for item in ocr_results:
        text = item.get("text", "").strip()
        if not text:
            continue
        box = item.get("box", [0, 0, 0, 0])
        x, y, bw, bh = box          # ← 无 len(box)>=4 校验
        results.append(ElementInfo(...))
```

#### 1.1 根因分析

- **直接原因**：`box = item.get("box", [0, 0, 0, 0])` 之后直接 `x, y, bw, bh = box`。当 OCRManager 返回的某个元素的 `box` 字段长度不为 4（例如仅 `[x, y]`、`[x1, y1, x2, y2]` 被误作列表解包时若长度恰为 4 不会崩但语义错；而长度 2/3 或 box 为 dict/`None` 时长度检查缺位）时，`x, y, bw, bh = box` 抛出 `ValueError: not enough values to unpack (expected 4, got N)`。
- **根本原因**：`try/except` 仅包裹了 `self._ocr_manager.run_ocr(...)` 调用（行 280-284），**未覆盖**其后对 `ocr_results` 的遍历与解包（行 287-303）。这与同文件 `_run_maafw_ocr`（行 112-173）和 `_run_maaend_ocr`（行 179-235）在 `_parse_ocr_box` 中做了多格式容错形成了鲜明对比——Route 3 是三条 OCR 路由中**唯一没有任何 box 格式/长度防护**的一条。
- **调用链**：`OcrBackend.recognize()` 行 87 `if not results and self._ocr_manager is not None: results.extend(self._run_ocr_manager(...))` 也未在调用处捕获异常 → 异常沿 `EndfieldElementRecognizer.recognize()` 向上传播。部分调用点（如 `scene_service.identify`，`scene_service.py:54-58`）虽被 `try/except` 包裹并降级为 `unknown`，但 GUI 识别、任务流水线中直接调用 `recognizer.recognize()` 的其它路径会因该 `ValueError` 中断本次 OCR 识别；且 Route 3 仅在 Route 1/2 均失败时才被触达，问题定位困难、表现不稳定。

#### 1.2 修改方案

将 box 解析统一收口到已有的 `_parse_ocr_box`（该函数已支持 `[x,y,w,h]`、`[x1,y1,x2,y2]`、四点多边形、`None` 回退），并对结果做长度兜底：

```python
# ocr_backend.py _run_ocr_manager 结果循环
for item in ocr_results:
    text = item.get("text", "").strip()
    if not text:
        continue
    box = item.get("box")
    # 复用统一解析器：box 为 None / 长度!=4 / 字典 / 多边形 均安全回退
    x, y, bw, bh = self._parse_ocr_box(item, 0, 0, w, h, w, h) \
        if box is not None else (0, 0, w, h)
    results.append(ElementInfo(
        element_type="text",
        label=text,
        bbox=(x / w, y / h, (x + bw) / w, (y + bh) / h),
        center=(item.get("cx", x + bw // 2) / w,
                item.get("cy", y + bh // 2) / h),
        confidence=item.get("score", 0.5),
        source="ocr",
        action="none",
        metadata={"ocr_engine": "OCRManager"},
    ))
```

> 说明：`_parse_ocr_box` 当前签名要求传入 `roi_*` 参数用于回退；对于 OCRManager 路径可传 `(0,0,w,h)` 作为屏幕级回退基准，或新增一个不依赖 roi 的重载。核心是**消除无校验解包**。

#### 1.3 影响面

- 函数：`OcrBackend._run_ocr_manager`（ocr_backend.py:275）。
- 上游：`OcrBackend.recognize()`（行 87 调用点）、`EndfieldElementRecognizer.recognize()`、`scene_service.identify()` / `analyze_elements()`。
- 触发条件：OCRManager 作为回退后端被启用（`self._ocr_manager is not None` 且 Route 1/2 未产出结果），且其返回任一元素 `box` 长度≠4。
- 相关：`_parse_ocr_box`（行 237-273）被复用，无需改动即可承接。

#### 1.4 非期待变化（回退策略）

- 复用 `_parse_ocr_box` 后，原本因解包崩溃而被丢弃的「整个 OCRManager 结果集」将改为「逐元素安全解析、畸形 box 退化为屏幕级回退框」。这会使个别文本块的 bbox 精度下降（回退到整屏），但不会再整体崩溃——精度损失远小于崩溃，属正向权衡。
- 若担心静默劣化，可在 `_parse_ocr_box` 回退分支补一行 `logger.debug("OCRManager 返回畸形 box，已回退: %r", box)`，便于后期调参。
- 回退策略：若暂不引入 `_parse_ocr_box`，最小修复为 `if not (isinstance(box, (list, tuple)) and len(box) >= 4): continue`，先跳过畸形元素而非崩溃。

## 2. 对既往报告的审计（写入本批次，重点纠错）

### 2.1 审计对象：`reports/auto/20260711_FINAL.md` 的 `D1`（High）与 `C01`（Critical）

`FINAL.md` 中两处论断（行 36、行 103 的 `D1`；行 312 的 `C01`）：

- **D1**："`recovery.py:72` `_force_stop` 将 `"am force-stop"` 作为单个参数，导致强制停止从未生效"（High，功能失效）。
- **C01**："`recovery.py:70-72` `_force_stop` 将 `"am force-stop"` 作为单个字符串参数传递，`self._package` 未过滤，可注入任意 shell 命令"（🔴 Critical，命令注入）。

#### 2.1.1 结合实际代码复核

`recovery.py` 实际实现（行 60-79）：

```python
def _resolve_adb(self, serial):
    cmd = [self._adb_path]
    if serial:
        cmd += ["-s", serial]
    return cmd

def _run(self, args, serial):
    cmd = self._resolve_adb(serial) + args
    subprocess.check_output(cmd, text=True, timeout=30)   # 以 list 传入

def _force_stop(self, serial):
    self._run(["shell", "am force-stop", self._package], serial)
```

**关键事实**：`_run` 将参数列表直接交给 `subprocess.check_output`。`adb shell` 的语义是把 `shell` 之后的**所有参数用单个空格拼接**后作为一条命令字符串送往设备端 shell 执行。因此设备端实际收到的命令是：

```
am force-stop com.hypergryph.endfield
```

这正是 `am force-stop` 的**正确语法**，强制停止**会正常生效**。

#### 2.1.2 审计结论

| 论断 | 复核结果 | 修正 |
|------|----------|------|
| **D1（强制停止从未生效）** | **错误**。`adb shell` 拼接参数后命令正确，`am force-stop <pkg>` 生效。原论断混淆了「列表中的单个字符串元素」与「shell 引号」，误以为参数被当成一个带空格的 token；实际 adb 不做 shell 引号解析，只做空格拼接。 | 从风险台账**移除 D1**（或降为 Info/风格建议）。 |
| **C01（命令注入 Critical）** | **严重性被严重高估，实质不成立**。`self._package` 在 `AndroidAppRestartPolicy.__init__` 中硬编码为常量 `"com.hypergryph.endfield"`（行 21），且全仓无用户可控路径写入该字段；常量不含任何 shell 元字符，`;`/`&&`/`` ` `` 等注入向量无从插入。唯一理论注入面（pkg 含元字符）在工程上不可达。 | 从 Critical **降级/移除**；若保留，仅作 Info 级「纵深防御」建议（对 `self._package` 做包名正则校验 `^[a-z0-9_.]+$`）。 |

#### 2.1.3 为什么原论断会写错（避免后人误报）

- 误判根源：把 `["shell", "am force-stop", pkg]` 中的 `"am force-stop"` 当成「被引号包住、整体作为一个不可分参数」。但 `adb` 并非把每个 list 元素当 shell 引号字符串，而是用空格 `join` 成 `am force-stop pkg`。`adb shell "am force-stop" pkg` 与 `adb shell am force-stop pkg` 在设备端等价。
- 注：拆分 `"am force-stop"` 为 `["am", "force-stop"]`（FINAL 行 266、355 的修复建议）**不改变任何运行时行为**，纯属风格调整；将其列为「修复命令注入/功能失效」会虚高修复收益，并误导后续维护者认为此处曾有真实漏洞。

> 历史对齐：本结论与更早批次（memory 记录的 0659 批次 A1）一致，但 `FINAL.md` 作为「最终整合版」**仍原样保留 D1/C01 并计入 High/Critical 统计**（行 233、245、368、379），构成权威报告内部的自相矛盾。本次审计正式建议修订 `FINAL.md`。
